📋 Cheatsheet — SQL

Ordre des clauses

SQL
SELECT col1, col2        -- colonnes à afficher
  FROM table             -- source principale
  JOIN other ON ...      -- jointure (optionnelle)
 WHERE condition         -- filtre sur les lignes
 GROUP BY col1           -- regroupement
HAVING condition         -- filtre sur l'agrégat
 ORDER BY col1 DESC      -- tri du résultat
 LIMIT 10                -- nombre de lignes à retourner

Agrégations

SQL
SELECT COUNT(*)                          -- nombre total de lignes
       MIN(weight), MAX(weight),         -- min et max
       AVG(weight), SUM(weight)          -- moyenne et somme
  FROM Player

SELECT country_id, COUNT(*)              -- agrégat par groupe
  FROM "Match"
 GROUP BY country_id

Arithmétique & fonctions

SQL
ROUND(MIN(weight) / 2.205, 0)           -- division + arrondi
POWER(height / 100, 2)                  -- mise au carré
SUBSTR(name, 1, 3)                      -- extrait les 3 premiers caractères
UPPER(name), LOWER(name)                -- mise en majuscules / minuscules
LENGTH(name)                            -- longueur de la chaîne
REPLACE(name, 'old', 'new')             -- remplacement de sous-chaîne
COALESCE(value, 'default')              -- remplace NULL par une valeur par défaut
STRFTIME('%Y-%m', DATE(col))            -- formatage de date

Jointures

SQL
-- INNER JOIN : lignes communes aux deux tables
SELECT * FROM League
  JOIN Country ON Country.id = League.country_id

-- LEFT JOIN : toutes les lignes de gauche, NULL si pas de correspondance
SELECT * FROM League
  LEFT JOIN Country ON Country.id = League.country_id

-- SELF JOIN : une table jointe à elle-même
SELECT e.name, m.name AS manager
  FROM employees e
  LEFT JOIN employees m ON m.id = e.manager_id

Window Functions

SQL
SUM(amount) OVER()                               -- agrégat sur toute la table
SUM(amount) OVER(PARTITION BY customer_id)       -- agrégat par partition
SUM(amount) OVER(PARTITION BY customer_id        -- cumul ordonné
                 ORDER BY ordered_at)
RANK() OVER(PARTITION BY customer_id             -- rang dans chaque partition
            ORDER BY ordered_at)

CTE avec WITH

SQL
WITH tmp AS (
    SELECT AVG(weight) AS avg_weight    -- sous-requête nommée
      FROM Player
)
SELECT AVG(POWER(weight - tmp.avg_weight, 2)) AS var_weight
  FROM Player, tmp                      -- utilisation du CTE

CRUD (DML)

SQL
INSERT INTO table (col1, col2) VALUES (val1, val2)   -- insertion
UPDATE table SET col1 = val1 WHERE condition          -- mise à jour
DELETE FROM table WHERE condition                     -- suppression

⚠️ Récapitulatif — Erreurs clés à éviter

❌

Mauvais ordre des clauses SQL

L'ordre SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT est obligatoire.

❌ WHERE ... GROUP BY ... FROM → erreur de syntaxe

✅ SELECT ... FROM ... WHERE ... GROUP BY ...

❌

SELECT avec GROUP BY : colonnes non agrégées

Toute colonne dans SELECT doit être dans GROUP BY ou dans une fonction d'agrégation.

❌ SELECT country_id, name, COUNT(*) FROM Match GROUP BY country_id

✅ SELECT country_id, COUNT(*) FROM Match GROUP BY country_id

❌

Utiliser f-string pour les paramètres SQL (injection)

L'interpolation de chaîne dans les requêtes SQL expose à des injections malveillantes.

❌ query = f"SELECT * FROM users WHERE name = '{name}'"

✅ pd.read_sql(query, con=conn, params=[name]) avec WHERE name = ?

❌

Confondre WHERE et HAVING

WHERE filtre les lignes avant l'agrégation, HAVING filtre après.

❌ SELECT country_id, COUNT() FROM Match WHERE COUNT() > 10 GROUP BY country_id

✅ SELECT country_id, COUNT() FROM Match GROUP BY country_id HAVING COUNT() > 10

❌

Oublier de fermer la connexion à la base

Sans with, la connexion reste ouverte si le code plante.

❌ conn = connect('db.sqlite') sans conn.close() ni with

✅ with connect('db.sqlite') as conn: — fermeture automatique garantie

3) Jointures SQL

3.1 INNER JOIN

Retourne uniquement les lignes qui ont une correspondance dans les deux tables.

SQL
SELECT *
  FROM League
  JOIN Country
    ON Country.id = League.country_id

3.2 LEFT JOIN

Conserve toutes les lignes de la table de gauche, même sans correspondance à droite (valeur NULL).

SQL
SELECT *
  FROM League
  LEFT JOIN Country
    ON Country.id = League.country_id

Remplacer les NULL avec COALESCE :

SQL
SELECT League.id,
       League.name,
       COALESCE(Country.name, 'Unknown') AS country_name   -- 'Unknown' si NULL
  FROM League
  LEFT JOIN Country
    ON Country.id = League.country_id

3.3 RIGHT JOIN

💡

RIGHT JOIN est rarement utilisé — on préfère inverser les tables dans un LEFT JOIN, qui est plus lisible.

3.4 FULL OUTER JOIN

Retourne toutes les lignes des deux tables, avec NULL là où il n'y a pas de correspondance.

Une window function effectue un calcul sur un ensemble de lignes lié à la ligne courante sans regrouper les lignes en une seule.

4.2 Agrégation globale vs window

Part de chaque commande dans le total des ventes :

SQL
-- Sans window function : une seule ligne retournée
SELECT SUM(amount) FROM orders

-- Avec OVER() : agrégat calculé pour chaque ligne
SELECT amount,
       SUM(amount) OVER() AS total,                           -- total global
       amount / SUM(amount) OVER() AS share                   -- part de chaque commande
  FROM orders

4.3 Partition par client

Part de chaque commande dans le total par client :

SQL
SELECT amount,
       SUM(amount) OVER(PARTITION BY customer_id) AS total,   -- total par client
       amount / SUM(amount) OVER(PARTITION BY customer_id) AS share
  FROM orders

4.4 Cumul dans le temps

Montant cumulé des commandes par client, dans l'ordre chronologique :

SQL
-- Étape 1 : cumul par client
SELECT orders.customer_id,
       orders.ordered_at,
       orders.amount,
       SUM(orders.amount) OVER (
         PARTITION BY orders.customer_id
         ORDER BY orders.ordered_at          -- cumul dans l'ordre des dates
       ) AS cumulative_amount
  FROM orders
SQL
-- Étape 2 : ajout des infos client via JOIN
SELECT customers.first_name,
       customers.last_name,
       orders.ordered_at,
       SUM(orders.amount) OVER (
         PARTITION BY orders.customer_id
         ORDER BY orders.ordered_at
       ) AS cumulative_amount
  FROM customers
  LEFT JOIN orders ON orders.customer_id = customers.id

4.5 Rang des commandes

Rang de chaque commande pour chaque client (utile comme feature ML) :

SQL
SELECT orders.id,
       orders.ordered_at,
       orders.customer_id,
       RANK() OVER (
         PARTITION BY orders.customer_id
         ORDER BY orders.ordered_at          -- 1ère, 2ème commande...
       ) AS order_rank
  FROM orders

5) Clause WITH — CTE (Common Table Expression)

👉 Documentation SQLite WITH

5.1 Pourquoi utiliser WITH ?

💡

Les CTEs permettent de nommer des sous-requêtes pour les réutiliser, rendant le code SQL plus lisible et évitant les sous-requêtes imbriquées.

5.2 Exemple : calcul de la variance

En BigQuery/PostgreSQL, VARIANCE() existe nativement. En SQLite, il faut le calculer manuellement.

⚠️

Toujours préciser son dialecte SQL lors des recherches — les fonctions varient selon la base de données.

Avec sous-requête imbriquée (peu lisible) :

SQL
SELECT AVG(
         POWER(weight - (SELECT AVG(weight) FROM Player), 2)  -- difficile à lire
       ) AS var_weight
  FROM Player

Avec CTE (recommandé) :

SQL
WITH tmp AS (
    SELECT AVG(weight) AS avg_weight   -- étape 1 : calcul de la moyenne
      FROM Player
)
SELECT AVG(
         POWER(weight - tmp.avg_weight, 2)   -- étape 2 : variance
       ) AS var_weight
  FROM Player, tmp

Avec window function :

SQL
WITH tmp AS (
    SELECT POWER(weight - AVG(weight) OVER(), 2) AS var_weight  -- écart au carré
      FROM Player
)
SELECT AVG(var_weight) AS var_weight   -- variance finale
  FROM tmp

5.3 Exemple avancé : nombre cumulé de matchs par mois

SQL
-- Étape 1 : nombre de matchs par mois
SELECT STRFTIME('%Y-%m', DATE(matches.date)) AS period,
       COUNT(*) AS cnt
  FROM "Match" AS matches
 GROUP BY period
 ORDER BY period
SQL
-- Étape 2 : cumul avec window function via CTE
WITH matches_per_month AS (
    SELECT STRFTIME('%Y-%m', DATE(matches.date)) AS period,
           COUNT(*) AS cnt
      FROM "Match" AS matches
     GROUP BY period
     ORDER BY period
)
SELECT matches_per_month.period,
       SUM(matches_per_month.cnt) OVER (
         ORDER BY matches_per_month.period    -- cumul dans l'ordre chronologique
       ) AS cumulative_count
  FROM matches_per_month

8) Annexe — SQL Injection 😈

L'injection SQL exploite une requête construite par f-string pour exécuter du code malveillant.

Python
## Requête vulnérable
def authenticate(username, password):
    query = f"""
        SELECT * FROM users
        WHERE users.username = '{username}'
        AND users.password = '{password}'   -- vulnérable
    """
    c.execute(query)
    return "Authorized" if c.fetchone() else "Unauthorized"

authenticate("john", "' OR 1=1 --")   # contourne le mot de passe !

La requête exécutée devient :

SQL
SELECT * FROM users
WHERE users.username = 'john'
AND users.password = '' OR 1=1 --'    -- la condition est toujours vraie !

✅ Solution : paramètre substitution

Python
def authenticate_safe(username, password):
    query = """
        SELECT * FROM users
        WHERE users.username = ?
        AND users.password = ?          -- ? injecté de façon sécurisée
    """
    c.execute(query, (username, password))
    return "Authorized" if c.fetchone() else "Unauthorized"

authenticate_safe("john", "' OR 1=1 --")   # retourne 'Unauthorized' ✅
📚

👉 Documentation sqlite3 — execute vs executescript — l'injection ne fonctionne pas avec execute() en SQLite, mais fonctionne avec executescript()

🧭 Introduction

SQL ("Structured Query Language") est le langage universel pour parler à une base de données. Dans ce cours, on va au-delà du simple SELECT * FROM table : on apprend à combiner des tables, à faire des calculs sophistiqués ligne par ligne, à organiser des requêtes complexes et à connecter SQL à Python.

  5. En Python, on utilise toujours la substitution de paramètres (?) — jamais les f-strings — pour éviter les injections SQL.

1) Les 8 clauses fondamentales

Toute requête SQL suit toujours ce même squelette, dans cet ordre précis :

SQL
SELECT col1, col2        -- 1. quelles colonnes afficher
  FROM table             -- 2. depuis quelle table
  JOIN autre ON ...      -- 3. fusionner avec une autre table (optionnel)
 WHERE condition         -- 4. filtrer les lignes
 GROUP BY col1           -- 5. regrouper par valeur
HAVING condition         -- 6. filtrer sur les groupes
 ORDER BY col1 DESC      -- 7. trier le résultat
 LIMIT 10               -- 8. limiter le nombre de lignes
⚠️

L'ordre est imposé par le langage SQL. Écrire WHERE avant FROM, par exemple, produit une erreur de syntaxe. Mémorisez ce squelette une bonne fois pour toutes.

La différence cruciale : WHERE vs HAVING

WHERE filtre avant le regroupement (sur les lignes brutes).

HAVING filtre après le regroupement (sur les résultats agrégés).

SQL
-- MAUVAIS : on ne peut pas utiliser COUNT(*) dans WHERE
SELECT country_id, COUNT(*)
  FROM commandes
 WHERE COUNT(*) > 10       -- erreur ! COUNT n'est pas encore calculé ici
 GROUP BY country_id

-- BON : on utilise HAVING pour filtrer après le GROUP BY
SELECT country_id, COUNT(*)
  FROM commandes
 GROUP BY country_id
HAVING COUNT(*) > 10       -- filtre appliqué après le regroupement
💡

Règle simple : si tu veux filtrer sur un résultat d'agrégation (COUNT, SUM, AVG...), utilise HAVING. Sinon, utilise WHERE.

4) Window Functions — calculs sans écraser les lignes

Une window function effectue un calcul sur un groupe de lignes, mais contrairement à GROUP BY, elle ne fusionne pas les lignes — chaque ligne garde son existence propre dans le résultat.

Le mot-clé clé : OVER. Dès que tu vois OVER(...), c'est une window function.

4.1 Comprendre la différence

Imaginons une table orders :

5) CTEs avec WITH — organiser les requêtes

5.1 C'est quoi un CTE ?

Un CTE (Common Table Expression) est une sous-requête que l'on nomme pour pouvoir l'utiliser ensuite, comme si c'était une vraie table. On l'écrit avec le mot-clé WITH.

Sans CTE, les requêtes complexes deviennent vite illisibles. Avec WITH, on découpe le problème en étapes nommées.

5.2 Exemple simple : calculer la variance du poids des joueurs

Sans CTE (difficile à lire) :

SQL
SELECT AVG(
         POWER(weight - (SELECT AVG(weight) FROM Player), 2)  -- sous-requête imbriquée
       ) AS variance_poids
  FROM Player
⚠️

La sous-requête (SELECT AVG(weight) FROM Player) est exécutée pour chaque ligne — c'est valide mais illisible et potentiellement lent.

Avec CTE (recommandé) :

SQL
WITH poids_moyen AS (
    -- Étape 1 : on calcule la moyenne du poids, on lui donne un nom
    SELECT AVG(weight) AS avg_weight
      FROM Player
)
-- Étape 2 : on utilise ce résultat nommé comme une table normale
SELECT AVG(
         POWER(weight - poids_moyen.avg_weight, 2)  -- écart au carré
       ) AS variance_poids
  FROM Player, poids_moyen   -- on "croise" Player avec notre CTE (1 seule ligne)
💡

Lire un CTE comme une recette de cuisine : "d'abord, prépare ceci (WITH), puis utilise-le dans la recette principale (SELECT)."

5.3 Exemple avancé : nombre cumulé de matchs par mois

On veut savoir combien de matchs ont été joués en tout depuis le début, mois par mois.

Étape 1 : compter les matchs par mois

SQL
SELECT STRFTIME('%Y-%m', DATE(matches.date)) AS periode,  -- format "2024-03"
       COUNT(*) AS nb_matchs                               -- nombre de matchs ce mois
  FROM "Match" AS matches
 GROUP BY periode
 ORDER BY periode

Étape 2 : ajouter le cumul avec une window function via CTE

WITH matchs_par_mois AS (
    -- Étape 1 nommée : matchs regroupés par mois
    SELECT STRFTIME('%Y-%m', DATE(matches.date)) AS periode,
           COUNT(*) AS nb_matchs
      FROM "Match" AS matches
     GROUP BY periode
     ORDER BY periode
)
SELECT matchs_par_mois.periode,
       matchs_par_mois.nb_matchs,
       SUM(matchs_par_mois.nb_matchs) OVER (
         ORDER BY matchs_par_mois.periode   -- cumul dans l'ordre chronologique
       ) AS total_cumule
  FROM matchs_par_mois

On ne peut pas utiliser une window function directement sur un GROUP BY — le CTE résout ce problème en séparant les deux opérations en deux étapes distinctes.

Le with connect(...) as conn est un "context manager" : il garantit que la connexion à la base est fermée automatiquement même si une erreur survient dans le code. C'est la bonne façon de faire.

Sans with, si une erreur se produit, la connexion reste ouverte indéfiniment — ce qui peut provoquer des blocages ou des fuites de ressources.

6.2 Requêtes dynamiques — le danger des f-strings

Parfois, on veut passer une variable dans une requête (par exemple, un nom saisi par l'utilisateur). Il y a une bonne et une mauvaise façon de faire.

Mauvaise pratique — f-string (dangereux) :

Python
name = "John%"

with connect('data/soccer.sqlite') as conn:
    query = f"""
        SELECT *
          FROM Player
         WHERE player_name LIKE '{name}'   -- DANGER : name est injecté directement
    """
    players_df = pd.read_sql(query, con=conn)
⚠️

Si name vient d'un utilisateur externe (formulaire, API...), il peut contenir du code SQL malveillant. C'est ce qu'on appelle une injection SQL (voir section 8).

Bonne pratique — substitution de paramètres (sécurisé) :

Python
name = "Paul%"

with connect('data/soccer.sqlite') as conn:
    query = """
        SELECT *
          FROM Player
         WHERE player_name LIKE ?          -- le '?' est un emplacement réservé
    """
    players_df = pd.read_sql(
        query,
        con=conn,
        params=[name]    # la valeur est passée séparément, jamais concaténée
    )

Avec plusieurs paramètres :

Python
name = "John%"
min_weight = 200

with connect('data/soccer.sqlite') as conn:
    query = """
        SELECT *
          FROM Player
         WHERE player_name LIKE ?   -- 1er '?' = premier paramètre de la liste
           AND weight > ?           -- 2ème '?' = deuxième paramètre
    """
    players_df = pd.read_sql(
        query,
        con=conn,
        params=[name, min_weight]   -- les valeurs dans l'ordre des '?'
    )
✅

Règle : toujours passer les valeurs variables via params=[...]. Le ? dans la requête est simplement un emplacement réservé que pandas/sqlite3 remplira de façon sécurisée.

8) Sécurité — Les injections SQL

Qu'est-ce qu'une injection SQL ?

Une injection SQL, c'est quand un utilisateur malveillant insère du code SQL dans une valeur pour manipuler ta requête.

Voici une fonction d'authentification vulnérable :

Python
def authenticate(username, password):
    query = f"""
        SELECT * FROM users
        WHERE users.username = '{username}'   -- valeurs injectées directement
        AND users.password = '{password}'
    """
    c.execute(query)
    return "Autorisé" if c.fetchone() else "Refusé"

# Appel normal
authenticate("alice", "secret123")   -- fonctionne normalement

# Appel malveillant
authenticate("john", "' OR 1=1 --") -- contourne complètement le mot de passe !

La requête exécutée devient :

SQL
SELECT * FROM users
WHERE users.username = 'john'
AND users.password = '' OR 1=1 --'  -- '1=1' est toujours vrai, '--' commente le reste
                                     -- résultat : l'utilisateur est toujours "autorisé" !
⚠️

OR 1=1 rend la condition toujours vraie. -- est un commentaire SQL qui ignore tout ce qui suit. L'attaquant est connecté sans connaître le mot de passe.

La solution : substitution de paramètres

Python
def authenticate_safe(username, password):
    query = """
        SELECT * FROM users
        WHERE users.username = ?   -- '?' = emplacement sécurisé
        AND users.password = ?
    """
    c.execute(query, (username, password))   -- les valeurs sont passées séparément
    return "Autorisé" if c.fetchone() else "Refusé"

authenticate_safe("john", "' OR 1=1 --")   -- retourne 'Refusé' : l'injection ne fonctionne plus
✅

Avec la substitution de paramètres, le driver SQL traite les valeurs comme de simples chaînes de caractères, jamais comme du code SQL. L'injection est impossible.

🎯 Questions de vérification

1. Quelle est la différence entre WHERE et HAVING ? Donne un exemple où tu utiliserais chacun.

WHERE filtre les lignes avant le groupement (sur les données brutes). HAVING filtre après le groupement (sur les résultats agrégés). Exemple : WHERE age > 18 pour garder seulement les adultes, HAVING COUNT(*) > 5 pour garder seulement les groupes avec plus de 5 lignes.

2. Explique avec tes mots la différence entre GROUP BY et une window function avec OVER.

Avec GROUP BY, les lignes sont fusionnées en une ligne par groupe — on perd le détail. Avec OVER, chaque ligne reste intacte et le calcul est ajouté comme une colonne supplémentaire. On garde le détail ET l'agrégat.

3. Pourquoi est-il dangereux d'utiliser une f-string pour construire une requête SQL avec une valeur externe ?

Parce que la valeur externe peut contenir du code SQL malveillant (injection SQL) qui sera exécuté. Un attaquant peut ainsi contourner une authentification, accéder à toutes les données, ou même supprimer des tables entières.

4. À quoi sert un CTE (WITH) ? Dans quel cas est-il particulièrement utile ?

Un CTE permet de nommer une sous-requête pour la réutiliser ensuite. Il est particulièrement utile quand on doit combiner deux opérations incompatibles directement (comme un GROUP BY suivi d'une window function), ou pour rendre une requête complexe plus lisible en la découpant en étapes.

📌 TL;DR

  • L'ordre des clauses SQL est obligatoire : SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT.
  • JOIN fusionne deux tables sur une condition ; LEFT JOIN garde toutes les lignes de gauche (avec NULL si pas de correspondance) ; SELF JOIN joint une table sur elle-même (utile pour les hiérarchies).
  • Une window function (OVER) calcule un agrégat sans fusionner les lignes — PARTITION BY divise en groupes, ORDER BY crée un cumul, RANK() attribue des rangs.
  • Un CTE (WITH nom AS (...)) nomme une sous-requête pour la réutiliser : il rend le code lisible et permet de chaîner des opérations complexes.
  • En Python, utiliser with connect(...) as conn + pd.read_sql(query, conn, params=[...]) : le with ferme la connexion automatiquement, le params protège contre les injections SQL.
  • Ne jamais construire une requête SQL avec une f-string à partir d'une valeur externe : toujours utiliser ? et params=[...].
  • WHERE filtre avant le groupement, HAVING filtre après — les confondre provoque une erreur.
  • COALESCE(colonne, 'valeur_par_défaut') est pratique pour remplacer les NULL issus des jointures.

A. Concepts du cours

JOINs avancés et auto-join

Au-delà de l'INNER/LEFT/RIGHT JOIN basique, plusieurs patterns sont essentiels pour écrire des requêtes de vrai niveau data analyst.

Rappel express : un JOIN fusionne deux tables selon une condition. INNER ne garde que les lignes qui matchent des deux côtés ; LEFT garde toutes les lignes de gauche (avec NULL à droite si pas de match).

Self-join (auto-jointure) : joindre une table avec elle-même. Cas typique : hiérarchie employé/manager, où manager est lui-même un employé.

SQL
-- Chaque employé + le nom de son manager
SELECT
    e.name AS employee,           -- alias "e" pour l'employé
    m.name AS manager             -- alias "m" pour le manager (même table !)
FROM employees e
LEFT JOIN employees m             -- on joint employees à elle-même
  ON e.manager_id = m.id;         -- condition de jointure hiérarchique
-- LEFT JOIN car le CEO n'a pas de manager (manager_id NULL)

L'astuce : utiliser des alias différents (e et m) pour distinguer les deux instances de la table. Sans alias, SQL ne saurait pas à quelle copie faire référence.

Exemple chiffré : avec une table de 1000 employés dont 100 managers, la requête retourne 1000 lignes (chaque employé + son manager ou NULL).

JOIN sur intervalle : utile pour matcher une date avec une période de validité.

SQL
-- Pour chaque commande, trouver la promo active à ce moment
SELECT o.order_id, p.promo_name
FROM orders o
JOIN promotions p
  ON o.created_at BETWEEN p.start_date AND p.end_date;
-- Attention : si plusieurs promos se chevauchent, une commande apparaît plusieurs fois

Anti-join (LEFT JOIN + WHERE NULL) : trouver les lignes de A qui n'ont pas de correspondance dans B.

SQL
-- Tous les utilisateurs qui n'ont JAMAIS commandé
SELECT u.*
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;               -- le filtre clé : pas de match = NULL du côté droit
-- Résultat : utilisateurs "dormants"

Semi-join (EXISTS) : retourner les lignes de A qui ont au moins une correspondance dans B, sans dupliquer.

SQL
-- Tous les utilisateurs qui ont commandé au moins une fois (1 ligne par user)
SELECT u.*
FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o        -- SELECT 1 : on ne lit pas les colonnes, juste "existe ?"
    WHERE o.user_id = u.id        -- corrélation avec la requête extérieure
);
-- Contrairement à un INNER JOIN, pas de duplication si l'user a 50 commandes
💡

Analogie : INNER JOIN multiplie les lignes (un user avec 5 orders apparaît 5 fois). EXISTS dédoublonne (un user avec ≥1 order apparaît une fois). Pour "qui a fait au moins une commande", utilisez EXISTS. Pour "afficher chaque commande avec son utilisateur", utilisez JOIN. Confondre les deux donne des résultats faux ou des lignes en trop.

Subqueries et CTE

Une sous-requête est une SELECT imbriquée dans une autre. Utile pour les petits filtres, vite illisible pour les requêtes complexes.

SQL
-- Top 3 utilisateurs par dépenses, avec leurs infos
SELECT * FROM users
WHERE id IN (                     -- filtre sur les ids retournés par la sous-requête
    SELECT user_id FROM orders
    GROUP BY user_id
    ORDER BY SUM(amount) DESC     -- tri par dépenses décroissantes
    LIMIT 3                        -- garde les 3 premiers
);

Les sous-requêtes deviennent vite illisibles dès qu'elles s'imbriquent sur 2-3 niveaux. La CTE (Common Table Expression, syntaxe WITH) est plus claire et permet de réutiliser des blocs :

SQL
-- Même résultat, mais lisible étape par étape
WITH user_totals AS (             -- étape 1 : calcul des totaux par user
    SELECT user_id, SUM(amount) AS total
    FROM orders
    GROUP BY user_id
),
top_users AS (                    -- étape 2 : top 3 par total
    SELECT user_id FROM user_totals
    ORDER BY total DESC
    LIMIT 3
)
SELECT u.*, ut.total              -- étape 3 : join final avec la table users
FROM users u
INNER JOIN user_totals ut ON u.id = ut.user_id
INNER JOIN top_users t ON u.id = t.user_id
ORDER BY ut.total DESC;

Lire ça comme une suite de transformations : "construis user_totals, puis top_users à partir de la première, puis fais le SELECT final". Cette discipline rend les requêtes complexes maintenables, versionnables et testables.

Récursive CTE : pour parcourir des hiérarchies (graphes, arborescences).

SQL
-- Tous les descendants d'une catégorie (sous-catégories, sous-sous-catégories, ...)
WITH RECURSIVE descendants AS (
    -- Cas de base : la catégorie de départ
    SELECT id, name, parent_id, 1 AS level
    FROM categories
    WHERE id = 42                  -- point de départ

    UNION ALL                      -- concaténation avec l'étape récursive

    -- Cas récursif : enfants des catégories déjà trouvées
    SELECT c.id, c.name, c.parent_id, d.level + 1
    FROM categories c
    INNER JOIN descendants d ON c.parent_id = d.id  -- jointure avec "soi-même" (itération)
)
SELECT * FROM descendants;
-- Résultat : toute la sous-arborescence, avec un niveau de profondeur

Window functions

LE concept SQL avancé qui change la donne quand on fait de l'analytics. Une window function calcule une valeur agrégée sans regrouper les lignes.

Syntaxe : function() OVER (PARTITION BY col1 ORDER BY col2).

  • PARTITION BY : définit les groupes (comme un GROUP BY interne)
  • ORDER BY : définit l'ordre dans chaque groupe (utile pour ROW_NUMBER, LAG, cumulés)
SQL
-- Rang des commandes par utilisateur, du plus récent au plus ancien
SELECT
    user_id,
    order_id,
    created_at,
    ROW_NUMBER() OVER (
        PARTITION BY user_id      -- redémarre le compteur à chaque user
        ORDER BY created_at DESC  -- la plus récente = rang 1
    ) AS rn
FROM orders;
-- Chaque commande conserve ses colonnes + gagne une colonne "rn"

Le résultat conserve toutes les lignes d'origine, plus une nouvelle colonne rn. Pour ne garder que la commande la plus récente par user :

SQL
WITH ranked AS (
    SELECT *,
        ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
    FROM orders
)
SELECT * FROM ranked WHERE rn = 1;     -- ne garde que la 1ère de chaque partition
-- Pattern canonique : "dernier enregistrement par groupe"

Fonctions de fenêtre courantes :

FonctionUsage
ROW_NUMBER()Numéro unique par ligne dans la partition (1, 2, 3, 4)
RANK()Rang avec gaps en cas d'égalité (1, 2, 2, 4)
DENSE_RANK()Rang sans gaps (1, 2, 2, 3)
LAG(col) / LEAD(col)Valeur de la ligne précédente / suivante
SUM(col) OVER (...)Somme cumulative ou par groupe
AVG(col) OVER (...)Moyenne mobile
FIRST_VALUE(col) / LAST_VALUE(col)Première / dernière valeur

Exemple : calcul de rétention avec LAG :

SQL
-- Pour chaque commande, calculer le gap avec la commande précédente du même user
SELECT
    user_id,
    created_at,
    LAG(created_at) OVER (
        PARTITION BY user_id
        ORDER BY created_at
    ) AS prev_order,             -- date de la commande précédente
    created_at - LAG(created_at) OVER (
        PARTITION BY user_id
        ORDER BY created_at
    ) AS gap                       -- intervalle entre les 2
FROM orders;
-- Résultat : user_id | created_at | prev_order | gap (ex: 14 days)
-- Utile pour calculer la fréquence d'achat
💡

Analogie : un GROUP BY classique écrase les détails (vous obtenez une ligne par groupe). Les window functions sont un GROUP BY qui garde tous les détails et ajoute juste une colonne d'agrégat. C'est exactement comme df.groupby('city')['salary'].transform('mean') en pandas — ça enrichit le DataFrame sans perdre de lignes.

GROUP BY avancé : HAVING, ROLLUP, CUBE

HAVING filtre les groupes après agrégation (vs WHERE qui filtre les lignes avant).

SQL
-- Villes d'engineering avec un salaire moyen > 50k
SELECT city, AVG(salary) AS avg_sal
FROM employees
WHERE department = 'engineering'   -- étape 1 : filtre les lignes (avant group)
GROUP BY city                       -- étape 2 : regroupe par ville
HAVING AVG(salary) > 50000;         -- étape 3 : filtre les groupes (après agrégation)
-- Ordre d'exécution : WHERE → GROUP BY → HAVING → SELECT → ORDER BY

GROUP BY ROLLUP(a, b) génère des sous-totaux hiérarchiques :

SQL
-- Totaux par (region, country), puis par region, puis total global
SELECT region, country, SUM(sales)
FROM orders
GROUP BY ROLLUP(region, country);
-- Résultat :
-- ('EMEA', 'France', 1000), ('EMEA', 'Germany', 1500),  -- détails par pays
-- ('EMEA', NULL, 2500),                                  -- sous-total EMEA
-- (NULL, NULL, 10000)                                    -- grand total

CUBE(a, b) génère tous les croisements possibles :

SQL
-- Totaux par (region, country), par region, par country, et global
SELECT region, country, SUM(sales)
FROM orders
GROUP BY CUBE(region, country);
-- Ajoute par rapport à ROLLUP : ligne "par country seul" (tous pays tous regions confondus)

C'est ce qui sous-tend les pivot tables d'Excel et les "tableaux croisés dynamiques" en BI (Tableau, Power BI).

B. Compléments avancés

Index et plan d'exécution

Un index est une structure auxiliaire (généralement un B-tree) qui maintient une ou plusieurs colonnes triées avec un pointeur vers les lignes. Conséquence : les requêtes WHERE indexed_col = X passent de $O(n)$ à $O(\log n)$.

Lecture : en notation big-O, $O(n)$ signifie que le temps est proportionnel au nombre de lignes (1 million de lignes = 1 million d'opérations). $O(\log n)$ signifie que le temps croît très lentement (1 million de lignes ≈ 20 opérations seulement, grâce au parcours dichotomique du B-tree).

Exemple chiffré : sur une table de 10M lignes, trouver une ligne spécifique :

  • Sans index : ~10M opérations, ~10 secondes
  • Avec index B-tree : ~$\log_2(10^7) \approx 23$ opérations, ~1 ms
  • Facteur d'accélération : ~10 000×
SQL
-- Index simple sur une colonne
CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_date ON orders(created_at);

-- Index composite (ordre des colonnes important !)
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);

-- Index partiel (PostgreSQL) : n'indexe qu'un sous-ensemble
CREATE INDEX idx_orders_active ON orders(user_id) WHERE status = 'active';
-- Bien plus petit si seulement 5% des orders sont "active"

-- Index unique (impose la contrainte d'unicité en plus d'accélérer)
CREATE UNIQUE INDEX idx_users_email ON users(email);

Règle de l'index composite : (a, b) accélère les requêtes filtrant sur a ou (a, b) mais pas sur b seul. C'est comme un annuaire trié par "nom puis prénom" — vous trouvez vite "Dupont Marie" mais pas tous les "Marie".

Pour comprendre ce que la base fait :

SQL
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 42 AND created_at > '2024-01-01';

Sortie typique :

Code
Index Scan using idx_orders_user_date on orders  (cost=0.42..8.45 rows=10 width=128)
  Index Cond: (user_id = 42 AND created_at > '2024-01-01')
  Actual time: 0.123..0.145 ms

Cherchez dans le plan :

  • Index Scan : utilise un index — bon
  • Seq Scan : parcourt toute la table — mauvais (sauf si la table est petite)
  • Hash Join vs Nested Loop : selon la taille des tables jointes
  • actual time : temps réel d'exécution (le plus important à optimiser)
⚠️

Piège des index : chaque index accélère les SELECT mais ralentit les INSERT/UPDATE/DELETE (il faut mettre à jour le B-tree). Sur une table très écrite, trop d'index tuent les performances d'écriture. Règle : créer un index seulement quand une requête lente est identifiée par EXPLAIN, pas "au cas où".

Optimisation des requêtes lentes

Cinq leviers, par ordre d'impact :

1. Ajouter un index sur les colonnes de WHERE, JOIN, ORDER BY. Effet souvent x100 à x1000.

2. Réécrire avec des CTE ou éviter les sous-requêtes corrélées (qui sont exécutées une fois par ligne extérieure).

SQL
-- Mauvais : subquery corrélée, O(n²)
SELECT u.*, (
    SELECT COUNT(*) FROM orders WHERE user_id = u.id
) AS n_orders                       -- exécuté pour CHAQUE utilisateur = n exécutions
FROM users u;
-- Avec 10 000 users et 1M orders : 10 000 × 1M opérations = 10 milliards

-- Bon : un seul GROUP BY puis JOIN, O(n + m)
SELECT u.*, COALESCE(o.n, 0) AS n_orders
FROM users u
LEFT JOIN (
    SELECT user_id, COUNT(*) AS n
    FROM orders
    GROUP BY user_id                -- un seul parcours des orders
) o ON u.id = o.user_id;
-- Avec 10 000 users et 1M orders : ~1M opérations. Facteur 10 000.

3. Limiter les colonnes lues : SELECT * lit toutes les colonnes, ce qui peut multiplier la taille du résultat. Lisez seulement ce dont vous avez besoin — impact I/O significatif sur des colonnes BLOB/TEXT.

4. LIMIT tôt : si vous voulez seulement le top 10, ajoutez LIMIT 10 plutôt que de tout calculer puis filtrer côté Python.

5. Matérialiser les agrégats coûteux : si la même agrégation est calculée 100 fois par jour, créez une vue matérialisée qui la précalcule.

SQL
-- Vue matérialisée : calculée une fois, stockée comme une table
CREATE MATERIALIZED VIEW user_stats AS
SELECT user_id, COUNT(*) AS n_orders, SUM(amount) AS total_spent
FROM orders
GROUP BY user_id;

-- À rafraîchir périodiquement (cron, trigger, ou manuellement)
REFRESH MATERIALIZED VIEW user_stats;
-- Désormais SELECT user_stats est instantané au lieu d'un GROUP BY à chaque appel
👉

Bonne pratique : avant d'optimiser, mesurez. EXPLAIN ANALYZE est votre meilleur ami. Beaucoup d'optimisations "intuitives" n'apportent rien parce que la base est déjà très intelligente. Les vraies victoires viennent de comprendre le plan réel et ses chiffres.

Partitioning et sharding

Partitioning : découper une grosse table en plusieurs sous-tables physiques selon une clé (souvent la date). Tout reste sur le même serveur.

SQL
-- Table maître partitionnée par date
CREATE TABLE orders (
    id BIGINT,
    user_id BIGINT,
    created_at TIMESTAMP,
    amount NUMERIC
) PARTITION BY RANGE (created_at);

-- Une partition par mois
CREATE TABLE orders_2024_01 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
CREATE TABLE orders_2024_02 PARTITION OF orders
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');
-- ... et ainsi de suite

Une requête WHERE created_at > '2024-02-15' ne touche que les partitions concernées (partition pruning), ce qui peut accélérer d'un facteur 100 sur des historiques de plusieurs années.

Exemple chiffré : table orders de 500M lignes sur 5 ans.

  • Sans partitionnement : une requête "commandes de février 2024" scanne 500M lignes
  • Avec partitionnement mensuel : scanne seulement ~8M lignes (la partition de février)
  • Gain : ×60 sur cette requête

Sharding : répartir une table sur plusieurs serveurs. Utilisé pour passer à l'échelle au-delà de ce qu'une machine peut stocker ou traiter. Le shard key (souvent un user_id hashé) détermine sur quel serveur va chaque ligne. Plus complexe : les jointures cross-shard sont coûteuses, les transactions distribuées sont fragiles (problème du "two-phase commit").

C'est ce que font Cassandra, MongoDB sharded clusters, Vitess (MySQL sharded), CockroachDB, Google Spanner.

⚠️

Conseil : ne shardez pas tant que vous pouvez l'éviter. Une seule machine PostgreSQL peut tenir des centaines de millions de lignes avec de bons index. Le sharding ajoute une complexité opérationnelle énorme (topologie, reshard, failover, observabilité). Question d'entretien architecture data : "Quand passer au sharding ?" — réponse : "quand le scale-up vertical (CPU/RAM/SSD) plus l'optimisation des requêtes ne suffisent plus".

C. Questions de compréhension

Q1 : Pourquoi WHERE created_at > '2024-01-01' AND status = 'paid' peut-il être beaucoup plus lent que WHERE status = 'paid' AND created_at > '2024-01-01' sur certaines bases ?

💡 Voir l'indice

Ordre des conditions dans un index composite.

✅ Voir la réponse

Sur les bases modernes (PostgreSQL, MySQL 8+, SQL Server), l'ordre des conditions dans le WHERE n'a aucune importance : l'optimiseur réorganise intelligemment. Cependant, l'ordre des colonnes dans un index composite est crucial.

  • Avec INDEX (status, created_at) : les deux versions de la requête utilisent l'index efficacement, peu importe l'ordre des conditions dans le WHERE.
  • Avec INDEX (created_at, status) : la condition status = 'paid' ne peut être utilisée qu'après avoir filtré par date. Si la date matche beaucoup de lignes (par exemple 80% de votre historique), l'index devient presque inutile pour le statut.

Règle : dans un index composite, mettez en première position la colonne qui filtre le plus drastiquement (la plus sélective). Calculez la sélectivité comme n_valeurs_distinctes / n_lignes_totales — plus c'est proche de 1, plus c'est sélectif.

SQL
-- Bon index pour cette requête (status = 2-3 valeurs, date = large plage)
CREATE INDEX idx_status_date ON orders(status, created_at);

-- Vérifier que l'index est bien utilisé
EXPLAIN ANALYZE SELECT * FROM orders
WHERE status = 'paid' AND created_at > '2024-01-01';
-- Doit afficher : Index Scan using idx_status_date

Pour des bases anciennes (Oracle 9, certains MySQL ≤ 5.6), l'ordre dans le WHERE pouvait avoir un impact réel — mais ce n'est plus le cas en 2025. Question d'entretien data engineering : "Comment concevoir un index pour cette requête ?" — la réponse révèle si le candidat a vraiment optimisé des requêtes en prod.

Q2 : Vous voulez calculer le pourcentage du chiffre d'affaires total représenté par chaque commande. Comment écrire cela en SQL ?

💡 Voir l'indice

Window function sans PARTITION BY.

✅ Voir la réponse
SQL
SELECT
    order_id,
    amount,
    100.0 * amount / SUM(amount) OVER () AS pct_of_total
    --                             ^^^^ window vide = total global
FROM orders;

L'astuce est SUM(amount) OVER () sans PARTITION BY ni ORDER BY : c'est une window function qui calcule la somme sur toutes les lignes sans regrouper. Chaque ligne reçoit la même valeur de total dans une nouvelle colonne (le grand total du dataset).

Exemple chiffré : 3 commandes de 100, 200, 700 (total = 1000).

order_idamountpct_of_total
110010.0
220020.0
370070.0

Sans window function, il faudrait faire un sous-SELECT qui calcule le total, puis joindre. La version window est plus lisible et souvent plus rapide (un seul scan de la table).

Variante par catégorie : pourcentage au sein de chaque catégorie plutôt qu'au total.

SQL
SELECT
    order_id,
    category,
    amount,
    100.0 * amount / SUM(amount) OVER (PARTITION BY category) AS pct_of_category
FROM orders;

Variante cumul : revenu cumulé jour par jour, essentiel pour les dashboards.

SQL
SELECT
    date,
    daily_revenue,
    SUM(daily_revenue) OVER (ORDER BY date) AS cumulative_revenue
FROM daily_sales;

C'est l'idiome canonique pour les calculs de pourcentages, de cumulés, ou de moyennes mobiles. Question d'entretien data analyst : "Comment calculer le revenu cumulé jour par jour ?" — réponse : SUM(daily_revenue) OVER (ORDER BY date).

Q3 : Vous avez une requête qui prend 30 secondes et EXPLAIN ANALYZE montre Seq Scan on orders (cost=0.00..1500000.00 rows=10000000). Quel est le diagnostic et la première chose à essayer ?

💡 Voir l'indice

Index manquant sur la colonne de WHERE.

✅ Voir la réponse

Seq Scan signifie que la base parcourt toutes les lignes de la table — 10 millions ici — pour appliquer le filtre. C'est presque toujours le signe d'un index manquant sur la colonne du WHERE.

Première action : créer un index sur la ou les colonnes du WHERE.

SQL
-- Avant : Seq Scan sur 10M lignes, ~30 secondes
SELECT * FROM orders WHERE user_id = 42 AND created_at > '2024-01-01';

-- Action : créer l'index
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at);

-- Après : Index Scan, ~50 ms
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42 AND created_at > '2024-01-01';
-- Index Scan using idx_orders_user_date on orders (cost=0.42..8.45 rows=10)

Effet typique x600. Sur des requêtes très sélectives, l'accélération peut dépasser x10 000.

Cas particulier : si le WHERE retourne plus de ~5-10% des lignes de la table, l'optimiseur peut volontairement choisir un Seq Scan parce que c'est plus rapide qu'un Index Scan (les pages contiguës se lisent plus vite que des accès aléatoires sur l'index, à cause du fonctionnement des disques). Dans ce cas, vous devez réduire la sélectivité du filtre, ou repenser le schéma (partitionner par date par exemple).

Checklist complète quand vous voyez un Seq Scan lent :

  1. Y a-t-il un index sur la colonne du WHERE ?
  2. L'index existe-t-il mais n'est pas utilisé ? (stats obsolètes → ANALYZE orders;)
  3. Le filtre est-il trop peu sélectif ? (envisager un index partiel)
  4. La table est-elle trop grosse pour un seul serveur ? (envisager partitioning)

Toujours mesurer après : un index ne sert à rien si l'optimiseur l'ignore pour de bonnes raisons. EXPLAIN ANALYZE avant/après est le seul juge fiable.
