Règles fondamentales :

  1. Splitter AVANT tout preprocessing
  2. Ne jamais optimiser le random_state.

"Optimiser le random_state" veut dire : essayer random_state=0, 1, 2, 3... et garder celui qui donne le meilleur score. En faisant ça, tu ne choisis pas un meilleur modèle — tu choisis un découpage chanceux. Ton score ne reflète plus la vraie performance du modèle, il reflète ta chance au tirage.

  1. Le test set ne sert qu'UNE FOIS à la fin
  2. Classification déséquilibrée → stratify=y

❌ Erreur fatale n°1 : scaler/encoder AVANT le split = data leakage.

Exemples typiques

Ex 2 — OneHotEncoder (catégories sans ordre) :

Python
from sklearn.preprocessing import OneHotEncoder
ohe = OneHotEncoder(sparse_output=False, drop='first')
ohe.fit(X_train[['city']])

encoded_train = ohe.transform(X_train[['city']])
encoded_test = ohe.transform(X_test[['city']])
print(ohe.get_feature_names_out())   # → ['city_London', 'city_Paris']
Code
**sparse_output=False** — Par défaut, le OneHotEncoder retourne une matrice sparse (creuse). 
Une matrice sparse est un format optimisé pour stocker des matrices avec beaucoup de zéros 
— ce qui est le cas du OHE (chaque ligne n'a qu'un seul 1, tout le reste est 0). 
C'est efficace en mémoire quand tu as des centaines de catégories, mais c'est difficile à manipuler : 
tu ne peux pas le mettre directement dans un DataFrame, tu ne peux pas le visualiser avec print() facilement. 
Avec sparse_output=False, tu récupères un array numpy classique, plus pratique pour travailler.

drop='first' — C'est la solution au piège de la multicolinéarité. Prenons l'exemple de embark_town avec 3 villes :
Sans drop='first', le OHE crée 3 colonnes

Le problème : si tu connais deux colonnes, tu déduis la troisième (Cherbourg + Southampton + Queenstown = 1 toujours). 
C'est une redondance parfaite. 

Sans régularisation, le modèle ne sait pas comment répartir les coefficients entre les 3 → ils deviennent instables et explosent.
Avec drop='first', on supprime la première catégorie (Cherbourg) :
On ne perd aucune information — Cherbourg est implicitement représenté quand les deux autres sont à 0. Mais on élimine la redondance. 
Cherbourg devient la catégorie de référence et les coefficients des autres villes s'interprètent par rapport à elle.

Ex 3 — OHE SANS drop='first' → piège multicolinéarité :

Python
## embark_town : Cherbourg, Southampton, Queenstown
## OHE crée 3 colonnes : C + S + Q = 1 TOUJOURS → multicolinéarité parfaite
## → Sans régularisation : coefficients instables, explosent
## → Avec Lasso (L1) : met les redondants à 0 automatiquement
## → Solution : drop='first' OU régulariser

Python

## ============================================================

#

##    3 villes → 3 colonnes → C1 + C2 + C3 = 1 toujours

#

6.2 Modèles de régression

Ex 1 — LinearRegression (OLS)

Python
from sklearn.linear_model import LinearRegression
model = LinearRegression()
model.fit(X_train, y_train)
print(f"R²: {model.score(X_test, y_test):.3f}")

## ŷ = β₀ + β₁X₁ + ... | Minimise Σ(y-ŷ)²

Ex 2 — Ridge (L2 — réduit coefs vers 0)

Python
from sklearn.linear_model import Ridge
ridge = Ridge(alpha=1)   # alpha = force de régularisation
ridge.fit(X_train_scaled, y_train)   # ⚠️ scaling obligatoire!
## Loss = Σ(y-ŷ)² + α·Σβⱼ²

Ex 3 — Lasso (L1 — annule coefs, feature selection)

Python
from sklearn.linear_model import Lasso
lasso = Lasso(alpha=1)
lasso.fit(X_train_scaled, y_train)
## Loss = Σ(y-ŷ)² + α·Σ|βⱼ| → certains coefs = exactement 0
print(f"Features éliminées: {(lasso.coef_ == 0).sum()}")

Ex 4 — ElasticNet (mix L1+L2)

Python
from sklearn.linear_model import ElasticNet
elastic = ElasticNet(alpha=1, l1_ratio=0.5)
## l1_ratio=0 → Ridge | l1_ratio=1 → Lasso
Python
Ridge → tu veux garder toutes les features, juste calmer les coefficients.
Lasso → tu veux que le modèle élimine lui-même les features inutiles (coefs = 0).
ElasticNet → tu hésites, ou tu as des features corrélées en groupes. Tu laisses l1_ratio décider via GridSearch.
En cas de doute → Ridge. C'est rarement nuisible.

💡 On ne régularise JAMAIS β₀. Scaling OBLIGATOIRE pour Ridge/Lasso/ElasticNet.

Ridge (L2)Lasso (L1)
Effet α++Coefs → 0Coefs = 0 (sélection)
MulticolinéaritéGère bienGarde 1 feature
QuandToutes features utilesSélection features

C vs alpha — dans Ridge/Lasso, alpha grand = forte régularisation. 

Dans LogisticRegression, le paramètre s'appelle C et c'est l'inverse : C = 1/alpha. 
Donc C grand = peu régularisé, C petit = forte régularisation. Le réflexe est inversé.

Interprétation des coefficients — la LogisticRegression prédit des log-odds en interne. 
Un coef positif signifie que la feature augmente la probabilité de la classe positive. 
Pour rendre ça concret, on prend exp(coef) qui donne l'odds ratio : de combien les chances sont multipliées. 
Exemple sur le Titanic : exp(2.67) = 14 → une femme avait 14 fois plus de chances de survivre qu'un homme, toutes choses égales par ailleurs.

🧭 Introduction

Le machine learning (ML), c'est l'art d'apprendre à une machine à faire des prédictions à partir de données. Mais entre le moment où tu récupères tes données brutes et celui où ton modèle prédit correctement, il y a tout un pipeline à suivre — une suite d'étapes dans un ordre précis.

Ce cours couvre ce pipeline en entier, du début à la fin.

🧠

Les 5 choses absolument à retenir avant de commencer :

1. Le pipeline a un ordre strict — ne jamais modéliser sans avoir exploré et nettoyé les données.

2. La règle d'or du split — toutes les transformations (scaling, imputation) doivent être apprises sur le train set uniquement, puis appliquées sur le test set.

3. Data leakage = ennemi public n°1 — si le modèle "voit" le test avant l'évaluation, le score est bidon.

4. Baseline d'abord — un modèle bête sert de référence. S'il fait mieux que le tien, il y a un problème.

5. Cross-validation > test set unique — évaluer sur 5 folds donne une estimation bien plus fiable.

Analogie fil rouge : la ligne de production

Imagine une usine qui fabrique des voitures. Les pièces brutes arrivent (COLLECT), on les inspecte (EDA), on enlève les défauts (CLEAN), on partage le stock en deux (SPLIT), on usine les pièces aux bonnes dimensions (PREPROCESS), on assemble le moteur (MODEL), on règle les réglages fins (TUNE), puis on teste le véhicule fini (EVALUATE). Si tu usines les pièces avant de les trier, tu gâches du travail. Si tu testes en utilisant les pièces que tu as déjà inspectées, ton test ne vaut rien.

3.3 Outliers

Un outlier est une valeur anormalement éloignée des autres. Il peut être :

  • une erreur de saisie → supprimer
  • une valeur extrême réelle → garder, mais adapter le scaler

Détection par la méthode IQR (Inter-Quartile Range) :

$$\text{IQR} = Q_3 - Q_1$$
$$\text{Seuil bas} = Q_1 - 1.5 \times \text{IQR}, \quad \text{Seuil haut} = Q_3 + 1.5 \times \text{IQR}$$

Intuition : tout ce qui est au-delà de 1.5 fois l'écart interquartile est considéré suspect.

Exemple numérique : si $Q_1 = 100$, $Q_3 = 200$, alors $\text{IQR} = 100$, seuil bas = $-50$, seuil haut = $350$.

Python
Q1 = data['col'].quantile(0.25)   # 1er quartile
Q3 = data['col'].quantile(0.75)   # 3ème quartile
IQR = Q3 - Q1
lower = Q1 - 1.5 * IQR
upper = Q3 + 1.5 * IQR
data = data[(data['col'] >= lower) & (data['col'] <= upper)]  # garder l'intervalle
⚠️

Toujours comprendre le contexte métier avant de supprimer un outlier. Une maison à 600 000 $ dans un dataset moyen à 180 000 $ est une vraie maison chère — pas une erreur. Dans ce cas, utilise RobustScaler plutôt que de supprimer la valeur.

Les 4 règles du split :

1. Splitter AVANT tout preprocessing

2. Ne jamais choisir le random_state qui donne le meilleur score (c'est tricher)

3. Le test set ne sert qu'UNE SEULE FOIS, à la toute fin

4. Dataset déséquilibré → toujours utiliser stratify=y

Étape 5 — PREPROCESS : préparer les features

La règle absolue : .fit() uniquement sur le train → .transform() sur train ET test.

5.1 Feature Scaling (mise à l'échelle)

Certains modèles (KNN, SVM, Ridge, Lasso, LogisticRegression régularisée) sont sensibles aux échelles des features. Si une feature vaut entre 0 et 1 000 000 et une autre entre 0 et 1, le modèle va accorder trop d'importance à la première.

ScalerFormuleQuand l'utiliser
StandardScaler$z = \frac{x - \mu}{\sigma}$Par défaut, distributions proches du normal
MinMaxScaler$x' = \frac{x - x_{min}}{x_{max} - x_{min}}$ (dans [0,1])KNN, réseaux de neurones, features positives
RobustScaler$x' = \frac{x - \text{mediane}}{\text{IQR}}$Quand il y a des outliers

Intuition de StandardScaler : $\mu$ est la moyenne, $\sigma$ l'écart-type. Après transformation, la moyenne devient 0 et l'écart-type devient 1. Exemple : températures [10, 20, 30] → $\mu=20$, $\sigma=10$ → [-1, 0, 1].

Python
from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # apprend μ et σ sur le train, et transforme
X_test_scaled = scaler.transform(X_test)         # utilise les mêmes μ et σ appris, PAS fit_transform!
Python
from sklearn.preprocessing import RobustScaler

scaler = RobustScaler()                          # utilise médiane et IQR → résistant aux outliers
X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)
⚠️

scaler.fit_transform(X_test) = DATA LEAKAGE. On apprendrait les paramètres du scaler sur les données de test — interdit. Seul .transform() sur le test.

5.2 Encodage des variables catégorielles

Les modèles ML ne savent pas lire du texte ("Paris", "Lyon"). Il faut transformer les catégories en nombres.

OrdinalEncoder — pour les catégories avec un ordre naturel :

Python
from sklearn.preprocessing import OrdinalEncoder

# "quality" a un ordre : bad < average < good
encoder = OrdinalEncoder(categories=[['bad', 'average', 'good']])
encoder.fit(X_train[['quality']])
X_train['quality_enc'] = encoder.transform(X_train[['quality']])  # bad=0, average=1, good=2
X_test['quality_enc'] = encoder.transform(X_test[['quality']])

OneHotEncoder — pour les catégories SANS ordre :

Python
from sklearn.preprocessing import OneHotEncoder

# "city" : Paris, Lyon, Marseille — aucun ordre
ohe = OneHotEncoder(sparse_output=False, drop='first')
# sparse_output=False → tableau numpy classique (plus facile à manipuler)
# drop='first' → évite la multicolinéarité parfaite (voir explication ci-dessous)
ohe.fit(X_train[['city']])
encoded_train = ohe.transform(X_train[['city']])
encoded_test = ohe.transform(X_test[['city']])
print(ohe.get_feature_names_out())   # → ['city_Lyon', 'city_Paris']
🧠

Pourquoi drop='first' ? Si tu as 3 villes (Paris, Lyon, Marseille), OHE crée 3 colonnes de 0/1. Mais la somme des 3 colonnes vaut toujours 1 : si tu sais que Paris=0 et Lyon=0, tu déduis forcément Marseille=1. C'est une redondance parfaite qui rend les coefficients instables. En supprimant la première colonne, Marseille devient implicite (les deux autres à 0 = Marseille), et la redondance disparaît.

LabelEncoder — UNIQUEMENT pour encoder la target y :

Python
from sklearn.preprocessing import LabelEncoder

le = LabelEncoder()
y_encoded = le.fit_transform(y)         # 'Adelie'=0, 'Chinstrap'=1, 'Gentoo'=2
y_original = le.inverse_transform(y_encoded)  # décodage
⚠️

Résumé des erreurs classiques d'encodage :

- LabelEncoder sur une feature X → invente un faux ordre (Paris=2 > Lyon=0, ce qui n'a aucun sens)

- OrdinalEncoder sur des catégories sans ordre → même problème

- OneHotEncoder sans drop='first' → multicolinéarité parfaite

- LabelEncoder attend une Series 1D — pour les features X, utiliser OrdinalEncoder ou OHE

Étape 6 — MODEL : construire et évaluer le modèle

6.1 Commencer par la baseline (obligatoire)

Avant tout modèle sérieux, on entraîne un modèle "bête" qui sert de référence minimum.

Python
from sklearn.dummy import DummyRegressor

💡

Si votre modèle ne fait pas mieux que la baseline (qui prédit toujours la moyenne), il y a un problème fondamental — dans les données ou dans le code.

6.2 Modèles de régression (prédire un nombre)

LinearRegression — le plus simple :

$$\hat{y} = \beta_0 + \beta_1 X_1 + \beta_2 X_2 + \ldots + \beta_n X_n$$

Le modèle cherche les coefficients $\beta$ qui minimisent la somme des erreurs au carré : $\sum (y - \hat{y})^2$.

Python
from sklearn.linear_model import LinearRegression

model = LinearRegression()
model.fit(X_train, y_train)              # apprend les coefficients β
print(f"R²: {model.score(X_test, y_test):.3f}")
print(f"Coefficients: {model.coef_}")   # un β par feature
print(f"Intercept: {model.intercept_}") # β₀

Ridge (régression avec régularisation L2) :

$$\text{Loss} = \underbrace{\sum (y - \hat{y})^2}_{\text{erreur classique}} + \underbrace{\alpha \sum \beta_j^2}_{\text{pénalité L2}}$$

Intuition : on pénalise les coefficients trop grands. Plus $\alpha$ est élevé, plus les coefficients sont poussés vers 0 (mais jamais exactement 0).

Python
from sklearn.linear_model import Ridge

ridge = Ridge(alpha=1)                          # alpha = force de la pénalité
ridge.fit(X_train_scaled, y_train)              # ⚠️ scaling OBLIGATOIRE

Lasso (régularisation L1 — fait de la sélection de features) :

$$\text{Loss} = \sum (y - \hat{y})^2 + \alpha \sum |\beta_j|$$

Différence clé avec Ridge : la pénalité en valeur absolue pousse certains coefficients à être exactement 0 → le modèle élimine lui-même les features inutiles.

Python
from sklearn.linear_model import Lasso

lasso = Lasso(alpha=1)
lasso.fit(X_train_scaled, y_train)
print(f"Features éliminées: {(lasso.coef_ == 0).sum()}")  # combien de coefs = 0

ElasticNet — mix Ridge + Lasso :

Python
from sklearn.linear_model import ElasticNet

elastic = ElasticNet(alpha=1, l1_ratio=0.5)  # l1_ratio=0 → Ridge, l1_ratio=1 → Lasso
elastic.fit(X_train_scaled, y_train)
🧠

Quand utiliser quoi ?

- Ridge : tu veux garder toutes les features, juste "calmer" les coefficients → cas général

- Lasso : tu veux que le modèle élimine lui-même les features inutiles

- ElasticNet : tu hésites, ou tu as des groupes de features corrélées

- En cas de doute → Ridge. C'est rarement nuisible.

⚠️

Scaling OBLIGATOIRE pour Ridge, Lasso, ElasticNet. Sans ça, les pénalités sont appliquées de façon inégale selon les échelles des features.

6.3 Modèles de classification (prédire une classe)

LogisticRegression :

Malgré son nom, c'est bien un modèle de classification. Il prédit une probabilité entre 0 et 1 via la fonction sigmoïde :

$$P(y=1) = \frac{1}{1 + e^{-(\beta_0 + \beta_1 X_1 + \ldots)}}$$
Python
from sklearn.linear_model import LogisticRegression

# Sans régularisation
model = LogisticRegression(penalty=None, max_iter=5000)
model.fit(X_scaled, y)

# Avec régularisation L2 (C = 1/alpha → C petit = forte régularisation)
model = LogisticRegression(penalty='l2', C=0.1, max_iter=5000)
model.fit(X_scaled, y)

# Avec L1 (feature selection) — solver='liblinear' requis pour L1 !
model = LogisticRegression(penalty='l1', C=0.1, solver='liblinear', max_iter=5000)
model.fit(X_scaled, y)
⚠️

Attention au paramètre C : dans Ridge/Lasso, alpha grand = forte régularisation. Dans LogisticRegression, C est l'inverse : C = 1/alpha. Donc C petit = forte régularisation. Le réflexe est inversé !

KNN (K plus proches voisins) :

Pour prédire, on regarde les K voisins les plus proches dans l'espace des features et on vote (classification) ou on moyenne (régression).

Python
from sklearn.neighbors import KNeighborsRegressor
from sklearn.model_selection import cross_validate

# Trouver le bon K avec la méthode du coude
scores = []
for k in range(1, 51):
    model = KNeighborsRegressor(n_neighbors=k)
    cv = cross_validate(model, X_train_scaled, y_train, cv=5)
    scores.append(cv['test_score'].mean())

plt.plot(range(1, 51), scores)
plt.xlabel('K (nombre de voisins)'); plt.ylabel('R²')
# Chercher le "coude" : le K à partir duquel le score se stabilise
⚠️

K=1 = overfit garanti (le modèle mémorise les données). K trop grand = underfit (trop lissé). Scaling OBLIGATOIRE — KNN mesure des distances et est totalement faussé par des échelles différentes.

SVM (Support Vector Machine) :

Le SVM cherche la frontière de décision qui sépare les classes avec la marge maximale. Le "kernel" permet de gérer des frontières non-linéaires.

Python
from sklearn.svm import SVC

svc = SVC(kernel='rbf', C=1, gamma=0.1)

KernelFormuleQuand
linear$a^Tb$Données linéairement séparables
rbf$e^{-\gamma \a-b\^2}$Cas général (défaut)
poly$(a^Tb + c)^d$Relations polynomiales

6.4 Cross-Validation

Au lieu d'évaluer sur un seul split train/test, la cross-validation coupe le train set en K folds et évalue K fois. On obtient K scores → on prend la moyenne.

Python

from sklearn.pipeline import make_pipeline

print(f"Écart-type: {cv['test_score'].std():.3f}")

# Multi-métriques
cv = cross_validate(model, X, y, cv=5,
    scoring=['r2', 'neg_mean_absolute_error'])
⚠️

cross_validate ne retourne PAS de modèle entraîné. Il sert uniquement à estimer la performance. Pour obtenir les prédictions finales, faire model.fit() séparément après.

Étape 7 — TUNE : optimiser les hyperparamètres

Params vs Hyperparamètres

  • Paramètres ($\beta_0, \beta_1, \ldots$) : appris automatiquement par .fit() — tu ne les choisis pas
  • Hyperparamètres (alpha, C, K, gamma) : choisis par toi avant l'entraînement — c'est ça le tuning

Le tuning cherche les hyperparamètres qui maximisent la métrique (par CV sur le train set — jamais sur le test).

7.1 GridSearchCV — exploration exhaustive

On définit une grille de valeurs à tester, GridSearch les teste toutes.

Python
from sklearn.model_selection import GridSearchCV
from sklearn.neighbors import KNeighborsRegressor

# Étape 1 — grille grossière (coarse) : trouver la zone
search = GridSearchCV(
    KNeighborsRegressor(),
    {'n_neighbors': [1, 5, 10, 20, 50], 'p': [1, 2, 3]},  # grille à tester
    scoring='r2',   # métrique d'optimisation
    cv=5,           # 5-fold CV pour chaque combinaison
    n_jobs=-1       # paralléliser sur tous les CPU
)
search.fit(X_train_scaled, y_train)
print(search.best_params_)    # → {'n_neighbors': 10, 'p': 2}
print(search.best_score_)     # → 0.72

# Étape 2 — grille fine : affiner autour du best
search2 = GridSearchCV(
    KNeighborsRegressor(),
    {'n_neighbors': list(range(6, 16)), 'p': [1, 2]},  # zone resserrée
    scoring='r2', cv=5, n_jobs=-1
)
search2.fit(X_train_scaled, y_train)

Fonctions de perte (Loss functions)

⚠️

Loss ≠ Métrique ! La loss est ce que .fit() minimise (elle doit être dérivable pour le gradient descent). La métrique est ce qu'on utilise pour évaluer (lisible par un humain). R² est une métrique, MSE peut être les deux.

  • Régression : MSE (sensible aux outliers) | MAE (robuste) | Huber (compromis)
  • Classification : Log-Loss (LogisticRegression) | Hinge (SVM)

Descente de gradient

Comment le modèle apprend : il met à jour les coefficients $\beta$ dans la direction qui réduit l'erreur.

$$\beta^{(k+1)} = \beta^{(k)} - \eta \cdot \nabla L(\beta^{(k)})$$

où $\eta$ est le learning rate (taille du pas) et $\nabla L$ est le gradient de la loss.

MéthodeObservations par étapeTrade-off
Gradient Descent completToutesStable mais très lent
Mini-Batch GD16–64Compromis (le plus utilisé)
SGD (stochastique)1 seuleRapide mais bruité

🎯 Questions de vérification

Q1 — Pipeline et data leakage :

Tu charges un dataset, tu appliques un StandardScaler sur l'intégralité des données, puis tu fais le train/test split. Quel est le problème ? Comment le corriger ?

Réponse

C'est du data leakage : le scaler a "vu" les données de test pour calculer μ et σ. Le test set est contaminé. Solution : toujours splitter AVANT de scaler. Mieux encore : utiliser un Pipeline sklearn pour que le scaling soit automatiquement fait après le split dans chaque fold de CV.

Q2 — Choix d'encodeur :

Tu as une feature "niveau d'éducation" avec les valeurs : "lycée", "licence", "master", "doctorat". Quel encodeur utilises-tu et pourquoi ? Qu'est-ce qui se passerait avec un OneHotEncoder ?

Réponse

OrdinalEncoder avec l'ordre explicite ['lycée', 'licence', 'master', 'doctorat']. Il y a un ordre naturel. Un OneHotEncoder créerait 4 colonnes binaires, perdant l'information d'ordre et augmentant la dimensionnalité inutilement.

Q3 — Biais/Variance :

Un modèle Ridge donne R²=0.92 sur le train et R²=0.61 sur le test. Quel est le diagnostic ? Quelles sont deux solutions possibles ?

Réponse

Overfit (forte variance). Solutions : augmenter alpha pour renforcer la régularisation, obtenir plus de données d'entraînement, supprimer des features peu informatives, utiliser la cross-validation pour mieux estimer le vrai score.

Q4 — Classification déséquilibrée :

Dataset de détection de fraude : 99% de transactions légitimes, 1% frauduleuses. Ton modèle atteint 99% d'accuracy. Est-il bon ? Quelle métrique regarder en priorité et pourquoi ?

Réponse

Non — un modèle qui prédit "légitime" à chaque fois atteint 99% d'accuracy. Il est inutile. La métrique prioritaire est le Recall (ou F1) : on veut surtout ne pas rater les fraudes (Recall = TP/(TP+FN)). Un Recall de 0% est catastrophique même avec 99% d'accuracy.

📌 TL;DR

  1. Pipeline strict : COLLECT → EDA → CLEAN → SPLIT → PREPROCESS → MODEL → TUNE → EVALUATE.
  2. Règle d'or : fit sur le train uniquement, transform sur train ET test — jamais l'inverse.
  3. Data leakage : scaler/encoder avant le split = résultats biaisés → utiliser Pipeline sklearn.
  4. Split : toujours avant tout preprocessing ; stratify=y si classes déséquilibrées.
  5. Scaling obligatoire pour KNN, SVM, Ridge, Lasso, ElasticNet, LogisticRegression régularisée.
  6. Encodage : OrdinalEncoder pour l'ordre, OneHotEncoder(drop='first') pour sans ordre, LabelEncoder uniquement pour y.
  7. Biais/Variance : underfit → modèle plus complexe ; overfit → régulariser ou plus de données.
  8. Tuning : GridSearchCV coarse puis fine, ou RandomizedSearchCV — toujours sur le train (CV), jamais sur le test.
  9. Classification : l'accuracy est trompeuse sur dataset déséquilibré → Precision, Recall, F1, AUC.
  10. Test set : utilisé UNE SEULE FOIS à la fin pour la mesure de performance définitive.

A. Concepts du cours

Le pipeline ML end-to-end

Tout projet de machine learning sérieux suit le même pipeline, indépendamment du domaine ou de l'algorithme. Que vous fassiez de la détection de fraude bancaire, de la recommandation de films ou de la prédiction de maintenance industrielle, vous retrouverez les mêmes 5 grandes étapes. Les confondre ou en sauter une est la cause numéro un des échecs de projets ML en entreprise.

Code
1. Définition du problème  (clarifier l'objectif business)
   ↓
2. Collecte + exploration des données  (EDA)
   ↓
3. Préparation + feature engineering  (cleaning, encoding, scaling)
   ↓
4. Modélisation (train + validation)  (baseline → modèles complexes)
   ↓
5. Évaluation et tuning  (CV, métriques, interprétation)
   ↓
6. Déploiement et monitoring  (API, batch, drift)
   ↓
7. Maintenance et amélioration continue  (retraining, versioning)

Sans ce cadre, on bricole. Avec ce cadre, on industrialise. Chaque étape a ses outils, ses pièges, et ses livrables.

💡

Analogie : un projet ML, c'est comme construire une maison. Vous ne commencez pas par les murs sans plan. Vous étudiez le terrain (EDA), dessinez les plans (définition du problème), posez les fondations (preprocessing), puis montez les murs (modèle) et finissez l'intérieur (déploiement). Sauter une étape — "on commence à entraîner avant de regarder la donnée" — garantit que vous reconstruirez tout deux fois.

Étape 1 — Définition du problème (10% du temps, 50% de l'impact)

C'est l'étape la plus courte et la plus décisive. Quatre questions à clarifier avant d'ouvrir un notebook :

  • Quel est l'objectif business ? "Réduire le churn de 5% d'ici 6 mois" est précis. "Améliorer la rétention" est une lettre au père Noël.
  • Quelle est la métrique de succès ? F1, RMSE, ROC-AUC, lift sur le top 10%, ROI estimé ?
  • Quelle est la baseline actuelle ? Sans ML, comment décide-t-on aujourd'hui ? Si le métier fait déjà 82% de bonnes prédictions à la main, un modèle à 83% n'apporte rien.
  • Quelle est la valeur business attendue ? Si elle est inférieure au coût du projet (compute + salaires + maintenance), abandonnez.

Exemple concret — churn télécom :

ÉlémentValeur
ObjectifRéduire le churn mensuel de 4.5% à 3.5%
Métrique techniqueRecall@top-10% (capturer 80% des churners dans le top 10%)
Baseline métierCiblage aléatoire : recall ≈ 10%
Baseline ML trivialeRègle "client inactif 30j" : recall ≈ 35%
Gain estimé1 point de churn évité ≈ 2M€/an
Budget projet120k€ (2 DS × 3 mois + infra)

ROI estimé = 2M / 0.12 = 16×. Le projet a du sens. Si le gain était de 100k€, il faudrait refuser.

Étape 2 — Données et EDA (40% du temps)

C'est l'étape la plus longue et souvent la plus ingrate, mais celle qui apporte le plus de valeur. Trois sous-étapes :

(a) EDA rapide (premier coup d'œil) — 1 journée max :

Python
import pandas as pd

df = pd.read_csv("customers.csv")

# Taille et types : repérer immédiatement si on a affaire à 1k ou 10M lignes
print(df.shape)                      # ex : (50_000, 42)
print(df.dtypes.value_counts())      # combien de num / cat / date

# Statistiques de base : détecter les ordres de grandeur suspects
df.describe(include="all").T

# Missing values : toute colonne > 30% de manquants est suspecte
print(df.isnull().mean().sort_values(ascending=False).head(10))

# Cible : équilibrée ou pas ? Si 1% de positifs, c'est du très déséquilibré
print(df["churned"].value_counts(normalize=True))

(b) Cleaning — doublons, types, dates, valeurs aberrantes.

(c) Feature engineering — c'est ici que se gagnent les gains les plus importants. Pour le churn télécom, ajouter ratio_appels_semaine_vs_mois, jours_depuis_derniere_connexion, ou delta_facture_M-1_vs_M-3 peut faire gagner 5 à 10 points de F1 — beaucoup plus que passer de RF à XGBoost.

⚠️

Piège classique : négliger l'EDA et plonger directement dans la modélisation. Résultat typique : un modèle qui atteint 99% d'accuracy… parce qu'il a appris que customer_id est corrélé à la cible (fuite de données). En EDA, vous auriez vu customer_id.nunique() == len(df) et éliminé cette feature.

Étape 3 — Modélisation (20% du temps)

Discipline de fer : toujours commencer par une baseline triviale, puis monter en complexité uniquement si ça apporte quelque chose.

Exemple chiffré typique (projet churn, F1-score sur classe positive) :

ModèleF1 validationTemps d'entraînementCommentaire
DummyClassifier(strategy="most_frequent")0.00< 1 sPoint zéro obligatoire
Règle métier "inactif 30j"0.420 sÀ battre impérativement
Régression logistique0.585 sPremière vraie baseline
Random Forest (défauts)0.7130 sGrand saut
XGBoost (défauts)0.7445 s+3 pts sur RF
XGBoost tuné (Optuna 100 essais)0.772 h tuning+3 pts supplémentaires
Stacking RF+XGB+LogReg0.783 h+1 pt pour 3× la complexité — à éviter en prod

Le bon choix ici est XGBoost tuné : meilleur ratio performance / complexité.

Python
from sklearn.dummy import DummyClassifier
from sklearn.linear_model import LogisticRegression
from sklearn.ensemble import RandomForestClassifier
from xgboost import XGBClassifier
from sklearn.model_selection import cross_val_score

# Baseline triviale : répond toujours la classe majoritaire
# → utile pour savoir quel score "gratuit" on bat
dummy = DummyClassifier(strategy="most_frequent")

# LogReg = première vraie baseline explicable
logreg = LogisticRegression(max_iter=1000, class_weight="balanced")

# RF = robuste, peu de tuning
rf = RandomForestClassifier(n_estimators=300, random_state=42, n_jobs=-1)

# XGBoost = performance state-of-the-art sur tabulaire
xgb = XGBClassifier(eval_metric="logloss", tree_method="hist")

for name, model in [("Dummy", dummy), ("LogReg", logreg), ("RF", rf), ("XGB", xgb)]:
    scores = cross_val_score(model, X_train, y_train, cv=5, scoring="f1")
    # On affiche moyenne ± écart-type : un modèle "meilleur" mais instable
    # (gros écart-type) est un drapeau rouge
    print(f"{name:8s}: {scores.mean():.3f} ± {scores.std():.3f}")

Étape 4 — Évaluation rigoureuse (10% du temps)

Une bonne évaluation repose sur trois piliers : cross-validation, métrique business, et interprétation.

  • Cross-validation : jamais un seul split. Toujours 5-fold stratifié minimum. Pour des séries temporelles, utilisez TimeSeriesSplit — un split aléatoire entraîne du leakage temporel.
  • Métriques business : accuracy ne veut rien dire sur un dataset à 1% de positifs. Préférez F1, PR-AUC, ou mieux encore une métrique $ (ex : "€ économisés par prédiction correcte moins € perdus par erreur").
  • Interprétation : SHAP values sur quelques centaines d'exemples pour voir ce que le modèle a appris. Si la feature numéro 1 est customer_id, vous avez du leakage.

Étape 5 — Déploiement et monitoring (20% du temps)

Un modèle qui n'est pas en production n'existe pas. Le plus petit déploiement utile c'est :

  1. Sérialiser le pipeline complet (preprocessing + modèle) avec joblib.dump(pipe, "model.pkl").
  2. Exposer via FastAPI ou un batch job Airflow.
  3. Monitorer deux choses : (a) la performance (accuracy qui baisse = retraining), (b) le drift des inputs (distribution des features qui change = alerte).
⚠️

Erreur classique : passer 80% du temps sur la modélisation et 5% sur les données. La vraie répartition d'un projet pro est ~40% data, 20% modélisation, 40% production. Les juniors inversent cette proportion et leurs projets échouent en prod.

Si une régression logistique atteint 85% et XGBoost 87%, gardez la régression : elle est plus simple, plus rapide, plus interprétable, et 2 points ne justifient probablement pas la complexité supplémentaire en production.

Méthodologie d'itération

Un bon data scientist itère rapidement et discipliné. Pattern type :

  1. Hypothèse : "Je pense que ajouter la feature jours_depuis_derniere_connexion va améliorer le F1 de 3%."
  2. Implémentation : ajouter la feature dans le pipeline.
  3. Mesure : cross-validation 5-fold.
  4. Décision : keep ou rollback selon le résultat.
  5. Documentation : tracker dans MLflow le résultat de l'expérience pour ne pas retester la même chose dans 2 semaines.
Python
import mlflow
from sklearn.model_selection import cross_val_score

# On isole CHAQUE expérience pour pouvoir la comparer ensuite
with mlflow.start_run(run_name="add_feature_days_since_login"):
    # Tracer l'hypothèse testée : essentiel pour relire l'historique 3 mois plus tard
    mlflow.log_param("hypothesis", "days_since_login improves F1 by 3%")
    mlflow.log_param("baseline_f1", 0.74)

    # Entraîner avec la nouvelle feature
    pipe = build_pipeline(extra_features=["days_since_login"])
    scores = cross_val_score(pipe, X_train, y_train, cv=5, scoring="f1")

    # Moyenne + écart-type : une moyenne seule peut cacher un modèle instable
    mlflow.log_metric("cv_f1_mean", scores.mean())
    mlflow.log_metric("cv_f1_std", scores.std())

    # Résultat attendu : 0.74 → 0.77. Si non, rollback.
💡

Analogie : l'itération en ML est comme l'optimisation d'une recette de pâtisserie. Vous changez un ingrédient à la fois et goûtez. Si vous changez 5 ingrédients en même temps et que c'est meilleur, vous ne savez pas lequel a aidé. C'est exactement pareil avec les hyperparamètres et le feature engineering : un changement à la fois, mesure rigoureuse, conclusion claire.

B. Compléments avancés

Le bon ordre des optimisations

Quand votre baseline est insuffisante, dans quel ordre essayer les améliorations ? Voici l'ordre par rendement décroissant, tel qu'observé dans la plupart des projets industriels :

OrdreLevierGain typiqueCoût
1Plus de données (si possible)+3 à +10 ptsSouvent très cher / impossible
2Meilleur feature engineering+5 à +10 ptsFaible (une journée de cerveau)
3Meilleur preprocessing (outliers, encoding)+1 à +3 ptsFaible
4Modèle plus puissant (RF → XGBoost)+2 à +5 ptsMoyen
5Hyperparameter tuning (Optuna)+1 à +3 ptsMoyen (compute)
6Ensemble / stacking+1 à +2 ptsÉlevé (complexité prod)
7Architecture custom / deep learning+0 à +2 ptsTrès élevé

Commencer par tuner finement avant d'avoir fait du feature engineering est l'erreur la plus courante : on gagne 1% au prix de 2 semaines, au lieu de gagner 8% en une journée de réflexion sur les features.

Top 10 des erreurs en ML applicatif

  1. Pas de baseline : impossible de savoir si votre modèle apporte quelque chose.
  2. Métrique mal choisie : 99% d'accuracy sur un dataset à 1% de positifs… le modèle qui dit toujours "non" a aussi 99%.
  3. Data leakage : train/test pas vraiment séparés (split aléatoire sur du temporel, scaler fitté sur tout le dataset, feature calculée après la cible).
  4. Pas de cross-validation : un seul split train/val donne un score bruité à ±3%.
  5. Tuning sur le test set : on overfit l'évaluation, le score reporté est mensonger.
  6. Pipeline pas reproductible : "ça marche chez moi" en production.
  7. Pas de monitoring en prod : on ne voit pas la dégradation, jusqu'au jour où le CEO appelle.
  8. Modèle trop complexe pour le besoin business (réseau de neurones pour 500 lignes).
  9. Communication insuffisante : les stakeholders ne comprennent pas et n'utilisent pas.
  10. Pas d'A/B test : on ne sait pas si le modèle apporte vraiment de la valeur en production.
⚠️

L'erreur la plus coûteuse : entraîner un modèle pendant 3 mois, le déployer, et découvrir que le métier ne l'utilisera jamais parce qu'il n'est pas intégré dans le workflow opérationnel. La technique sans le business est inutile. Avant de commencer à coder, parlez aux utilisateurs finaux pour comprendre comment ils prendraient la décision de l'utiliser ou non.

Communication des résultats

Présenter un modèle ML à des non-techniques demande de traduire les concepts. Quatre règles :

  1. Pas de jargon : "F1-score" devient "qualité moyenne des prédictions positives". "Feature importance" devient "ce qui compte le plus pour la décision".
  2. Métriques business : "Notre modèle détecte 85% des fraudes" est plus parlant que "Recall = 0.85". "Économie estimée : 200k€/an" est encore mieux.
  3. Visualisations claires : confusion matrix avec couleurs, courbes ROC, SHAP summary plot avec les 10 features les plus importantes.
  4. Limites explicites : "Le modèle est moins fiable pour les nouveaux clients (< 3 mois d'historique)". Ne pas cacher les faiblesses, c'est ce qui crée la confiance.
👉

Bonne pratique : à la fin de chaque projet, écrivez un document de 1-2 pages qui explique le modèle à un non-technique. Si vous n'arrivez pas à le faire, c'est que vous ne le comprenez pas vraiment. C'est aussi le livrable qui distinguera votre travail d'un Kaggle isolé.

C. Questions de compréhension

Q1 : Quelle est la première chose à faire avant de commencer un projet ML ?

💡 Voir l'indice

Pas du code. Ce n'est ni un notebook, ni une ligne de Python. C'est une conversation.

✅ Voir la réponse

Définir précisément le problème business et la métrique de succès, AVANT toute ligne de code. Questions à clarifier : (1) Quel est l'objectif business ? (réduire le churn de 5%, améliorer la satisfaction de 10 points NPS, économiser 200k€/an). (2) Quelle métrique technique reflète cet objectif ? (F1, ROC-AUC, RMSE, lift sur le top 10%). (3) Quelle est la baseline actuelle ? (comment décident-ils aujourd'hui sans ML ?). (4) Quel niveau de performance serait "utile" ? (un modèle à +1% par rapport à la baseline ne justifie probablement pas le coût). (5) Comment le modèle sera-t-il utilisé ? (real-time, batch, par qui, dans quel workflow ?). (6) Quelles données sont disponibles, lesquelles devraient l'être ? (7) Combien de temps, quelle équipe ? Sans cette clarification, vous risquez de passer 3 mois sur un modèle inutile. Les meilleures équipes ML passent 1-2 semaines à clarifier ces questions avec les stakeholders avant d'écrire la première ligne. Question d'entretien data science senior : "Par où commencez-vous un nouveau projet ?" — la réponse révèle l'expérience pratique.

Q2 : Vous avez 6 semaines pour un projet ML. Comment répartiriez-vous le temps ?

💡 Voir l'indice

Pas 80% sur le modèle. Pensez "end-to-end d'abord, optimisation ensuite".

✅ Voir la réponse

Répartition typique réaliste pour un projet de 6 semaines (~30 jours ouvrés) : (1) Semaine 1 (5 jours) : compréhension du problème, parler aux stakeholders, premier coup d'œil sur les données, définition de la métrique de succès et baseline. (2) Semaines 2-3 (10 jours) : EDA approfondie, cleaning, feature engineering, premier modèle baseline. C'est la phase la plus longue. (3) Semaine 4 (5 jours) : modélisation sérieuse, comparaison de plusieurs algorithmes, tuning des plus prometteurs. (4) Semaine 5 (5 jours) : évaluation rigoureuse (cross-validation, test sur données out-of-time, interprétation SHAP), itération sur les features les plus prometteuses. (5) Semaine 6 (5 jours) : déploiement (API, dashboard, ou batch), documentation, présentation aux stakeholders. Anti-pattern junior : passer 5 semaines sur le modèle et 1 semaine sur le reste — résultat : un super modèle qui n'est jamais utilisé. Pro tip : commencer par construire une end-to-end version dégradée dès la semaine 1 (baseline triviale + déploiement minimal), puis itérer. Cela force à résoudre les problèmes d'intégration tôt et garantit qu'il y aura quelque chose à montrer en cas de retard. Question d'entretien data science senior très révélatrice.

Q3 : Vous présentez votre modèle au métier qui dit "85% d'accuracy ce n'est pas assez". Comment réagir ?

💡 Voir l'indice

Ne pas argumenter techniquement. Creuser le besoin.

✅ Voir la réponse

Ne pas argumenter techniquement. Creuser le besoin business. Questions à poser : (1) Quelle est leur baseline actuelle ? S'ils décident à 60% d'accuracy aujourd'hui, votre 85% est déjà une amélioration massive. (2) Quelle est la conséquence d'une erreur ? Selon que c'est un email marketing (peu grave) ou un diagnostic médical (très grave), 85% est suffisant ou pas. (3) Quel serait un "bon" niveau ? 95% ? 99% ? Demandez le chiffre cible. Souvent ils répondent "le plus haut possible" — ce n'est pas une réponse. Demandez ce qui est acceptable. (4) Sont-ils prêts à accepter plus de coût pour plus d'accuracy ? Passer de 85% à 90% peut nécessiter 10x plus de données ou un modèle 100x plus cher en compute. (5) Quel est le coût de NE PAS utiliser le modèle ? Si la décision actuelle est aléatoire (50%), votre 85% est déjà une victoire énorme. Action concrète : décomposer le 85% en confusion matrix et discuter ligne par ligne. Souvent le métier réalise que les "erreurs" sont sur des cas spécifiques qu'ils peuvent traiter manuellement, et que le modèle est utile sur les 80% de cas standards. Exemple concret : un modèle de scoring de prospect à 85% est utile pour prioriser l'effort commercial — vous appelez d'abord les top 20% au lieu d'appeler tout le monde. Personne ne s'attend à ce que ce soit parfait. Question d'entretien data science : "Comment gérez-vous des attentes irréalistes du métier ?" — la réponse révèle la maturité.
