Guide visuel pour choisir le bon modèle sklearn selon ta situation. Lis de haut en bas, réponds aux questions, arrive au modèle.

Régression (target continue)

  • 💡 J'ai besoin d'interpréter les coefficients
  • ✅ Le modèle linéaire ne surappend pas (train ≈ test)
🎯

→ LinearRegression()

Scaling : non requis

Pourquoi : OLS classique, coefficients directement interprétables, p-values via statsmodels. Baseline de référence.

Tuning : aucun hyperparamètre.

  • ⚠️ Le modèle overfit (train >> test) et je veux réduire les features
🎯

→ Lasso(alpha=1.0) ou ElasticNet(alpha=1, l1_ratio=0.5)

Scaling : OUI obligatoire

Pourquoi : L1 annule les coefficients non-significatifs → feature selection automatique. ElasticNet = compromis L1+L2 si features corrélées.

Tuning : alpha via GridSearch sur [0.001, 0.01, 0.1, 1, 10]. Pour ElasticNet, ajouter l1_ratio via stats.uniform(0, 1).

  • ⚠️ Le modèle overfit mais je veux garder toutes les features
🎯

→ Ridge(alpha=1.0)

Scaling : OUI obligatoire

Pourquoi : L2 réduit tous les coefficients vers 0 sans les annuler. Gère bien la multicolinéarité.

Tuning : alpha via GridSearch sur [0.01, 0.1, 1, 10, 100].

  • 📈 La relation n'est pas linéaire (pattern dans les résidus)
🎯

→ make_pipeline(PolynomialFeatures(degree=2), StandardScaler(), Ridge())

Scaling : OUI obligatoire

Pourquoi : Ajoute x² et interactions x₁·x₂ tout en gardant un modèle interprétable + régularisé.

Tuning : degree (2 max en général) + alpha du Ridge. Degree 3+ = explosion combinatoire → overfit.

  • 🚀 Je veux la meilleure performance (boîte noire OK)
  • 📊 Dataset très grand (n > 100K ou p > 1000)
🎯

→ SGDRegressor(loss='squared_error')

Scaling : OUI obligatoire

Pourquoi : O(n) vs O(p³) pour LinearRegression. loss='huber' si outliers.

Tuning : loss, eta0, penalty (l1/l2/elasticnet), max_iter.

  • 📉 Relation linéaire, dataset de taille raisonnable
🎯

→ Ridge(alpha=1.0)

Scaling : OUI obligatoire

Pourquoi : Régularisation presque toujours bénéfique même sans overfit visible.

Tuning : alpha via GridSearch.

  • 🌀 Relation non-linéaire + outliers ou données bruitées
🎯

→ RandomForestRegressor(n_estimators=100, n_jobs=-1)

Scaling : NON requis

Pourquoi : Bagging d'arbres = robuste aux outliers, réduit la variance. OOB score = validation gratuite (~37% non tirés).

Tuning : n_estimators (100+), max_depth (5–15), min_samples_leaf.

  • 🌀 Relation non-linéaire + données propres → performance max
🎯

→ XGBRegressor(n_estimators=100, learning_rate=0.1, early_stopping_rounds=5)

Scaling : NON requis

Pourquoi : Boosting séquentiel sur les résidus. Régularisation L1/L2 intégrée. Souvent le meilleur score brut.

Tuning : learning_rate (0.01–0.1), n_estimators, max_depth, early_stopping_rounds. Attention : sensible aux outliers.

  • 🌀 Relation non-linéaire + peu de features + dataset petit
🎯

→ SVR(kernel='rbf', C=1.0, gamma='scale')

Scaling : OUI obligatoire

Pourquoi : Kernel trick = projection implicite en haute dimension. Lent si n > 10K.

Tuning : C + gamma toujours ensemble via GridSearch. epsilon = largeur du tube.

Classification (target catégorielle)

  • 💡 J'ai besoin d'interpréter les coefficients (log-odds)

> → LogisticRegression(solver='lbfgs', max_iter=1000)

>

> Scaling : OUI obligatoire

>

> Pourquoi : Classifieur calibré (predict_proba fiable). Coefficients = log-odds interprétables. Ajustement du seuil de classification possible.

>

> Tuning : C (= 1/alpha), penalty (l1/l2), solver. Pour sélection de features : penalty='l1', solver='saga'.

>

  • 🌲 J'ai besoin d'un arbre visualisable (stakeholder non-tech)

> → DecisionTreeClassifier(max_depth=5)

>

> Scaling : NON requis

>

> Pourquoi : Visualisable avec graphviz, feature_importances_ intégré. Seul : très sujet à l'overfit.

>

> Tuning : max_depth (crucial !), min_samples_split, min_samples_leaf. Sans max_depth → R² train=1.0, test≈0.

>

  • 🚀 Je veux la meilleure performance (boîte noire OK)
  • 📊 Dataset très grand (n > 100K)

> → SGDClassifier(loss='log_loss')

>

> Scaling : OUI obligatoire

>

> Pourquoi : loss='log_loss' = LogisticRegression, loss='hinge' = SVM. Très rapide sur grands datasets.

>

> Tuning : loss, eta0, penalty, max_iter.

>

  • 📏 Frontière de décision linéaire (classes bien séparées)

> → SVC(kernel='linear', C=1.0)

>

> Scaling : OUI obligatoire

>

> Pourquoi : Maximise la marge entre les classes. C élevé = marge stricte, C faible = marge souple.

>

> Tuning : C via GridSearch sur [0.01, 0.1, 1, 10, 100].

>

  • 🌀 Frontière non-linéaire + données bruitées / outliers

> → RandomForestClassifier(n_estimators=100, n_jobs=-1)

>

> Scaling : NON requis

>

> Pourquoi : Vote majoritaire entre arbres bootstrapés. Robuste au bruit et aux outliers. OOB score gratuit.

>

> Tuning : n_estimators (100+), max_depth, min_samples_leaf.

>

  • 🌀 Frontière non-linéaire + données propres → performance max

> → XGBClassifier(n_estimators=100, learning_rate=0.1, early_stopping_rounds=5)

>

> Scaling : NON requis

>

> Pourquoi : Boosting séquentiel. Souvent le meilleur score brut. Early stopping évite l'overfit.

>

> Tuning : learning_rate (0.01–0.1), n_estimators, max_depth, early_stopping_rounds.

>

  • 🌀 Frontière non-linéaire + peu de données (n < 10K)

> → SVC(kernel='rbf', C=1.0, gamma='scale')

>

> Scaling : OUI obligatoire

>

> Pourquoi : Kernel trick gaussien. Très efficace sur petits datasets. Lent si n > 10K.

>

> Tuning : C + gamma toujours ensemble. Petit gamma = lisse (underfit), grand gamma = irrégulier (overfit).

>

Règle générale : les modèles basés sur la distance (KNN, SVM, régularisation) exigent du scaling. Les modèles basés sur des arbres (DT, RF, XGB) n'en ont pas besoin car les splits sont basés sur l'ordre, pas la magnitude.

🧭 Introduction

Imagines-toi guide touristique. Un touriste te dit : "Je veux aller à Lyon." Tu ne lui proposes pas systématiquement le même transport. Tu lui poses des questions : c'est loin ? Il a des bagages ? Il veut aller vite ou pas cher ? Il a un budget limité ? Selon ses réponses, tu lui dis : "Prends le TGV", "Loue une voiture", "Prend le bus".

En Machine Learning, c'est pareil. On ne choisit pas un modèle au hasard. On suit un arbre de décision : une série de questions qui nous guide vers le bon modèle selon la situation réelle.

A retenir

  • Il n'existe pas un "meilleur modèle universel" — tout dépend du contexte.
  • La première question à se poser : est-ce que je prédit un nombre ou une catégorie ? (régression vs classification)
  • La deuxième question : est-ce que j'ai besoin de comprendre ce que fait le modèle ? (interprétabilité vs performance brute)
  • Certains modèles exigent un scaling (mise à l'échelle des données), d'autres non.
  • Le surapprentissage (overfit) — quand un modèle marche très bien sur les données d'entraînement mais mal sur de nouvelles données — est un signal clé pour choisir le bon modèle.

Mini-plan

  1. Régression — quand la cible est un nombre
  2. Classification — quand la cible est une catégorie
  3. Questions de vérification
  4. TL;DR

📈 Partie 1 — Régression (la cible est un nombre continu)

La régression répond à des questions du type :

  • "Combien va coûter cet appartement ?"
  • "Quelle sera la température demain ?"
  • "Combien d'unités va-t-on vendre ce mois-ci ?"

La cible (ce qu'on prédit) est un nombre, pas une catégorie.

💡

Un nombre "continu" peut prendre n'importe quelle valeur : 42, 42.7, 42.73… C'est différent d'une classe comme "chat" ou "chien".

🔍 Branche 1 — Tu as besoin de comprendre les coefficients

"Je veux pouvoir expliquer à mon patron ce qui influence le prix."

Tu te trouves dans ce cas si tu dois justifier les résultats ou identifier quelles variables comptent le plus.

✅ Cas 1.1 — Le modèle ne surappend pas (train ≈ test)

La situation concrète : Tu entraînes ton modèle, tu compares le score sur les données d'entraînement (train) et sur les données de test. Les deux sont proches. Tout va bien.

Exemple : Tu prédis le prix d'une maison avec 5 variables (surface, nombre de pièces, étage, ville, année). Le modèle simple marche déjà bien.

Python
from sklearn.linear_model import LinearRegression

model = LinearRegression()
model.fit(X_train, y_train)

Pourquoi ce modèle ?

  • Chaque coefficient est directement lisible : "chaque m² supplémentaire ajoute 3 000 € au prix".
  • C'est la baseline (le point de départ) : si un modèle plus complexe ne fait pas mieux, inutile de le compliquer.
  • Pas d'hyperparamètre à régler — c'est prêt à l'emploi.
✅

Scaling requis : NON. La régression linéaire classique n'est pas sensible à l'échelle des variables.

⚠️ Cas 1.2 — Le modèle surappend ET tu veux éliminer les variables inutiles

La situation concrète : Score train = 0.95, score test = 0.60. Ton modèle a "mémorisé" les données au lieu d'apprendre. Et en plus, tu as beaucoup de variables dont certaines sont probablement sans intérêt.

Exemple : Tu prédis le salaire d'un employé avec 50 variables. Beaucoup sont redondantes ou inutiles. Tu veux que le modèle fasse lui-même le tri.

Python
from sklearn.linear_model import Lasso, ElasticNet
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

# Lasso : élimine les variables inutiles
model = make_pipeline(StandardScaler(), Lasso(alpha=1.0))

# ElasticNet : si les variables sont corrélées entre elles
model = make_pipeline(StandardScaler(), ElasticNet(alpha=1.0, l1_ratio=0.5))

Pourquoi ce modèle ?

  • Lasso utilise une pénalité L1 qui force certains coefficients à tomber exactement à zéro — c'est une sélection automatique de features.
  • ElasticNet combine L1 et L2 : utile quand plusieurs variables sont corrélées (ex : taille et poids vont souvent ensemble).
⚠️

Scaling requis : OUI obligatoire. Si une variable est en euros (0–100 000) et une autre en pourcentage (0–100), la pénalité va faussement privilégier la deuxième. Le scaling met tout à la même échelle.

🧠

Tuning : cherche la valeur optimale d'alpha avec un GridSearch sur [0.001, 0.01, 0.1, 1, 10]. Plus alpha est grand, plus les coefficients sont écrasés.

⚠️ Cas 1.3 — Le modèle surappend MAIS tu veux garder toutes tes variables

La situation concrète : Même problème d'overfit, mais toutes tes variables sont importantes et tu ne veux pas en perdre.

Exemple : Tu analyses l'impact de 20 indicateurs économiques sur le PIB. Chacun a un sens métier, tu ne veux pas en sacrifier.

Python
from sklearn.linear_model import Ridge
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(StandardScaler(), Ridge(alpha=1.0))

Pourquoi ce modèle ?

  • Ridge utilise une pénalité L2 qui réduit les coefficients vers zéro sans jamais les annuler complètement.
  • Gère très bien la multicolinéarité (quand des variables sont corrélées).
⚠️

Scaling requis : OUI obligatoire. Même raison que pour Lasso.

📈 Cas 1.4 — La relation n'est pas linéaire (il y a une courbe)

La situation concrète : En traçant tes résidus (erreurs de prédiction), tu vois un pattern — une courbe, des vagues. Ce n'est pas du bruit aléatoire. Le modèle linéaire est structurellement insuffisant.

Exemple : La consommation d'énergie d'une ville augmente de façon exponentielle en été (climatiseurs). Une droite ne peut pas capturer ça.

Python
from sklearn.preprocessing import PolynomialFeatures, StandardScaler
from sklearn.linear_model import Ridge
from sklearn.pipeline import make_pipeline

model = make_pipeline(
    PolynomialFeatures(degree=2),
    StandardScaler(),
    Ridge(alpha=1.0)
)

Pourquoi ce modèle ?

  • PolynomialFeatures crée de nouvelles variables : x², x₁·x₂, etc. On garde ainsi l'interprétabilité tout en capturant des courbes.
  • Ridge en fin de pipeline pour éviter l'overfit qui arrive vite avec des features polynomiales.
⚠️

Ne jamais dépasser degree=2 ou 3 sans raison forte. Avec degree=4 ou plus, le nombre de variables explose et le modèle overfit massivement.

🚀 Branche 2 — Tu veux la meilleure performance (boîte noire acceptable)

"Je m'en fiche de comprendre comment ça marche, je veux le meilleur score."

📊 Cas 2.1 — Dataset très grand (plus de 100 000 lignes ou 1 000 colonnes)

La situation concrète : Ton fichier est énorme. Les modèles classiques mettent des heures à s'entraîner.

Exemple : Tu analyses les transactions d'une banque — 5 millions de lignes, 200 colonnes.

Python
from sklearn.linear_model import SGDRegressor
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(
    StandardScaler(),
    SGDRegressor(loss='squared_error', max_iter=1000)
)
# Si tu as des outliers, utilise loss='huber'

Pourquoi ce modèle ?

  • SGDRegressor apprend par petits paquets (stochastic gradient descent), pas sur tout le dataset d'un coup.
  • Complexité O(n) au lieu de O(p³) pour la régression linéaire classique — incomparablement plus rapide sur de très grands volumes.
🧠

SGD = Stochastic Gradient Descent. "Stochastic" = aléatoire. Au lieu de regarder toutes les données pour faire un pas, il en regarde un tout petit échantillon. C'est moins précis à chaque pas, mais beaucoup plus rapide.

⚠️

Scaling requis : OUI obligatoire. SGD est particulièrement sensible à l'échelle.

📉 Cas 2.2 — Relation linéaire, dataset de taille normale

La situation concrète : Pas besoin d'interpréter les coefficients, les données sont "normales" (quelques milliers à dizaines de milliers de lignes).

Exemple : Prédire le prix d'une voiture d'occasion à partir de 10 caractéristiques.

Python
from sklearn.linear_model import Ridge
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(StandardScaler(), Ridge(alpha=1.0))

Pourquoi ce modèle ?

  • Même sans overfit visible, une légère régularisation (Ridge) est presque toujours bénéfique — ça ne fait pas de mal et souvent ça aide un peu.

🌀 Cas 2.3 — Relation non-linéaire + données bruitées ou outliers

La situation concrète : Les données ont des valeurs aberrantes (outliers), des erreurs de mesure, beaucoup de bruit. Tu veux un modèle robuste.

Exemple : Prédire la consommation électrique horaire d'un bâtiment sur un an. Il y a des pics bizarres (pannes, événements), beaucoup de variabilité.

Python
from sklearn.ensemble import RandomForestRegressor

model = RandomForestRegressor(n_estimators=100, n_jobs=-1)
model.fit(X_train, y_train)

Pourquoi ce modèle ?

  • Random Forest = une "forêt" de nombreux arbres de décision. Chacun est entraîné sur un sous-échantillon aléatoire des données.
  • Le vote (ou la moyenne) de tous ces arbres rend le modèle robuste aux outliers et réduit la variance.
  • OOB score (Out-Of-Bag) : environ 37% des données ne sont jamais vus par chaque arbre → validation quasi-gratuite sans cross-validation séparée.
✅

Scaling requis : NON. Les arbres fonctionnent par seuils (splits), pas par distance. L'échelle des valeurs ne change pas l'ordre, donc ça n'a pas d'importance.

🌀 Cas 2.4 — Relation non-linéaire + données propres → performance maximale

La situation concrète : Tes données sont bien préparées, pas trop d'outliers. Tu veux le meilleur score possible. On est en mode compétition.

Exemple : Kaggle. Prédire les prix des maisons (la compétition classique).

Python
from xgboost import XGBRegressor

model = XGBRegressor(
    n_estimators=100,
    learning_rate=0.1,
    early_stopping_rounds=5
)
model.fit(X_train, y_train, eval_set=[(X_val, y_val)], verbose=False)

Pourquoi ce modèle ?

  • XGBoost = boosting séquentiel. Chaque nouvel arbre corrige les erreurs résiduelles du précédent.
  • Régularisation L1/L2 intégrée. Early stopping = stoppe automatiquement quand ça n'améliore plus.
  • C'est souvent le modèle qui donne le meilleur score brut.
⚠️

XGBoost est sensible aux outliers (contrairement à Random Forest). Si tes données ont beaucoup d'aberrations, préfère Random Forest. Scaling requis : NON.

🌀 Cas 2.5 — Relation non-linéaire + peu de features + petit dataset

La situation concrète : Tu n'as pas beaucoup de données (moins de 10 000 lignes) et peu de variables. C'est un cas souvent rencontré en sciences ou en médecine.

Exemple : Prédire la résistance d'un matériau à partir de 4 propriétés chimiques mesurées sur 500 échantillons.

Python
from sklearn.svm import SVR
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(StandardScaler(), SVR(kernel='rbf', C=1.0, gamma='scale'))

Pourquoi ce modèle ?

  • Le kernel trick du SVR projette implicitement les données dans un espace de très haute dimension où la relation devient linéaire.
  • Très efficace sur des petits datasets bien structurés.
⚠️

SVR devient très lent au-delà de 10 000 lignes. Si tu as plus de données, reviens au Random Forest ou XGBoost. Scaling requis : OUI obligatoire.

🏷️ Partie 2 — Classification (la cible est une catégorie)

La classification répond à des questions du type :

  • "Cet email est-il un spam ou non ?"
  • "Ce client va-t-il partir (churn) ?"
  • "Ce patient a-t-il une maladie A, B ou C ?"

La cible est une étiquette (0/1, "chat"/"chien", "A"/"B"/"C"…).

💡

En classification, on prédit une appartenance à une classe. En régression, on prédit une valeur. C'est la distinction fondamentale qui oriente tout le reste.

💡 Cas 3.1 — Tu as besoin d'interpréter les coefficients

La situation concrète : Tu veux comprendre comment chaque variable influence la probabilité d'appartenir à une classe.

Exemple : Prédire si un client va rembourser son crédit. Tu dois pouvoir justifier le refus auprès du régulateur ("le refus est basé sur le revenu et la durée d'emploi").

Python
from sklearn.linear_model import LogisticRegression
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(
    StandardScaler(),
    LogisticRegression(solver='lbfgs', max_iter=1000)
)

Pourquoi ce modèle ?

  • predict_proba() donne une vraie probabilité (0.73 = "73% de chance d'être dans la classe 1").
  • Coefficients = log-odds interprétables : "un revenu supplémentaire de 1 000 € multiplie les chances par 1.3".
  • On peut ajuster le seuil de décision (par défaut 0.5) selon le contexte (ex : en médecine on préfère être prudent et descendre le seuil à 0.3).
🧠

Pour la sélection de features en classification : utilise penalty='l1', solver='saga'. C'est le Lasso de la classification.

⚠️

Scaling requis : OUI obligatoire.

🌲 Cas 3.2 — Tu as besoin d'un arbre visualisable pour un non-technique

La situation concrète : Tu dois présenter ton modèle à des gens qui n'y connaissent rien en ML. Tu veux un schéma lisible que tout le monde peut suivre.

Exemple : Tu développes un outil d'aide au diagnostic pour des médecins. Ils veulent voir exactement la logique de décision : "si fièvre > 38.5 ET toux persistante → suspicion grippe".

Python
from sklearn.tree import DecisionTreeClassifier, export_graphviz
import graphviz

model = DecisionTreeClassifier(max_depth=5)
model.fit(X_train, y_train)

# Visualisation de l'arbre
dot_data = export_graphviz(model, feature_names=feature_names, class_names=class_names, filled=True)
graph = graphviz.Source(dot_data)
graph.render("arbre_decision")

Pourquoi ce modèle ?

  • L'arbre est visuellement compréhensible : chaque nœud est une question, chaque feuille est une décision.
  • feature_importances_ indique quelles variables comptent le plus.
⚠️

ATTENTION : sans max_depth, un arbre seul va mémoriser parfaitement les données d'entraînement (score train = 1.0) mais sera nul en généralisation (score test ≈ 0). Toujours limiter la profondeur. Scaling requis : NON.

🚀 Branche 4 — Tu veux la meilleure performance (boîte noire acceptable)

📊 Cas 4.1 — Dataset très grand (plus de 100 000 lignes)

La situation concrète : Même logique que pour la régression : le volume est tel que les modèles classiques sont trop lents.

Exemple : Classer 10 millions de tweets selon leur sentiment (positif / négatif / neutre).

Python
from sklearn.linear_model import SGDClassifier
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(
    StandardScaler(),
    SGDClassifier(loss='log_loss', max_iter=1000)
)
# loss='log_loss' = équivalent LogisticRegression
# loss='hinge' = équivalent SVM linéaire
💡

SGDClassifier est un "couteau suisse" : en changeant juste le paramètre loss, tu changes complètement de modèle. C'est le même algorithme d'optimisation, appliqué à des fonctions de perte différentes.

⚠️

Scaling requis : OUI obligatoire.

📏 Cas 4.2 — Frontière linéaire (les classes sont bien séparées)

La situation concrète : En visualisant tes données, tu vois que les classes peuvent être séparées par une droite (ou un hyperplan). Les classes sont relativement bien distinctes.

Exemple : Classer des emails comme spam / non-spam à partir de mots-clés. Les emails "gagnez de l'argent" vs "réunion d'équipe" sont souvent bien séparés.

Python
from sklearn.svm import SVC
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(
    StandardScaler(),
    SVC(kernel='linear', C=1.0)
)

Pourquoi ce modèle ?

  • SVM linéaire maximise la marge entre les classes : il cherche la séparatrice la plus éloignée possible des deux groupes.
  • C élevé = marge stricte (peu d'erreurs tolérées), C faible = marge souple (quelques erreurs acceptées mais plus robuste).
⚠️

Scaling requis : OUI obligatoire. SVM est basé sur des distances.

🌀 Cas 4.3 — Frontière non-linéaire + données bruitées ou outliers

La situation concrète : La séparation entre les classes est complexe (une courbe, des zones imbriquées), et les données contiennent du bruit.

Exemple : Détecter des fraudes bancaires. Les fraudes sont rares et peuvent se cacher partout parmi des transactions normales. Plein de faux positifs et faux négatifs potentiels.

Python
from sklearn.ensemble import RandomForestClassifier

model = RandomForestClassifier(n_estimators=100, n_jobs=-1)
model.fit(X_train, y_train)

Pourquoi ce modèle ?

  • Vote majoritaire entre des centaines d'arbres → robuste au bruit.
  • OOB score disponible gratuitement.
  • Moins sensible aux outliers qu'XGBoost.
✅

Scaling requis : NON.

🌀 Cas 4.4 — Frontière non-linéaire + données propres → performance maximale

La situation concrète : Tes données sont bien nettoyées. Tu veux le meilleur score possible.

Exemple : Compétition Kaggle — classer des images de maladies des plantes (données labellisées, bien nettoyées, tu veux gagner).

Python
from xgboost import XGBClassifier

model = XGBClassifier(
    n_estimators=100,
    learning_rate=0.1,
    early_stopping_rounds=5
)
model.fit(X_train, y_train, eval_set=[(X_val, y_val)], verbose=False)

Pourquoi ce modèle ?

  • Même logique que XGBRegressor : boosting séquentiel sur les erreurs.
  • Souvent le meilleur score brut sur des datasets tabulaires propres.
✅

Scaling requis : NON.

🌀 Cas 4.5 — Frontière non-linéaire + peu de données (moins de 10 000 lignes)

La situation concrète : Peu de données, frontière complexe.

Exemple : Classer des types de roches à partir de 5 mesures géochimiques sur 800 échantillons de terrain.

Python
from sklearn.svm import SVC
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import make_pipeline

model = make_pipeline(
    StandardScaler(),
    SVC(kernel='rbf', C=1.0, gamma='scale')
)

Pourquoi ce modèle ?

  • Le kernel RBF (gaussien) permet de séparer des classes qui ne sont pas linéairement séparables, via une transformation implicite en haute dimension.
  • Très efficace sur de petits datasets bien structurés.
🧠

gamma contrôle la "portée" de l'influence d'un point : petit gamma = influence large (frontière lisse, risque underfit), grand gamma = influence courte (frontière irrégulière, risque overfit). Toujours tuner C et gamma ensemble.

⚠️

SVC est lent sur de grands datasets (n > 10 000). Scaling requis : OUI obligatoire.

📋 Résumé des règles de scaling

Type de modèleExemplesScaling requis ?
Basés sur la distance ou la magnitudeSVM, SGD, LogisticRegression, Ridge, LassoOUI
Basés sur des arbres (splits par ordre)DecisionTree, RandomForest, XGBoostNON
🧠

La règle d'or : les arbres séparent par des seuils ("est-ce que x > 5 ?"). La valeur absolue ne change pas l'ordre, donc l'échelle est sans importance. En revanche, les modèles basés sur des distances ou des gradients sont directement affectés par l'échelle des valeurs.

🎯 Questions de vérification

Essaie de répondre seul avant de lire la correction !

Cas 1. Tu travailles pour une agence immobilière. Tu veux prédire le prix d'une maison (en euros). Tu as 15 variables (surface, quartier, année de construction…). Ton modèle de base donne train score = 0.89, test score = 0.88. Ton patron veut comprendre l'impact de chaque variable sur le prix.

Quel modèle choisir ?

Correction

LinearRegression(). La cible est continue (régression), le modèle ne surappend pas (train ≈ test), et on a besoin d'interpréter les coefficients. C'est exactement le cas de la baseline. Pas de scaling nécessaire.

Cas 2. Tu travailles dans une startup fintech. Tu veux détecter si une transaction est frauduleuse (oui/non). Tu as 2 millions de transactions et 50 variables. Tu veux le meilleur score possible sans te soucier de l'interprétabilité.

Quel modèle choisir ? Faut-il scaler ?

Correction

Dataset > 100K lignes → SGDClassifier(loss='log_loss'). Scaling OUI obligatoire. Alternatively, si les données sont propres et tu peux te permettre un temps d'entraînement plus long, XGBClassifier (sans scaling) peut donner un meilleur score mais sera plus lent sur 2M lignes.

Cas 3. Tu analyses des données médicales pour prédire si un patient a une maladie rare (classe A, B ou C). Tu n'as que 600 patients, 8 mesures biologiques. La relation entre les mesures et la maladie est complexe (non-linéaire).

Quel modèle choisir ? Faut-il scaler ?

Correction

Peu de données (n < 10K), relation non-linéaire → SVC(kernel='rbf', C=1.0, gamma='scale'). Scaling OUI obligatoire. Penser à tuner C et gamma ensemble via GridSearch.

Cas 4. Tu dois présenter un modèle de scoring client à la direction marketing. Ils veulent voir exactement comment les décisions sont prises. Le dataset a 5 000 clients et 12 variables.

Quel modèle choisir ?

Correction

Besoin de visualisation et explicabilité pour non-technique → DecisionTreeClassifier(max_depth=5). Pas de scaling nécessaire. Bien limiter max_depth pour éviter l'overfit. Utiliser graphviz pour générer une image de l'arbre.

📌 TL;DR

  • La première question est toujours : régression (nombre) ou classification (catégorie) ?
  • Ensuite : as-tu besoin d'interpréter les résultats ou veux-tu la meilleure performance ?
  • En régression interprétable : LinearRegression si pas d'overfit, Ridge pour garder toutes les features, Lasso pour éliminer les inutiles.
  • En régression performance max : RandomForest si données bruitées, XGBoost si données propres, SGDRegressor si dataset gigantesque.
  • En classification interprétable : LogisticRegression. Besoin d'un visuel pour non-tech : DecisionTreeClassifier.
  • En classification performance max : RandomForestClassifier (robuste), XGBClassifier (score max), SVC (peu de données), SGDClassifier (grande échelle).
  • Règle de scaling : modèles basés sur la distance ou les gradients (SVM, SGD, régularisation) → scaler OUI. Modèles basés sur des arbres (DT, RF, XGB) → scaler NON.
  • En cas de doute sur le modèle, commence toujours par la baseline (LinearRegression ou LogisticRegression) : si elle suffit, inutile de compliquer.
  • L'overfit (train >> test) est un signal fort : ajoute de la régularisation (Ridge, Lasso) ou limite la complexité (max_depth).

A. Concepts du cours

Le bon modèle dépend du problème

Il n'y a pas de "meilleur algorithme ML". Le choix dépend de quatre dimensions fondamentales :

  1. Type de problème : classification, régression, clustering, recommandation, génération
  2. Type de données : tabulaire, image, texte, audio, séries temporelles, graphe
  3. Taille du dataset : 100 lignes ou 100 millions ? Ce seul critère change tout
  4. Contraintes : interprétabilité, latence, mémoire, budget, régulation

L'arbre de décision suivant vous guide vers le bon modèle pour 90% des cas. Les 10% restants demandent du jugement contextuel — mais après 5-10 projets, ce jugement devient automatique.

💡

Analogie : choisir un modèle ML, c'est comme choisir un outil dans une boîte à outils. Vous ne plantez pas un clou avec une scie. Vous ne coupez pas une planche avec un marteau. L'arbre de décision est le mode d'emploi qui vous dit quel outil pour quel travail.

B. Compléments avancés

Arbre de décision express — cheatsheet

Pour un projet tabulaire standard (cas le plus fréquent), voici la décision en 3 questions :

Code
1. Combien de lignes ?
   < 1 000     → Random Forest (robuste, peu de tuning)
   1 k – 100 k → XGBoost / LightGBM (standard)
   100 k – 1 M → LightGBM (plus rapide que XGB)
   > 1 M       → LightGBM + GPU, ou DL tabulaire si vraiment complexe

2. Interprétabilité nécessaire ?
   Critique   → LogReg ou arbre profondeur 4
   Importante → XGB/LightGBM + SHAP
   Peu        → N'importe quel modèle

3. Latence < 10 ms ?
   Oui → LogReg ou LightGBM léger exporté en ONNX
   Non → Plus de liberté

Exemples concrets :

  • Churn télécom, 50k lignes, pas de contrainte latence → LightGBM + SHAP. Baseline F1 ≈ 0.74.
  • Scoring de crédit, 500k lignes, régulation AI Act → LogReg + XGBoost + SHAP en challenger. Docs complètes.
  • Détection spam, 30k emails → TF-IDF + LogReg. Baseline accuracy 0.95. Pas besoin de BERT.
  • Classification défauts images, 50k photos → Transfer learning EfficientNet-B0, gel du backbone, fine-tune tête. Recall > 0.95.
  • Prédiction trafic web horaire, 2 ans de données → Prophet (saisonnalité claire) ou LightGBM avec features laggées.
  • Résumé automatique de docs juridiques → LLM via API (Claude ou GPT), prompt engineering, pas d'entraînement.
  • Segmentation clients RFM, 100k clients → K-Means k=5 pour baseline, HDBSCAN si clusters non sphériques.

Construire votre arbre personnel

L'arbre ci-dessus est un point de départ. En pratique, votre arbre dépend de votre expertise, votre infrastructure, et les patterns de votre domaine.

Conseils pour construire le vôtre :

  1. Commencez par votre baseline préférée : "pour tout problème de classification tabulaire, je commence toujours par LightGBM avec hyperparams par défaut".
  1. Identifiez les cas où la baseline échoue : "si l'AUC est < 0.7, je passe à l'investigation des features. Si l'AUC est > 0.9, c'est probablement du leakage à vérifier".
  1. Connaître 3-5 modèles à fond est plus utile que de connaître 30 modèles superficiellement. Pour la majorité des cas tabulaires, maîtriser LightGBM, Random Forest et la régression logistique suffit.
  1. Documentez vos choix : pour chaque projet passé, notez le modèle final, les alternatives essayées, et pourquoi vous avez choisi celui-ci. Au bout de 10 projets, vous avez votre méta-modèle de décision.
👉

Bonne pratique : un data scientist senior n'essaie pas 20 modèles à chaque projet. Il en essaie 2-3 (sa baseline + 1-2 alternatives basées sur le contexte) et passe la majorité du temps sur la qualité des données et le feature engineering. C'est ce qui différencie l'expérience.

Cas particuliers et exceptions

Quand le tabulaire devient deep learning : quand vous avez > 1M lignes ET des relations très non linéaires (interactions complexes entre features), un MLP ou TabNet peut battre XGBoost. Mais c'est l'exception, pas la règle — sur 95% des benchmarks tabulaires, LightGBM gagne encore en 2025.

Quand le deep learning n'est pas optimal en NLP : pour de la classification simple sur peu de données, TF-IDF + LogReg reste compétitif et 100× plus rapide que BERT. Ne sortez pas l'artillerie pour une simple détection de spam de 1 000 emails.

Quand le clustering est inapproprié : quand vos clusters n'existent pas vraiment dans la donnée. K-means trouvera toujours K clusters, même sur du bruit pur. Vérifiez avec silhouette score : < 0.3 signifie probablement pas de structure, et aucun algorithme ne sauvera la situation.

Quand le RL est tentant mais inutile : quand vous pouvez transformer le problème en supervisé avec des labels existants, c'est presque toujours plus efficace que le RL. Le RL brille quand il y a un simulateur et pas de labels (jeux, robotique).

Quand un LLM est overkill : pour de l'extraction de données structurées simple (regex, patterns fixes), un script Python basique est meilleur qu'un LLM. Plus rapide, gratuit, déterministe. Le LLM brille uniquement quand le problème exige du raisonnement ou de la flexibilité sur des inputs variables.

⚠️

Anti-pattern fréquent en 2025 : "utilisons un LLM pour tout". Un LLM coûte $0.001-1 par appel, prend 1-10 secondes, et n'est pas 100% fiable. Pour la plupart des problèmes ML structurés, un classifieur sklearn coûte $0, prend 1 ms, et est 99% fiable. Le LLM est l'outil ultime, pas l'outil par défaut.

⚠️

Piège de taille : avec < 1 000 lignes, éviter les réseaux de neurones profonds et le stacking agressif. Même si vous obtenez un super score en train, la variance est trop forte pour que ce soit fiable en prod. Sur petit dataset, la simplicité est la régularisation.

C. Questions de compréhension

Q1 : Quelqu'un vous demande "quel est le meilleur algorithme de machine learning ?". Comment répondez-vous ?

💡 Voir l'indice

Pas un nom. Une question en retour.

✅ Voir la réponse

La bonne réponse est "ça dépend" suivi des questions clarificatrices : (1) quel est le type de problème ? Classification, régression, clustering ? (2) Quel type de données ? Tabulaire, image, texte ? (3) Combien de données ? 100 lignes ou 100 millions ? (4) Quelles contraintes ? Latence, interprétabilité, budget ? Sans ces informations, donner un nom d'algo serait du dogmatisme. En mentionnant le théorème No-Free-Lunch (Wolpert, 1996), vous montrez que vous comprenez pourquoi : aucun algorithme n'est universellement meilleur. Pour donner une réponse pragmatique malgré tout, vous pouvez ajouter : "sur du tabulaire en 2025, LightGBM est mon point de départ. Sur des images, transfer learning EfficientNet. Sur du texte, fine-tuning DistilBERT. Mais ces choix sont contextuels, pas absolus." Question d'entretien data science classique : un candidat qui répond "deep learning" ou "XGBoost" sans nuance révèle un manque de maturité. Un candidat qui demande des précisions montre l'expérience du terrain.

Q2 : Vous travaillez sur un problème de classification médicale critique. Quels modèles éliminez-vous d'office ?

💡 Voir l'indice

Interprétabilité et régulation.

✅ Voir la réponse

Trois familles à éliminer ou justifier explicitement. (1) Modèles boîte noire profonds sans explicabilité : un grand neural network sans SHAP est inutilisable en médical, pour deux raisons : régulation (RGPD, AI Act, FDA aux US) et confiance médicale (un médecin n'utilisera pas un outil qu'il ne comprend pas). (2) Stacking de plusieurs modèles complexes : difficile à expliquer, à auditer, à débugger en cas de problème. (3) Modèles avec beaucoup de features dérivées non interprétables : embeddings, PCA très réduits — perdent la connexion avec les concepts médicaux. À privilégier en médical : (a) Régression logistique avec features cliniquement interprétables, (b) XGBoost ou LightGBM avec analyse SHAP systématique, (c) Decision Tree peu profond quand les règles doivent être directement lisibles, (d) Generalized Additive Models (GAM, EBM de InterpretML) qui combinent flexibilité et interprétabilité native. Précautions supplémentaires : (i) calibration des probabilités (pour donner une vraie probabilité, pas un score arbitraire), (ii) validation sur des données out-of-time ET out-of-hospital pour mesurer la généralisation, (iii) audit trail complet (qui a entraîné, sur quelles données, quelle version), (iv) reentraînement contrôlé. Question d'entretien data science régulé : la réponse exhaustive parle de SHAP, calibration, et compliance.

Q3 : Vous avez un dataset de 50 000 images de défauts industriels (binaire : OK / défaut). Quel modèle et pourquoi ?

💡 Voir l'indice

Transfer learning.

✅ Voir la réponse

Transfer learning sur un CNN pré-entraîné est le bon choix. Pourquoi : (1) 50 000 images = "moyen" : suffisant pour fine-tuner un modèle pré-entraîné, mais insuffisant pour entraîner from scratch (qui demande des millions). (2) Domaine standard : les défauts industriels (rayures, fissures, taches) ont des features visuelles que les CNN pré-entraînés sur ImageNet capturent déjà bien (bords, textures, formes). Pipeline canonique : (a) Charger un EfficientNet-B0 ou ResNet-50 pré-entraîné. (b) Geler le base model. (c) Ajouter une nouvelle tête : GlobalAveragePooling → Dropout(0.3) → Dense(2, softmax) (ou Dense(1, sigmoid) pour binaire). (d) Entraîner pour 5-10 epochs avec learning rate 1e-3 et data augmentation aggressive (rotation, flip, brightness, blur). (e) Optionnel : dégeler les dernières couches et fine-tuner avec LR 1e-5 pour adapter les features. Métriques à suivre : précision, recall, et particulièrement recall sur la classe "défaut" (rater un défaut est plus grave que générer un faux positif). Probablement viser recall > 95% au prix d'une précision moindre, puis recalibrer si nécessaire. Considérations avancées : (i) data augmentation très importante pour la généralisation, (ii) class balancing si les classes sont déséquilibrées (ex : 95% OK, 5% défaut), (iii) explainability avec Grad-CAM pour montrer où le modèle "regarde" — crucial en industriel pour valider la décision.
