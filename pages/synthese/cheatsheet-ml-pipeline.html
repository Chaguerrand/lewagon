1. Data Collection & Exploration (EDA)

1.1 Objectif

Charger, explorer et comprendre les données avant toute modélisation. Identifier les distributions, corrélations, anomalies.

2. Preprocessing

2.1 Doublons

Python
data.duplicated().sum()          # détecter
data = data.drop_duplicates()    # supprimer — toujours avant le train/test split

Pourquoi c'est critique

Des doublons présents dans le train ET le test provoquent du data leakage et faussent l'évaluation.

2.2 Valeurs manquantes

Python

from sklearn.impute import SimpleImputer

imputer = SimpleImputer(strategy="median")  # mean | median | most_frequent | constant
imputer.fit(X_train[['col']])               # fit sur le TRAIN uniquement

Erreur fréquente

imputer.fit(X_test) — le test set ne doit JAMAIS être utilisé pour fitter un transformer.

⚠️

Un outlier peut être une nouveauté (information légitime). Toujours comprendre la signification métier avant de supprimer.

2.4 Feature Scaling

ScalerFormuleQuand l'utiliserSensibilité outliers
StandardScalerz = (x - μ) / σDistribution ≈ normale, défaut sûrSensible
MinMaxScaler(x - min) / (max - min)Matrice sparse, pixels RGB, KNNTrès sensible
RobustScaler(x - median) / IQRFeatures skewed avec outliersRobuste
Python
from sklearn.preprocessing import StandardScaler, MinMaxScaler, RobustScaler

scaler = StandardScaler()
scaler.fit(X_train[['col']])          # apprend sur le train
X_train['col'] = scaler.transform(X_train[['col']])
X_test['col']  = scaler.transform(X_test[['col']])  # même scaler
💡

Arbre de décision scaling : Feature très skewed → Feature Engineering d'abord (log) — Skewed + outliers → RobustScaler — Distribution ≈ normale → StandardScaler — Matrice sparse / KNN → MinMaxScaler

❌

Data leakage : scaler.fit_transform(X) sur tout le dataset AVANT le cross-validation → chaque fold de validation a "vu" les données de tous les autres folds via le scaler. Solution : utiliser un Pipeline sklearn.

3. Feature Engineering & Selection

3.1 Création de features

Introduire de la connaissance métier pour créer de nouveaux signaux : ratios (bedroom/total_rooms), interactions (BMI = weight/height²), délais (delivered_date - dispatch_date), catégorisation temporelle (weekday vs weekend).

Python
## Transformation log — réduit le skew d'une feature
import numpy as np
data['log_price'] = np.log1p(data['price'])   # log(1+x) gère le cas x=0

## FunctionTransformer pour intégrer dans un Pipeline
from sklearn.preprocessing import FunctionTransformer
log_transformer = FunctionTransformer(np.log1p)   # stateless → compatible Pipeline

3.2 Polynomial Features — Augmenter la complexité d'un modèle linéaire

Ajoute des termes polynomiaux et d'interaction aux features existantes. Permet à un modèle linéaire de capturer des relations non-linéaires.

Python
from sklearn.preprocessing import PolynomialFeatures

## degree=2 → ajoute x², x₁·x₂ pour chaque paire de features
poly = PolynomialFeatures(degree=2, include_bias=False)
X_poly = poly.fit_transform(X)   # [x₁, x₂] → [x₁, x₂, x₁², x₁·x₂, x₂²]
poly.get_feature_names_out()     # noms des colonnes générées

## Dans un Pipeline complet
from sklearn.pipeline import make_pipeline
from sklearn.linear_model import Ridge

pipe = make_pipeline(PolynomialFeatures(degree=2), StandardScaler(), Ridge())
cross_val_score(pipe, X, y, cv=5).mean()
degreeNb de features générées (p features en entrée)Risque
1p (aucun ajout)Underfitting si relation non-linéaire
2p + p(p+1)/2Bon compromis — le plus courant
3+Croissance combinatoireOverfitting rapide, explosion dimensionnelle
⚠️

Toujours combiner PolynomialFeatures avec une régularisation (Ridge ou Lasso) pour contrôler l'overfitting. Plus le degré est élevé, plus l'alpha doit être fort.

3.3 Corrélation entre features

Python
corr_matrix = data.select_dtypes('number').corr()
corr_df = corr_matrix.stack().reset_index()
corr_df.columns = ['f1', 'f2', 'correlation']

corr_df.sort_values('abs_corr', ascending=False).head(10)

Features très corrélées entre elles → redondantes → supprimer l'une des deux.

3.5 Multicolinéarité — VIF

Python
from statsmodels.stats.outliers_influence import variance_inflation_factor as vif

X_scaled = (X - X.mean()) / X.std()   # toujours scaler avant
df_vif = pd.DataFrame()
df_vif['feature'] = X_scaled.columns
df_vif['vif'] = [vif(X_scaled.values, i) for i in range(X_scaled.shape[1])]
💡

Règle : VIF ≥ 10 → multicolinéarité préoccupante. La matrice de corrélation ne détecte que la colinéarité bivariée — le VIF détecte la multicolinéarité multiple.

5.3 K-Fold Cross-Validation

Python
from sklearn.model_selection import cross_validate, cross_val_score

## cross_val_score — retourne uniquement les test scores (usage simple)
scores = cross_val_score(model, X, y, cv=5, scoring='r2')
scores.mean()   # score moyen cross-validé

## cross_validate — retourne test_score + fit_time + score_time (plus complet)
cv_results = cross_validate(model, X, y, cv=5)
cv_results['test_score'].mean()
⚠️

cross_validate et cross_val_score ne retournent que les scores — pas de modèle entraîné en sortie. Le modèle final doit être entraîné séparément avec model.fit(X, y) sur toutes les données.

7. Under the Hood — Gradient Descent & Loss Functions

7.1 Ce qui se passe dans .fit()

Tout modèle ML : y = h(X, β) + error où h = hypothèse, β = paramètres appris.

.fit() trouve β = argmin L(β, X, y, h) — les paramètres qui minimisent la loss.

7.2 Loss Functions — Régression

LossPropriétésCode SGD
L2 (MSE)Pénalise fort les grandes erreurs, sensible outliersSGDRegressor(loss='squared_error')
L1 (MAE)Robuste aux outliers, non diff. en 0SGDRegressor(loss='absolute_error')
HuberMix L1/L2, réglable via epsilonSGDRegressor(loss='huber', epsilon=0.1)

7.3 Loss Functions — Classification

Stateless (pas de mémoire) → FunctionTransformer(np.log1p)

Stateful (mémorise des valeurs du train) → Classe avec TransformerMixin + BaseEstimator :

from sklearn.base import TransformerMixin, BaseEstimator

class MyTransformer(TransformerMixin, BaseEstimator):
    def fit(self, X, y=None):
        self.mean_ = X.mean()   # stocke sur le train
        return self
    def transform(self, X, y=None):
        return X - self.mean_   # applique sur train ET test

10.2 Bagging — Random Forest

Plusieurs arbres en parallèle sur des bootstraps distincts → réduit la variance.

Bootstrapping : chaque arbre s'entraîne sur un échantillon tiré avec remplacement. En moyenne, ~63% des données originales sont représentées dans chaque bootstrap. Les ~37% restants (Out-of-Bag) servent de pseudo-validation.

Python

forest.feature_importances_    # importance de chaque feature (basée sur réduction Gini, somme = 1)
💡

OOB Score : chaque arbre est évalué sur les échantillons qu'il n'a PAS vus (les ~37% out-of-bag). La moyenne de ces scores donne une estimation de la performance sans avoir besoin d'un validation set séparé. Très utile quand le dataset est petit.

2. Preprocessing

Le preprocessing, c'est le nettoyage de tes données avant de les donner à un modèle. Un modèle ne peut pas apprendre correctement sur des données sales (doublons, valeurs manquantes, outliers, colonnes non-numériques).

2.1 Supprimer les doublons

Des lignes identiques dans le dataset faussent les statistiques et peuvent provoquer du data leakage si la même ligne se retrouve à la fois dans le train et le test.

Python
data.duplicated().sum()       # combien de doublons ?
data = data.drop_duplicates() # on les supprime
⚠️

Pourquoi c'est critique : si une ligne apparaît en double et se retrouve dans le train ET dans le test, ton modèle a "vu" la réponse avant l'examen. C'est du data leakage — tes scores seront artificiellement bons, mais le modèle sera mauvais en production. Toujours supprimer les doublons AVANT le split train/test.

2.2 Gérer les valeurs manquantes (NaN)

Les valeurs manquantes (NaN = Not a Number) sont courantes dans les datasets réels. Tu as deux options : supprimer la colonne ou imputer (remplacer par une valeur).

Python
data.isnull().sum() / len(data)  # pourcentage de NaN par colonne

Python
from sklearn.impute import SimpleImputer

# FIT uniquement sur le train — l'imputer apprend la médiane du train

# TRANSFORM sur les deux — applique la même médiane apprise sur le train

X_test['col']  = imputer.transform(X_test[['col']])   # applique la MÊME médiane sur le test
⚠️

Erreur classique — data leakage : si tu fais imputer.fit(X_test), tu calcules la médiane du test. Résultat : le test "informe" le preprocessing, ce qui fausse l'évaluation. La règle absolue : fit uniquement sur le train, transform sur les deux.

2.4 Feature Scaling (normalisation)

La plupart des modèles ML sont sensibles à l'échelle des features. Si une feature va de 0 à 1 000 000 et une autre de 0 à 1, le modèle va sur-pondérer la première. Le scaling ramène tout à la même échelle.

ScalerFormuleQuand l'utiliserSensibilité aux outliers
StandardScaler$z = \frac{x - \mu}{\sigma}$Distribution approximativement normale, choix par défaut sûrSensible
MinMaxScaler$\frac{x - \min}{\max - \min}$Matrices creuses, pixels RGB, KNNTrès sensible
RobustScaler$\frac{x - \text{médiane}}{\text{IQR}}$Features avec beaucoup d'outliersRobuste
Python
from sklearn.preprocessing import StandardScaler, MinMaxScaler, RobustScaler

scaler = StandardScaler()
scaler.fit(X_train[['col']])           # apprend μ et σ sur le train

💡

Arbre de décision pour choisir ton scaler :

- Feature très asymétrique (skewed) → appliquer d'abord une transformation log, puis StandardScaler

- Asymétrique + outliers → RobustScaler

- Distribution approximativement normale → StandardScaler

- Matrices creuses (beaucoup de zéros) ou KNN → MinMaxScaler

⚠️

Data leakage classique : faire scaler.fit_transform(X) sur tout le dataset AVANT la cross-validation fait fuir les statistiques du test dans le preprocessing. La solution propre : utiliser un Pipeline sklearn (voir section 9).

3.2 PolynomialFeatures — Rendre un modèle linéaire plus puissant

3.5 VIF — Détecter la multicolinéarité multiple

La corrélation entre deux features est bivariée — elle ne détecte pas quand une feature est une combinaison linéaire de plusieurs autres. Le VIF (Variance Inflation Factor) détecte cette multicolinéarité multiple.

Python
from statsmodels.stats.outliers_influence import variance_inflation_factor as vif

X_scaled = (X - X.mean()) / X.std()  # toujours scaler avant le calcul VIF
df_vif = pd.DataFrame()
df_vif['feature'] = X_scaled.columns
df_vif['vif'] = [vif(X_scaled.values, i) for i in range(X_scaled.shape[1])]
💡

Règle : VIF ≥ 10 → multicolinéarité préoccupante, envisage de supprimer la feature. La heatmap de corrélation ne détecte que la colinéarité entre deux features — le VIF détecte les combinaisons de plusieurs features.

5. Training — Fit, Split, Validate

5.1 Le workflow sklearn en 4 lignes

Tout modèle sklearn suit exactement le même schéma : instancier → entraîner → évaluer → prédire.

Python
from sklearn.linear_model import LinearRegression

model = LinearRegression()       # 1. Instancier : créer l'objet modèle vide
model.fit(X_train, y_train)      # 2. Entraîner : le modèle apprend sur le train
model.score(X_test, y_test)      # 3. Évaluer : R² pour régression, accuracy pour classification
model.predict(X_new)             # 4. Prédire : sur de nouvelles données jamais vues

5.2 Train/Test Split

Avant tout preprocessing, on coupe le dataset en deux : une partie pour entraîner (train), une partie pour évaluer (test). Le test simule des données "inconnues" — comme un examen après l'entraînement.

Python
from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(
    X, y,
    test_size=0.3,    # 30% des données pour le test, 70% pour le train
    random_state=42   # graine aléatoire pour la reproductibilité
)
⚠️

Erreur fréquente : tester plusieurs valeurs de random_state et garder celle qui donne le meilleur score — c'est du data leakage ! Tu as en réalité optimisé sur le test. Fixe un random_state a priori et ne le change plus.

5.3 K-Fold Cross-Validation

Le split simple train/test donne un score qui dépend du hasard du split. La cross-validation fait K splits différents et moyennise les scores — bien plus fiable.

Python
from sklearn.model_selection import cross_validate, cross_val_score

# cross_val_score : simple et rapide — retourne uniquement les scores des K folds
scores = cross_val_score(model, X, y, cv=5, scoring='r2')
scores.mean()   # score moyen sur les 5 folds

# cross_validate : plus complet — retourne aussi les temps de fit et de score
cv_results = cross_validate(model, X, y, cv=5)
cv_results['test_score'].mean()  # score moyen
🧠

cross_validate et cross_val_score ne retournent que des scores — ils n'entraînent pas un modèle final. Pour obtenir un modèle utilisable après la cross-validation, tu dois le réentraîner séparément avec model.fit(X, y) sur toutes les données disponibles.

5.4 Baseline — Toujours commencer ici

Avant tout modèle sophistiqué, commence par un modèle stupide (DummyRegressor/Classifier). Il te donne le plancher minimum à dépasser. Si ton modèle n'est pas meilleur que ça, quelque chose ne va pas.

Python
from sklearn.dummy import DummyRegressor, DummyClassifier

# DummyRegressor(strategy="mean") : prédit toujours la moyenne du train
baseline = DummyRegressor(strategy="mean").fit(X_train, y_train)
baseline.score(X_test, y_test)  # R² ≈ 0 — le plancher absolu

7. Under the Hood — Gradient Descent & Loss Functions

7.1 Ce qui se passe dans .fit()

Tout modèle ML cherche les paramètres $\beta$ qui font les meilleures prédictions :

$$y = h(X, \beta) + \varepsilon$$

où $h$ est la fonction hypothèse et $\varepsilon$ l'erreur résiduelle.

.fit() résout : $\beta^* = \underset{\beta}{\arg\min} \; L(\beta, X, y, h)$ — trouver les paramètres qui minimisent la fonction de perte (loss).

7.2 Loss Functions — Régression

HuberMix L1/L2 réglable, bonne robustesseSGDRegressor(loss='huber', epsilon=0.1)

Loss ≠ Métrique : la loss sert à entraîner le modèle (elle doit être différentiable pour que le gradient descent fonctionne). Les métriques comme accuracy ou R² servent à évaluer après l'entraînement. On n'entraîne pas sur l'accuracy.

7.4 Gradient Descent — Variantes

L'algorithme qui minimise la loss en ajustant les paramètres pas à pas, en descente de gradient.

Quand utiliser SGD :

  • n ≥ 100 000 lignes (l'inversion matricielle devient trop coûteuse)
  • Pour sortir d'un minimum local
  • Deep Learning par défaut

LinearRegression vs SGDRegressor :

  • LinearRegression = inversion matricielle, complexité $O(p^3)$ — exact mais lent si p est grand
  • SGDRegressor = gradient descent itératif — bien plus rapide sur grands n ou p

8. Tuning — Régularisation & Hyperparameter Search

8.1 Régularisation

La régularisation consiste à ajouter une pénalité sur les coefficients dans la loss pour éviter l'overfitting :

$$\text{Loss régularisée} = L(X, y, \beta) + \text{Pénalité}(\beta)$$
MéthodePénalitéEffetQuand l'utiliser

from sklearn.linear_model import Ridge, Lasso, ElasticNet

ridge   = Ridge(alpha=0.2).fit(X_train, y_train)    # alpha élevé = plus de régularisation
lasso   = Lasso(alpha=0.2).fit(X_train, y_train)    # annule des coefficients
elastic = ElasticNet(alpha=1, l1_ratio=0.2).fit(X_train, y_train)
# l1_ratio=0 → comportement Ridge pure
# l1_ratio=1 → comportement Lasso pure
⚠️

Toujours scaler avant Ridge, Lasso, SVM : si une feature est en millions (salaire) et une autre en unités (âge), la pénalité sera disproportionnée. Le scaling est obligatoire pour que la régularisation soit équitable.

8.2 GridSearchCV

GridSearch teste toutes les combinaisons d'hyperparamètres possible et garde la meilleure via cross-validation.

Python
from sklearn.model_selection import GridSearchCV

grid = {'alpha': [0.01, 0.1, 1], 'l1_ratio': [0.2, 0.5, 0.8]}
# 3 × 3 = 9 combinaisons testées, chacune évaluée en cross-validation à 5 folds = 45 fits

search = GridSearchCV(ElasticNet(), grid, scoring='r2', cv=5, n_jobs=-1)
search.fit(X_train, y_train)  # fit sur le train uniquement

search.best_score_      # meilleur score moyen en cross-validation
search.best_params_     # combinaison d'hyperparamètres gagnante
search.best_estimator_  # modèle entraîné avec ces hyperparamètres

8.3 RandomizedSearchCV

Quand l'espace de recherche est grand, GridSearch prend trop de temps. RandomizedSearch tire aléatoirement un nombre fixé de combinaisons — bien plus rapide.

Python
from sklearn.model_selection import RandomizedSearchCV
from scipy import stats

grid = {
    'alpha':    [0.001, 0.01, 0.1, 1],
    'l1_ratio': stats.uniform(0, 1)   # distribution continue entre 0 et 1
}

search = RandomizedSearchCV(ElasticNet(), grid, scoring='r2', n_iter=100, cv=5, n_jobs=-1)
# n_iter=100 → 100 combinaisons tirées aléatoirement, pas toutes
search.fit(X_train, y_train)

Distributions scipy utiles pour RandomizedSearch :

  • stats.norm(10, 2) — si tu as un a priori sur la valeur probable
  • stats.randint(1, 100) — entier uniforme entre 1 et 100
  • stats.uniform(1, 100) — flottant uniforme entre 1 et 100
  • stats.loguniform(0.01, 1) — explore plusieurs ordres de grandeur (idéal pour alpha)
💡

Bonne pratique : commence par RandomizedSearchCV avec loguniform sur une large plage pour localiser la bonne zone, puis affine avec GridSearchCV sur une plage plus restreinte.

⚠️

Erreur : tuner les hyperparamètres en regardant le score sur le test set → le test est contaminé. Toujours tuner en cross-validation sur le train set uniquement. Le test set est utilisé UNE seule fois, à la toute fin.

8.4 SVM — Support Vector Machines

Les SVM cherchent la frontière de décision qui maximise la marge entre les classes.

Python
from sklearn.svm import SVC, SVR

SVC(kernel='linear', C=10)           # frontière linéaire, C élevé = peu régularisé
SVC(kernel='rbf', C=1, gamma=0.1)    # noyau gaussien pour données non-linéaires
SVC(kernel='poly', degree=2, C=1)    # noyau polynomial
SVR(epsilon=0.1, C=1, kernel='rbf')  # version régression du SVM

Hyperparamètre C :

  • C élevé → marge stricte, peu d'erreurs tolérées → moins régularisé → risque d'overfitting
  • C faible → marge souple, plus d'erreurs tolérées → très régularisé → risque d'underfitting

Hyperparamètre gamma (noyau RBF) :

  • Petit gamma → frontière lisse (underfitting)
  • Grand gamma → frontière très irrégulière (overfitting)
  • Toujours tuner C et gamma ensemble.
⚠️

Toujours scaler avant SVM : le SVM est basé sur les distances entre points — des features non-normalisées le rendent inutilisable.

9. Pipelines sklearn — Le workflow professionnel

9.1 Pourquoi utiliser des Pipelines ?

Sans Pipeline, il est facile d'oublier d'appliquer le même preprocessing sur le test, ou de fitter un scaler sur les deux splits. Le Pipeline garantit :

  • Un ordre garanti des étapes
  • Un code lisible et reproductible
  • Une protection automatique contre le data leakage en cross-validation (le fit ne se fait que sur le fold train)

9.2 Pipeline séquentiel simple

Un Pipeline chaîne des transformateurs l'un après l'autre, puis un modèle final. make_pipeline nomme les étapes automatiquement.

Python
from sklearn.pipeline import Pipeline, make_pipeline
from sklearn.impute import SimpleImputer
from sklearn.preprocessing import StandardScaler

# Pipeline : imputer → scaler → (potentiellement un modèle)
pipeline = make_pipeline(
    SimpleImputer(strategy="median"),  # étape 1 : combler les NaN
    StandardScaler()                   # étape 2 : normaliser
)
pipeline.fit(X_train[['age']])         # fit uniquement sur le train
pipeline.transform(X_test[['age']])    # transform sur le test

9.3 ColumnTransformer — Traitement différencié par type de colonne

Dans la réalité, les colonnes numériques et catégorielles ne se traitent pas de la même façon. ColumnTransformer applique un Pipeline différent selon le type de colonne.

Python
from sklearn.compose import ColumnTransformer, make_column_transformer, make_column_selector
from sklearn.preprocessing import OneHotEncoder

# Pipeline pour colonnes numériques : imputer les NaN + scaler
num_transformer = make_pipeline(SimpleImputer(strategy="mean"), StandardScaler())

# Pipeline pour colonnes catégorielles : juste OHE (les NaN sont gérés via handle_unknown)
cat_transformer = OneHotEncoder(handle_unknown='ignore')

# Assembler les deux : auto-détection des types de colonnes
preprocessor = make_column_transformer(
    (num_transformer, make_column_selector(dtype_include=['float64'])),      # → colonnes float
    (cat_transformer, make_column_selector(dtype_include=['object', 'bool'])), # → colonnes texte/bool
    remainder='passthrough'  # conserver les colonnes non traitées tel quel
)
⚠️

Sans remainder='passthrough', toutes les colonnes non mentionnées explicitement sont supprimées silencieusement — erreur difficile à déboguer.

9.4 FeatureUnion — Branches parallèles concaténées

FeatureUnion applique des transformateurs en parallèle et concatène leurs sorties. Utile pour ajouter des features calculées au preprocessing standard.

Python
from sklearn.pipeline import FeatureUnion
from sklearn.preprocessing import FunctionTransformer
import pandas as pd

# Créer une nouvelle feature (ratio BMI/âge) à partir du DataFrame
bmi_age_ratio = FunctionTransformer(lambda df: pd.DataFrame(df['bmi'] / df['age']))

# Concaténer le preprocessing standard + la nouvelle feature
union = FeatureUnion([
    ('preprocess',    preprocessor),    # colonnes 0 à 7 : preprocessing normal
    ('bmi_age_ratio', bmi_age_ratio)    # colonne 8 : nouvelle feature calculée
])
X_transformed = union.fit_transform(X_train)  # shape = (n_samples, 9)

9.5 Pipeline complet avec modèle — La façon propre

Python
from sklearn.linear_model import Ridge

# Pipeline complet : preprocessing → modèle
pipeline = make_pipeline(preprocessor, Ridge())

pipeline.fit(X_train, y_train)    # fit TOUT d'un coup sur le train
pipeline.predict(X_test)          # transform + predict automatiquement
pipeline.score(X_test, y_test)    # évaluation finale

# Cross-validation sans data leakage : le preprocessing est refitté à chaque fold
cross_val_score(pipeline, X_train, y_train, cv=5, scoring='r2').mean()

9.6 GridSearch sur Pipeline

La syntaxe pour accéder aux paramètres d'un Pipeline dans GridSearch utilise des doubles underscores pour naviguer dans la hiérarchie des étapes.

Python
from sklearn.model_selection import GridSearchCV

grid_search = GridSearchCV(
    pipeline,
    param_grid={
        # Syntaxe : 'nom_étape__sous_étape__paramètre'

    },
    cv=5, scoring='r2'
)
grid_search.fit(X_train, y_train)
pipeline_tuned = grid_search.best_estimator_  # pipeline complet avec les meilleurs hyperparamètres
💡

Utilise pipeline.get_params() pour voir tous les paramètres accessibles avec leur chemin exact.

9.7 Custom Transformers

Quand les transformateurs sklearn ne suffisent pas, tu peux créer les tiens.

Stateless (sans mémoire, pas de fit nécessaire) → FunctionTransformer :

Python
from sklearn.preprocessing import FunctionTransformer
log_transformer = FunctionTransformer(np.log1p)  # s'intègre directement dans un Pipeline

Stateful (mémorise des valeurs du train pour les appliquer au test) → classe avec TransformerMixin :

Python
from sklearn.base import TransformerMixin, BaseEstimator

class MyTransformer(TransformerMixin, BaseEstimator):
    def fit(self, X, y=None):
        self.mean_ = X.mean()  # calcule et mémorise la moyenne du train
        return self            # toujours retourner self dans fit()

    def transform(self, X, y=None):
        return X - self.mean_  # applique la soustraction avec la moyenne apprise sur le train

10. Ensemble Methods

Les méthodes d'ensemble combinent plusieurs modèles pour obtenir de meilleures performances qu'un seul modèle.

10.1 Decision Trees — La brique de base

Un arbre de décision pose des questions binaires successives sur les features pour séparer les données en groupes homogènes (feuilles).

Python
from sklearn.tree import DecisionTreeClassifier, DecisionTreeRegressor

tree = DecisionTreeClassifier(
    max_depth=3,          # profondeur maximale de l'arbre
    min_samples_split=7,  # nombre minimum d'exemples pour créer une division
    min_samples_leaf=3    # nombre minimum d'exemples dans chaque feuille
)
tree.fit(X_train, y_train)
tree.feature_importances_  # importance de chaque feature (basée sur réduction de l'indice de Gini)

Indice de Gini : mesure l'impureté d'un noeud : $\text{Gini} = 1 - \sum p_i^2$

  • Gini = 0 → noeud pur (toutes les observations appartiennent à la même classe)
  • Gini = 0.5 → noeud maximalement impur (classification binaire)
HyperparamètreEffet si augmentéRisque si trop bas
max_depthRéduit la variance, augmente le biaisOverfitting (arbre trop profond mémorise)
min_samples_splitRéduit la varianceOverfitting
min_samples_leafRéduit la varianceOverfitting
⚠️

DecisionTreeRegressor() sans max_depth → l'arbre mémorise chaque point du train, R² train = 1.0 mais R² test ≈ 0. Toujours limiter la profondeur.

10.2 Bagging — Random Forest

Le Random Forest entraîne des centaines d'arbres en parallèle, chacun sur un sous-échantillon différent des données (bootstrapping), puis agrège leurs prédictions. Objectif : réduire la variance.

Bootstrapping : chaque arbre s'entraîne sur un échantillon tiré avec remplacement. En moyenne, ~63% des données originales sont représentées dans chaque bootstrap. Les ~37% restants (Out-of-Bag) servent de pseudo-validation gratuite.

Python
from sklearn.ensemble import RandomForestRegressor

forest = RandomForestRegressor(
    n_estimators=100,  # nombre d'arbres dans la forêt
    max_depth=5,       # profondeur maximale de chaque arbre
    n_jobs=-1,         # utiliser tous les cœurs CPU disponibles
    oob_score=True     # activer le score Out-of-Bag
)
forest.fit(X_train, y_train)
forest.oob_score_           # score OOB : évaluation gratuite sur les ~37% non tirés
forest.feature_importances_ # importance de chaque feature (somme = 1)
💡

OOB Score : chaque arbre est évalué sur les exemples qu'il n'a PAS vus lors de son entraînement. La moyenne de ces scores donne une estimation fiable de la performance sans avoir besoin d'un validation set séparé — précieux quand le dataset est petit.

10.3 Boosting — Gradient Boosting & XGBoost

Le Boosting entraîne des arbres séquentiellement : chaque nouvel arbre apprend les erreurs (résidus) du précédent. Objectif : réduire le biais.

Python
from sklearn.ensemble import GradientBoostingRegressor
from xgboost import XGBRegressor

# Gradient Boosting sklearn
gbr = GradientBoostingRegressor(n_estimators=100, learning_rate=0.1, max_depth=3)

# XGBoost : implémentation optimisée, bien plus rapide
xgb = XGBRegressor(
    n_estimators=100,
    learning_rate=0.1,
    max_depth=10,
    early_stopping_rounds=5  # stoppe si le score ne s'améliore pas après 5 rounds
)
xgb.fit(X_train, y_train, eval_set=[(X_val, y_val)])  # surveille le score sur un val set
💡

Règle pratique XGBoost : learning_rate petit (0.01 à 0.1) + n_estimators grand + early_stopping_rounds pour éviter l'overfitting. Plus le learning rate est petit, plus tu peux augmenter le nombre d'estimateurs.

10.5 Stacking et Voting — Combiner des modèles hétérogènes

Python
from sklearn.ensemble import StackingClassifier, VotingClassifier
from sklearn.ensemble import RandomForestClassifier as RF
from sklearn.linear_model import LogisticRegression as LR
from sklearn.neighbors import KNeighborsClassifier as KNN

)
⚠️

Erreur à éviter : BaggingRegressor(RandomForestRegressor()) → tu bags un Random Forest qui est déjà une forêt. Résultat : 100 × 100 = 10 000 arbres pour un gain marginal. Ne bague pas un modèle qui est déjà un ensemble.

11.6 Vérifier les conditions d'inférence

Python
predicted = model.predict(X)

11.7 Régression logistique — Interpréter les coefficients

Python
model = smf.logit(formula='survived ~ fare + C(sex) + age', data=titanic).fit()
model.params                         # coefficients en LOG-ODDS (pas en probabilité directe)
np.exp(model.params['fare'])         # odds ratio : ex 1.015 = +1.5% d'odds par dollar supplémentaire
model.prsquared                      # pseudo R² de McFadden (pas comparable au R² OLS)

Conversion log-odds → probabilité :

$$\text{log-odds} \xrightarrow{\exp} \text{odds} = e^{\text{log-odds}} \xrightarrow{} p = \frac{\text{odds}}{1 + \text{odds}}$$
⚠️

Erreur classique d'interprétation : $\beta = 0.015$ ne signifie PAS "fare augmente la probabilité de 1.5%". Le coefficient augmente les log-odds, pas la probabilité directement. La conversion probabilité ↔ log-odds est non-linéaire — l'effet réel sur la probabilité dépend du niveau de départ.

12. Checklist Projet ML

Voici la liste complète des étapes dans le bon ordre, à suivre pour tout projet ML :

  1. Explorer → describe(), info(), corr(), boxplots, histogrammes
  2. Nettoyer → supprimer doublons, gérer les NaN (imputer ou drop), traiter les outliers
  3. Splitter → train_test_split AVANT tout preprocessing — c'est non-négociable
  4. Preprocesser dans un Pipeline → scaler si nécessaire, encoder les catégories, feature engineering
  5. Baseline → DummyRegressor ou DummyClassifier — établir le plancher minimal
  6. Modèle simple → LinearRegression ou LogisticRegression — établir une référence
  7. Évaluer → cross_validate, métriques adaptées au contexte métier
  8. Itérer → learning curves, feature selection, modèle plus complexe si nécessaire
  9. Tuner → GridSearchCV ou RandomizedSearchCV sur le Pipeline, sur le train set
  10. Évaluation finale → score sur le test set UNE seule fois — c'est l'examen final
  11. Exporter → pickle.dump(pipeline_tuned, f) — sérialiser le Pipeline complet

📌 TL;DR

  • Un pipeline ML se lit dans l'ordre : EDA → Preprocessing → Feature Engineering → Choix du modèle → Training → Évaluation → Tuning → Export.
  • Toujours splitter train/test AVANT tout preprocessing — le test simule des données inconnues en production.
  • fit() uniquement sur le train, transform() sur les deux — toujours, sans exception.
  • Le Pipeline sklearn est la solution propre pour éviter le data leakage et rendre le code reproductible.
  • Commencer par une baseline (DummyRegressor/Classifier), puis un modèle simple, puis progresser.
  • Bagging (Random Forest) réduit la variance via des arbres parallèles ; Boosting (XGBoost) réduit le biais via des arbres séquentiels sur les résidus.
  • Le test set est utilisé UNE seule fois à la toute fin — l'utiliser pour tuner les hyperparamètres le contamine.
  • sklearn optimise la performance prédictive ; statsmodels fournit l'inférence statistique (p-values, IC).
  • Loss ≠ Métrique : la loss sert à entraîner (doit être différentiable) ; les métriques servent à évaluer.
  • La régularisation (Ridge/Lasso/ElasticNet) ne fonctionne correctement que si les features sont scalées au préalable.

# 1. Doublons exacts — presque toujours une erreur
df = df.drop_duplicates()

# 2. Missing values : stratégies différentes selon le type
#    - numérique : médiane (robuste aux outliers, contrairement à la moyenne)
#    - catégoriel : string "missing" (garde l'info "c'était manquant")
num_cols = df.select_dtypes(include="number").columns
cat_cols = df.select_dtypes(include=["object", "category"]).columns

df[num_cols] = df[num_cols].fillna(df[num_cols].median())
df[cat_cols] = df[cat_cols].fillna("missing")

# 3. Outliers : IQR rule (plus robuste que mean±3σ)
def cap_outliers(s, factor=1.5):
    """Remplace les valeurs < Q1-1.5*IQR et > Q3+1.5*IQR par ces bornes."""
    q1, q3 = s.quantile([0.25, 0.75])
    iqr = q3 - q1
    return s.clip(q1 - factor*iqr, q3 + factor*iqr)

df["price"] = cap_outliers(df["price"])

# Pourquoi capper plutôt que supprimer : supprimer des lignes perd de l'info
# et biaise la distribution. Capping garde le signal ("c'est cher") sans laisser
# un outlier à 10^9 € détruire le scaler.
⚠️

Piège : ne JAMAIS faire ce cleaning sur l'ensemble du dataset avant le split train/test. La médiane et les quantiles doivent être calculés uniquement sur le train, sinon vous avez du data leakage. Utilisez SimpleImputer dans un Pipeline sklearn pour respecter cette règle automatiquement.

Python
from sklearn.compose import ColumnTransformer
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.ensemble import RandomForestClassifier

num_features = ["age", "income", "tenure"]
cat_features = ["city", "department"]

# ColumnTransformer = applique des traitements DIFFÉRENTS selon le type de colonne
preprocessor = ColumnTransformer([
    ("num", Pipeline([
        ("imputer", SimpleImputer(strategy="median")),   # Médiane : robuste
        ("scaler", StandardScaler()),                     # Centrage-réduction
    ]), num_features),
    ("cat", Pipeline([
        ("imputer", SimpleImputer(strategy="constant", fill_value="missing")),
        ("encoder", OneHotEncoder(handle_unknown="ignore")),  # ignore = robuste aux catégories inconnues en test
    ]), cat_features),
])

# Pipeline final : preprocessing + modèle en une seule entité
pipe = Pipeline([
    ("preprocess", preprocessor),
    ("model", RandomForestClassifier(n_estimators=100, random_state=42)),
])

# Un seul .fit() qui fait TOUT : imputation, scaling, encoding, entraînement
pipe.fit(X_train, y_train)

# Un seul .score() qui applique TOUT le preprocessing avant de prédire
pipe.score(X_test, y_test)

# Pourquoi ce pattern est crucial : en encapsulant tout dans un Pipeline,
# vous garantissez que le test set reçoit EXACTEMENT le même traitement que le train,
# et vous pouvez sauvegarder un seul objet pour le déploiement. Zéro risque de leakage.

Q3 : Vous avez 100 features et vous voulez un modèle interprétable. Quelle approche ?

💡 Voir l'indice

Lasso ou tree-based avec SHAP.

✅ Voir la réponse

Deux bonnes approches. Approche 1 — Lasso pour la sélection : Lasso(alpha=...) sur des features standardisées met à zéro les features non informatives. Gardez seulement celles à coefficient non nul, puis entraînez votre modèle final sur ce sous-ensemble. Avantage : vous obtenez à la fois un modèle linéaire interprétable et une liste claire des features qui comptent. Inconvénient : Lasso est linéaire, peut rater des relations non linéaires. Approche 2 — Random Forest + SHAP : entraînez un RF (capture les interactions et relations non linéaires), puis utilisez SHAP pour expliquer chaque prédiction et identifier les features les plus importantes globalement. SHAP donne une décomposition par feature qui est interprétable même pour un non-statisticien. Approche hybride (recommandée) : (1) Lasso pour la sélection brute (passer de 100 à ~20 features), (2) Random Forest sur les 20 features sélectionnées pour gagner en performance, (3) SHAP pour expliquer le modèle final. Cette pipeline combine sélection automatique, performance non linéaire, et interprétabilité. Bonus : si les stakeholders demandent une explication "simple", générer un dashboard SHAP avec les top 5 features importantes globalement et un sample d'explications individuelles. C'est plus convaincant qu'une équation Lasso à 20 termes.
