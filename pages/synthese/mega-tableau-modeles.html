LinearRegression

CritèreDétail
QuandRelation linéaire, besoin d'interprétabilité, premier modèle à tester
TypeRégression
HyperparamètresAucun (pas de régularisation)
LossMSE (OLS) : $\sum(y_i - \hat{y}_i)^2$
SolverInversion matricielle (SVD) — $O(p^3)$
MétriquesR², RMSE, MAE
ScalingNon obligatoire (n'affecte pas les résultats) mais recommandé pour interprétabilité
Diagnostic overfitLearning curves : si train score >> test score → ajouter régularisation (Ridge/Lasso)
Diagnostic underfitLearning curves : plateau bas train ET test → ajouter des features
Erreurs fréquentesMulticolinéarité → coefs instables. Utiliser np.linalg.matrix_rank(X) pour vérifier
PreprocessingEncoder les catégorielles (OHE). Imputer les NaN
TuningAucun hyperparamètre à tuner → passer directement à Ridge/Lasso
SGD equiv.SGDRegressor(loss='squared_error') — plus rapide si n ou p grands
Python
from sklearn.linear_model import LinearRegression
model = LinearRegression()
model.fit(X_train, y_train)
print(f"R²: {model.score(X_test, y_test):.3f}")
print(f"Coefs: {model.coef_}, Intercept: {model.intercept_}")
Python
from sklearn.pipeline import make_pipeline
pipe = make_pipeline(StandardScaler(), LinearRegression())

Ridge (L2)

CritèreDétail
QuandToutes features potentiellement utiles, multicolinéarité, défaut quand on régularise
TypeRégression
Hyperparamètresalpha (défaut=1) : alpha↑ = + régularisation, coefs → 0
Loss$\sum(y-\hat{y})^2 + \alpha \sum \beta_j^2$
SolverInversion matricielle ou SGD
MétriquesR², RMSE, MAE
ScalingOUI obligatoire — pénalité sensible à l'échelle
Diagnostic overfitalpha trop petit → augmenter alpha
Diagnostic underfitalpha trop grand → diminuer alpha
Erreurs fréquentesOublier le scaling. Confondre avec C de LogisticRegression (inverse!)
PreprocessingScaling + encoding + imputation
Tuningalpha: GridSearch sur [0.001, 0.01, 0.1, 1, 10, 100] ou loguniform(0.001, 100)
SGD equiv.SGDRegressor(loss='squared_error', penalty='l2')
Python
from sklearn.linear_model import Ridge
from sklearn.preprocessing import StandardScaler
scaler = StandardScaler()
X_train_sc = scaler.fit_transform(X_train)
X_test_sc = scaler.transform(X_test)
ridge = Ridge(alpha=1).fit(X_train_sc, y_train)
print(f"R²: {ridge.score(X_test_sc, y_test):.3f}")
Python
from sklearn.pipeline import make_pipeline
pipe = make_pipeline(StandardScaler(), Ridge(alpha=1))

Lasso (L1)

CritèreDétail
QuandFeature selection automatique, interprétabilité, beaucoup de features
TypeRégression
Hyperparamètresalpha (défaut=1) : alpha↑ = + features éliminées (coefs = 0)
Loss$sum(y-hat{y})^2 + alpha sum
SolverCoordinate descent
MétriquesR², RMSE, MAE
ScalingOUI obligatoire
Diagnostic overfitalpha trop petit → augmenter alpha
Diagnostic underfitalpha trop grand → trop de features éliminées → diminuer alpha
Erreurs fréquentesOublier le scaling. Alpha trop fort → élimine des features importantes
PreprocessingScaling + encoding + imputation
Tuningalpha: GridSearch [0.001, 0.01, 0.1, 1, 10] ou loguniform(0.001, 10)
SGD equiv.SGDRegressor(loss='squared_error', penalty='l1')
Python
from sklearn.linear_model import Lasso
lasso = Lasso(alpha=1).fit(X_train_sc, y_train)
print(f"R²: {lasso.score(X_test_sc, y_test):.3f}")
print(f"Features éliminées: {(lasso.coef_ == 0).sum()}/{len(lasso.coef_)}")
Python
from sklearn.pipeline import make_pipeline
pipe = make_pipeline(StandardScaler(), Lasso(alpha=1))

LogisticRegression

CritèreDétail
QuandClassification linéaire, interprétabilité (odds ratios), premier classifieur à tester
TypeClassification
Hyperparamètrespenalty (None/l1/l2/elasticnet), C (=1/α, défaut=1), solver, max_iter
LossLog-Loss : $-\frac{1}{n}\sum[y\log\hat{y} + (1-y)\log(1-\hat{y})]$
Solverlbfgs (défaut, L2), liblinear (requis pour L1), saga (tous)
MétriquesAccuracy, Precision, Recall, F1, ROC-AUC
ScalingOUI si régularisé (penalty ≠ None)
Diagnostic overfitC trop grand → diminuer C (= augmenter régul)
Diagnostic underfitC trop petit → augmenter C (= diminuer régul)
Erreurs fréquentesConfondre C et alpha (C = INVERSE!). penalty='l1' sans solver='liblinear'. Non convergence → augmenter max_iter
PreprocessingScaling + encoding + imputation. SMOTE si déséquilibré
TuningC: loguniform(0.001, 1000), penalty: ['l1','l2'], solver: dépend du penalty
SGD equiv.SGDClassifier(loss='log_loss')
Interprétationcoef > 0 → log-odds augmentent. exp(coef) = odds ratio
Python
from sklearn.linear_model import LogisticRegression
## Sans régularisation (vrais coefs)
model = LogisticRegression(penalty=None, max_iter=5000, tol=1e-9)
## Avec L2
model = LogisticRegression(penalty='l2', C=1)
## Avec L1 (feature selection)
model = LogisticRegression(penalty='l1', C=0.1, solver='liblinear')
model.fit(X_train_sc, y_train)
print(f"Accuracy: {model.score(X_test_sc, y_test):.3f}")
Python
from sklearn.pipeline import make_pipeline
pipe = make_pipeline(StandardScaler(), LogisticRegression(C=1))

KNeighborsRegressor / KNeighborsClassifier

CritèreDétail
QuandRelations non-linéaires, peu de features, dataset pas trop grand
TypeRégression ET Classification
Hyperparamètresn_neighbors (1-50, K↑=underfit, K↓=overfit), p (1=Manhattan, 2=Euclid), weights (uniform/distance)
LossAucune loss classique — basé sur la distance aux voisins
SolverCalcul de distances (brute force ou KD-tree)
MétriquesReg: R², RMSE, MAE. Classif: Accuracy, F1, ROC-AUC
ScalingOUI OBLIGATOIRE — distance-based, les features avec grandes valeurs dominent
Diagnostic overfitK trop petit (surtout K=1) → augmenter K
Diagnostic underfitK trop grand → diminuer K. Elbow method pour trouver l'optimal
Erreurs fréquentesOublier le scaling! K=1 overfit toujours. Lent sur grands datasets
PreprocessingScaling obligatoire + encoding + imputation
Tuningn_neighbors: range(1, 50), p: [1, 2, 3], weights: ['uniform','distance']
SGD equiv.Aucun — KNN n'est pas un modèle linéaire

SVC (Classification)

CritèreDétail
QuandClassification non-linéaire, frontières complexes, dataset moyen
TypeClassification
Hyperparamètreskernel (linear/poly/rbf/sigmoid), C (coût erreurs), gamma (rayon RBF), degree (poly), coef0 (poly/sigmoid)
LossHinge : $\max(0, 1 - y \cdot \hat{y})$
SolverOptimisation quadratique (libsvm)
MétriquesAccuracy, Precision, Recall, F1, ROC-AUC
ScalingOUI OBLIGATOIRE — distance-based
Diagnostic overfitC trop grand → diminuer C. gamma trop grand → diminuer gamma
Diagnostic underfitC trop petit → augmenter C. gamma trop petit → augmenter gamma
Erreurs fréquentesOublier le scaling. Ne pas tuner C et gamma ensemble. Gridsearch poly en même temps que rbf (sklearn bug)
PreprocessingScaling obligatoire + encoding + imputation
Tuningrbf/sigmoid: C loguniform(0.1, 1000), gamma loguniform(0.001, 10). poly séparé: degree randint(2,6), coef0 uniform(0,10)
SGD equiv.SGDClassifier(loss='hinge')
Kernel TrickSimilarité en haute dimension sans créer les features

Effet des hyperparamètres :

ParamPetitGrand
CMarge souple, underfit, forte régulMarge dure, overfit, peu régul
gammaVision large, lisse, underfitMyope, irrégulier, overfit
degreeFrontière simpleFrontière complexe, overfit

from sklearn.svm import SVC
svc = SVC(kernel='rbf', C=1, gamma=0.1)
svc.fit(X_train_sc, y_train)
print(f"Accuracy: {svc.score(X_test_sc, y_test):.3f}")

🧭 Introduction

Tu as passé des semaines à apprendre des modèles de machine learning un par un. LinearRegression, Ridge, Lasso, KNN, SVM… À un moment, tu te retrouves à te demander : "Mais lequel je prends dans mon cas ?" C'est exactement l'objet de ce cours.

L'idée ici est simple : avoir une vue d'ensemble de tous les modèles vus jusqu'ici, côte à côte, pour que tu puisses les comparer et choisir le bon au bon moment.

Analogie fil rouge : imagine que tu es menuisier. Tu as une boîte à outils. Un marteau, une scie, un tournevis, une perceuse. Chaque outil a un usage précis. Tu n'utilises pas une perceuse pour enfoncer un clou. En machine learning, c'est pareil : chaque modèle est un outil, et ce cours est ton catalogue d'outils.

À retenir avant de commencer

  • Il n'existe pas de modèle universellement supérieur — ça dépend toujours du problème.
  • La première chose à faire est toujours de poser une baseline (un modèle stupide), pour savoir si ton vrai modèle apprend quelque chose.
  • Certains modèles exigent absolument que tu normalises tes données. D'autres s'en fichent. C'est un critère clé de choix.
  • Overfitting et underfitting se corrigent : la régularisation est ton levier principal pour les modèles linéaires.
  • Avant de chercher le meilleur modèle, vérifie que ta métrique d'évaluation est la bonne.

🔑 Comprendre les colonnes du tableau

Avant de plonger dans les modèles, expliquons le vocabulaire utilisé dans les tableaux. Ces colonnes reviennent pour chaque modèle — il faut les comprendre une bonne fois pour toutes.

"Quand" — dans quelle situation l'utiliser

C'est la question la plus importante. Le tableau te dit dans quel contexte ce modèle brille. Lis cette colonne en premier.

"Type" — Régression ou Classification

  • Régression : prédire une valeur continue. Exemple : prédire le prix d'une maison (150 000 €, 320 000 €...).
  • Classification : prédire une catégorie. Exemple : spam ou pas spam, chat ou chien.
💡

Si tu confonds les deux, tu utilises le mauvais modèle et les mauvaises métriques. C'est l'erreur de débutant numéro 1.

"Hyperparamètres" — les réglages du modèle

Ce sont les paramètres que TU fixes avant l'entraînement (contrairement aux paramètres que le modèle apprend tout seul). Exemples : le nombre de voisins dans KNN, la force de régularisation dans Ridge.

🧠

Pense à un four : la température est un hyperparamètre. La cuisson du gâteau est ce que le modèle apprend. Toi tu règles la température, le four fait le reste.

"Loss" (ou fonction de coût) — ce que le modèle minimise

La loss est la mesure de l'erreur que le modèle cherche à minimiser pendant l'entraînement. En régression, c'est souvent l'écart entre la prédiction et la vraie valeur, au carré (MSE). En classification, c'est souvent la log-loss.

Tu n'as pas besoin de la calculer toi-même — sklearn s'en charge. Mais la connaître t'aide à comprendre pourquoi le modèle se comporte ainsi.

"Scaling" — faut-il normaliser les données ?

Le scaling consiste à ramener toutes tes variables sur la même échelle (par exemple, entre 0 et 1, ou avec moyenne 0 et écart-type 1). C'est critique pour les modèles qui calculent des distances ou qui appliquent une pénalité sur les coefficients.

⚠️

Si une variable va de 0 à 1 et une autre de 0 à 1 000 000, la deuxième va "écraser" la première dans les calculs. Le modèle sera biaisé sans que tu comprennes pourquoi.

Le scaler standard en sklearn : StandardScaler. Il transforme chaque feature pour qu'elle ait une moyenne de 0 et un écart-type de 1.

Python
from sklearn.preprocessing import StandardScaler
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # fit ET transform sur le train
X_test_scaled = scaler.transform(X_test)        # transform seulement sur le test !
⚠️

JAMAIS de fit_transform sur le test set. Ça provoquerait du data leakage (le modèle "verrait" les données de test pendant l'entraînement). Toujours : fit sur train, transform sur train ET test.

"Diagnostic overfit / underfit"

  • Overfit : le modèle a trop bien appris les données d'entraînement (y compris le bruit). Il performe bien sur le train, mal sur le test.
  • Underfit : le modèle est trop simple pour capturer les patterns. Il performe mal partout.

"Métriques" — comment mesurer la performance

Ce sont les indicateurs que tu regardes après l'entraînement. Ils diffèrent selon le type de problème. On les détaille dans la section dédiée plus bas.

"SGD equiv." — la version rapide pour grands datasets

SGD = Stochastic Gradient Descent. Certains modèles ont une version "SGD" qui fait exactement la même chose mais beaucoup plus vite sur de très grands datasets (> 100 000 lignes). C'est une information pour plus tard — quand tu seras confronté à des volumes importants.

1. DummyRegressor / DummyClassifier — le point de départ obligatoire

À quoi ça sert ? Ce modèle ne "réfléchit" pas du tout. Il prédit toujours la même chose : la moyenne (en régression) ou la classe la plus fréquente (en classification). Son seul rôle est de servir de référence de base.

Analogie : imagine un médecin qui, pour diagnostiquer tout le monde, dit systématiquement "vous n'avez rien de grave" parce que c'est la réponse vraie 90% du temps. C'est stupide comme médecin — mais ça te donne un seuil minimum à dépasser.

Points forts :

  • Instantané à entraîner
  • Donne la baseline absolue

Points faibles :

  • Ne prédit rien d'intelligent
  • Inutile en production
✅

Règle d'or : ton premier modèle est TOUJOURS un Dummy. Si ton vrai modèle ne fait pas mieux, il y a un problème fondamental dans tes données ou ton pipeline.

Python
from sklearn.dummy import DummyRegressor, DummyClassifier

# Régression : prédit toujours la moyenne
baseline = DummyRegressor(strategy='mean')
baseline.fit(X_train, y_train)
print(f"Baseline R²: {baseline.score(X_test, y_test):.3f}")

# Classification : prédit toujours la classe majoritaire
baseline_clf = DummyClassifier(strategy='most_frequent')
baseline_clf.fit(X_train, y_train)
print(f"Baseline Accuracy: {baseline_clf.score(X_test, y_test):.3f}")

2. LinearRegression — la régression classique

À quoi ça sert ? Prédire une valeur continue en traçant la droite (ou l'hyperplan) qui passe au mieux à travers les données. C'est le modèle le plus simple qui existe pour la régression.

Analogie : trouver la droite de tendance sur un graphique. Si les prix d'appartements augmentent avec la surface, tu traces la droite qui résume cette tendance.

Points forts :

  • Très rapide
  • Résultats interprétables (chaque coefficient a un sens)
  • Bon point de départ

Points faibles :

  • Ne fonctionne que si la relation est linéaire
  • Sensible à la multicolinéarité (deux features qui se ressemblent trop)
  • Pas de régularisation → peut overfitter avec beaucoup de features

La loss : la régression linéaire minimise la somme des carrés des erreurs (MSE) :

$$\text{Loss} = \sum_{i=1}^{n}(y_i - \hat{y}_i)^2$$
Python
from sklearn.linear_model import LinearRegression
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

model = LinearRegression()
model.fit(X_train, y_train)
print(f"R²: {model.score(X_test, y_test):.3f}")
print(f"Coefs: {model.coef_}")
print(f"Intercept: {model.intercept_}")

# Dans un pipeline (recommandé)
pipe = make_pipeline(StandardScaler(), LinearRegression())
pipe.fit(X_train, y_train)
💡

Les coefficients sont directement lisibles : si le coef de "surface" est 1500, ça veut dire que chaque m² supplémentaire ajoute 1500 € au prix prédit (toutes choses égales par ailleurs).

3. Ridge (régularisation L2) — quand LinearRegression overfitte

À quoi ça sert ? Exactement comme LinearRegression, mais avec une pénalité qui empêche les coefficients de devenir trop grands. C'est le "LinearRegression raisonnable".

Analogie : tu fais de la régression linéaire, mais tu ajoutes une règle : "les coefficients ne peuvent pas être trop extrêmes". C'est comme un professeur qui corrige les copies mais pénalise les réponses excessives.

L'hyperparamètre clé : alpha

  • alpha petit (0.001) = peu de régularisation, proche de LinearRegression
  • alpha grand (100) = forte régularisation, coefficients très petits

La loss Ridge :

$$\text{Loss} = \sum(y - \hat{y})^2 + \alpha \sum \beta_j^2$$

Le deuxième terme pénalise les gros coefficients. Plus alpha est grand, plus la pénalité est forte.

Points forts :

  • Gère la multicolinéarité
  • Stabilise les coefficients
  • Scaling obligatoire → force de bonnes pratiques

Points faibles :

  • N'élimine pas les features inutiles (garde tous les coefs, juste petits)
  • Un hyperparamètre à tuner
Python
from sklearn.linear_model import Ridge
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

pipe = make_pipeline(StandardScaler(), Ridge(alpha=1))
pipe.fit(X_train, y_train)
print(f"R²: {pipe.score(X_test, y_test):.3f}")
⚠️

Scaling OBLIGATOIRE avec Ridge. La pénalité est appliquée sur les coefficients — si une feature est en millions et une autre en unités, leurs coefficients sont à des échelles totalement différentes, et la pénalité sera injuste.

4. Lasso (régularisation L1) — quand tu veux éliminer des features

À quoi ça sert ? Comme Ridge, mais la pénalité force certains coefficients à être exactement zéro. Résultat : Lasso fait de la sélection de features automatique.

Analogie : comme Ridge, sauf que le professeur ne penalise pas juste les réponses excessives — il efface carrément les réponses qu'il juge inutiles. Lasso barre les features non pertinentes.

L'hyperparamètre clé : alpha

  • alpha petit = peu de features éliminées
  • alpha grand = beaucoup de features éliminées (potentiellement trop)

La loss Lasso : au lieu de pénaliser le carré des coefs (comme Ridge), Lasso pénalise leur valeur absolue — ce qui force certains coefs à zéro.

Points forts :

  • Sélection automatique de features (pratique quand tu en as beaucoup)
  • Modèle plus simple et plus lisible
  • Utile pour identifier les features vraiment importantes

Points faibles :

  • Peut éliminer des features utiles si alpha est trop grand
  • Moins stable que Ridge si features corrélées entre elles
Python
from sklearn.linear_model import Lasso
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

pipe = make_pipeline(StandardScaler(), Lasso(alpha=1))
pipe.fit(X_train, y_train)

# Voir combien de features ont été éliminées
lasso_coefs = pipe.named_steps['lasso'].coef_
print(f"Features éliminées: {(lasso_coefs == 0).sum()} / {len(lasso_coefs)}")
🧠

Résumé Ridge vs Lasso : Ridge rend les coefficients petits mais pas nuls (garde toutes les features). Lasso met certains coefficients à zéro (élimine des features). Si tu as beaucoup de features inutiles → Lasso. Si toutes tes features sont potentiellement utiles → Ridge.

5. ElasticNet — le compromis Ridge + Lasso

À quoi ça sert ? ElasticNet combine les deux pénalités (L1 + L2). Il fait un peu de sélection de features ET stabilise les coefficients.

Analogie : tu hésites entre Ridge et Lasso ? ElasticNet est le modèle "les deux mon capitaine". Tu règles le curseur l1_ratio entre 0 (= Ridge pur) et 1 (= Lasso pur).

Les hyperparamètres clés :

  • alpha : force totale de régularisation
  • l1_ratio : dosage entre L1 et L2 (0 = tout Ridge, 1 = tout Lasso, 0.5 = moitié-moitié)

Quand l'utiliser : quand des features sont corrélées par groupes (Ridge seul garde tout, Lasso choisit arbitrairement une feature du groupe — ElasticNet garde le groupe entier tout en réduisant les coefs).

Python
from sklearn.linear_model import ElasticNet
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

pipe = make_pipeline(StandardScaler(), ElasticNet(alpha=1, l1_ratio=0.5))
pipe.fit(X_train, y_train)
print(f"R²: {pipe.score(X_test, y_test):.3f}")

6. LogisticRegression — le classifieur linéaire de référence

À quoi ça sert ? Prédire une classe (0 ou 1, spam ou non-spam, etc.). Malgré son nom, c'est bien un modèle de classification, pas de régression. Il prédit une probabilité, puis décide en fonction d'un seuil (par défaut 50%).

Analogie : un thermomètre avec une alarme. Il mesure une température (une probabilité entre 0 et 1), et si ça dépasse 50°, il dit "fièvre". En dessous, il dit "pas de fièvre".

Points forts :

  • Rapide, interprétable
  • Donne des probabilités (pas juste oui/non)
  • Baseline naturelle pour la classification

Points faibles :

  • Suppose une frontière de décision linéaire
  • Pas adapté si les classes ne sont pas séparables linéairement

La loss : Log-Loss (entropie croisée)

$$\text{Loss} = -\frac{1}{n}\sum\left[y\log\hat{y} + (1-y)\log(1-\hat{y})\right]$$

L'hyperparamètre C : c'est l'INVERSE de alpha des modèles précédents !

  • C grand = peu de régularisation (risque d'overfit)
  • C petit = forte régularisation
⚠️

Piège classique : dans Ridge/Lasso, alpha grand = forte régularisation. Dans LogisticRegression, C grand = FAIBLE régularisation. C = 1/alpha. Ne pas les confondre !

Python
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

# Sans régularisation (pour voir les vrais coefficients)
model = LogisticRegression(penalty=None, max_iter=5000)

# Avec régularisation L2 (défaut)
model = LogisticRegression(penalty='l2', C=1)

# Avec L1 (feature selection) — attention au solver !
model = LogisticRegression(penalty='l1', C=0.1, solver='liblinear')

pipe = make_pipeline(StandardScaler(), LogisticRegression(C=1))
pipe.fit(X_train, y_train)
print(f"Accuracy: {pipe.score(X_test, y_test):.3f}")
💡

Interprétation des coefficients en LogisticRegression : exp(coef) = odds ratio. Si exp(coef_age) = 1.5, chaque année d'âge supplémentaire multiplie les odds d'avoir la maladie par 1.5.

7. KNeighborsRegressor / KNeighborsClassifier — le modèle "par ressemblance"

À quoi ça sert ? KNN (K-Nearest Neighbors) ne construit aucun modèle. Pour prédire, il cherche les K points d'entraînement les plus proches de la nouvelle observation, et fait une moyenne (régression) ou un vote majoritaire (classification).

Analogie : tu veux savoir si un film va te plaire. Tu demandes à tes 5 amis les plus proches de toi en termes de goûts. S'ils ont tous aimé, tu vas probablement aimer. KNN fait exactement ça, avec des données.

L'hyperparamètre clé : n_neighbors (K)

  • K petit (1, 2, 3) = le modèle colle trop aux données → overfit
  • K grand (30, 40, 50) = le modèle est trop lisse → underfit
  • L'outil pour trouver le bon K : la méthode du coude (Elbow method)
Python
import matplotlib.pyplot as plt
from sklearn.neighbors import KNeighborsRegressor
from sklearn.model_selection import cross_validate

scores = []
for k in range(1, 51):
    model = KNeighborsRegressor(n_neighbors=k)
    cv = cross_validate(model, X_train_scaled, y_train, cv=5)
    scores.append(cv['test_score'].mean())

plt.plot(range(1, 51), scores)
plt.xlabel('K')
plt.ylabel('Score CV')
plt.title('Elbow method — trouver le bon K')
plt.show()

Points forts :

  • Pas d'hypothèse sur la forme des relations (non-linéaire OK)
  • Simple à comprendre

Points faibles :

  • Très lent sur grands datasets (calcule toutes les distances à chaque prédiction)
  • Scaling absolument obligatoire
⚠️

Sans scaling, une feature en millions (ex: salaire) va dominer les distances par rapport à une feature en unités (ex: nombre d'enfants). Le modèle ignorerait de fait toutes les autres features.

Python
from sklearn.neighbors import KNeighborsRegressor, KNeighborsClassifier
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

pipe = make_pipeline(StandardScaler(), KNeighborsRegressor(n_neighbors=5, weights='distance'))
pipe.fit(X_train, y_train)

8. SVC (Support Vector Classifier) — la frontière optimale

À quoi ça sert ? Trouver la frontière qui sépare les classes en maximisant la marge entre les deux groupes. Avec le "kernel trick", SVC peut tracer des frontières non-linéaires complexes.

Analogie : tu veux séparer des chats et des chiens sur un graphique. SVC trouve la ligne (ou courbe) qui les sépare avec le plus d'espace possible de chaque côté. Plus la marge est grande, plus le modèle est confiant.

Hyperparamètres clés :

ParamètrePetitGrand
C (coût erreurs)Marge souple, tolère des erreurs, underfitMarge dure, veut tout classer, overfit
gamma (portée du RBF)Influence large, décision lisseInfluence locale, décision tordue, overfit
kernellinear / poly / rbf / sigmoid—

Points forts :

  • Efficace avec des frontières non-linéaires complexes (kernel RBF)
  • Robuste si bien tuné

Points faibles :

  • Lent sur grands datasets
  • Scaling absolument obligatoire
  • Deux hyperparamètres à tuner ensemble (C et gamma)
Python
from sklearn.svm import SVC
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

pipe = make_pipeline(StandardScaler(), SVC(kernel='rbf', C=1, gamma=0.1))
pipe.fit(X_train, y_train)
print(f"Accuracy: {pipe.score(X_test, y_test):.3f}")
🧠

Le "kernel trick" est une astuce mathématique : au lieu de créer des milliers de nouvelles features (polynomiales, etc.), le kernel calcule directement la similarité entre points comme si ces features existaient. Résultat : un modèle non-linéaire à faible coût.

9. SVR (Support Vector Regression) — SVC version régression

À quoi ça sert ? Même principe que SVC, mais pour prédire une valeur continue. SVR essaie de faire passer le maximum de points dans un "tube" autour de la courbe prédite. Les points dans le tube ne sont pas pénalisés.

Hyperparamètre supplémentaire : epsilon — la demi-largeur du tube. Plus epsilon est grand, plus le modèle tolère des erreurs sans les pénaliser.

Python
from sklearn.svm import SVR
from sklearn.pipeline import make_pipeline
from sklearn.preprocessing import StandardScaler

pipe = make_pipeline(StandardScaler(), SVR(kernel='rbf', C=1, gamma=0.1, epsilon=0.1))
pipe.fit(X_train, y_train)
print(f"R²: {pipe.score(X_test, y_test):.3f}")

🗺️ Arbre de décision — quel modèle choisir ?

Voici la démarche à suivre pas à pas pour choisir ton modèle.

Étape 0 : Toujours poser une baseline

Code
Quel que soit le problème :
  → Régression    : DummyRegressor(strategy='mean')
  → Classification: DummyClassifier(strategy='most_frequent')

Étape 1 : Régression ou Classification ?

Code
Le target (y) est...
├── Une valeur continue (prix, température, score)
│   → RÉGRESSION
│
└── Une catégorie (oui/non, chat/chien, type A/B/C)
    → CLASSIFICATION

Étape 2 (Régression) — la relation est-elle linéaire ?

Code
Régression :
├── Relation linéaire (nuage de points en droite)
│   └── LinearRegression
│       ├── Overfit ? → Ridge
│       ├── Trop de features ? → Lasso
│       └── Pas sûr ? → ElasticNet
│
└── Relation non-linéaire (courbes, interactions)
    ├── Peu de features, dataset moyen → KNeighborsRegressor
    └── Relation complexe → SVR(kernel='rbf')

Étape 2 (Classification) — les classes sont-elles linéairement séparables ?

Code
Classification :
├── Frontière linéaire possible
│   └── LogisticRegression
│       ├── Sélection de features ? → penalty='l1', solver='liblinear'
│       └── Cas général → penalty='l2' (défaut)
│
└── Frontière non-linéaire
    ├── Peu de features → KNeighborsClassifier
    └── Frontière complexe → SVC(kernel='rbf')

Étape 3 : Dataset très grand (n > 100 000 ou p > 1 000) ?

Code
→ Utiliser les équivalents SGD (beaucoup plus rapides) :
  SGDRegressor(loss='squared_error')    # ≈ LinearRegression / Ridge
  SGDClassifier(loss='log_loss')        # ≈ LogisticRegression
  SGDClassifier(loss='hinge')           # ≈ SVC
✅

Résumé de la démarche : Dummy → Modèle linéaire simple → Si ça ne suffit pas → Modèle non-linéaire (KNN ou SVM). Toujours dans cet ordre.

🎯 Questions de vérification

Question 1 : Tu veux prédire le chiffre d'affaires mensuel d'une boutique à partir de sa surface, son emplacement et le nombre d'employés. Tu as 500 exemples. Par quoi commences-tu, et quel modèle utilises-tu ensuite ?

Réponse

C'est un problème de régression (target continu). Tu commences par DummyRegressor(strategy='mean') pour avoir une baseline. Ensuite, LinearRegression (ou Ridge si tu suspectes de l'overfit). Si la relation n'est pas linéaire, KNeighborsRegressor.

Question 2 : Tu entraînes un LogisticRegression pour détecter des fraudes bancaires. 99% des transactions sont légitimes, 1% sont des fraudes. Ton modèle affiche 99% d'accuracy. Est-ce que c'est un bon modèle ?

Réponse

Non. Un DummyClassifier qui dit "jamais de fraude" aurait aussi 99% d'accuracy. Ce qui compte ici, c'est le Recall sur la classe "fraude" : est-ce que le modèle détecte vraiment les fraudeurs ? Vérifie le Recall et le F1 score sur la classe minoritaire.

Question 3 : Tu utilises Ridge sur un dataset. Tu observes que le score sur le train est bien meilleur que sur le test. Que fais-tu ?

Réponse

C'est de l'overfitting. Avec Ridge, la solution est d'augmenter alpha (plus de régularisation). Tu peux utiliser GridSearchCV ou RandomizedSearchCV pour trouver la bonne valeur.

Question 4 : Tu as 50 features dans ton dataset, mais tu penses que seulement une dizaine sont vraiment utiles. Quel modèle de régression choisir ?

Réponse

Lasso (régularisation L1). Il va automatiquement mettre à zéro les coefficients des features inutiles, laissant seulement les features importantes dans le modèle.

📌 TL;DR

  • Commence toujours par un Dummy pour avoir une baseline — si ton modèle ne fait pas mieux, il y a un problème.
  • Régression vs Classification : décide en premier. Régression = target continu, Classification = target catégoriel.
  • Modèles linéaires d'abord : LinearRegression (régression) ou LogisticRegression (classification) → simple, rapide, interprétable.
  • Ajoute de la régularisation si ça overfitte : Ridge (garde tout) ou Lasso (élimine des features) — les deux exigent le scaling.
  • Modèles non-linéaires si la frontière est complexe : KNN (simple, lent) ou SVM (puissant, plus lent) — les deux exigent le scaling.
  • Scaling = obligatoire dès qu'il y a une pénalité ou un calcul de distance. Toujours dans un Pipeline pour éviter le data leakage.
  • C dans LogisticRegression/SVM est l'INVERSE de alpha : C grand = peu de régularisation (contrairement à Ridge/Lasso).
  • Ne te fie pas à l'accuracy seule en classification, surtout si les classes sont déséquilibrées. Regarde Recall, Precision, F1.
  • Pour les grands datasets (> 100 000 lignes) : utilise les équivalents SGD (SGDRegressor, SGDClassifier).
  • Ordre de test recommandé : Dummy → Linéaire simple → Linéaire régularisé → Non-linéaire (KNN/SVM).

A. Concepts du cours

Comment lire ce méga-tableau

Aucun modèle ML n'est universellement le meilleur — c'est le théorème No-Free-Lunch (Wolpert, 1996). Choisir un modèle, c'est faire un trade-off entre :

  • Performance : précision, recall, F1, RMSE sur vos données
  • Vitesse : entraînement (batch) et inférence (real-time)
  • Mémoire : taille du modèle sur disque et empreinte RAM
  • Interprétabilité : peut-on expliquer une décision à un humain ?
  • Complexité de tuning : combien d'hyperparamètres à régler ?
  • Robustesse : résistance aux outliers, au bruit, au drift
  • Capacité : quelle complexité de relations peut-il apprendre ?

Ce méga-tableau résume les principales familles. Ce n'est pas exhaustif mais ce sont les modèles que vous croiserez 95% du temps en pratique.

💡

Analogie : choisir un modèle ML, c'est comme choisir un mode de transport. Marcher (régression linéaire) : simple, gratuit, vous comprenez tout. Vélo (Random Forest) : plus rapide, fiable, pas trop cher. Voiture (XGBoost) : rapide, polyvalent, demande un peu d'apprentissage. Avion (deep learning) : excellent pour les longues distances, mais cher et inutile pour aller acheter du pain.

Modèles supervisés — fiches détaillées

Pour chaque modèle : forces, faiblesses, hyperparamètres clés, quand l'éviter, et un exemple chiffré d'usage typique.

Régression linéaire

  • Forces : simple, interprétable (chaque coefficient a un sens), rapide (entraînement O(n·d²), inférence en microsecondes), baseline universelle pour la régression.
  • Faiblesses : suppose une relation linéaire, très sensible aux outliers, ne capture aucune interaction entre features.
  • Hyperparams clés : aucun (vanilla) ou alpha si régularisée (Ridge/Lasso).
  • Quand l'éviter : relations très non linéaires, interactions fortes, features colinéaires sans régularisation.
  • Exemple chiffré : prédiction du prix d'un bien immobilier (Boston Housing), RMSE ≈ 5.2 k$, entraînement < 0.1 s sur 10k lignes. Une baseline à battre.

Régression logistique

  • Forces : comme la régression linéaire mais pour la classification, produit des probabilités calibrées, très interprétable via les coefficients (odds-ratios).
  • Faiblesses : frontière de décision linéaire, pas d'interactions automatiques, sensible aux features non scalées.
  • Hyperparams clés : C (inverse de la régularisation), penalty (L1/L2/ElasticNet), class_weight.
  • Quand l'éviter : dataset avec beaucoup d'interactions non linéaires, sauf si vous faites du feature engineering polynomial manuel.
  • Exemple chiffré : détection de spam sur Enron, ROC-AUC ≈ 0.93 avec TF-IDF, entraînement en 2 s sur 30k emails. Souvent meilleure qu'un BERT non fine-tuné pour des problèmes simples.

Ridge / Lasso / ElasticNet

  • Forces : régularisation pour gérer la multicolinéarité, Lasso fait de la sélection automatique de features (coefficients mis à zéro).
  • Faiblesses : toujours linéaires, choix de alpha critique (CV obligatoire).
  • Hyperparams clés : alpha, l1_ratio (ElasticNet).
  • Quand l'éviter : quand vous n'avez que 5 features propres et non corrélées — c'est inutile.
  • Exemple chiffré : prédire le revenu avec 150 features économiques ; Lasso réduit à 23 features non nulles, R² = 0.68 au lieu de 0.71 pour LinReg, mais bien plus interprétable.

K-Nearest Neighbors (KNN)

  • Forces : aucun entraînement (lazy learner), capture des relations locales, facile à expliquer ("on regarde les voisins").
  • Faiblesses : lent à l'inférence (O(n·d) par prédiction), catastrophique en haute dimension (malédiction de la dimensionnalité), très sensible au scaling.
  • Hyperparams clés : n_neighbors, metric (euclidean/manhattan/cosine), weights.
  • Quand l'éviter : datasets > 100k points, inférence real-time, dimensions > 50.
  • Exemple chiffré : classification de chiffres manuscrits (MNIST réduit à 1k images), accuracy ≈ 96% avec k=5. Mais sur 70k images, l'inférence prend plusieurs secondes — éliminatoire en prod.

Naive Bayes

  • Forces : extrêmement rapide, marche avec peu de données, probabiliste, très utilisé en NLP.
  • Faiblesses : hypothèse d'indépendance des features — presque toujours fausse en pratique, mais ça marche quand même.
  • Hyperparams clés : alpha (lissage Laplace).
  • Quand l'éviter : petites dimensions avec fortes corrélations, ou quand l'interprétabilité des interactions compte.
  • Exemple chiffré : filtre anti-spam, accuracy ≈ 97% sur un corpus de 10k emails en < 1 s. Le classique historique.

SVM linéaire

  • Forces : robuste à l'overfit même en haute dimension, marche bien avec peu de samples et beaucoup de features.
  • Faiblesses : lent à entraîner sur > 10k samples, pas de probabilités natives (doit passer par Platt scaling).
  • Hyperparams clés : C, class_weight.
  • Quand l'éviter : datasets > 100k lignes, besoin de probabilités rapides et calibrées.
  • Exemple chiffré : classification de textes avec TF-IDF (20 newsgroups), accuracy ≈ 88%, entraînement en 1 min sur 18k documents.

SVM avec kernel RBF

  • Forces : capture des frontières de décision non linéaires, théoriquement élégant.
  • Faiblesses : très lent au-delà de ~10k samples (O(n²) voire O(n³)), tuning de C et gamma délicat, peu interprétable.
  • Hyperparams clés : C, gamma, kernel.
  • Quand l'éviter : gros datasets, besoin d'interprétabilité, production real-time.
  • Exemple chiffré : petit dataset médical de 500 patients, F1 ≈ 0.82 avec kernel RBF, là où RF atteint 0.78. Le SVM brille sur les petits datasets non linéaires.

Decision Tree

  • Forces : interprétable (règles lisibles), pas besoin de scaling, gère catégories et numériques.
  • Faiblesses : overfit massif si non régulé, instable (un changement mineur dans les données change tout l'arbre).
  • Hyperparams clés : max_depth, min_samples_leaf, ccp_alpha.
  • Quand l'éviter : dès que la performance prime — utilisez RF ou XGBoost à la place.
  • Exemple chiffré : arbre de profondeur 4 pour expliquer une décision de prêt bancaire. Accuracy 75% (contre 85% pour RF), mais les 4 règles sont présentables au comité de conformité.

Random Forest

  • Forces : très robuste out-of-the-box, peu de tuning, interprétable via feature_importances_, parallélisable, marche presque partout sur tabulaire.
  • Faiblesses : modèles volumineux (plusieurs centaines de Mo possible), inférence lente avec beaucoup d'arbres, moins performant qu'XGBoost en pointe.
  • Hyperparams clés : n_estimators, max_depth, min_samples_leaf, max_features.
  • Quand l'éviter : contraintes de latence < 5 ms, contraintes mémoire fortes (mobile), datasets > 10M lignes.
  • Exemple chiffré : churn télécom 50k lignes, F1 = 0.71 avec 300 arbres et hyperparams par défaut, entraînement 30 s. Excellente baseline forte.

Gradient Boosting (sklearn)

  • Forces : plus performant que RF, API sklearn standard.
  • Faiblesses : plus lent à entraîner que XGBoost/LightGBM, pas de support GPU.
  • Hyperparams clés : n_estimators, learning_rate, max_depth.
  • Quand l'éviter : si vous pouvez installer XGBoost ou LightGBM (toujours supérieurs).
  • Exemple chiffré : mêmes données que RF, F1 ≈ 0.73, entraînement 2 min. Largement remplacé par XGBoost aujourd'hui.

XGBoost

  • Forces : state-of-the-art tabulaire, support GPU, gère nativement les valeurs manquantes, support des matrices sparses, mature et battle-tested.
  • Faiblesses : une dizaine d'hyperparamètres sensibles, peut overfitter avec des paramètres trop agressifs.
  • Hyperparams clés : n_estimators, learning_rate, max_depth, min_child_weight, subsample, colsample_bytree, reg_alpha, reg_lambda.
  • Quand l'éviter : très petits datasets (< 500 lignes), besoin de très faible latence si vous ne quantisez pas.
  • Exemple chiffré : même dataset churn, F1 = 0.74 par défaut, 0.77 après tuning Optuna (100 trials, 2 h). Standard en compétition Kaggle.

LightGBM

  • Forces : 2 à 10× plus rapide qu'XGBoost sur gros datasets, empreinte mémoire réduite, support natif des catégorielles.
  • Faiblesses : plus sensible au sur-apprentissage sur petits datasets, attention à min_child_samples (trop bas = overfit).
  • Hyperparams clés : num_leaves, learning_rate, min_child_samples, feature_fraction.
  • Quand l'éviter : très petits datasets (< 10k lignes) où la variance domine.
  • Exemple chiffré : dataset de crédit à 1M lignes, AUC = 0.86 en 3 min avec LightGBM contre 15 min pour XGBoost. Standard 2025 sur tabulaire.

CatBoost

  • Forces : excellent avec features catégorielles à haute cardinalité (pas besoin de OneHot ou Target Encoding manuel), peu de tuning, ordered boosting évite le target leakage.
  • Faiblesses : plus lent en inférence qu'XGBoost/LightGBM, communauté plus petite.
  • Hyperparams clés : iterations, learning_rate, depth, cat_features.
  • Quand l'éviter : datasets purement numériques (XGBoost ou LightGBM seront plus rapides).
  • Exemple chiffré : dataset e-commerce avec city (5000 valeurs), product_category (800), user_segment (200). CatBoost atteint F1 = 0.79 en natif contre 0.75 pour XGBoost+OneHot.

MLP (neural network tabulaire)

  • Forces : capture des relations complexes, flexibilité d'architecture.
  • Faiblesses : quasi-toujours battu par XGBoost/LightGBM sur tabulaire, demande beaucoup de tuning (architecture, LR, dropout, batch size).
  • Hyperparams clés : nb de couches, nb de neurones, LR, dropout, batch size, optimizer.
  • Quand l'éviter : tout cas tabulaire < 1M lignes. Vraiment.
  • Exemple chiffré : sur 500k lignes, MLP bien tuné F1 = 0.75 vs LightGBM F1 = 0.76 en 10× moins de temps. Le MLP ne vaut le coût que pour du multimodal.

Modèles non supervisés — fiches détaillées

K-Means

  • Forces : simple, rapide (O(n·k·i)), scalable, standard universel.
  • Faiblesses : K à choisir a priori, suppose des clusters sphériques et de tailles égales, sensible aux outliers et à l'initialisation.
  • Hyperparams clés : n_clusters, init (k-means++ par défaut), n_init.
  • Quand l'éviter : clusters de formes arbitraires, densités très variables, présence d'anomalies.
  • Exemple chiffré : segmentation client RFM sur 100k clients, silhouette = 0.42 avec k=5. Utilisable pour un premier ciblage marketing.

DBSCAN

  • Forces : trouve le nombre de clusters automatiquement, gère les formes arbitraires, identifie les outliers (label -1).
  • Faiblesses : très sensible à eps, échoue en haute dimension (> 20), performance dégradée sur densités variables.
  • Hyperparams clés : eps, min_samples.
  • Quand l'éviter : datasets haute dimension, clusters de densités très différentes.
  • Exemple chiffré : détection d'anomalies sur 50k transactions bancaires 2D (montant × heure), trouve 47 anomalies en 10 s.

HDBSCAN

  • Forces : améliore DBSCAN sur densités variables, plus robuste au choix d'hyperparam.
  • Faiblesses : plus lent que DBSCAN, un peu moins connu.
  • Hyperparams clés : min_cluster_size, min_samples.
  • Quand l'éviter : très gros datasets (> 1M points).
  • Exemple chiffré : clustering de documents sur 20k articles avec UMAP en 2D d'abord, puis HDBSCAN identifie 12 thématiques naturelles.

Hierarchical clustering

  • Forces : dendrogramme exploitable, pas besoin de K a priori.
  • Faiblesses : complexité O(n³) ou O(n² log n), inutilisable au-delà de 10k points.
  • Hyperparams clés : linkage (ward/complete/average), n_clusters ou distance_threshold.
  • Quand l'éviter : tout dataset > 10k points.
  • Exemple chiffré : analyse exploratoire de 500 produits, dendrogramme permet de visualiser 3 familles principales.

GMM (Gaussian Mixture Models)

  • Forces : soft assignment (probabilités d'appartenance), généralise K-means.
  • Faiblesses : hypothèse gaussienne, sensible à l'initialisation.
  • Hyperparams clés : n_components, covariance_type.
  • Quand l'éviter : clusters clairement non gaussiens.
  • Exemple chiffré : modèle de revenu avec 3 composantes gaussiennes (low/mid/high earners), BIC optimal à 3.

PCA

  • Forces : standard absolu de réduction de dimension, déterministe, rapide.
  • Faiblesses : linéaire uniquement, composantes peu interprétables.
  • Hyperparams clés : n_components (ou seuil de variance cumulée).
  • Quand l'éviter : relations fortement non linéaires — préférer UMAP ou t-SNE pour la viz.
  • Exemple chiffré : 150 features corrélées réduites à 10 composantes expliquant 92% de la variance. Puis modèle linéaire dessus.

t-SNE

  • Forces : excellent pour visualisation 2D, préserve les structures locales.
  • Faiblesses : lent (O(n²)), non déterministe, ne marche QUE pour visualisation (ne projette pas de nouveaux points), perplexity à tuner.
  • Hyperparams clés : perplexity, learning_rate, n_iter.
  • Quand l'éviter : tout usage autre que la visualisation.
  • Exemple chiffré : 10k embeddings BERT projetés en 2D en 2 min pour visualiser les clusters thématiques.

UMAP

  • Forces : plus rapide que t-SNE, préserve structure locale ET globale, projette de nouveaux points.
  • Faiblesses : un peu de tuning, stochastique.
  • Hyperparams clés : n_neighbors, min_dist.
  • Quand l'éviter : quand PCA suffit et que vous voulez du déterministe.
  • Exemple chiffré : 100k points MNIST en 2D en 30 s, clusters de chiffres très bien séparés.

Isolation Forest

  • Forces : détection d'anomalies multivariée efficace, fonctionne en haute dimension, O(n log n).
  • Faiblesses : pas de probabilités calibrées, seuil de contamination à choisir.
  • Hyperparams clés : n_estimators, contamination, max_features.
  • Quand l'éviter : quand vous avez des labels (auquel cas un classifieur supervisé sera meilleur).
  • Exemple chiffré : 1M de logs serveur, détecte 0.1% d'accès anormaux en 2 min.

B. Compléments avancés

Trade-offs en pratique

Performance vs interprétabilité : c'est presque toujours un trade-off. Régression linéaire = très interprétable, performance moyenne. Deep learning = excellent, opaque.

Performance vs vitesse : XGBoost avec 10 000 arbres et profondeur 20 sera meilleur que 100 arbres et profondeur 5, mais 100× plus lent en inférence.

Performance vs simplicité : stacking de 5 modèles peut gagner 1% de F1 mais multiplie la complexité opérationnelle par 5. Rarement rentable.

Choix selon le contexte :

ContexteCritère prioritaireModèle type
Recherche académiquePerformance pureSOTA, deep learning
Compétition KagglePerformance pureStacking / blending
Production critique (médical, finance)Interprétabilité + robustesseLogReg, XGB + SHAP

Évolution des standards 2020-2025

Tabulaire : XGBoost a longtemps dominé. LightGBM est maintenant souvent préféré pour sa vitesse. CatBoost pour les catégorielles. Les neural networks restent rarement compétitifs sur du tabulaire pur.

Vision : CNN ont dominé jusqu'en 2020. Vision Transformers les ont rattrapés et dépassés sur les très grands datasets (> 100M images). En pratique, les CNN modernes pré-entraînés (EfficientNet) restent le bon choix pour 90% des cas.

NLP : BERT/GPT ont remplacé les RNN en 2018-2019. Les LLM de plusieurs milliards de paramètres dominent depuis 2022 pour la génération. Pour la classification, DistilBERT + fine-tuning reste un bon compromis coût/performance.

Génération d'images : GAN ont dominé jusqu'en 2021. Diffusion (Stable Diffusion, DALL-E, Midjourney) les ont remplacés depuis 2022. Plus stables à entraîner, qualité supérieure.

RL : limité à des cas où il y a un simulateur. PPO et SAC sont les standards pour les nouveaux projets. RLHF est utilisé pour aligner les LLMs.

👉

Recommandation 2025 : pour démarrer un projet, identifiez le standard du domaine au moment T, ne réinventez pas. Sur tabulaire = LightGBM, vision = EfficientNet pré-entraîné, NLP = transformer pré-entraîné, génération = LLM API ou diffusion. Le SOTA évolue, mais ces choix sont solides depuis 2-3 ans et le resteront probablement pour 1-2 ans encore.

C. Questions de compréhension

Q1 : Quelle est la différence pratique entre Random Forest et XGBoost, et lequel choisir par défaut sur du tabulaire ?

💡 Voir l'indice

Bagging vs boosting. Indépendants vs séquentiels.

✅ Voir la réponse

Random Forest est un bagging d'arbres profonds : on entraîne N arbres indépendants sur des sous-échantillons, on moyenne les prédictions. Robuste, peu de tuning, parallélisable. XGBoost est un gradient boosting d'arbres peu profonds : on entraîne séquentiellement, chaque arbre corrige les erreurs du précédent. Plus performant mais plus de tuning. Différences pratiques : (1) Performance : XGBoost gagne souvent 2-5% sur les benchmarks tabulaires. (2) Tuning : RF marche presque out-of-the-box, XGBoost a 10+ hyperparams sensibles. (3) Vitesse d'entraînement : RF est parallélisable (chaque arbre indépendant), XGBoost est séquentiel mais utilise GPU. (4) Robustesse au bruit : RF est plus robuste, XGBoost peut overfitter avec des hyperparams trop agressifs. Recommandation pratique : commencez par Random Forest comme baseline. Si vous voulez gagner 2-5%, passez à XGBoost ou LightGBM (plus rapide qu'XGBoost). Pour 95% des projets industriels, RF est suffisant et plus simple à maintenir. Pour Kaggle ou la recherche de la perfection, LightGBM est le standard. Question d'entretien data science classique : "RF ou XGBoost ?" — la réponse mature parle du contexte plutôt que d'un absolu.

Q2 : Vous avez 50 features dont 10 catégorielles à haute cardinalité (> 1000 valeurs uniques). Quel modèle privilégier ?

💡 Voir l'indice

Encoding natif des catégorielles.

✅ Voir la réponse

CatBoost est probablement le meilleur choix. Trois raisons. (1) Encoding natif : CatBoost gère les features catégorielles directement, sans avoir besoin de OneHot (qui exploserait la dimension à 10×1000 = 10 000 features) ni de Target Encoding manuel (qui nécessite une CV interne pour éviter le leakage). Vous lui passez juste cat_features=[...]. (2) Ordered boosting : CatBoost utilise une technique qui évite le "target leakage" inhérent au Target Encoding classique. C'est mathématiquement plus correct. (3) Bonnes performances par défaut : moins de tuning nécessaire qu'XGBoost ou LightGBM. Alternatives : (a) LightGBM avec categorical_feature : a aussi un support natif, presque aussi efficace. (b) XGBoost + Target Encoding manuel via category_encoders.TargetEncoder : ça marche mais c'est plus de code et plus risqué. (c) OneHot + n'importe quel modèle : à éviter, explose la dimension. (d) Embeddings appris dans un neural network : très puissant pour de très gros datasets, mais overkill pour la plupart des cas. Règle pratique : pour des datasets tabulaires avec beaucoup de catégorielles, CatBoost est l'outil le plus rapide à mettre en place et donne d'excellents résultats.

Q3 : Vous voulez prédire quelque chose en temps réel avec une latence < 10 ms. Quels modèles éliminez-vous immédiatement ?

💡 Voir l'indice

Inference time, pas training time.

✅ Voir la réponse

À éliminer : (1) K-NN : O(n) par requête (parcourir tous les points d'entraînement). Sur un dataset de 1M de points, plusieurs secondes. (2) Random Forest avec 1000 arbres profonds : il faut évaluer 1000 arbres, peut prendre 10-100 ms. (3) Gros transformers (BERT large, GPT) : ~50-500 ms par inférence sur CPU. (4) Stacking de plusieurs modèles : la latence est la somme des composants. À privilégier : (a) Régression linéaire/logistique : O(d) où d est le nombre de features. Quelques microsecondes. (b) Decision Tree unique : un chemin d'arbre = très rapide. (c) LightGBM avec arbres peu profonds : optimisé pour l'inférence, < 1 ms. (d) MLP petit bien optimisé. (e) DistilBERT quantizé : pour du NLP rapide, ~10-30 ms. Optimisations possibles : (i) Quantization (FP32 → INT8) : 2-4× plus rapide. (ii) ONNX runtime ou TensorRT : 2-5× plus rapide qu'un modèle PyTorch nu. (iii) Caching des prédictions répétitives. (iv) Batching si possible. Pour < 10 ms en production, le standard est : LightGBM ou Logistic Regression, exporté en ONNX, déployé en C++ ou Rust derrière un load balancer. Question d'entretien production ML : la réponse révèle la pratique réelle.
