Gestion du flux et des erreurs en Python

⚠️ Récapitulatif — Erreurs clés à éviter

📋 Cheatsheet — Gestion des erreurs & flux

Python
## ── TRY / EXCEPT ─────────────────────────────────────
try:
    résultat = 10 / x
except ZeroDivisionError:
    print("Division par zéro")
except ValueError as e:
    print(f"Valeur invalide : {e}")
except Exception as e:          # filet de sécurité (toujours en dernier)
    print(f"Erreur inattendue : {e}")
else:
    print("Aucune erreur !")    # exécuté si PAS d'erreur
finally:
    print("Toujours exécuté")   # nettoyage, fermeture de fichier...

## ── ERREURS COURANTES ────────────────────────────────
## ValueError        → int("abc")
## TypeError         → "a" + 1
## KeyError          → dict["clé_absente"]
## IndexError        → liste[99]
## ZeroDivisionError → 10 / 0
## FileNotFoundError → open("inexistant.txt")
## AttributeError    → 42.upper()

## ── RAISE ────────────────────────────────────────────
raise ValueError("Message explicite obligatoire")

## ── CONTRÔLE DE FLUX ─────────────────────────────────
## break     → quitte la boucle immédiatement
for i in range(10):
    if i == 5:
        break           # sort à i=5

## continue  → saute au tour suivant
for i in range(6):
    if i == 3:
        continue        # ignore i=3, continue avec i=4

## pass      → placeholder (ne fait rien)
def a_implementer():
    pass

## return    → quitte la fonction et retourne une valeur
def f(x):
    if x < 0:
        return None     # sortie anticipée
    return x * 2

## assert    → vérifie une hypothèse (debug)
assert age >= 0, "L'âge ne peut pas être négatif"

Python
def diviser(a, b):
    if b == 0:
        raise ValueError("Le diviseur ne peut pas être zéro !")

diviser(10, 0)  # ValueError: Le diviseur ne peut pas être zéro !

Utile pour valider des données et signaler un problème explicitement.

pass — ne rien faire

Python
for i in range(5):
    if i == 3:
        pass    # on ignore, on continue normalement
    print(i)    # affiche 0, 1, 2, 3, 4

pass est un placeholder — il dit à Python "ici il n'y a rien à faire" sans provoquer d'erreur. Utile aussi pour les fonctions/classes vides :

Python
def fonction_a_implementer():

def premier_positif(liste):

Exemple combiné

Python
def lire_entier(texte):
    while True:
        try:
            valeur = int(input(texte))
            assert valeur > 0, "Doit être positif"
            return valeur                       # succès → on sort
        except ValueError:
            print("Ce n'est pas un entier !")
        except AssertionError as e:
            print(e)
        finally:
            print("--- tentative terminée ---")

Cette fonction redemande un nombre jusqu'à obtenir un entier positif valide — un pattern très utilisé en pratique.

🧭 Introduction

Quand on écrit du code, les choses ne se passent pas toujours comme prévu. L'utilisateur tape une lettre là où on attendait un chiffre, un fichier est introuvable, on divise par zéro... Sans outil pour gérer ça, le programme plante brutalement. Python offre une boîte à outils complète pour anticiper ces situations et y réagir proprement.

À retenir avant de commencer :

  • Une erreur (aussi appelée "exception") en Python n'est pas une catastrophe — c'est un signal qu'on peut intercepter.
  • Le bloc try / except est le mécanisme central pour gérer les erreurs.
  • raise permet de déclencher une erreur soi-même, quand on veut signaler un problème explicitement.
  • break, continue, return et pass permettent de contrôler le flux d'exécution d'une boucle ou d'une fonction.
  • assert est un outil de vérification utile pendant le développement.

Analogie fil rouge : Imagine que ton programme est un cuisinier qui suit une recette. Si un ingrédient manque, il ne jette pas tout — il note le problème, utilise un substitut ou prévient poliment. C'est exactement ce que fait la gestion des erreurs en Python.

try / except — Intercepter une erreur

Le problème sans try

Sans protection, une erreur arrête tout le programme d'un coup :

Python
resultat = 10 / 0   # ZeroDivisionError: division by zero
print("Cette ligne ne s'affiche jamais.")

La solution avec try / except

Le bloc try dit à Python : "essaie ce code, et si ça plante, voici quoi faire à la place."

Python
try:
    resultat = 10 / 0      # on essaie ce code risqué
except ZeroDivisionError:  # si CETTE erreur précise survient...
    print("Division par zéro !")  # ...on affiche ce message au lieu de planter
💡

Le programme ne plante plus — il intercepte l'erreur et continue proprement. C'est comme attraper une balle avant qu'elle casse une vitre.

Structure complète : try / except / else / finally

Python propose quatre blocs complémentaires pour couvrir tous les cas de figure :

Python
try:
    nombre = int(input("Entrez un nombre : "))  # code risqué : l'utilisateur peut taper n'importe quoi
except ValueError:
    print("Ce n'est pas un nombre !")           # exécuté si int() échoue (ex: l'user tape "abc")
else:
    print(f"Vous avez entré : {nombre}")        # exécuté UNIQUEMENT si aucune erreur n'a eu lieu dans try
finally:
    print("Fin du programme.")                  # exécuté TOUJOURS, qu'il y ait eu une erreur ou non

Récapitulatif des quatre blocs :

BlocQuand s'exécute-t-il ?
tryToujours — c'est là qu'on met le code à risque
exceptSeulement si une erreur survient dans try
elseSeulement si aucune erreur n'a eu lieu
finallyToujours, erreur ou pas — idéal pour le nettoyage
⚠️

finally s'exécute même si tu utilises return à l'intérieur du try. C'est un comportement surprenant pour les débutants — garde-le en tête !

Python
def f():
    try:
        return "résultat"          # on retourne ici...
    finally:
        print("finally s'exécute quand même !")  # ...mais finally passe QUAND MÊME avant

Gérer plusieurs types d'erreurs

On peut empiler plusieurs blocs except pour réagir différemment selon l'erreur :

Python
try:
    x = int(input("Entrez un nombre : "))   # risque 1 : l'user tape une lettre → ValueError
    print(10 / x)                           # risque 2 : l'user tape 0 → ZeroDivisionError
except ValueError:                          # cas : mauvais type de valeur
    print("Pas un nombre")
except ZeroDivisionError:                   # cas : division par zéro
    print("Division par zéro")
except Exception as e:                      # filet de sécurité : attrape tout le reste
    print(f"Erreur inattendue : {e}")       # "e" contient le message d'erreur détaillé
⚠️

L'ordre des except est crucial : toujours mettre les erreurs les plus spécifiques en premier, et Exception (le filet) en dernier. Si tu mets Exception en premier, il attrape tout et les blocs suivants ne sont jamais atteints.

Python
# Mauvais ordre — NE PAS FAIRE :
except Exception:       # attrape tout, masque les vraies causes
    print("Erreur")
except ValueError:      # cette ligne n'est jamais atteinte !
    print("Pas un entier")

# Bon ordre :
except ValueError:      # d'abord le plus spécifique
    print("Pas un entier")
except Exception as e:  # ensuite le filet de sécurité
    print(f"Erreur inattendue : {e}")
❌

Ne jamais faire except Exception: pass — cela "avale" l'erreur en silence. Le programme continue comme si de rien n'était, mais le problème est toujours là. Impossible à déboguer ensuite.

Python
try:
    risque()
except Exception:
    pass  # l'erreur est ignorée silencieusement — très difficile à déboguer

raise — Déclencher une erreur manuellement

Parfois, tu veux toi-même signaler qu'un problème a eu lieu — par exemple pour valider des données en entrée :

Python
def diviser(a, b):
    if b == 0:                                          # on vérifie la condition problématique
        raise ValueError("Le diviseur ne peut pas être zéro !")  # on déclenche une erreur avec un message clair
    return a / b                                        # si tout va bien, on calcule normalement

diviser(10, 0)  # → ValueError: Le diviseur ne peut pas être zéro !

raise interrompt la fonction et envoie l'erreur vers le code qui a appelé la fonction. Si personne ne l'attrape avec un except, le programme plante avec ton message.

✅

Toujours inclure un message explicite dans ton raise — c'est ce message que lira la personne qui débogue le code (peut-être toi dans 3 semaines) !

Python
raise ValueError()                                   # message vide — inutile pour déboguer
raise ValueError("Le diviseur ne peut pas être zéro !")  # message clair — excellent

Contrôle de flux : pass, break, continue, return, assert

Ces mots-clés ne gèrent pas les erreurs au sens strict, mais ils contrôlent comment Python avance dans le code.

pass — Ne rien faire

pass est un placeholder : il dit à Python "ici, intentionnellement, il n'y a rien à faire". Utile pour les blocs vides temporaires :

Python
for i in range(5):
    if i == 3:
        pass         # on ne fait rien de spécial pour i=3, on continue normalement
    print(i)         # affiche 0, 1, 2, 3, 4 — le 3 est bien affiché aussi
Python
def fonction_a_implementer():
    pass   # la fonction existe déjà dans le code mais n'est pas encore codée — pas d'erreur
💡

pass est différent de continue : avec pass, le reste du bloc s'exécute quand même (ici, print(i) affiche aussi le 3). Avec continue, on saute directement au tour suivant.

break — Quitter une boucle immédiatement

break sort complètement de la boucle, peu importe combien de tours il reste :

Python
for i in range(10):    # boucle prévue de 0 à 9
    if i == 5:
        break          # on sort immédiatement quand i vaut 5
    print(i)           # affiche 0, 1, 2, 3, 4 — on ne verra jamais 5, 6, 7, 8, 9

Analogie : break, c'est comme appuyer sur le bouton "STOP" d'un tapis roulant — tout s'arrête là.

continue — Sauter au tour suivant

continue ignore le reste du code du tour actuel et passe directement à l'itération suivante :

Python
for i in range(6):     # boucle de 0 à 5
    if i == 3:
        continue       # quand i vaut 3, on saute le print et on passe à i=4
    print(i)           # affiche 0, 1, 2, 4, 5 — le 3 est absent

Analogie : continue, c'est comme passer son tour dans un jeu — on revient au prochain tour normalement.

⚠️

Confusion fréquente — retiens bien la différence :

- break → quitte la boucle complètement

- continue → saute l'itération courante et continue la boucle

Python
# Exemple pour visualiser la différence :
for i in range(5):
    if i == 3:
        break      # résultat : 0, 1, 2 — la boucle s'arrête à 3
        continue   # résultat si on remplaçait break : 0, 1, 2, 4 — la boucle continue après 3

return — Quitter une fonction (et retourner une valeur)

return sort immédiatement d'une fonction et renvoie optionnellement une valeur :

Python
def premier_positif(liste):
    for n in liste:              # on parcourt la liste
        if n > 0:
            return n             # dès qu'on trouve un positif, on sort de la fonction avec cette valeur
    return None                  # si on arrive ici, c'est qu'il n'y avait aucun positif
🧠

Sans return explicite, Python retourne automatiquement None. C'est parfois source de bugs — si tu récupères la valeur d'une fonction qui ne contient pas de return, tu obtiendras None sans erreur visible.

assert — Vérifier une hypothèse (outil de debug)

assert permet de vérifier qu'une condition est vraie pendant le développement. Si elle est fausse, une AssertionError est levée :

Python
age = -5
assert age >= 0, "L'âge ne peut pas être négatif !"
# → AssertionError: L'âge ne peut pas être négatif !
Python
# Syntaxe générale :
assert condition, "Message affiché si la condition est fausse"
💡

assert est surtout utile pour déboguer : tu t'assures qu'une variable a bien la valeur attendue à un moment donné. En production, les assertions peuvent être désactivées — ne les utilise pas pour valider des données utilisateur (préfère raise pour ça).

Récapitulatif des mots-clés

Python
try / except  → intercepter et gérer les erreurs
else          → si aucune erreur dans le try
finally       → s'exécute toujours (nettoyage, fermeture de fichier...)
raise         → déclencher une erreur manuellement
pass          → ne rien faire (placeholder)
break         → quitter une boucle immédiatement
continue      → sauter au tour suivant dans une boucle
return        → quitter une fonction et retourner une valeur

Exemple combiné — Tout ensemble

Voici une fonction qui utilise presque tous ces outils à la fois. Elle demande un entier positif à l'utilisateur et recommence jusqu'à en obtenir un valide :

Python
def lire_entier(texte):
    while True:                                    # boucle infinie — on ne sort que via return
        try:
            valeur = int(input(texte))             # risque : l'user peut taper une lettre → ValueError
            assert valeur > 0, "Doit être positif" # on vérifie que c'est bien positif
            return valeur                          # succès : on sort de la fonction avec la valeur
        except ValueError:
            print("Ce n'est pas un entier !")      # l'user a tapé une lettre → on demande à nouveau
        except AssertionError as e:
            print(e)                               # l'user a tapé 0 ou négatif → on affiche le message
        finally:
            print("--- tentative terminée ---")    # affiché après chaque essai, réussi ou non

Comment ça marche :

  1. On entre dans la boucle et on essaie de lire un entier.
  2. Si l'utilisateur tape "abc" → ValueError → on affiche le message et finally s'exécute → nouvelle tentative.
  3. Si l'utilisateur tape -3 → AssertionError → on affiche "Doit être positif" et finally s'exécute → nouvelle tentative.
  4. Si l'utilisateur tape 7 → pas d'erreur → return 7 → finally s'exécute quand même → la fonction se termine.
✅

Ce pattern "boucle + try/except + return" est très utilisé en pratique pour s'assurer qu'une entrée utilisateur est valide avant de continuer.

📌 TL;DR

  • try / except intercepte les erreurs sans planter le programme — le code risqué va dans try, la réaction dans except.
  • else s'exécute si aucune erreur n'a eu lieu ; finally s'exécute toujours (même après un return).
  • Toujours ordonner les except du plus spécifique au plus général — Exception en dernier comme filet de sécurité.
  • Ne jamais faire except Exception: pass — cela cache les erreurs silencieusement.
  • raise ValueError("message clair") déclenche une erreur manuellement avec un message utile.
  • break quitte la boucle entièrement ; continue saute juste l'itération courante.
  • pass est un placeholder qui ne fait rien ; return quitte la fonction et renvoie une valeur.
  • assert condition, "message" vérifie une hypothèse en développement — ne pas utiliser en production pour valider des données.
  • Les erreurs les plus fréquentes : ValueError, TypeError, KeyError, IndexError, ZeroDivisionError.

A. Concepts du cours

Exceptions vs valeurs de retour

Deux philosophies coexistent dans les langages de programmation :

  • Go renvoie un tuple (result, err) à chaque appel et force l'utilisateur à tester explicitement.
  • Python (et Java, Ruby) utilise les exceptions : la fonction renvoie le résultat normal, et en cas d'erreur, elle lève une exception qui remonte la pile d'appels jusqu'à un try / except.

Cette approche s'appelle EAFP : Easier to Ask Forgiveness than Permission. Plutôt que de tester chaque condition possible avant d'agir (LBYL — Look Before You Leap), on tente l'opération et on rattrape l'échec.

Python
# LBYL — verbeux ET race-conditioné
if os.path.exists("data.csv"):
    with open("data.csv") as f:
        data = f.read()
# ↑ Entre path.exists et open(), un autre processus peut supprimer le fichier !

# EAFP — pythonique ET atomique
try:
    with open("data.csv") as f:
        data = f.read()
except FileNotFoundError:
    data = ""

Pourquoi LBYL est piégeux : entre os.path.exists et open, un autre processus pourrait supprimer le fichier. C'est une race condition. EAFP fait l'opération atomiquement — soit elle réussit, soit elle lève FileNotFoundError, mais il n'y a pas de fenêtre temporelle entre les deux.

💡

Analogie : LBYL c'est vérifier qu'il pleut avant de prendre son parapluie. EAFP c'est prendre son parapluie et accepter qu'il sera inutile s'il fait beau. Dans un monde où le temps change pendant que vous regardez par la fenêtre, EAFP est plus fiable.

try / except / else / finally

La syntaxe complète :

Python
try:
    result = risky_operation()
except ValueError as e:
    # Capturé : ValueError uniquement
    log.error(f"Bad value: {e}")
    result = None
except (KeyError, IndexError) as e:
    # Plusieurs types dans un même except
    log.error(f"Missing data: {e}")
    result = None
except Exception as e:
    # Catch-all — à éviter sauf cas exceptionnel
    log.exception("Unexpected error")
    raise                               # Re-raise pour ne pas masquer
else:
    # Exécuté SEULEMENT si aucune exception n'a été levée
    log.info(f"Success: {result}")
finally:
    # TOUJOURS exécuté, exception ou pas
    cleanup_resources()

Rôle de chaque clause :

  • try : le code potentiellement faillible.
  • except Type as e : capture du type spécifique. On peut avoir plusieurs except.
  • else : exécuté si aucune exception n'est levée dans le try. Utile pour séparer le code à risque du code post-traitement.
  • finally : toujours exécuté, même si on fait return dans le try. Parfait pour libérer des ressources (fichier, connexion, lock).

Quatre règles :

  1. Capturer le type le plus spécifique possible. except FileNotFoundError est mieux que except OSError qui est mieux que except Exception. Un except: nu est presque toujours un bug : il capture aussi KeyboardInterrupt et SystemExit, ce qui rend votre programme inarrêtable au Ctrl+C.
  1. Le else sert à séparer le code à risque du code qui dépend du succès. Évite de capturer accidentellement des exceptions qui viendraient du code post-traitement.
  1. Le finally est l'endroit pour libérer une ressource (fichier, connexion DB, lock). Il s'exécute même si vous faites return dans le try ou le except.
  1. raise sans argument relance l'exception courante en préservant le traceback complet. C'est crucial pour ne pas masquer un bug.
⚠️

Anti-pattern toxique : except: pass. Cela cache toutes les erreurs, y compris les bugs de programmation, et rend l'application impossible à débugger. Si vous trouvez ce pattern dans une code review, c'est un red flag majeur.

Hiérarchie des exceptions

Les exceptions Python sont organisées en arbre. Comprendre cette hiérarchie permet de capturer plusieurs types proches en une seule clause.

Code
BaseException
 ├─ SystemExit
 ├─ KeyboardInterrupt
 └─ Exception             ← presque toutes les exceptions héritent d'ici
     ├─ ArithmeticError
     │   ├─ ZeroDivisionError
     │   └─ FloatingPointError
     ├─ LookupError
     │   ├─ KeyError
     │   └─ IndexError
     ├─ ValueError
     ├─ TypeError
     ├─ OSError (alias IOError)
     │   ├─ FileNotFoundError
     │   ├─ PermissionError
     │   └─ ConnectionError
     │       ├─ ConnectionRefusedError
     │       └─ ConnectionResetError
     └─ RuntimeError
         └─ RecursionError

except Exception capture tout ce qui hérite de Exception mais pas KeyboardInterrupt ni SystemExit — c'est pourquoi c'est moins dangereux que except:.

Exceptions custom

Pour signaler une condition métier spécifique, créez vos propres classes d'exception :

Python
class DataValidationError(Exception):
    """Base class pour toutes les erreurs de validation de données."""
    pass

class InsufficientDataError(DataValidationError):
    """Raised when fewer than 100 samples are available."""
    pass

def train(X, y):
    if len(X) < 100:
        raise InsufficientDataError(
            f"Need at least 100 samples, got {len(X)}"
        )
    if X.isnull().any().any():
        raise DataValidationError("X contains NaN values")
    # ...

L'appelant peut alors capturer finement ou largement :

Python
try:
    train(X, y)
except InsufficientDataError:
    # Cas spécifique : on peut collecter plus de données
    collect_more_data()
except DataValidationError:
    # Cas général : on nettoie et on réessaye
    clean_and_retry()
👉

Bonne pratique : héritez vos exceptions custom d'une classe parent commune (ici DataValidationError) pour permettre une capture groupée. C'est le pattern qu'utilisent toutes les bibliothèques sérieuses : requests a requests.exceptions.RequestException parent de ConnectionError, Timeout, HTTPError, etc.

B. Compléments avancés

Context managers et with

L'instruction with garantit qu'une ressource est libérée même en cas d'exception. Le pattern le plus connu :

Python
with open("data.csv") as f:
    data = f.read()
# f est automatiquement fermé ici, MÊME si read() lève une exception

C'est strictement équivalent à :

Python
f = open("data.csv")
try:
    data = f.read()
finally:
    f.close()

Le with est plus court et plus safe : impossible d'oublier le close().

Créer son propre context manager, deux options :

Option 1 — classe avec __enter__ / __exit__ :

Python
class DatabaseConnection:
    def __enter__(self):
        self.conn = psycopg2.connect(...)
        return self.conn                       # Valeur renvoyée par `as`

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.conn.close()
        return False                           # False = ne pas supprimer l'exception

with DatabaseConnection() as conn:
    conn.execute("SELECT 1")
# conn est fermée ici, automatiquement

Option 2 — décorateur @contextmanager (plus simple) :

Python
from contextlib import contextmanager

@contextmanager
def timer(name):
    import time
    t0 = time.perf_counter()
    yield                                      # ↑ code avant = setup, ↓ code après = cleanup
    print(f"{name}: {time.perf_counter() - t0:.2f}s")

with timer("training"):
    model.fit(X, y)
# Affiche : "training: 12.34s"

Logging vs print

print() est interdit en production. Pourquoi ?

  • Pas de niveau de sévérité
  • Pas de timestamp
  • Va dans stdout (pollue la sortie du programme)
  • Non configurable à distance

Le module logging corrige tout cela :

Python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    handlers=[
        logging.FileHandler("app.log"),        # Fichier
        logging.StreamHandler(),               # Console
    ],
)
log = logging.getLogger(__name__)              # Logger local au module

log.debug("Variable detail: %s", x)            # Caché par défaut (level=INFO)
log.info("Training started")
log.warning("Deprecated parameter used")
log.error("Failed to load model")
log.critical("Database is down")

try:
    risky()
except Exception:
    log.exception("Operation failed")          # Log + traceback complet

Cinq niveaux de sévérité — DEBUG < INFO < WARNING < ERROR < CRITICAL — vous permettent de filtrer selon l'environnement (DEBUG en dev, WARNING en prod).

Règle importante : n'utilisez jamais de f-strings dans les arguments du logger :

Python
log.info(f"x={x}")          # ❌ Formate IMMÉDIATEMENT, même si le niveau est désactivé
log.info("x=%s", x)         # ✓ Formate PARESSEUSEMENT, seulement si le niveau est actif

Stratégies de retry et backoff

Quand une opération est intrinsèquement faillible (appel API, requête réseau, lecture de fichier sur NFS), il faut réessayer. Mais réessayer immédiatement spamme le service et n'aide pas. La stratégie canonique est l'exponential backoff with jitter :

Python
import time, random

def retry_with_backoff(func, max_attempts=5, base_delay=1.0, max_delay=60.0):
    for attempt in range(max_attempts):
        try:
            return func()
        except Exception as e:
            if attempt == max_attempts - 1:
                raise                                    # Dernier essai : propager
            delay = min(base_delay * (2 ** attempt), max_delay)
            jitter = random.uniform(0, delay * 0.1)     # ±10% de hasard
            time.sleep(delay + jitter)
            log.warning(f"Attempt {attempt+1} failed: {e}. Retry in {delay:.1f}s")

Décomposition :

  • base_delay * (2 ** attempt) : 1s, 2s, 4s, 8s, 16s → croissance exponentielle.
  • min(..., max_delay) : plafond pour ne pas attendre des heures.
  • jitter : ±10% de hasard pour éviter les "thundering herds" (1000 clients qui retentent au même instant).

Bibliothèque battle-tested : tenacity :

Python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(5), wait=wait_exponential(multiplier=1, max=60))
def fetch_from_api(url):
    return requests.get(url, timeout=10).json()
⚠️

Piège : ne pas confondre erreurs transitoires (timeout réseau, 503 Service Unavailable) qui méritent un retry, et erreurs permanentes (404 Not Found, 401 Unauthorized, validation) qui n'en méritent pas. Réessayer une 401 n'a aucune chance de réussir et fait gaspiller du temps. Distinguez les deux explicitement dans votre code.

C. Questions de compréhension

Q1 : Pourquoi try: ... except Exception: pass est-il considéré comme un anti-pattern majeur, et dans quels cas exceptionnels est-il acceptable ?

💡 Voir l'indice

Observabilité et debug.

✅ Voir la réponse

C'est un anti-pattern parce qu'il avale silencieusement toutes les erreurs, y compris celles qui révèlent un bug de programmation (NameError, TypeError, etc.). Le programme continue à tourner dans un état incohérent et vous ne saurez jamais qu'il y a un problème — jusqu'à ce que les données en sortie soient fausses, au moment où il est trop tard pour corriger.

Cas exceptionnel acceptable : quand vous nettoyez des ressources dans un finally et que vous devez vous assurer que l'erreur de cleanup ne masque pas l'erreur originale. Mais même là, on log l'erreur avalée :

Python
try:
    main_operation()
finally:
    try:
        cleanup()
    except Exception as e:
        log.exception("Cleanup failed, ignoring")    # ✓ On log au moins

La règle : si vous ne pouvez pas expliquer pourquoi une exception devrait être ignorée, vous ne devez pas l'ignorer. Et si vous l'ignorez vraiment, loggez-la.

Q2 : Quelle est la différence sémantique entre raise NewError("msg"), raise NewError("msg") from original_error, et raise NewError("msg") from None ?

💡 Voir l'indice

Exception chaining.

✅ Voir la réponse

Python 3 chaîne automatiquement les exceptions : si une exception est levée pendant la gestion d'une autre, le traceback affiche les deux séparées par "During handling of the above exception, another exception occurred".

  • raise NewError("msg") : chaînage implicite. Le traceback affiche les deux exceptions dans l'ordre.
  • raise NewError("msg") from original_error : chaînage explicite avec le message "The above exception was the direct cause of the following exception". Utile pour signaler que vous transformez intentionnellement une exception bas niveau en exception métier :
Python
  try:
      data = json.loads(raw)
  except json.JSONDecodeError as e:
      raise DataValidationError("Invalid JSON in config") from e
  • raise NewError("msg") from None : supprime complètement le contexte, masquant l'exception d'origine. Utile pour des messages d'erreur publics qui ne doivent pas exposer la stack interne (par exemple dans une API publique, pour ne pas révéler la structure de votre code).

En entretien backend, cette nuance révèle un dev qui maîtrise vraiment l'observabilité.

Q3 : Vous appelez une API qui renvoie parfois une 503 (service unavailable). Vous mettez un retry avec exponential backoff. Pourquoi ajoute-t-on systématiquement du jitter, et que se passe-t-il sans lui ?

💡 Voir l'indice

1000 clients qui retentent au même instant.

✅ Voir la réponse

Sans jitter, tous les clients qui ont reçu une 503 au même instant attendront exactement le même délai (1 s, 2 s, 4 s, 8 s…) et tenteront leur retry exactement en même temps — ce qu'on appelle un thundering herd. Le service à peine remis va recevoir un déluge de requêtes simultanées et planter à nouveau.

Le jitter (un petit délai aléatoire ajouté) désynchronise les retries et lisse la charge. C'est exactement ce que font DynamoDB et la plupart des AWS SDK en interne.

Règle empirique : ajoutez un jitter de ±10-25% du délai calculé.

Python
# Sans jitter : tous les clients retentent à t=1s, t=2s, t=4s...
delay = base_delay * (2 ** attempt)

# Avec jitter : chaque client retente à t=1s±0.1s, t=2s±0.2s...
delay = base_delay * (2 ** attempt)
jitter = random.uniform(-delay * 0.25, delay * 0.25)
time.sleep(delay + jitter)

Sans cette mécanique, votre retry rend la situation pire au lieu de la résoudre. C'est pourquoi tenacity et les SDKs cloud ajoutent le jitter par défaut.
