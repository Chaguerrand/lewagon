Git push & commit

Sauvegarder et envoyer son travail sur GitHub via les commandes Git essentielles.

🎯 Notions cles a retenir
Working tree
Vos fichiers locaux non encore mis en scène.
Stage / index
Zone tampon où vous préparez ce qui ira dans le prochain commit (`git add`).
Commit
Snapshot horodaté et signé de votre code, identifié par un hash SHA-1.
Remote
Copie distante du dépôt (souvent `origin` sur GitHub).

🧭 Introduction

Contexte

Quand tu codes, tu crées et modifies des fichiers en permanence. Mais que se passe-t-il si tu casses quelque chose ? Ou si tu veux partager ton travail avec quelqu'un ? C'est là qu'intervient Git, un outil qui te permet de sauvegarder l'historique de ton code et de l'envoyer sur internet (via GitHub).

3 à 5 choses à retenir absolument

  • Git sauvegarde ton code en 3 étapes : add → commit → push
  • git add prépare les fichiers que tu veux sauvegarder
  • git commit crée un "snapshot" (une photo instantanée) de ton travail
  • git push envoie ce snapshot sur GitHub (en ligne)
  • git status est ton meilleur ami : il te dit toujours où tu en es

Analogie

Imagine que tu prépares un colis à envoyer par la Poste :

  1. Tu choisis les affaires à mettre dans la boîte → c'est git add
  2. Tu fermes et étiquettes la boîte avec une description → c'est git commit
  3. Tu deposes la boîte à la Poste pour qu'elle parte → c'est git push

Si tu sautes une étape, ton colis n'arrive jamais !

1) Le workflow standard — les 3 étapes incontournables

Voici la séquence que tu utiliseras tous les jours :

Bash
# Étape 1 — on prépare le fichier modifié
git add string_methods.py

# Étape 2 — on crée le snapshot avec un message descriptif
git commit -m "Progress on string_methods: 5 tests passing"

# Étape 3 — on envoie tout sur GitHub
git push origin master
💡

Ces 3 commandes forment un seul enchaînement logique. Pense-y comme un rituel : tu ne pousses jamais sans avoir d'abord ajouté et commité.

2) Les commandes expliquées une par une

2.1 git add — choisir ce qu'on sauvegarde

Avant de créer un snapshot, tu dois dire à Git quels fichiers tu veux inclure. On appelle ça "stager" un fichier.

Bash
# Ajouter un seul fichier précis (recommandé quand tu débutes)
git add mon_fichier.py

# Ajouter TOUS les fichiers modifiés d'un coup
git add .
⚠️

git add . ajoute absolutement tout ce qui a changé dans le dossier. C'est pratique, mais dangereux si tu as modifié des fichiers par erreur. Prends l'habitude de faire git status avant pour voir exactement ce qui sera ajouté.

2.2 git commit — prendre la photo

Une fois les fichiers stagés, tu crées le snapshot. Le flag -m te permet d'écrire un message directement dans la commande.

Bash
# La structure de base
git commit -m "Description claire de ce qui a été fait"

Bien écrire ses messages de commit

Un bon message de commit est court, précis, et en anglais (convention universelle). Il décrit ce qui a changé, pas comment tu l'as fait.

Bash
# Bons messages ✅
git commit -m "Add Morse encoder function"
git commit -m "Fix IndexError in caesar cipher"
git commit -m "Remove duplicate lines in data cleaning script"

# Mauvais messages ❌
git commit -m "fix"
git commit -m "wip"
git commit -m "update"
🧠

Imagine que tu regardes l'historique de tes commits dans 6 mois. Un message comme "fix" ne te dira rien du tout. Un message comme "Fix IndexError in caesar cipher" te rappellera immédiatement ce que tu as corrigé et pourquoi.

2.3 git push — envoyer sur GitHub

push envoie ton ou tes commits locaux vers le dépôt distant (GitHub). origin désigne l'adresse distante, master (ou main) est le nom de ta branche principale.

Bash
# Si ta branche principale s'appelle "master"
git push origin master

# Si ta branche principale s'appelle "main" (GitHub utilise "main" par défaut depuis 2020)
git push origin main
💡

Tu peux vérifier le nom de ta branche avec git branch — la branche active est précédée d'une étoile *.

3) Les commandes utiles en plus

Ces commandes ne font pas partie du workflow de sauvegarde, mais elles t'aident à comprendre ce qui se passe et à corriger des erreurs.

Inspecter l'état de ton dépôt

Bash
git status          # affiche les fichiers modifiés, stagés, ou non trackés
git log --oneline   # affiche l'historique des commits sur une seule ligne chacun
git diff            # montre les modifications non encore stagées (ligne par ligne)
git diff --staged   # montre les modifications déjà stagées (prêtes à être commitées)

Exemple de git status : avant de faire quoi que ce soit, tape toujours git status. Tu verras quelque chose comme :

Bash
On branch master
Changes not staged for commit:
  modified:   string_methods.py   # <-- ce fichier a été modifié mais pas encore stagé

Untracked files:
  new_script.py                   # <-- ce fichier est nouveau, Git ne le suit pas encore

Annuler des modifications

Bash
# Annuler les modifications d'un fichier (revenir à la dernière version commitée)
git restore fichier.py

# Déstaguer un fichier (le retirer du "add" sans perdre les modifications)
git restore --staged fichier.py

# Corriger le message du tout dernier commit (uniquement si pas encore pushé)
git commit --amend -m "Nouveau message corrigé"
⚠️

git restore fichier.py efface définitivement tes modifications locales sur ce fichier. Il n'y a pas de "annuler" après ça — utilise-le avec précaution.

Les branches (aperçu rapide)

Une branche, c'est comme une copie parallèle de ton code où tu peux expérimenter sans toucher à la version principale.

Bash
git branch                        # lister toutes les branches (l'active est marquée *)
git checkout -b nouvelle-branche  # créer une nouvelle branche et basculer dessus
git checkout master               # revenir sur la branche principale
git merge ma-branche              # fusionner "ma-branche" dans la branche actuelle
💡

Au début, tu n'auras probablement qu'une seule branche (master ou main). Les branches deviennent utiles quand tu travailles en équipe ou sur des fonctionnalités expérimentales.

4) Les erreurs classiques à éviter

Erreur 1 — Faire git push sans avoir ajouté ni commité

Bash
# ❌ Ça ne pousse rien si tu n'as pas fait add + commit avant
git push origin master

# ✅ La bonne séquence
git add .
git commit -m "Mon message"
git push origin master

Erreur 2 — Écrire un message de commit trop vague

Bash
git commit -m "fix"        # ❌ incompréhensible dans 6 mois
git commit -m "Fix index out of range in caesar cipher"  # ✅ précis et utile

Erreur 3 — Commiter sans vérifier ce qu'on commit

Bash
git commit -m "Add feature"  # ❌ tu ne sais pas exactement ce que tu inclus
git status                   # ✅ commence toujours par vérifier l'état
git add fichier_specifique.py
git commit -m "Add feature"  # ✅ maintenant tu sais exactement ce qui est commité

Erreur 4 — Utiliser git add . sans vérifier d'abord

Bash
git add .   # ⚠️ ajoute TOUT — y compris des fichiers que tu ne voulais pas inclure
git status  # ✅ vérifie d'abord quels fichiers sont modifiés avant d'ajouter en masse

🎯 Questions de vérification

Question 1 — Quelle est la bonne séquence pour sauvegarder et envoyer son travail sur GitHub ? (Mets les commandes dans l'ordre)

git push, git commit, git add

Question 2 — Tu as modifié analyse.py mais quand tu fais git push, rien ne se passe. Pourquoi ? Que dois-tu faire ?

Question 3 — Parmi ces messages de commit, lequel est le meilleur et pourquoi ?

a) "update" — b) "Fix TypeError in data cleaning function" — c) "blabla" — d) "wip2"

Question 4 — Tu viens de stager accidentellement un fichier avec git add . que tu ne voulais pas inclure. Quelle commande utilises-tu pour le "déstaguer" sans perdre tes modifications ?

📌 TL;DR

  • Git sauvegarde ton code en 3 étapes : git add (préparer) → git commit (snapshot) → git push (envoyer sur GitHub).
  • git status est la commande à taper en premier, toujours, avant de faire quoi que ce soit.
  • Un bon message de commit est court et précis : il décrit ce qui a changé ("Add Morse encoder function"), pas l'état d'esprit ("fix", "wip").
  • git add . ajoute tout d'un coup — pratique mais risqué : vérifie avec git status d'abord.
  • git log --oneline te montre l'historique de tous tes commits en un coup d'oeil.
  • git restore fichier.py annule les modifications locales de façon irréversible.
  • git restore --staged fichier.py retire un fichier du stage sans effacer tes changements.
  • origin = l'adresse distante (GitHub) ; master ou main = ta branche principale.
  • Ne fais jamais git push sans avoir d'abord fait add + commit — sinon il n'y a rien à envoyer.
  • En résumé : status → add → commit → push. Dans cet ordre, à chaque fois.

A. Concepts du cours

Le modèle mental de Git

Git n'est pas un système de versionnement comme SVN ou Dropbox. C'est un graphe acyclique orienté (DAG) de snapshots immuables. Chaque commit est un snapshot complet de votre projet à un instant donné, identifié par un hash SHA-1 (en pratique, ses 7 premiers caractères suffisent : a3f2c91).

Trois zones à distinguer absolument :

ZoneAussi appeléeCommande qui y déplace
Working treedossier de travail(vos modifications brutes)
Staging areaindexgit add
Repositoryhistory, .git/git commit

Diagramme ASCII :

Code
Working tree         Staging area         Repository
(.py modifié)        (index)              (commits)
     |                    |                    |
     |  git add           |  git commit        |
     |------------------->|------------------->|
     |                    |                    |
     |  git checkout      |  git reset         |
     |<-------------------|<-------------------|
     |                    |                    |
💡

Analogie : pensez à la prise de photos de famille. Le working tree est la scène (tout le monde bouge). Le stage est quand vous demandez "tout le monde se met en place pour la photo". Le commit est le clic sur le déclencheur — la photo est sauvegardée pour toujours, immuable. Le push est l'envoi du tirage à la famille distante.

Le triptyque add / commit / push

Bash
git status                       # Affiche : quelles modifications, quoi est staged
git add file.py                  # Met UNE modification en stage
git add .                        # Stage TOUTES les modifications du dossier courant
git add -p                       # Stage interactivement, hunk par hunk (très puissant)
git commit -m "Ajoute pipeline preprocessing"
git push origin main             # Envoie les commits locaux vers le remote

Le message de commit est crucial. Convention conventional commits largement adoptée :

Code
feat: ajoute le scaler StandardScaler dans le pipeline
fix: corrige le bug de division par zéro dans normalize()
refactor: extrait load_data dans un module dédié
test: ajoute les tests unitaires pour TrainSplit
docs: met à jour le README avec les instructions d'install
chore: bump scikit-learn de 1.3 à 1.4
perf: vectorise compute_features, 5x plus rapide

Préfixes : feat, fix, refactor, test, docs, chore, perf. Ce format permet à des outils comme semantic-release de générer automatiquement le changelog et le numéro de version (semver).

👉

Bonne pratique : un commit = une seule intention. Si votre message contient un "et" ("ajoute X et corrige Y"), c'est probablement deux commits différents. La règle d'or : un commit doit pouvoir être annulé proprement avec git revert sans casser le reste.

Les branches comme pointeurs

Une branche n'est pas un dossier, ce n'est même pas une copie de votre code. C'est un pointeur mobile vers un commit. Quand vous faites un commit sur une branche, le pointeur avance d'un cran.

C'est pourquoi créer une branche est une opération O(1) en Git — contrairement à SVN où ça duplique tout.

Bash
git branch                       # Liste les branches locales
git branch feature-pca           # Crée une branche (sans y basculer)
git checkout feature-pca         # Bascule dessus
git checkout -b feature-pca      # Combine les deux (shortcut)
git switch feature-pca           # Syntaxe moderne (Git 2.23+)
git switch -c feature-pca        # Idem -c = create

Workflow standard pour ajouter une feature :

Bash
git checkout main && git pull            # Partir d'un main à jour
git switch -c feat/pca-pipeline          # Nouvelle branche
# ... travail, add, commit ...
git push -u origin feat/pca-pipeline     # Push avec tracking de la branche distante
# Ouvrir une Pull Request sur GitHub

Diagramme avant/après création de branche :

Code
Avant :               Après création :        Après commit sur feat :

  A---B---C main        A---B---C main          A---B---C main
                                 \                        \
                                  feat                     D feat

Pull, merge, conflits

git pull est en réalité git fetch && git merge : récupère les nouveaux commits du remote, puis les fusionne dans votre branche locale.

Quand vos changements et ceux du remote modifient le même fichier au même endroit, Git ne sait pas lequel choisir et marque le fichier en conflit. Le fichier ressemble à :

Python
def compute_loss(y_true, y_pred):
<<<<<<< HEAD
    return np.mean((y_true - y_pred) ** 2)
=======
    return np.sqrt(np.mean((y_true - y_pred) ** 2))
>>>>>>> origin/main

Les marqueurs <<<<<<<, =======, >>>>>>> délimitent les deux versions concurrentes. Vous devez :

  1. Ouvrir le fichier et choisir ce qui vous va (ou un mélange)
  2. Supprimer les marqueurs
  3. git add fichier pour marquer le conflit résolu
  4. git commit (le message de merge est généré automatiquement)
⚠️

Piège : ne jamais committer un fichier qui contient encore les marqueurs <<<<<<<. Vos tests vont peut-être passer (Python tolère parfois les < en parsing selon le contexte) mais en production le code sera cassé. Ajoutez un pre-commit hook qui rejette ces marqueurs : grep -r '<<<<<<<' . && exit 1.

B. Compléments avancés

git stash et changements en attente

Vous êtes au milieu d'une feature, mais on vous demande un hotfix urgent sur main. Vos modifications ne sont pas prêtes à être committées. Solution :

Bash
git stash                        # Met TOUT ce qui n'est pas committé de côté
git checkout main
# ... fix urgent, commit, push ...
git checkout feat/pca
git stash pop                    # Restaure les modifs mises de côté et supprime le stash

Variantes utiles :

Bash
git stash list                   # Voir tous les stashes (vous pouvez en avoir plusieurs)
git stash show stash@{0}         # Voir le contenu d'un stash
git stash apply stash@{1}        # Restaure SANS supprimer (pop = apply + drop)
git stash drop stash@{0}         # Supprime un stash
git stash -u                     # Inclut aussi les fichiers non trackés (untracked)
git stash -m "WIP on PCA"        # Avec un message (comme un mini-commit)

Réécrire l'historique : reset, rebase, amend

Trois opérations dangereuses mais essentielles à maîtriser pour ne pas paniquer.

git commit --amend modifie le dernier commit (message, contenu, ou les deux). Utile quand vous avez oublié un fichier ou fait une typo :

Bash
git add fichier_oublie.py
git commit --amend --no-edit     # Ajoute le fichier au dernier commit, garde le message
git commit --amend -m "Nouveau message"

git reset déplace le pointeur HEAD en arrière. Trois modes selon ce qu'on garde :

Bash
git reset --soft HEAD~1     # Annule le commit, garde tout staged (prêt à re-commiter)
git reset --mixed HEAD~1    # Annule le commit, garde le working tree (défaut)
git reset --hard HEAD~1     # ☢️ Annule le commit ET les modifications (destructif)

git rebase rejoue vos commits sur une autre base. Utile pour intégrer les changements de main dans votre branche feature sans créer un commit de merge :

Bash
git switch feat/pca
git fetch origin
git rebase origin/main           # Rejoue vos commits sur le main à jour

Diagramme — merge vs rebase :

Code
Avant :              Après merge :        Après rebase :

main:  A---B---C     A---B---C---M        A---B---C
             \                 /                    \
feat:         D---E    D---E--/                      D'---E' feat
⚠️

Règle d'or : ne jamais rebase ou amend une branche déjà poussée et partagée. Vous réécrivez l'historique, et toutes les copies locales de vos collègues divergent. Conséquence : conflits horribles. Sur une branche perso non partagée, rebase autant que vous voulez.

git bisect pour trouver le commit fautif

Un test passait il y a une semaine, ne passe plus aujourd'hui, et vous avez fait 50 commits entre temps. Plutôt que de les tester un par un :

Bash
git bisect start
git bisect bad                   # Le commit courant est cassé
git bisect good a3f2c91          # Ce commit-là marchait

# Git checkout automatiquement le commit médian
# Vous testez :
pytest tests/test_model.py
git bisect good                  # ou 'bad' selon le résultat

# Git affine par dichotomie. log₂(50) ≈ 6 itérations suffisent.
git bisect reset                 # Quand fini

C'est une recherche dichotomique automatisée. Sur 1000 commits, ça prend ~10 vérifications. Vous pouvez même automatiser totalement :

Bash
git bisect run pytest tests/test_model.py
# Git lance pytest à chaque étape, et s'arrête tout seul sur le coupable

C. Questions de compréhension

Q1 : Vous avez fait git reset --hard HEAD~3 puis réalisé que vous avez perdu trois commits importants. Sont-ils définitivement perdus ?

💡 Voir l'indice

git reflog.

✅ Voir la réponse

Non — pas pendant ~90 jours. Git conserve un journal de tous les déplacements de HEAD dans .git/logs/HEAD, accessible via git reflog :

Bash
git reflog
# abc1234 HEAD@{0}: reset: moving to HEAD~3
# def5678 HEAD@{1}: commit: feat: add normalization   ← voilà vos commits perdus
# 91011ab HEAD@{2}: commit: fix: edge case
# ...

Vous voyez tous les états par lesquels HEAD est passé, y compris ceux d'avant le reset. Pour récupérer :

Bash
git reset --hard HEAD@{1}        # Retour à l'état d'avant le reset
# OU
git checkout -b recovery HEAD@{1}  # Nouvelle branche pour isoler

La leçon plus profonde : Git ne supprime presque rien immédiatement. Les commits "supprimés" deviennent inaccessibles mais restent dans .git/objects/ jusqu'au prochain git gc qui tourne automatiquement après ~30-90 jours. Cette propriété explique pourquoi Git est si sûr : presque toutes les opérations destructrices sont récupérables tant qu'on agit vite.

Q2 : Quelle est la différence entre git pull et git pull --rebase, et lequel préférer en équipe ?

💡 Voir l'indice

Forme de l'historique.

✅ Voir la réponse
  • git pull fait fetch + merge. S'il y a des commits divergents, ça crée un commit de merge supplémentaire. L'historique prend une forme de spaghetti avec des branches qui se recroisent.
  • git pull --rebase fait fetch + rebase. Vos commits locaux sont rejoués au-dessus de la nouvelle pointe du remote. L'historique reste linéaire.

Comparaison visuelle :

Code
Après git pull (merge)          Après git pull --rebase

  A---B---C---M  main             A---B---C---D'---E'  main
       \     /
        D---E (local)

La plupart des équipes préfèrent le rebase pour la lisibilité de git log. Configuration globale :

Bash
git config --global pull.rebase true

Attention : si vos commits locaux sont déjà partagés (push public), ne rebase pas. La règle est : rebase tant que c'est privé, merge dès que c'est public.

Q3 : Vous avez accidentellement committé un fichier secrets.env contenant une clé API et vous l'avez poussé sur GitHub. Que faut-il faire ?

💡 Voir l'indice

La clé est immédiatement compromise.

✅ Voir la réponse

Étape 1 (la plus urgente) : révoquer immédiatement la clé API auprès du fournisseur (OpenAI, AWS, Stripe…). Tout secret poussé sur GitHub doit être considéré comme compromis dans les minutes qui suivent — des bots scannent en continu les commits publics à la recherche de clés.

Étape 2 : générer une nouvelle clé et l'ajouter à votre .env local (pas en dur dans le code).

Étape 3 : supprimer le fichier de l'historique avec git filter-repo :

Bash
pip install git-filter-repo
git filter-repo --path secrets.env --invert-paths
git push --force-with-lease origin main

Note : cela réécrit l'historique, à coordonner avec l'équipe.

Étape 4 : prévenir les collègues — leurs clones contiennent encore la clé compromise. Ils doivent re-cloner ou rebase par-dessus.

Étape 5 (préventif) : ajouter au .gitignore et installer un pre-commit hook :

Bash
# .gitignore
*.env
secrets/
credentials.json

# Pre-commit hook (gitleaks ou detect-secrets)
pre-commit install

C'est une question d'entretien classique pour les rôles DevOps / sécurité. Le candidat qui répond "je supprime le fichier dans un nouveau commit" a tout faux : le fichier reste dans l'historique, et la clé reste compromise.
