🧭 Introduction

Le Terminal est une fenêtre noire (ou blanche) dans laquelle on tape des commandes textuelles pour piloter son ordinateur. Pas de clic, pas d'icône : uniquement du texte. Cela peut sembler intimidant au premier abord, mais c'est en réalité un outil extrêmement puissant et rapide, utilisé quotidiennement par tous les développeurs et data scientists.

À quoi ça sert concrètement ? Quand tu travailles sur un projet de data science, tu vas régulièrement avoir besoin de : naviguer entre des dossiers de fichiers, créer ou supprimer des scripts Python, lancer des programmes, installer des bibliothèques. Tout cela se fait beaucoup plus vite et plus précisément via le Terminal qu'en cliquant dans le Finder.

Principes clés à garder en tête :

  • Le Terminal travaille toujours à partir d'un emplacement précis sur ton disque dur (le "répertoire courant").
  • Chaque commande suit la même logique : un mot-clé, puis optionnellement des options et des cibles.
  • Certaines actions sont irréversibles (notamment rm qui supprime sans corbeille).
  • Le Terminal est ton allié : il ne fait que ce que tu lui demandes.

À retenir absolument :

  • pwd — "où suis-je ?" → affiche le dossier dans lequel tu te trouves
  • ls — "qu'est-ce qu'il y a ici ?" → liste les fichiers et dossiers
  • cd — "je me déplace" → change de répertoire
  • rm — "je supprime définitivement" → sans corbeille, sans retour possible
  • Tab — ton meilleur ami → autocomplétion des noms de fichiers et commandes

Analogie fil rouge : Imagine que ton ordinateur est un immeuble géant avec des centaines de salles (les dossiers) et des objets dans chaque salle (les fichiers). Le Terminal, c'est ton badge d'accès universel : au lieu de chercher visuellement chaque salle, tu donnes des instructions précises — "va au 3e étage, couloir B, ouvre la salle 12, prends le fichier rouge".

Dans ce cours reformulé, nous allons d'abord comprendre comment se repérer et naviguer dans l'arborescence de fichiers, puis apprendre à créer, copier, déplacer et supprimer des fichiers et dossiers, puis voir comment lire le contenu d'un fichier et faire des recherches, puis explorer quelques commandes système utiles, et enfin maîtriser les raccourcis clavier qui font gagner un temps précieux au quotidien.

1) Navigation — Se repérer dans l'arborescence

Avant de faire quoi que ce soit, il faut savoir où tu es. Dans le Terminal, tu es toujours "positionné" dans un dossier précis. C'est ce qu'on appelle le répertoire courant.

Savoir où l'on est : pwd

pwd signifie print working directory — "affiche le répertoire de travail". C'est la première commande à connaître.

Ici, on demande au Terminal de nous dire dans quel dossier on se trouve :

Bash
pwd                  # affiche le chemin complet du répertoire courant

Résultat typique : /Users/thibaud/Desktop/mon-projet — c'est le "chemin" vers ton dossier actuel, comme une adresse postale dans l'ordinateur.

Voir ce qu'il y a dans un dossier : ls

ls signifie list — "liste". Il affiche le contenu du dossier courant.

Bash
ls           # liste les fichiers et dossiers visibles
ls -la       # liste TOUT, y compris les fichiers cachés (qui commencent par .)
             # et affiche les détails : permissions, taille, date de modification
💡

Les fichiers dont le nom commence par un point (.bashrc, .gitignore) sont des fichiers cachés. Ils n'apparaissent pas avec ls simple, mais bien avec ls -la.

Se déplacer : cd

cd signifie change directory — "change de dossier". C'est la commande de déplacement.

Bash
cd projets/         # entre dans le sous-dossier "projets" (qui doit exister ici)
cd ..               # remonte d'un niveau (va dans le dossier parent)
cd ~                # va directement au dossier "home" (ton espace personnel)
cd -                # revient au répertoire où tu étais juste avant
⚠️

Si tu tapes cd projets mais que ce dossier n'existe pas, le Terminal affiche No such file or directory. Utilise d'abord ls pour vérifier ce qui existe avant de te déplacer.

Exemple concret : Tu es sur le bureau (/Users/thibaud/Desktop) et tu veux aller dans ton dossier de projet. Tu tapes :

Bash
ls                  # tu vois : "lewagon-data-science", "photos", "cv.pdf"
cd lewagon-data-science/   # tu entres dans le projet
pwd                 # confirme : /Users/thibaud/Desktop/lewagon-data-science
✅

Récap express — Navigation : pwd pour se situer, ls pour voir le contenu, cd pour se déplacer. Ces trois commandes sont la base de tout.

2) Fichiers & Dossiers — Créer, copier, déplacer, supprimer

Une fois qu'on sait se repérer, on peut commencer à agir sur les fichiers et dossiers.

Créer : touch et mkdir

Bash
touch notes.txt         # crée un fichier vide nommé "notes.txt"
                        # si le fichier existe déjà, met à jour sa date de modification
mkdir mon_projet/       # crée un nouveau dossier nommé "mon_projet"
mkdir -p data/raw/csv/  # crée plusieurs dossiers imbriqués d'un seul coup
                        # (-p = "parents" : crée tous les dossiers intermédiaires si besoin)

Exemple concret : Tu veux initialiser la structure d'un projet data :

Bash
mkdir -p mon_projet/data/raw    # crée mon_projet/, puis data/ dedans, puis raw/ dedans
touch mon_projet/README.txt     # crée un fichier README vide dans mon_projet/

Copier : cp

cp signifie copy — "copie". L'original reste intact.

Bash
cp script.py script_backup.py   # copie "script.py" → crée "script_backup.py"
                                 # l'original "script.py" existe toujours
cp -r dossier_source/ dossier_copie/  # copie tout un dossier (avec son contenu)
                                       # (-r = "récursif" : descend dans les sous-dossiers)

Déplacer ou renommer : mv

mv signifie move — "déplace". Contrairement à cp, l'original disparaît.

Bash
mv rapport.txt archives/rapport.txt   # déplace "rapport.txt" dans le dossier "archives/"
mv ancien_nom.py nouveau_nom.py       # renomme le fichier (même dossier, nouveau nom)
💡

mv fait deux choses : il déplace ET il renomme. Si la destination est dans le même dossier mais avec un nom différent, c'est un renommage. Si c'est dans un autre dossier, c'est un déplacement.

⚠️

Piège fréquent : confondre cp et mv. Avec cp, l'original reste. Avec mv, l'original disparaît. Quand tu veux juste renommer, utilise mv. Quand tu veux garder une copie de secours, utilise cp.

Supprimer : rm

rm signifie remove — "supprime".

Bash
rm fichier.txt       # supprime "fichier.txt" — définitivement, sans corbeille
rm -r mon_dossier/   # supprime tout un dossier et son contenu
                     # (-r = "récursif" : supprime tous les fichiers et sous-dossiers)
❌

rm est IRRÉVERSIBLE. Il n'y a pas de corbeille, pas d'annulation. Le fichier est supprimé pour de bon. Bonne habitude : toujours faire ls juste avant pour confirmer que tu cibles le bon fichier.

❌

Ne jamais taper rm -r / — c'est une commande qui supprimerait l'intégralité du système de fichiers de ton ordinateur. Cible toujours un dossier spécifique : rm -r mon_dossier_specifique/.

✅

Récap express — Fichiers & Dossiers : touch crée un fichier, mkdir crée un dossier, cp copie (l'original reste), mv déplace/renomme (l'original disparaît), rm supprime définitivement.

3) Lecture & Recherche — Lire et chercher dans les fichiers

Le Terminal te permet aussi de lire et de fouiller dans tes fichiers sans ouvrir un éditeur.

Afficher le contenu d'un fichier : cat

cat signifie concatenate — à l'origine pour assembler des fichiers, mais on l'utilise surtout pour afficher un fichier.

Bash
cat notes.txt        # affiche tout le contenu de "notes.txt" dans le terminal
head -n 5 notes.txt  # affiche seulement les 5 premières lignes
                     # (-n 5 = "nombre de lignes = 5")

Exemple concret : Tu as un fichier CSV de données et tu veux juste voir les premières lignes :

Bash
head -n 3 ventes.csv   # affiche les 3 premières lignes du fichier CSV

Chercher un mot dans un fichier : grep

grep permet de chercher un mot ou un motif dans un fichier. Imagine Ctrl+F mais en ligne de commande.

Bash
grep "erreur" logs.txt      # affiche toutes les lignes de "logs.txt" qui contiennent "erreur"
grep "import" script.py     # trouve toutes les lignes d'import dans un script Python

Chercher des fichiers par nom : find

find cherche des fichiers dans un dossier (et ses sous-dossiers) selon leur nom.

Bash
find . -name "*.py"         # cherche tous les fichiers .py à partir du dossier courant (.)
                            # (* = n'importe quel nom, .py = extension Python)
find . -name "rapport*"     # cherche tous les fichiers dont le nom commence par "rapport"
🧠

Pour aller plus loin : grep peut aussi chercher dans plusieurs fichiers à la fois. Par exemple grep "TODO" *.py cherche la chaîne "TODO" dans tous les fichiers Python du dossier courant. Très utile pour retrouver des notes laissées dans son code.

4) Commandes Système — Quelques outils pratiques

Ouvrir un fichier ou un dossier (macOS)

Bash
open .               # ouvre le Finder à l'emplacement courant (très pratique !)
open fichier.pdf     # ouvre "fichier.pdf" avec l'application par défaut (Aperçu, etc.)
💡

open . est très pratique quand tu veux passer du Terminal au Finder pour voir tes fichiers graphiquement. Le point . signifie "le dossier courant".

Nettoyer l'écran et voir l'historique

Bash
clear                # vide l'affichage du terminal (les commandes précédentes disparaissent)
history              # affiche la liste de toutes les commandes tapées précédemment

Surveiller les processus en cours : top

Bash
top                  # affiche en temps réel les processus qui tournent sur ton ordinateur
                     # (CPU utilisé, mémoire, etc.) — appuie sur "q" pour quitter
kill 1234            # arrête le processus qui a l'identifiant numérique 1234
                     # (le PID est visible dans la colonne "PID" de la commande top)
⚠️

Si une commande tourne depuis trop longtemps ou est bloquée, appuie sur Ctrl + C pour l'interrompre. C'est l'équivalent de "forcer la fermeture" d'un programme.

A. Concepts du cours

Le shell, ce méta-langage

Le shell (bash, zsh, fish…) est un programme qui lit votre saisie ligne par ligne, l'interprète comme une commande et l'exécute. Ce n'est ni le terminal (l'application graphique qui affiche du texte), ni le système d'exploitation : c'est la couche d'orchestration qui transforme ls -la | grep .py en quatre opérations système distinctes (fork, exec, pipe, wait).

Exemple concret pour comprendre la distinction :

  • Terminal : l'application que vous lancez (Terminal.app, iTerm2, Alacritty…). Elle ne fait qu'afficher du texte et capturer les touches.
  • Shell : le programme que le terminal exécute à l'intérieur (bash, zsh…). C'est lui qui comprend ls, cd, les variables.
  • Programme : python, git, curl… sont des binaires indépendants que le shell lance.

Cette distinction explique pourquoi votre alias zsh ne fonctionne pas dans un script bash : ce n'est pas la même implémentation du langage.

Une commande typique a la forme commande [options] [arguments]. Les options courtes (-l) se combinent (-la = -l -a), les options longues (--all) sont plus lisibles dans les scripts. Le code de retour (0 = succès, ≠ 0 = erreur) est stocké dans $? et conditionne les chaînages && et ||.

Bash
# Crée data/raw si inexistant, s'y déplace, télécharge le dataset
mkdir -p data/raw && cd data/raw && curl -O https://example.com/dataset.csv

Commentaire ligne par ligne :

  • mkdir -p data/raw : -p crée aussi les parents et n'échoue pas si le dossier existe déjà.
  • && : "ET logique" — la commande suivante n'est exécutée QUE si la précédente a retourné 0.
  • cd data/raw : entre dans le dossier qu'on vient de créer.
  • curl -O : télécharge l'URL et enregistre avec le nom de fichier de l'URL (ici dataset.csv).
  • Résultat : si l'une des trois étapes échoue, les suivantes ne s'exécutent pas → pas de dossier vide, pas de download dans le mauvais endroit.
💡

Analogie : le shell est à votre OS ce que le REPL Python est à l'interpréteur. Vous tapez une expression, il l'évalue, vous voyez le résultat. La différence est que chaque "expression" du shell est en réalité un mini-programme qui peut chaîner des dizaines de processus, exactement comme une chaîne de production d'usine.

Le système de fichiers comme arbre

Tout est fichier sous Unix, et tout est rangé dans un arbre dont la racine est /. Les chemins absolus (/Users/alice/projects/lewagon) partent toujours de la racine. Les chemins relatifs (../data/raw) partent du répertoire courant (obtenu avec pwd). Le tilde ~ est un raccourci vers votre home (/Users/alice sur macOS, /home/alice sur Linux).

Exemple concret : depuis /Users/alice/projects/lewagon/notebooks :

  • cd .. vous amène à /Users/alice/projects/lewagon (un niveau au-dessus)
  • cd ../.. vous amène à /Users/alice/projects
  • cd ~/Documents vous amène à /Users/alice/Documents quelque soit le point de départ
  • cd / vous amène à la racine du disque

C'est pourquoi les conventions Unix recommandent toujours snake_case ou kebab-case pour les noms de fichiers (mon_dossier ou mon-dossier).

Pipes et redirections

C'est le concept le plus puissant d'Unix et celui qu'on sous-utilise le plus en débutant. Trois opérateurs à maîtriser :

  • commande > fichier : redirige la sortie standard (stdout) vers un fichier (écrase le contenu).
  • commande >> fichier : pareil, mais en mode append (ajoute à la fin).
  • commande1 | commande2 : la stdout de la première devient la stdin de la seconde.

Exemples concrets :

Bash
# Compter combien de fichiers Python ont été modifiés aujourd'hui
find . -name "*.py" -mtime 0 | wc -l

Décomposition :

  1. find . -name "*.py" -mtime 0 : cherche récursivement depuis . tous les fichiers .py modifiés dans les dernières 24 h (-mtime 0). Sortie : une liste de chemins, un par ligne.
  2. | wc -l : compte le nombre de lignes de ce qui arrive en entrée. Résultat : un entier.

Ce qu'on obtient : un chiffre, par exemple 7. Le tout en 1 ligne, sans variable intermédiaire, sans fichier temporaire.

Bash
# Trouver les 5 fichiers les plus volumineux d'un projet
du -ah . | sort -hr | head -5

Décomposition :

  1. du -ah . : affiche la taille (-a = tous les fichiers, pas juste les dossiers ; -h = human readable). Sortie : 12K ./file.py, 1.2M ./data.csv, …
  2. sort -hr : trie en ordre numérique human (-h, comprend K/M/G) et en reverse (-r, décroissant).
  3. head -5 : garde les 5 premières lignes.

Résultat : les 5 plus gros fichiers avec leur taille.

Bash
# Sauvegarder stdout et stderr séparément pendant un entraînement
python train.py > train.log 2> train.err
  • > train.log redirige stdout vers train.log.
  • 2> train.err redirige stderr (descriptor 2) vers train.err.
  • Pour tout mélanger dans un seul fichier : python train.py > out.log 2>&1 ou python train.py &> out.log.

L'idée fondatrice : chaque outil fait UNE chose, et on les compose pour créer un pipeline plus complexe. Cette philosophie a précédé Map-Reduce et le streaming Spark de 40 ans.

Permissions et propriété

Chaque fichier appartient à un utilisateur (owner) et à un groupe, et porte trois triplets de permissions : owner / group / others, chacun avec read (r), write (w), execute (x). ls -l affiche par exemple -rwxr-xr-- :

  • - : type (fichier régulier ; d pour dossier, l pour lien symbolique)
  • rwx : owner peut tout faire (lire, écrire, exécuter)
  • r-x : groupe peut lire et exécuter (mais pas modifier)
  • r-- : autres ne peuvent que lire

En notation octale, chaque triplet devient un chiffre (r=4, w=2, x=1) : rwx r-x r-- = 7 5 4 → 754.

Bash
chmod +x train.sh          # Rend le script exécutable pour tout le monde
chmod 755 train.sh         # Équivalent : rwxr-xr-x (owner tout, groupe et autres en lecture/exec)
chmod 644 secrets.env      # rw-r--r-- : owner lit/écrit, les autres lisent seulement
chown alice:staff train.sh # Change owner et group
⚠️

Piège : chmod 777 sur tout est l'équivalent unix de try: ... except: pass — ça fait disparaître l'erreur sans la comprendre. C'est aussi un trou de sécurité béant : n'importe qui peut modifier votre code et y injecter une backdoor. Préférez toujours des permissions minimales : 644 pour des fichiers de données, 755 pour des scripts exécutables, 600 pour des secrets.

B. Compléments avancés

Variables d'environnement et $PATH

Une variable d'environnement est une paire clé/valeur héritée par les processus enfants. La plus critique est $PATH : la liste des dossiers où le shell cherche un exécutable quand vous tapez son nom. Quand vous installez un nouvel outil et qu'il n'est "pas trouvé" (command not found), c'est presque toujours un problème de $PATH.

Bash
echo $PATH                       # Affiche la liste, colon-separated
# /usr/local/bin:/usr/bin:/bin:/Users/alice/.pyenv/shims

which python3                    # Trouve quel binaire est résolu en premier
# /Users/alice/.pyenv/shims/python3

export PATH="$HOME/.local/bin:$PATH"  # Ajoute un dossier EN TÊTE

Pourquoi "en tête" : le shell cherche les dossiers dans l'ordre et prend le premier match. En ajoutant votre dossier en tête, vos binaires prennent la priorité sur les binaires système.

Pour rendre un export permanent, ajoutez la ligne dans ~/.zshrc (zsh) ou ~/.bashrc (bash). Pour appliquer immédiatement : source ~/.zshrc.

En data science, deux variables d'environnement vitales :

  • PYTHONPATH : dossiers où Python cherche les modules à importer.
  • OPENAI_API_KEY (et consorts) : secrets d'API — jamais dans le code, toujours via l'environnement, et idéalement chargés depuis un .env via python-dotenv.

Exemple de .env pour un projet ML :

Bash
OPENAI_API_KEY=sk-proj-abc...xyz
DATABASE_URL=postgresql://alice:secret@localhost:5432/mydb
MLFLOW_TRACKING_URI=http://localhost:5000

Puis dans votre script Python :

Python
from dotenv import load_dotenv
import os

load_dotenv()                               # Lit .env et peuple os.environ
api_key = os.environ["OPENAI_API_KEY"]      # Lève KeyError si absent → fail-fast
👉

Bonne pratique : ajoutez .env à votre .gitignore dès le premier commit. Une fois qu'une clé d'API est dans l'historique git public, elle est compromise — il faut la révoquer ET réécrire l'historique. C'est une question d'entretien classique : "Qu'est-ce qui se passe si vous committez un secret par erreur ?"

Glob, expansion et quoting

Le shell expanse les motifs avant d'invoquer la commande. C'est puissant mais c'est une source de bugs subtils :

Bash
ls *.csv         # Le shell remplace *.csv par tous les .csv du dossier courant
rm -rf "$DIR"/*  # Supprime le contenu de $DIR mais pas $DIR lui-même

Trois niveaux de quoting à distinguer :

  • 'simple quotes' : aucun caractère n'est interprété, prend tout littéralement. Utile pour des chaînes contenant $, *, !.
  • "double quotes" : interpole $variables et $(commandes), mais protège des espaces.
  • Pas de quotes : tout est expansé, espaces compris. Source de bugs si une variable contient un espace.
Bash
NAME="Alice Wagon"
echo $NAME      # Affiche "Alice Wagon" — mais bash voit deux arguments
echo "$NAME"    # Affiche "Alice Wagon" — un seul argument, sûr
echo '$NAME'    # Affiche littéralement "$NAME", sans expansion

Exemple pratique :

Bash
# Mauvais : si un fichier contient un espace dans son nom, ça casse
for f in $(ls *.csv); do mv $f ${f%.csv}.txt; done

# Bon : le glob direct préserve les noms, les quotes protègent chaque variable
for f in *.csv; do mv "$f" "${f%.csv}.txt"; done

${f%.csv} signifie "$f sans le suffixe .csv" — syntaxe utile pour changer une extension.

⚠️

Piège classique en entretien DevOps : pourquoi for f in $(ls .csv); do ... casse-t-il sur des fichiers contenant des espaces ? Réponse : $(ls) est expansé en une suite de tokens découpés sur les espaces. for f in .csv; do ... est correct parce que le glob produit des tokens déjà séparés proprement et chaque token est un fichier complet.

Processus, jobs et signaux

Chaque commande lancée crée un processus avec un PID (process ID). Plusieurs façons de les manipuler :

Bash
python long_training.py &      # & lance en arrière-plan, retourne aussitôt
jobs                            # Liste les jobs du shell courant
fg %1                           # Ramène le job 1 au premier plan
# Ctrl+Z → suspend le processus courant
bg                              # Reprend en arrière-plan le job suspendu

Cas d'usage concret : vous lancez un entraînement long, vous voulez garder le terminal libre pour d'autres commandes :

Bash
python train.py > train.log 2>&1 &    # & à la fin : arrière-plan
[1] 12345                              # Le shell affiche le job id et le PID
# ... vous continuez à travailler ...
tail -f train.log                      # Pour voir les logs en live
# Ctrl+C sur tail ne tue pas train.py — c'est un autre processus

Signaux les plus utiles :

SignalTouche / cmdActionQuand l'utiliser
SIGINT (2)Ctrl+CInterrompre proprementArrêter un script Python qui boucle
SIGTSTP (20)Ctrl+ZSuspendreMettre en pause sans tuer
SIGTERM (15)kill PIDTerminer proprementDemander à un serveur de s'arrêter
SIGKILL (9)kill -9 PIDTuer immédiatementQuand SIGTERM échoue (dernier recours)

SIGKILL ne peut pas être intercepté par le programme : il n'a aucune chance de fermer ses fichiers, libérer ses verrous ou flusher ses buffers. C'est pourquoi kill -9 est à éviter sauf urgence — utilisez d'abord kill PID et laissez 5 secondes au processus pour se terminer proprement.

tmux et travail en remote

Quand vous entraînez un modèle sur un serveur distant via SSH, votre session se ferme dès que la connexion tombe — et votre python train.py part avec elle. La solution professionnelle est tmux (ou screen) : un multiplexeur de terminal qui maintient des sessions persistantes côté serveur, indépendamment de votre client.

Bash
# Sur le serveur (via ssh)
tmux new -s train          # Nouvelle session nommée "train"
python train.py            # Lancer l'entraînement
# Ctrl+b puis d : se détache (le job continue en arrière-plan du serveur)

# Vous pouvez fermer votre laptop, ssh se coupe — train.py tourne encore

# De retour plus tard
ssh user@server
tmux ls                    # Liste les sessions existantes
# train: 1 windows (created Mon Apr  1 10:00:00 2024) [80x24]
tmux attach -t train       # Réattacher pour voir la progression

Avantages secondaires : panneaux multiples (Ctrl+b " horizontal, Ctrl+b % vertical), historique scrollable (Ctrl+b <a href="https://mywiki.wooledge.org/BashPitfalls" target="_blank" rel="noopener">), copie/collage entre panneaux. C'est la première chose à installer sur n'importe quel serveur de prod.

C. Questions de compréhension

Q1 : Pourquoi cd new_dir dans un script bash ne change-t-il pas le dossier du shell parent ?

💡 Voir l'indice

Pensez à ce qui se passe quand vous lancez ./script.sh : quel processus exécute le script ?

✅ Voir la réponse

Quand vous exécutez un script avec ./script.sh, le shell parent crée un processus enfant qui exécute le script. Les variables d'environnement et le pwd sont locaux à ce processus enfant — ils sont détruits quand le script se termine. Le shell parent ne voit jamais ces changements.

Pour qu'un script affecte le shell parent, il faut le sourcer :

Bash
source script.sh    # ou, équivalent :
. script.sh

Le sourcing exécute le script dans le contexte du shell courant, pas dans un sous-processus. C'est exactement ce qu'on fait avec source venv/bin/activate : si on l'exécutait normalement (./venv/bin/activate), le PATH modifié serait perdu dès la fin de l'activation.

Règle : pour les scripts qui modifient l'environnement, documenter clairement qu'il faut les sourcer (ou fournir un alias qui le fait automatiquement).

Q2 : Quelle est la différence entre > et >>, et pourquoi command > file 2>&1 n'est-il pas équivalent à command 2>&1 > file ?

💡 Voir l'indice

Les redirections sont appliquées dans l'ordre où elles sont écrites. Que fait 2>&1 à chaque position ?

✅ Voir la réponse

> écrase, >> append. Pour la deuxième partie, c'est une question piège classique : les redirections sont évaluées de gauche à droite.

Version 1 : command > file 2>&1

  1. > file : stdout (fd 1) est redirigé vers file.
  2. 2>&1 : stderr (fd 2) est redirigé vers "ce vers quoi stdout pointe actuellement", c'est-à-dire file.
  3. Résultat : tout va dans file.

Version 2 : command 2>&1 > file

  1. 2>&1 : stderr est redirigé vers "ce vers quoi stdout pointe actuellement", c'est-à-dire le terminal (stdout n'a pas encore été redirigé).
  2. > file : stdout est redirigé vers file.
  3. Résultat : stdout va dans file, stderr va dans le terminal.

La syntaxe moderne command &> file évite l'ambiguïté et redirige stdout + stderr vers file en une seule fois. En entretien DevOps, c'est une question qui sépare ceux qui ont vraiment débuggé des scripts en prod de ceux qui ont juste lu un tutoriel.

Q3 : Vous lancez du -sh * dans un dossier de 200 Go et la commande est très lente. Comment l'accélérer sans changer d'outil ?

💡 Voir l'indice

du est séquentiel par défaut. Comment paralléliser sur plusieurs sous-dossiers en même temps ?

✅ Voir la réponse

du parcourt l'arbre séquentiellement. Sur un disque SSD ou un système de fichiers réseau, vous pouvez paralléliser avec GNU parallel ou xargs -P :

Bash
# Lance 8 'du' en parallèle, un par sous-dossier
ls | xargs -n1 -P8 du -sh
  • -n1 : un argument (un dossier) par invocation de du.
  • -P8 : 8 processus en parallèle.

Sur un HDD classique en revanche, paralléliser empire les choses (les têtes de lecture font des allers-retours). La vraie réponse d'entretien est : connaître le matériel sur lequel on tourne avant de paralléliser. C'est exactement le même raisonnement qu'avec le multiprocessing en Python : le GIL et le coût des fork peuvent rendre la version parallèle plus lente qu'une version séquentielle bien écrite.

Alternative moderne : dust (une réécriture rust de du) est beaucoup plus rapide sans avoir besoin de paralléliser manuellement.
