Coding as a Team

Maîtriser le workflow Git en équipe — branches, pull requests, code review, gestion des conflits, notebooks collaboratifs et organisation de projet Data Science.

🎯 Notions cles a retenir
Branch
Ligne de développement parallèle où vous travaillez sans casser `main`.
Pull Request
Demande de fusion d'une branche vers `main`, support de la code review.
Merge vs rebase
Fusion qui crée un commit vs réécriture linéaire de l'historique.
Code review
Relecture par un pair avant intégration ; outil de qualité et d'apprentissage.

📋 Cheatsheet — Coding as a Team

Créer un repo et inviter des collaborateurs

Bash
cd ~/code/YOUR_USERNAME
mkdir my-project && cd my-project
git init
echo '# my-project' >> README.md
git add README.md && git commit -m 'kickstart project'
gh repo create                         # crée le repo GitHub via CLI
git push origin master                 # premier et DERNIER push sur master

## Collaborateurs : Settings → Collaborators → Add people
## Le collaborateur clone ensuite :
git clone git@github.com:OWNER/PROJECT.git   # toujours SSH, pas HTTPS

Workflow branches

Bash
git branch                             # liste les branches locales
git checkout -b feature-encoding       # crée et switch sur une nouvelle branche
## ... coder, commiter ...
git push origin feature-encoding       # push la branche sur GitHub
## → Créer une Pull Request sur GitHub
## → Review → Merge → Delete branch

Mettre à jour master après un merge

Bash
git status                             # DOIT être clean avant pull/checkout/merge
git checkout master
git pull origin master                 # récupère les commits du merge
git sweep                              # nettoie les branches locales mergées
pip install -r requirements.txt        # au cas où des packages ont été ajoutés

Merger master dans sa branche (si besoin)

Bash
git add . && git commit -m 'WIP'       # commiter d'abord
git status                             # vérifier que c'est clean
git checkout master
git pull origin master
git checkout my-feature
git merge master                       # peut générer des conflits

Résoudre des conflits

Bash
## Après un merge avec conflits :
## 1. Ouvrir VS Code → Cmd+Shift+F → chercher "<<<<<"
## 2. Garder le bon code, supprimer les marqueurs <<<<< ===== >>>>>
## 3. Finaliser le merge :
git add .
git commit --no-edit                   # utilise le message de merge par défaut
git push origin my-branch

Git status — Les 3 niveaux

Bash
git add file.py                        # Modified → Staged
git restore --staged file.py           # Staged → Modified (garde les modifs)
git restore file.py                    # Modified → dernière version commitée (⚠️ perd les modifs)
git revert <commit_id>                 # annule un commit en créant un nouveau commit inverse

⚠️ Récapitulatif — Erreurs clés à éviter

❌

Commiter directement sur master

Le master doit toujours être stable — les modifications passent par des branches et des pull requests.

❌ git checkout master && git commit → casse le workflow de l'équipe

✅ Toujours créer une branche : git checkout -b my-feature → PR → review → merge

❌

Pull, checkout ou merge avec un git status non-clean

Des fichiers modifiés non commités peuvent créer des conflits silencieux ou des pertes de données.

❌ git pull origin master avec des fichiers modifiés → merge chaotique

✅ Toujours vérifier git status est clean avant toute opération de branche

❌

Modifier le notebook d'un coéquipier

Les notebooks génèrent des diffs illisibles et des conflits quasi impossibles à résoudre.

❌ Deux personnes éditent exploration.ipynb → conflit ingérable

✅ Chaque personne crée son propre notebook préfixé : alice_dataviz.ipynb

❌

Pusher des fichiers > 100 Mo sur GitHub

GitHub refuse les fichiers > 100 Mo — et une fois commités, ils sont difficiles à retirer de l'historique.

❌ git add data/big_dataset.csv → push bloqué

✅ Ajouter raw_data/ dans .gitignore dès le début du projet

❌

Cloner un repo avec HTTPS au lieu de SSH

HTTPS demande un mot de passe à chaque push — SSH utilise une clé et fonctionne sans friction.

❌ git clone https://github.com/... → mot de passe à chaque push

✅ git clone git@github.com:... (SSH)

3) Travailler en équipe — Branches et Pull Requests

3.1 Diviser le travail

  1. Découper le workflow en jobs parallèles (baseline, EDA, cleaning, feature engineering…)
  2. Découper chaque job en petites tâches
  3. Une tâche = une branche = une Pull Request

Bash
## 1. Créer une branche
git checkout -b feature-encoding

## 2. Coder, commiter
git add . && git commit -m 'add ordinal encoding'

## 3. Pusher la branche
git push origin feature-encoding

## 4. Créer une PR sur GitHub → Review → Merge → Delete branch

3.3 Pull Requests

  • Écrire un titre et une description clairs (ce qui a été fait, packages ajoutés…)
  • Le reviewer commente le diff, ligne par ligne
  • Si OK → "Merge Pull Request" → "Delete Branch"
  • Si pas OK → commentaire général "Review done", l'auteur corrige

Bash
git status                             # DOIT être clean
git checkout master
git pull origin master

Bash

4) Gérer les conflits

4.1 Quand ça arrive

Un conflit survient quand master a changé entre la création de la branche et le merge.

4.2 Résolution

Bash
git checkout master && git pull origin master
git checkout my-branch && git merge master
## → Conflits ! Normal.
## Ouvrir VS Code → Cmd+Shift+F → chercher "<<<<<"
## Garder le bon code, supprimer les marqueurs
git add .
git commit --no-edit
git push origin my-branch
💡

On peut aussi résoudre les conflits directement sur GitHub via le bouton "Resolve conflicts" dans la PR.

4.3 Deux règles d'or

  1. Ne jamais commiter sur master — toujours passer par une branche
  2. Toujours avoir un git status clean avant pull, checkout ou merge

🧭 Introduction

Jusqu'ici, tu as travaillé seul sur Git : tu commitais, tu pushais, et personne ne touchait à ton code entre-temps. En équipe, tout change. Plusieurs personnes modifient les mêmes fichiers en même temps, et sans organisation, c'est le chaos : qui a changé quoi ? Quelle version est la bonne ? Comment intégrer le travail de chacun sans tout casser ?

Ce cours répond à ces questions en t'apprenant le workflow Git en équipe : branches, pull requests, code review, et gestion des conflits. C'est la manière dont travaillent les équipes de développeurs professionnels — et les équipes Data Science aussi.

Analogie fil rouge. Imagine que tu co-écris un rapport Word avec deux collègues. Si vous éditez tous les trois le même fichier en même temps, vous allez vous écraser mutuellement. La solution ? Chacun fait une copie, travaille dessus, puis vous fusionnez à la fin en comparant les versions. C'est exactement ce que font les branches Git.

A retenir

  • master (ou main) est la version officielle et stable du projet — on n'y touche jamais directement.
  • Une branche est une copie de travail isolée — on y développe une fonctionnalité, puis on la fusionne.
  • Une Pull Request (PR) est une demande de fusion : "j'ai fini ma partie, quelqu'un peut vérifier et l'intégrer ?"
  • La code review permet à un coéquipier de valider le travail avant qu'il rejoigne master.
  • Un conflit survient quand deux personnes ont modifié le même endroit — Git demande alors qu'un humain choisisse quelle version garder.

1 — Créer et configurer un repo projet

Créer le repo depuis zéro

La première personne (le "owner") crée le repo et le met en ligne sur GitHub.

Bash
# Se placer dans son dossier de code
cd ~/code/YOUR_USERNAME

# Créer un nouveau dossier et y entrer
mkdir my-project && cd my-project

# Initialiser git dans ce dossier
git init

# Créer un fichier README minimal
echo '# my-project' >> README.md

# Premier commit
git add README.md && git commit -m 'kickstart project'

# Créer le repo sur GitHub via la CLI (répond aux questions interactives)
gh repo create

# Envoyer le code sur GitHub — c'est le SEUL push direct sur master
git push origin master
⚠️

Après ce premier push, plus personne ne pousse directement sur master. Tout passe par des branches et des Pull Requests.

Inviter des collaborateurs

Sur GitHub, aller dans Settings → Collaborators → Add people. Les collaborateurs reçoivent un email d'invitation. Une fois acceptée, ils clonent le repo :

Bash
# Créer un dossier au nom du owner pour s'organiser
mkdir ~/code/OWNER_USERNAME && cd ~/code/OWNER_USERNAME

# Cloner le repo en SSH (jamais en HTTPS)
git clone git@github.com:OWNER_USERNAME/PROJECT.git
💡

La différence SSH vs HTTPS : avec SSH, tu te connectes avec une clé déjà configurée — aucun mot de passe à saisir. Avec HTTPS, Git te le demande à chaque push. Toujours utiliser SSH.

2 — Rappels Git — Les 3 niveaux de statut

Avant de parler de travail en équipe, voici un rappel important sur les états d'un fichier dans Git. Il y en a 3 :

Code
Untracked / Modified  ──git add──▶  Staged  ──git commit──▶  Committed
                              ◀──git restore --staged──
                              ◀──git restore──  (⚠️ perd les changements)
  • Untracked / Modified : le fichier a été modifié, Git le voit mais ne le suit pas encore.
  • Staged : le fichier est "prêt à être commité" — tu l'as mis en file d'attente avec git add.
  • Committed : les changements sont enregistrés dans l'historique Git.
Bash
# Mettre un fichier en staged
git add file.py              # Modified → Staged

# Annuler le staging (les modifications sont conservées)
git restore --staged file.py # Staged → Modified

# Annuler les modifications (retourne à la dernière version commitée)
git restore file.py          # ⚠️ DESTRUCTIF — les changements sont perdus
🧠

git revert <commit_id> est une commande utile pour "annuler" un commit. Contrairement à d'autres méthodes, elle ne supprime rien de l'historique : elle crée un nouveau commit qui fait l'inverse du commit ciblé. C'est la méthode sûre pour revenir en arrière.

3 — Travailler en équipe — Branches et Pull Requests

3.1 Diviser le travail

La première étape est d'organisation, pas de code. On découpe le projet en jobs parallèles que les membres de l'équipe peuvent faire en même temps sans se marcher dessus.

Exemple sur un projet Data Science :

  • Alice travaille sur le nettoyage des données
  • Bob travaille sur l'analyse exploratoire (EDA)
  • Clara travaille sur le feature engineering

Chaque tâche devient une branche distincte. La règle d'or :

1 tâche = 1 branche = 1 Pull Request

3.2 Workflow Git complet (du début à la fin)

Voici ce que fait chaque développeur pour chaque tâche :

Bash
# 1. S'assurer d'être sur master à jour
git checkout master
git pull origin master

# 2. Créer une nouvelle branche pour la tâche
#    Le nom est explicite : on sait ce que fait la branche
git checkout -b feature-encoding

# 3. Travailler... puis commiter régulièrement
git add .
git commit -m 'add ordinal encoding for categorical columns'

# 4. Pusher la branche sur GitHub (pas master !)
git push origin feature-encoding

# 5. Aller sur GitHub et créer une Pull Request
#    GitHub propose souvent un bouton "Compare & pull request" automatiquement
💡

Une branche nommée feature-encoding indique clairement ce qu'elle contient. Évite les noms vagues comme test ou modif.

3.3 Pull Requests — Le processus de validation

Une Pull Request (PR) est une proposition de fusion. C'est aussi une conversation autour du code.

Créer une bonne PR :

  • Titre clair : "Add ordinal encoding for categorical columns"
  • Description : ce qui a été fait, pourquoi, packages éventuellement ajoutés
  • Assigner un reviewer (coéquipier)

Rôle du reviewer :

  • Il consulte le diff (les modifications) ligne par ligne sur GitHub
  • Il laisse des commentaires sur les lignes qu'il questionne
  • Si tout est OK → il clique "Merge Pull Request" puis "Delete Branch"
  • Si des corrections sont nécessaires → il laisse un commentaire général, l'auteur corrige et re-push
✅

Supprimer la branche après le merge est une bonne pratique. GitHub propose le bouton directement après le merge. Cela évite l'accumulation de branches mortes.

3.4 Mettre à jour master après un merge

Une fois la PR mergée, tous les coéquipiers doivent récupérer les changements sur leur machine :

Bash
# 1. Vérifier que le dossier de travail est propre (aucune modif en cours)
git status                             # doit afficher "nothing to commit"

# 2. Retourner sur master
git checkout master

# 3. Récupérer les nouveaux commits (ceux du merge)
git pull origin master

# 4. Nettoyer les branches locales déjà mergées
git sweep                              # commande custom Le Wagon

# 5. Installer les éventuels nouveaux packages
pip install -r requirements.txt
⚠️

Ne jamais faire git pull, git checkout ou git merge avec des fichiers modifiés non commités. Vérifie toujours avec git status que tout est propre avant. Des fichiers en cours peuvent créer des conflits silencieux ou être écrasés.

3.5 Mettre à jour sa branche avec master (si besoin)

Parfois, master a évolué pendant que tu travaillais sur ta branche. Tu veux intégrer ces nouveaux changements dans ta branche sans attendre le merge de la tienne.

Bash
# 1. Commiter ton travail en cours (même si c'est un WIP)
git add . && git commit -m 'WIP: encodage en cours'

# 2. Retourner sur master et le mettre à jour
git checkout master
git pull origin master

# 3. Retourner sur ta branche
git checkout my-feature

# 4. Fusionner master dans ta branche
#    Cela peut générer des conflits (voir section suivante)
git merge master

4 — Gérer les conflits

Quand un conflit survient-il ?

Un conflit arrive quand deux personnes ont modifié le même endroit dans le même fichier, et que Git doit choisir laquelle des deux versions garder — mais il ne peut pas décider seul.

Exemple concret : Alice et Bob ont tous les deux modifié la ligne 42 du fichier preprocessing.py. Quand Bob essaie de merger master (qui contient les changements d'Alice), Git s'arrête et demande : "je vois deux versions différentes de la même ligne — laquelle gardes-tu ?"

Résoudre un conflit pas à pas

Bash
# Mettre master à jour
git checkout master && git pull origin master

# Retourner sur sa branche et tenter le merge
git checkout my-branch && git merge master
# → Git affiche : "CONFLICT (content): Merge conflict in preprocessing.py"
# C'est normal. Ne pas paniquer.

# Ouvrir VS Code pour localiser les conflits
# Cmd+Shift+F (Mac) → chercher "<<<<<"
# Git a marqué les zones conflictuelles comme ceci :
#
# <<<<<<< HEAD
# mon code à moi
# =======
# le code de master (version d'Alice)
# >>>>>>> master
#
# → Garder le bon code (le tien, celui d'Alice, ou une combinaison)
# → Supprimer les marqueurs <<<<<<, =======, >>>>>>>

# Une fois tous les conflits résolus, finaliser le merge
git add .                              # marquer les fichiers comme résolus
git commit --no-edit                   # utilise le message de merge par défaut
git push origin my-branch              # pusher la branche résolue
💡

Sur GitHub, dans l'interface d'une Pull Request, un bouton "Resolve conflicts" permet parfois de résoudre des conflits simples directement dans le navigateur, sans avoir à toucher au terminal.

Les deux règles d'or

✅

Règle 1 — Ne jamais commiter directement sur master. Toujours créer une branche, même pour un changement minuscule. Master doit rester stable à tout moment.

✅

Règle 2 — Toujours avoir git status propre avant pull, checkout ou merge. Des fichiers modifiés non commités peuvent se mélanger avec les changements entrants et créer des situations très difficiles à démêler.

5 — Notebooks en équipe

Pourquoi les notebooks sont problématiques avec Git

Un fichier .ipynb n'est pas du Python pur — c'est un fichier JSON qui contient le code, les outputs, les métadonnées et l'état d'exécution. Cela pose trois problèmes :

  1. GitHub les affiche lentement et mal dans l'interface web.
  2. Les diffs sont illisibles : au lieu de voir "tu as changé cette ligne de code", tu vois des dizaines de lignes de JSON modifié.
  3. Les conflits sont quasi impossibles à résoudre manuellement dans ce format.

Bonne pratique : un notebook par personne

La solution la plus simple est d'éviter les conflits plutôt que de les résoudre.

Chaque personne a son propre notebook, préfixé par son pseudo GitHub.

Code
alice_eda.ipynb
bob_feature_engineering.ipynb
clara_dataviz.ipynb

Ainsi, personne ne touche jamais au notebook d'un autre — aucun conflit possible.

Comparer des notebooks avec nbdime

Si tu as besoin de comparer deux versions d'un notebook, utilise nbdime, un outil conçu pour ça :

Bash
# Installer nbdime
pip install nbdime

# Lancer une interface web pour comparer deux notebooks
nbdime diff-web base.ipynb updated.ipynb
# → Une page s'ouvre dans le navigateur avec une vue diff lisible
🧠

nbdime s'intègre aussi à Git pour remplacer le diff par défaut sur les .ipynb. Pratique dans les équipes qui travaillent beaucoup avec des notebooks.

6 — Fichiers volumineux

La limite GitHub

GitHub refuse catégoriquement les fichiers dépassant 100 Mo. En Data Science, c'est fréquent : datasets bruts, fichiers CSV volumineux, etc.

La solution : ne jamais les commiter. On les exclut avec .gitignore dès le début du projet.

Bash
# Ajouter le dossier raw_data/ à .gitignore
echo 'raw_data/' >> .gitignore         # tout ce dossier sera ignoré par Git

# Créer le dossier (il existera localement mais ne sera jamais pushé)
mkdir raw_data

# Chaque coéquipier télécharge les données séparément
# (depuis Kaggle, S3, Google Drive, etc.)
⚠️

Faire cela dès le début du projet. Si un gros fichier est commité par erreur, le retirer de l'historique est compliqué et destructif pour toute l'équipe.

Si un gros fichier a déjà été commité par erreur

C'est une situation délicate, mais réparable. Il faut réécrire l'historique Git pour effacer toute trace du fichier :

Bash
# 1. Déplacer le fichier dans raw_data/
mv path/to/big_file raw_data/

# 2. Commiter ce déplacement (mais le fichier est toujours dans l'historique Git)
git add path/to/big_file && git commit -m 'move big file to raw_data'

# 3. Réécrire l'historique pour supprimer le fichier de TOUS les commits passés
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch path/to/big_file" \
  --prune-empty --tag-name-filter cat -- --all

# 4. Pusher la branche (avec --force car l'historique a changé)
git push origin YOUR_BRANCH
⚠️

git filter-branch réécrit l'historique. Si d'autres personnes ont déjà cloné le repo, elles devront re-cloner. A éviter à tout prix — le .gitignore dès le début est la vraie solution.

🎯 Questions de vérification

1. Pourquoi ne doit-on jamais commiter directement sur master dans un projet d'équipe ?

Parce que master est la version de référence, stable et partagée. Si tout le monde pousse dessus directement, n'importe quelle erreur individuelle casse le travail de toute l'équipe. Les branches isolent les changements jusqu'à ce qu'ils soient validés.

2. Tu travailles sur ta branche feature-encoding depuis deux jours. Un coéquipier vient de merger sa branche dans master. Que fais-tu ?

Tu mets à jour master localement (git checkout master && git pull origin master), puis tu reviens sur ta branche et tu fais git merge master pour intégrer ses changements dans ta branche. Des conflits peuvent survenir — il faut les résoudre avant de continuer.

3. Que signifient les marqueurs <<<<<<<, =======, >>>>>>> dans un fichier ?

Ce sont des marqueurs de conflit insérés par Git. La zone entre <<<<<<< et ======= est ta version du code. La zone entre ======= et >>>>>>> est la version qui vient de l'autre branche. Il faut choisir ce qu'on garde, supprimer les marqueurs, puis finaliser le merge avec git add . && git commit --no-edit.

4. Pourquoi deux coéquipiers ne doivent-ils pas éditer le même notebook Jupyter ?

Les notebooks sont stockés en JSON. Deux personnes qui modifient le même notebook créent un conflit Git dans ce JSON — qui est illisible et presque impossible à résoudre manuellement. La solution est que chaque personne crée son propre notebook préfixé par son pseudo.

📌 TL;DR

  • master est sacré : stable, propre, jamais modifié directement — tout passe par des branches.
  • Workflow de base : git checkout -b ma-branche → coder → commiter → git push origin ma-branche → Pull Request → Review → Merge → Delete branch.
  • Avant tout pull, checkout ou merge : vérifier que git status est propre.
  • Un conflit n'est pas une catastrophe : Git marque les zones problématiques avec <<<<<<</=======/>>>>>>>, tu choisis ce qu'on garde, tu supprimes les marqueurs, tu commites.
  • Notebooks en équipe : un notebook par personne, préfixé par son pseudo — jamais toucher au notebook d'un autre.
  • Fichiers volumineux (> 100 Mo) : les exclure avec .gitignore dès le départ, jamais les commiter.
  • Organisation : tableau Kanban (Backlog → To do → Doing → Done) + une tâche = une branche = une PR.
  • Toujours cloner avec SSH (git@github.com:...), jamais HTTPS.

A. Concepts du cours

Git workflow en équipe

Travailler en équipe sur un projet logiciel demande une discipline qui n'existe pas en solo. Quatre principes :

  1. Une branche par feature : ne jamais coder directement sur main
  2. Petits commits : un commit = une intention claire, réversible
  3. Pull Requests : toute fusion vers main passe par une PR validée
  4. Communication asynchrone : les commentaires de PR remplacent les conversations
Bash
# Workflow standard
git checkout main && git pull               # Partir d'un main à jour
git switch -c feat/add-user-auth            # Créer une branche feature
# ... développer, committer ...
git push -u origin feat/add-user-auth       # Push avec tracking
# Ouvrir une PR sur GitHub

Vue graphique du flux :

Code
main         o───o───o───o───o───o ─────────── (branche partagée protégée)
                      \              \
feat/auth              o───o───o───o──●  (PR mergée)
                       ↑   ↑   ↑
                     commit commit commit

Chaque développeur "checkoute" depuis main, crée sa branche, y commit plusieurs fois, puis ouvre une PR. La PR est revue, testée par la CI, puis mergée dans main.

💡

Analogie : coder en solo, c'est cuisiner chez vous. Vous pouvez utiliser n'importe quelle marmite, ranger comme vous voulez. Coder en équipe, c'est cuisiner dans un restaurant. Tout doit être à sa place, étiqueté, et chaque action est visible par les autres. Les conventions ne sont pas du formalisme, elles évitent que la cuisine devienne ingérable à 5 cuisiniers.

Pull Requests et code review

Une Pull Request (PR) est une demande de fusion d'une branche vers main. Elle déclenche :

  1. Le build et les tests automatiques (CI)
  2. La review par un pair (code review)
  3. La fusion une fois tout validé

Anatomie d'une bonne PR :

  • Titre clair : "Add JWT authentication for /api/users endpoints" (pas "fix")
  • Description : pourquoi cette PR existe, ce qu'elle change, comment tester
  • Petite taille : < 400 lignes idéalement, jamais > 1000 lignes
  • Une seule intention : si la PR contient "add X and refactor Y", c'est probablement deux PR
  • Tests inclus : pas de PR sans test sur le nouveau code

Template typique :

Code
## What
Adds JWT authentication on /api/users endpoints.

## Why
Closes #123. The /users endpoints currently expose all user data publicly.

## How
- New middleware `auth_required` in `src/middleware/auth.py`
- New tests in `tests/test_auth.py`
- Updated docs in `README.md`

## How to test
1. `pytest tests/test_auth.py`
2. `curl -H "Authorization: Bearer ..." localhost:8000/api/users`

Code review : règles d'or

Pour le reviewer :

  • Lire la description avant le diff. Comprendre l'intention.
  • Commenter avec bienveillance. "Je trouve plus lisible de..." > "C'est moche".
  • Distinguer les types : nit (cosmétique), suggestion (optionnel), question, blocker.
  • Tester si possible : checkout la branche et essayer.
  • Approuver vite : un PR qui attend 3 jours frustre l'auteur et bloque l'équipe.

Pour l'auteur :

  • Auto-review d'abord : relisez votre propre PR avant de la demander.
  • Réagir à tous les commentaires : soit corriger, soit expliquer pourquoi.
  • Ne pas le prendre personnellement : la review critique le code, pas vous.
👉

Bonne pratique : adoptez la convention [nit], [suggestion], [blocker] dans les commentaires. Cela clarifie immédiatement ce qui doit être corrigé vs ce qui est optionnel. Un PR bloqué uniquement sur des nits est frustrant et inutile.

Conflits de merge

Quand votre branche et main ont modifié les mêmes lignes, git ne sait pas quel choisir. Vous devez résoudre manuellement.

Bash
git checkout main && git pull
git checkout feat/my-branch
git merge main
# CONFLICT (content): Merge conflict in src/auth.py

# Ouvrir src/auth.py — vous voyez :
<<<<<<< HEAD
def authenticate(user, password):
    return check_bcrypt(user.password, password)
=======
def authenticate(user, password):
    return check_argon2(user.password, password)
>>>>>>> main

Trois étapes pour résoudre :

  1. Ouvrir le fichier et choisir/combiner les versions
  2. Supprimer les marqueurs <<<<<<<, =======, >>>>>>>
  3. git add fichier && git commit
⚠️

Erreur catastrophique : committer avec les marqueurs de conflit encore dans le code. Le code passe les tests Python (les <<< sont des opérateurs valides) mais en runtime, c'est cassé. Toujours git diff ou git grep "<<<<<<<" après résolution. Un pre-commit hook peut bloquer ça automatiquement.

Stratégies pour minimiser les conflits :

  • Petites PRs : moins de chances de toucher aux mêmes fichiers
  • Merger main fréquemment dans votre branche (git merge main tous les jours)
  • Communication : "je touche à auth.py cette semaine"

Conventions et style guide

Une convention partagée vaut mieux qu'une "meilleure" pratique imposée par un seul. Cinq conventions à fixer en équipe :

  1. Style guide Python : PEP8 + ruff/black. Aucun débat.
  2. Format des commits : Conventional Commits (feat:, fix:, chore:...).
  3. Branches naming : feat/short-description, fix/bug-name, chore/cleanup.
  4. PR template : What, Why, How, How to test (cf. exemple plus haut).
  5. Code owners : qui doit reviewer quoi (CODEOWNERS file).
Code
# .github/CODEOWNERS
src/api/         @backend-team
src/ml/          @ml-team
src/frontend/    @frontend-team
*.md             @docs-team

Une PR qui modifie src/api/ sera automatiquement assignée à @backend-team pour review.

B. Compléments avancés

Git flow vs trunk-based

Deux philosophies de branching.

Git Flow (Vincent Driessen, 2010) : modèle complexe avec plusieurs branches longues.

Code
main           o-----------o-------o      (production)
develop        |  o-o-o-o-/-o-o-o-/-o     (intégration)
feature/X      |  |       |
hotfix/Y       |          |       |
release/v1.0   |              o-o-/
  • main : production, jamais touchée directement
  • develop : intégration des features
  • feature/* : par feature
  • release/* : préparation d'une release
  • hotfix/* : correctifs urgents en prod

Avantages : très structuré, permet de coder plusieurs versions en parallèle.

Inconvénients : complexe, lent à itérer, beaucoup de merges.

Trunk-based development : tout le monde commit dans main directement (ou via des PRs courtes). Adopté par Google, Facebook, Spotify.

  • Une seule branche longue : main (alias trunk)
  • Branches feature courtes (< 1-2 jours)
  • Feature flags pour cacher du code non terminé en production

Avantages : simple, itération rapide, moins de conflits.

Inconvénients : nécessite une bonne CI et de la discipline.

👉

Recommandation 2025 : pour 90% des équipes modernes, trunk-based est meilleur. Git Flow était adapté à des releases trimestrielles d'application desktop. Aujourd'hui, on déploie plusieurs fois par jour, donc une seule branche partagée + feature flags est plus efficace.

Rebase vs merge

Deux manières d'intégrer des changements de main dans votre branche feature.

Merge : crée un commit de merge. L'historique conserve la structure de branches.

Bash
git checkout feat/my-branch
git merge main
# Crée un commit "Merge branch 'main' into feat/my-branch"
Code
Avant merge :                      Après merge :
main     A───B───C                 main     A───B───C
              \                                 \   \
feat          D───E                 feat         D───E───M  (M = merge commit)

Rebase : rejoue vos commits par-dessus la pointe de main. Historique linéaire.

Bash
git checkout feat/my-branch
git rebase main
# Vos commits sont "déplacés" sur la nouvelle base
Code
Avant rebase :                     Après rebase :
main     A───B───C                 main     A───B───C
              \                                     \
feat          D───E                 feat             D'───E'  (recréés sur C)

Les commits D et E deviennent D' et E' (nouveaux hash) rejoués sur C. L'historique est linéaire, comme si vous aviez commencé votre branche depuis le C actuel.

MergeRebase
HistoriqueAvec branches (graphe)Linéaire
Lisibilité de git logMoins clairePlus claire
Préserve l'historique exactOuiNon (réécrit)
ConflitsUne foisPossiblement plusieurs fois
SécuritéSûrDangereux sur une branche partagée
⚠️

Règle d'or absolue : ne jamais rebase une branche déjà partagée. Si vous avez git push votre branche et que d'autres l'ont pullée, rebase + force-push casse leur historique local, conflits horribles. Rebase est OK uniquement sur une branche perso non encore push, ou avec une convention d'équipe ("toutes les branches feature/* sont rebasables").

Stratégie courante : rebase votre branche feature sur main avant de la merger, pour avoir un historique propre. Une fois mergée, ne touchez plus jamais à l'historique.

Ownership et code review distribuée

Sur de gros projets, personne ne connaît tout le codebase. La code ownership définit qui est responsable de quelle partie.

Avantages :

  • Review compétente : les bons experts revoient leur domaine
  • Connaissance maintenue : chaque module a un mainteneur identifié
  • Responsabilité claire : si un bug arrive en prod sur auth.py, on sait qui appeler

Outils :

  • CODEOWNERS (GitHub natif) : auto-assignation des reviewers
  • Reviewable ou Gerrit : pour de la review plus structurée
  • Crucible (Atlassian) : pour les enterprises

Sur des très gros codebases (Google, Facebook), il y a même des OWNERS files par dossier qui listent qui peut approuver des changements à chaque niveau.

💡

Analogie : sans ownership, votre code est comme un terrain communal — personne ne se sent responsable de l'entretenir, donc personne ne le fait. Avec ownership, chaque parcelle a un jardinier qui s'en occupe. Le résultat sur 5 ans est radicalement différent.

C. Questions de compréhension

Q1 : Pourquoi une PR de 1000 lignes est-elle un anti-pattern, même si le code est de qualité ?

💡 Voir l'indice

review.

✅ Voir la réponse

Trois raisons. (1) La review est inefficace : au-delà de ~400 lignes, l'attention du reviewer chute drastiquement. Une étude de SmartBear (2008) montre que la détection de défauts s'effondre après 200 lignes. Le reviewer va survoler, donner un "LGTM" superficiel, et laisser passer des bugs. (2) Difficile à debugger après merge : si la PR introduit un régression, vous devez chercher dans 1000 lignes laquelle est responsable. Une PR de 100 lignes vous donne immédiatement le coupable. (3) Frustrant pour l'auteur : passer 2 semaines sur une grosse PR puis recevoir des commentaires structurels qui demandent de tout refaire est démoralisant. Mieux vaut 5 PR de 200 lignes incrémentalement validées. Bonne pratique : viser ~200 lignes par PR. Si vous arrivez à 500, c'est un signal pour découper. Outils : git diff --stat montre la taille de votre PR. Décomposer une grosse feature en plusieurs PR demande un peu de planning au début mais accélère drastiquement le développement à terme. Question d'entretien dev senior : "Quelle est la bonne taille pour une PR ?" — la réponse révèle l'expérience d'équipe.

Q2 : Vous travaillez sur une feature depuis 1 semaine sur une branche, et main a beaucoup avancé. Faut-il merger main dans votre branche ou rebaser ?

💡 Voir l'indice

workflow d'équipe.

✅ Voir la réponse

Cela dépend de la convention de votre équipe, mais voici les recommandations modernes. Si votre branche n'est pas push (ou est privée à vous seul) : git rebase main. Cela donne un historique linéaire propre, plus facile à reviewer et à comprendre. Si votre branche est push et que d'autres y collaborent : git merge main. Le rebase + force-push casserait leur historique local. Compromis pragmatique : (a) pendant le développement, mergez main dans votre branche pour rester à jour (git merge main régulièrement). (b) Juste avant de merger votre PR vers main, faites un squash merge : tous vos commits sont fusionnés en un seul, qui est mergé proprement. C'est ce que GitHub propose en option "Squash and merge" — résultat : historique propre dans main, et vous n'avez jamais à rebaser. Note : Google et la plupart des grandes entreprises utilisent le squash merge par défaut pour cette raison. Ça simplifie la mental model. Question d'entretien git : "Rebase ou merge ?" — la réponse exhaustive parle de la politique d'équipe et du squash merge.

Q3 : Vous avez accidentellement committé un fichier .env contenant une clé API et l'avez poussé. Que faut-il faire ?

💡 Voir l'indice

la clé est compromise.

✅ Voir la réponse

Cinq étapes dans cet ordre, immédiatement : (1) Révoquer la clé chez le provider (OpenAI, AWS, Stripe...). C'est l'action la plus urgente. Toute clé poussée sur GitHub est compromise dans les minutes qui suivent — des bots scannent en continu. (2) Générer une nouvelle clé et l'ajouter à .env (ou à un secret manager). (3) Supprimer la clé de l'historique git avec git filter-repo --path .env --invert-paths puis git push --force-with-lease. Note : cela réécrit l'historique, à coordonner avec l'équipe. (4) Prévenir l'équipe : tous les clones existants contiennent encore la clé, ils doivent re-cloner. (5) Préventif : ajouter .env au .gitignore, installer un pre-commit hook (gitleaks, detect-secrets) qui bloque les commits contenant des secrets, ajouter un secret scanning dans la CI (GitHub fait ça automatiquement). Erreur critique : juste supprimer le fichier dans un nouveau commit. Le fichier reste accessible dans l'historique. La clé est toujours compromise. Attitude générale : tout secret committé est compromis, point. Pas de "j'ai supprimé tout de suite". Il faut toujours révoquer et régénérer. Question d'entretien sécurité : "Que faire si vous committez un secret par erreur ?" — la réponse doit être instantanée et exhaustive.
