CI / CD

Mettre en place l'intégration continue (CI) avec GitHub Actions pour tester automatiquement le code à chaque push, et le déploiement continu (CD) avec Streamlit Cloud ou Google Cloud Run pour livrer automatiquement en production.

🎯 Notions cles a retenir
CI
Continuous Integration : tests automatiques à chaque push.
CD
Continuous Delivery / Deployment : livraison automatisée jusqu'en prod.
GitHub Actions
Workflows YAML qui s'exécutent sur des événements Git.
Pre-commit
Hooks Git locaux qui formatent et lintent avant chaque commit.

📋 Cheatsheet — CI/CD

GitHub Actions — CI

Code
## .github/workflows/python-ci.yml
name: basic CI

on:
  push:
    branches: [ master, main ]
  pull_request:
    branches: [ master, main ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3                    # checkout la branche
    - name: Set up Python 3.10
      uses: actions/setup-python@v2
      with:
        python-version: "3.10"
    - name: Install dependencies
      run: |
        python -m pip install --upgrade pip
        pip install -r requirements.txt
    - name: Run tests
      run: make                                    # exécute pytest via Makefile
Bash
## Setup
git checkout -b ci-setup
mkdir -p .github/workflows
## Créer le fichier python-ci.yml ci-dessus
git add .github && git commit -m "Add CI workflow"
git push origin ci-setup
gh pr create --web                                 # crée la PR → CI se déclenche

⚠️ Récapitulatif — Erreurs clés à éviter

❌

Commiter sans lancer les tests localement

La CI rattrape les oublis, mais attendre le feedback GitHub est lent — tester localement d'abord.

❌ Push → attendre 3 min → CI échoue → re-push → re-attendre

✅ make test avant chaque push, la CI est un filet de sécurité

❌

Oublier que seuls les événements sur le remote déclenchent la CI

Un commit local sans push ne déclenche rien sur GitHub Actions.

❌ git commit sans git push → aucun workflow lancé

✅ Pousser la branche pour déclencher la CI : git push origin my-branch

❌

Merger une PR avec la CI en échec

Un merge avec des tests cassés introduit des bugs dans master.

❌ "Merge Pull Request" alors que le check CI est rouge

✅ Configurer la branch protection rule : merge bloqué si CI échoue

❌

Déployer en CD sans CI

Le CD sans CI signifie que du code non testé arrive en production à chaque push.

❌ Auto-deploy Streamlit Cloud sans aucun test → bugs en production

✅ CI (tests) → CD (deploy) — toujours dans cet ordre

❌

Hardcoder des secrets dans le workflow YAML

Les fichiers .github/workflows/ sont versionnés et visibles — les secrets y seraient exposés.

❌ env: API_KEY=sk-12345 dans le YAML

✅ Utiliser GitHub Secrets : Settings → Secrets → ${{ secrets.API_KEY }}

1) CI — Continuous Integration

1.1 Principe

La CI intègre automatiquement les changements de code dans la branche principale, en testant chaque commit ou PR dans un environnement isolé et frais.

Un workflow CI typique :

  1. Déclenché par un événement git (push, PR)
  2. Installe les dépendances dans un environnement propre
  3. Lance les tests (pytest)
  4. Reporte le résultat (✅ ou ❌) directement dans la PR

1.2 GitHub Actions

GitHub Actions est intégré nativement à GitHub — gratuit pour les repos publics, pas de configuration externe.

1.3 Mise en place

Bash
mkdir -p .github/workflows
touch .github/workflows/python-ci.yml

Contenu du fichier YAML (voir Cheatsheet ci-dessus).

Bash
git checkout -b ci-setup
git add .github
git commit -m "Configure GitHub Actions CI"
git push origin ci-setup
gh pr create --web

Le workflow se lance automatiquement dans la PR :

💡

Le statut du CI (pending → success/failure) apparaît directement dans la PR GitHub. Un développeur qui oublie de tester localement sera averti immédiatement.

2) CD — Continuous Deployment

2.1 Principe

Le CD déploie automatiquement en production chaque commit qui passe la CI sur la branche principale. Aucune intervention manuelle.

2.2 CD avec Streamlit Cloud

Streamlit Cloud est connecté à un repo GitHub. À chaque push sur main, l'app est automatiquement re-déployée.

2.3 CD avec Google Cloud Run + Cloud Build

  1. Créer un service Cloud Run depuis l'image Artifact Registry
  2. Activer "Continuously deploy new revisions from a source repository"
  3. Connecter le repo GitHub via Cloud Build
  4. Chaque push sur main → build Docker → deploy automatique
Bash

## Build et push initial
docker build -t $IMAGE_URI .
docker push $IMAGE_URI

🧭 Introduction

Imagine une usine qui fabrique des voitures. Avant qu'une voiture quitte la chaîne, elle passe par une série de contrôles qualité automatiques : vérification du moteur, des freins, de la carrosserie. Si un défaut est détecté, la voiture est bloquée — elle n'arrive jamais chez le client avec un problème.

Le CI/CD (Intégration Continue / Déploiement Continu), c'est exactement ce principe appliqué au code :

  • La CI (Continuous Integration) = les contrôles qualité automatiques (tests)
  • La CD (Continuous Deployment) = la livraison automatique chez le client (mise en production)

Chaque fois qu'un développeur pousse du code sur GitHub, une série d'étapes se déclenche automatiquement : on vérifie que le code fonctionne, puis on le déploie sans intervention humaine.

A retenir

  • La CI teste automatiquement le code à chaque git push
  • La CD déploie automatiquement le code en production si les tests passent
  • GitHub Actions est l'outil qui orchestre tout ça, directement dans GitHub
  • Un fichier YAML dans .github/workflows/ suffit à tout configurer
  • L'ordre est impératif : CI d'abord, CD ensuite — jamais l'inverse

Mini-plan

  1. La CI avec GitHub Actions : déclencher des tests automatiques
  2. La CD avec Streamlit Cloud ou Google Cloud Run : déployer automatiquement
  3. Les erreurs classiques à éviter absolument

1. CI — L'Intégration Continue

Qu'est-ce que c'est concrètement ?

Quand tu travailles en équipe (ou même seul), tu fais régulièrement des git push. Sans CI, rien ne garantit que ton code fonctionne encore après tes modifications. Tu pourrais avoir cassé quelque chose sans le savoir.

La CI, c'est un robot qui :

  1. Détecte ton git push
  2. Récupère ton code dans un environnement tout propre (une machine virtuelle vierge)
  3. Installe les dépendances
  4. Lance les tests automatiques (pytest)
  5. Te dit si tout va bien (✅) ou si quelque chose est cassé (❌)

Tout ça se passe sur les serveurs de GitHub — pas sur ton ordinateur.

💡

L'avantage d'un "environnement propre" : les tests tournent sur une machine Ubuntu fraîche, sans aucune dépendance installée en dehors de ce qui est dans requirements.txt. Ça évite le célèbre "ça marche chez moi mais pas ailleurs".

GitHub Actions : l'outil de CI intégré à GitHub

GitHub Actions est directement intégré à GitHub. Gratuit pour les repos publics. Pour l'activer, il suffit de créer un fichier YAML dans ton projet.

Mise en place pas à pas

Etape 1 — Créer la structure de dossiers

Bash
# Dans ton projet Git, créer le dossier pour les workflows
mkdir -p .github/workflows

# Créer le fichier de configuration CI
touch .github/workflows/python-ci.yml

Etape 2 — Ecrire le fichier YAML

C'est ce fichier qui dit à GitHub Actions quoi faire et quand le faire.

Code
# .github/workflows/python-ci.yml
# Ce fichier configure le workflow CI (Intégration Continue)

name: basic CI  # Nom affiché dans l'interface GitHub Actions

# "on" définit QUAND ce workflow se déclenche
on:
  push:
    branches: [ master, main ]   # Au moment d'un push sur main ou master
  pull_request:
    branches: [ master, main ]   # Ou quand une PR cible main/master

# "jobs" liste les tâches à exécuter
jobs:
  build-and-test:                # Nom du job (libre)
    runs-on: ubuntu-latest       # Sur quelle machine tourner (Ubuntu vierge)

    steps:                       # Liste des étapes, dans l'ordre
    - uses: actions/checkout@v3  # Etape 1 : récupérer le code du repo

    - name: Set up Python 3.10   # Etape 2 : installer Python
      uses: actions/setup-python@v2
      with:
        python-version: "3.10"

    - name: Install dependencies  # Etape 3 : installer les packages
      run: |
        python -m pip install --upgrade pip  # Mettre pip à jour
        pip install -r requirements.txt      # Installer les dépendances du projet

    - name: Run tests             # Etape 4 : lancer les tests
      run: make                   # Appelle le Makefile qui lance pytest

Lecture du fichier, section par section :

  • name : le nom du workflow, visible dans GitHub
  • on : les événements qui déclenchent le workflow (push, pull_request)
  • branches : sur quelles branches s'appliquent ces événements
  • jobs : les tâches à réaliser
  • runs-on : le type de machine virtuelle à utiliser
  • steps : chaque étape s'exécute l'une après l'autre
🧠

Un fichier YAML est très sensible à l'indentation. Chaque niveau d'imbrication correspond à 2 ou 4 espaces. Ne jamais utiliser de tabulations. Si le workflow ne se lance pas, commence par vérifier l'indentation.

Etape 3 — Pousser le workflow et créer une PR

Bash
# Créer une nouvelle branche pour cette configuration
git checkout -b ci-setup

# Ajouter et commiter le fichier de workflow
git add .github
git commit -m "Configure GitHub Actions CI"

# Pousser la branche sur GitHub
git push origin ci-setup

# Créer une Pull Request (la CI se déclenche automatiquement)
gh pr create --web

Une fois la PR créée, GitHub Actions se lance automatiquement. Dans l'interface GitHub, tu verras un badge vert (✅ tests OK) ou rouge (❌ tests échoués) directement sur la PR.

⚠️

Un git commit seul ne déclenche rien. Il faut absolument faire un git push pour que GitHub Actions soit informé et lance le workflow.

Bonus — Formater le code automatiquement avec Black

On peut ajouter une étape au workflow pour s'assurer que le code est bien formaté :

Code
# Ajouter cette étape dans la liste "steps" du workflow
- name: Format with Black
  run: black .   # Black reformate tout le code Python du projet
💡

Black est un formateur de code Python qui applique des règles de style de manière stricte et automatique. Plus besoin de débattre des espaces et des indentations en équipe : Black tranche.

2. CD — Le Déploiement Continu

Principe

La CD prend le relais après la CI. Une fois que les tests passent sur la branche principale (main), le code est automatiquement déployé en production — sans que personne n'ait besoin de cliquer sur quoi que ce soit.

⚠️

La CD sans CI est dangereuse. Si tu déploies automatiquement sans tester, chaque push peut envoyer du code cassé en production, visible par tes utilisateurs. Toujours : CI en premier, CD en second.

Option A — CD avec Streamlit Cloud (la plus simple)

Streamlit Cloud est une plateforme gratuite pour déployer des applications Streamlit directement depuis GitHub.

Bash
# La procédure complète en 3 étapes :

# 1. S'assurer que le code est bien poussé sur GitHub (branche main)
git push origin main

# 2. Aller sur share.streamlit.io et se connecter avec son compte GitHub

# 3. Sélectionner le repo et la branche → l'app est déployée

# Résultat : à chaque nouveau push sur main, Streamlit Cloud
# re-déploie automatiquement l'application.

C'est la solution la plus rapide à mettre en place. Idéal pour les projets de data science avec une interface Streamlit.

Option B — CD avec Google Cloud Run et Cloud Build (plus robuste)

Pour des applications plus complexes (APIs, modèles ML), Google Cloud Run est une option plus puissante. Le déploiement continu se fait via Cloud Build, qui surveille le repo GitHub.

Bash
# Variables de configuration (à adapter à ton projet)
PROJECT="le-wagon-project"        # Ton ID de projet Google Cloud
IMAGE="taxifare-api"              # Nom de ton image Docker
REGION="europe-west1"             # Région de déploiement
DOCKER_REPO_NAME="my-docker-repo" # Nom du repo dans Artifact Registry
TAG="0.1"                         # Tag de version

# Construire l'URI complète de l'image
IMAGE_URI=${REGION}-docker.pkg.dev/${PROJECT}/${DOCKER_REPO_NAME}/${IMAGE}:${TAG}

# Construire et pousser l'image Docker initiale
docker build -t $IMAGE_URI .   # Construire l'image localement
docker push $IMAGE_URI          # La pousser vers Artifact Registry

Configuration de Cloud Build (étapes dans la console Google Cloud) :

  1. Dans Cloud Run → créer un service depuis l'image Artifact Registry
  2. Activer "Continuously deploy new revisions from a source repository"
  3. Connecter le repo GitHub via Cloud Build
  4. Resultat : chaque push sur main → build Docker → deploy automatique

Tester que le CD fonctionne

Une fois le CD configuré, voici comment vérifier qu'il fonctionne :

Bash
# Créer une branche pour une modification de test
git checkout -b add-timestamp

# Faire une petite modification visible dans l'app
# Par exemple, ajouter un timestamp dans la réponse de l'API

# Commiter et pousser
git add . && git commit -m "Add timestamp to API response"
git push origin add-timestamp

# Créer et merger la PR sur main
# → Cloud Build détecte le merge et lance un nouveau build + deploy
# → L'app en production est mise à jour automatiquement
✅

Si tout est bien configuré, quelques minutes après le merge, ta modification est visible en production sans aucune action manuelle de ta part.

3. Erreurs classiques à éviter

Erreur 1 — Commiter sans tester localement

La CI va rattraper les erreurs, mais attendre 3 minutes de feedback GitHub à chaque oubli, c'est très lent.

Bash
# Mauvaise habitude
git add . && git commit -m "fix" && git push
# → Attendre 3 min → CI échoue → corriger → re-push → re-attendre

# Bonne habitude
make test          # Lancer les tests localement d'abord
git add . && git commit -m "fix" && git push
# → La CI est un filet de sécurité, pas le premier recours
⚠️

La CI est un filet de sécurité, pas un substitut aux tests locaux. Tester localement avant chaque push, c'est la base.

Erreur 2 — Croire que git commit déclenche la CI

Bash
# Ne déclenche PAS la CI
git commit -m "mon changement"

# Déclenche la CI
git push origin ma-branche

La CI est sur les serveurs GitHub. Elle ne peut être déclenchée que par des événements remote (push, ouverture de PR).

Erreur 3 — Merger une PR avec la CI en echec

Si les tests sont rouges et que tu merges quand même, tu introduis des bugs dans main — la branche de référence pour tout le monde.

✅

Solution : configurer une "branch protection rule" sur main dans GitHub. Cette règle empêche techniquement de merger si la CI est en echec. Settings → Branches → Add rule → "Require status checks to pass before merging".

Erreur 4 — Mettre des secrets dans le fichier YAML

Les fichiers dans .github/workflows/ sont versionnés et visibles par tout le monde (surtout sur un repo public).

Code
# DANGEREUX — la clé API est visible dans l'historique Git
env:
  API_KEY: "sk-12345-ma-super-cle-secrete"

# CORRECT — utiliser les GitHub Secrets
env:
  API_KEY: ${{ secrets.API_KEY }}
  # La valeur réelle est stockée dans GitHub Settings → Secrets and variables

Pour ajouter un secret : Settings → Secrets and variables → Actions → New repository secret.

4. Pourquoi le CI/CD change tout

AvantageSans CI/CDAvec CI/CD
Temps de livraisonDeploy manuel, lentDeploy automatique, immédiat
Qualité du codeTests oubliés = bugs en prodTests automatiques à chaque commit
Fiabilité"Ca marche chez moi"Environnement de test reproductible
Retour en arrièreDifficile, stressantFacile, chaque version est tracée
Charge mentale"Ai-je bien déployé la bonne version ?"L'automatisation s'en charge
🧠

Le CI/CD, c'est l'application du principe "automatiser ce qui peut l'être". Les humains font des erreurs sur les tâches répétitives. Les machines non. On délègue les vérifications et les déploiements à des machines, pour se concentrer sur ce qui nécessite vraiment de la réflexion.

🎯 Questions de vérification

1. Quelle est la difference entre la CI et la CD ?

La CI teste automatiquement le code à chaque push. La CD déploie automatiquement en production si les tests passent. La CI valide, la CD livre.

2. Que se passe-t-il si tu fais git commit sans git push ?

Rien du tout du côté de GitHub Actions. Le workflow ne se déclenche qu'au moment du git push, car c'est un événement remote que GitHub peut détecter.

3. Pourquoi ne faut-il jamais écrire un mot de passe ou une clé API directement dans un fichier YAML ?

Les fichiers .github/workflows/ sont versionnés dans Git et visibles dans le repo GitHub. Un secret écrit en clair serait accessible à n'importe qui. Il faut utiliser les GitHub Secrets, qui stockent les valeurs de manière chiffrée.

4. Dans quel ordre doit-on toujours mettre en place CI et CD ?

Toujours CI en premier, CD en second. Déployer sans tester (CD sans CI) risque d'envoyer du code cassé directement en production, visible par les utilisateurs.

📌 TL;DR

  • La CI (Intégration Continue) = tests automatiques déclenchés à chaque git push sur GitHub
  • La CD (Déploiement Continu) = mise en production automatique si les tests passent
  • Tout se configure avec un fichier YAML dans .github/workflows/
  • GitHub Actions orchestre la CI : checkout du code → install deps → pytest
  • Pour la CD : Streamlit Cloud (simple, gratuit) ou Google Cloud Run + Cloud Build (robuste)
  • Un commit seul ne déclenche rien : il faut un git push pour activer la CI
  • Ne jamais coller de secrets (clés API, mots de passe) dans le YAML — utiliser les GitHub Secrets
  • Ne jamais merger une PR avec la CI en echec — configurer une branch protection rule
  • L'ordre est sacré : CI d'abord, CD ensuite, jamais l'inverse
  • Analogie usine : CI = contrôle qualité, CD = expedition automatique vers le client

A. Concepts du cours

Continuous Integration

Continuous Integration (CI) : intégrer fréquemment le travail de plusieurs développeurs dans une même branche partagée, en exécutant automatiquement des tests à chaque push pour détecter les régressions tôt.

Sans CI : un développeur travaille 2 semaines sur une feature, merge dans main, casse 50 tests d'autres modules. Personne ne s'en aperçoit avant la prochaine release.

Avec CI : à chaque push, un serveur (GitHub Actions, GitLab CI, Jenkins) exécute la suite de tests. Si un test casse, le push est marqué en rouge et le merge est bloqué jusqu'à ce que ce soit corrigé.

Code
# .github/workflows/ci.yml
name: CI                              # nom affiché dans l'onglet Actions de GitHub
on: [push, pull_request]              # déclenché à chaque push OU ouverture de PR

jobs:
  test:                               # nom interne du job
    runs-on: ubuntu-latest            # machine virtuelle gratuite fournie par GitHub
    steps:
      - uses: actions/checkout@v4     # récupère le code du repo dans le runner
      - uses: actions/setup-python@v5 # installe Python (plus rapide que apt install)
        with:
          python-version: "3.11"
      - run: pip install -e ".[dev]"  # installe le package + deps dev (pytest, ruff...)
      - run: ruff check .             # lint ultra-rapide (remplace flake8 + black + isort)
      - run: mypy src/                # type checking statique
      - run: pytest tests/ --cov=src --cov-fail-under=80
      # ↑ lance les tests + calcule la couverture + échoue si < 80%
💡

Analogie : la CI, c'est comme un test antidopage avant chaque match. Même les meilleurs joueurs peuvent avoir consommé quelque chose de problématique sans s'en rendre compte. Le test attrape ça avant que ça ait des conséquences. Sans CI, vous découvrez le problème pendant le match (en production).

Continuous Delivery & Deployment

Continuous Delivery (CD) : automatiser la préparation de chaque version pour qu'elle soit déployable à tout moment. Le déploiement reste manuel (un humain valide).

Continuous Deployment (CD) : automatiser aussi le déploiement. Chaque commit qui passe la CI est automatiquement déployé en production.

CI seuleCI + Continuous DeliveryCI + Continuous Deployment

Build automatique✓✓
Déploiement staging✓✓
Déploiement productionmanuelmanuel (un clic)automatique

Les startups et les SaaS modernes (Netflix, Spotify, Vercel) pratiquent du Continuous Deployment, déployant des dizaines de fois par jour. Les domaines régulés (banques, santé) restent souvent en Continuous Delivery, avec un humain qui valide chaque release.

Code
# Exemple : déploiement auto vers Cloud Run après tests OK
deploy:
  needs: test                              # ne tourne que si le job "test" a réussi
  if: github.ref == 'refs/heads/main'      # seulement sur la branche main (pas les PR)
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: google-github-actions/auth@v2  # auth GCP via service account
      with:
        credentials_json: ${{ secrets.GCP_SA_KEY }}   # secret stocké dans GitHub Settings
    - run: gcloud run deploy api --source . --region us-central1
    # ↑ build l'image Docker + déploie Cloud Run en une commande

GitHub Actions

GitHub Actions est devenu le standard pour la CI/CD des projets open source et des startups. Configuration en YAML, exécution sur des runners GitHub (gratuits jusqu'à 2000 minutes/mois pour les repos privés).

Structure d'un workflow :

Code
name: My Workflow                # Nom affiché dans l'UI

on:                              # Quand déclencher
  push:
    branches: [main, develop]
  pull_request:
  schedule:
    - cron: "0 6 * * *"          # Tous les jours à 6h UTC
  workflow_dispatch:             # Déclenchement manuel

jobs:                            # Liste de jobs (parallèles par défaut)
  test:
    runs-on: ubuntu-latest       # Type de machine
    strategy:                    # Matrice d'environnements
      matrix:
        python: ["3.10", "3.11", "3.12"]
    steps:                       # Liste d'étapes (séquentielles)
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python }}
      - run: pip install -e .
      - run: pytest

Concepts clés :

  • uses : action réutilisable du marketplace
  • run : commande shell
  • secrets : variables sensibles stockées dans GitHub
  • needs : déclare une dépendance entre jobs
  • if : condition d'exécution
👉

Bonne pratique : commencez petit. Ajoutez seulement pytest au début. Une fois que ça marche, ajoutez ruff, puis mypy, puis le déploiement. Une CI surchargée et fragile décourage l'équipe — une CI minimale et verte est mieux.

Pre-commit hooks

Les pre-commit hooks sont des scripts qui s'exécutent localement avant chaque git commit. Ils attrapent les erreurs avant même que le commit ne parte sur GitHub.

Code
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.5.0
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.10.0
    hooks:
      - id: mypy
        additional_dependencies: [types-requests]

  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
        args: [--maxkb=1000]

  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks                # Bloque les commits contenant des secrets

Installation :

Bash
pip install pre-commit
pre-commit install
# Désormais, chaque git commit exécute ces hooks

Avantages :

  • Feedback instantané : pas besoin d'attendre la CI pour voir l'erreur
  • Cohérence : tout le monde a le même style automatiquement
  • Sécurité : gitleaks bloque les commits avec des secrets en clair
  • Économie : moins de runs CI inutiles à cause de typos
⚠️

Piège : un pre-commit qui prend > 30 secondes décourage l'équipe et finit par être contourné (git commit --no-verify). Gardez les hooks rapides : ruff (instantané), mypy si possible, gitleaks. Pas de tests lourds en pre-commit — laissez ça à la CI.

B. Compléments avancés

CI pour les pipelines ML

Tester un pipeline ML est plus complexe que tester du code classique. Cinq types de tests à considérer :

1. Tests de code : tests unitaires Python classiques (pytest).

2. Tests de données : valider que les données d'entraînement respectent un schéma.

Python
import pandera as pa

schema = pa.DataFrameSchema({
    "age": pa.Column(int, pa.Check.in_range(0, 120)),
    "income": pa.Column(float, pa.Check.greater_than(0)),
})
schema.validate(df)

3. Tests de modèle : sanity checks sur les prédictions.

Python
def test_model_baseline():
    """Le modèle doit être au moins aussi bon que de prédire la classe majoritaire."""
    score = model.score(X_test, y_test)
    baseline = (y_test == y_test.mode()[0]).mean()
    assert score > baseline + 0.05

4. Tests d'invariance : si je change un détail non significatif, la prédiction ne doit pas changer drastiquement.

Python
def test_robustness_to_capitalization():
    pred1 = model.predict(["This is great!"])
    pred2 = model.predict(["this is GREAT!"])
    assert pred1 == pred2

5. Tests de performance : la latence doit rester sous un seuil.

Python
def test_latency_below_100ms():
    start = time.perf_counter()
    model.predict(X_sample)
    assert (time.perf_counter() - start) * 1000 < 100

GitOps

GitOps est un pattern où l'état désiré de l'infrastructure est décrit dans un repo Git, et un agent (ArgoCD, Flux) maintient automatiquement l'état réel synchronisé avec ce qui est dans Git.

Avantages :

  • Audit trail : toutes les modifications sont des commits, traçables
  • Rollback : git revert pour annuler un déploiement
  • Code review : modifications d'infra passent par PR comme du code
  • Reproducibilité : recréer un environnement = re-appliquer le repo
Code
# Exemple : un fichier qui décrit le déploiement
apiVersion: apps/v1
kind: Deployment
metadata:
  name: churn-api
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: api
          image: gcr.io/my-project/churn-api:v3.2.1
          resources:
            limits:
              memory: "1Gi"
              cpu: "500m"

Avec ArgoCD/Flux, dès que vous changez v3.2.1 → v3.3.0 et committez, le cluster se met à jour automatiquement.

💡

Analogie : sans GitOps, déployer c'est appeler la cuisine pour leur dire "préparez la table 5 avec une pizza". Avec GitOps, c'est mettre à jour la commande dans le cahier de la cuisine — et la cuisine voit automatiquement qu'il faut faire une pizza pour la table 5. Le cahier (Git) est la source de vérité, pas les appels téléphoniques.

C. Questions de compréhension

Q1 : Pourquoi configurer une CI dès le premier jour d'un projet, même solo ?

💡 Voir l'indice

discipline et reproductibilité.

✅ Voir la réponse

Cinq raisons. (1) Discipline : une CI vous force à écrire des tests dès le début, ce qui produit un code plus modulaire et plus testable. Sans CI, vous procrastinez les tests "pour plus tard" qui n'arrive jamais. (2) Reproductibilité : la CI tourne dans un environnement neuf, vide. Si "ça marche en local" mais pas en CI, c'est que vous dépendez de quelque chose dans votre environnement local — bug à corriger immédiatement. (3) Documentation vivante : le fichier .github/workflows/ci.yml documente exactement comment installer et tester votre projet. Un nouveau contributeur le lit et sait quoi faire. (4) Sécurité : les linters (ruff, bandit) attrapent automatiquement des vulnérabilités courantes. (5) Trace de qualité : avoir des badges "CI passing" et "coverage 85%" sur votre README inspire confiance pour les recruteurs et les utilisateurs open source. Coût : 30 minutes de setup pour un workflow basique. Bénéfice : économisé chaque jour pendant des mois. C'est l'investissement avec le meilleur ROI en software engineering. Question d'entretien dev : "Vous configurez la CI quand sur un nouveau projet ?" — la bonne réponse est "dès le premier commit", pas "quand on aura le temps".

Q2 : Quelle est la différence entre Continuous Delivery et Continuous Deployment ?

💡 Voir l'indice

automatisation finale.

✅ Voir la réponse

Continuous Delivery : chaque commit qui passe la CI est prêt à être déployé en production, mais le déploiement reste déclenché manuellement (un humain clique sur "Deploy"). C'est l'approche prudente : un humain valide chaque release, peut décider de regrouper plusieurs commits en une seule release, ou de retarder un déploiement pour des raisons business. Continuous Deployment : chaque commit qui passe la CI est automatiquement déployé en production sans intervention humaine. C'est l'approche aggressive : Netflix, Vercel, Spotify déploient des centaines de fois par jour avec ce modèle. Conditions pour passer en Continuous Deployment : (1) tests complets et fiables (aucun flaky test), (2) monitoring + rollback automatique, (3) feature flags pour activer/désactiver une feature sans redéploiement, (4) discipline d'équipe — un commit cassé en main est cassé en prod. Recommandation : commencer par Continuous Delivery (un clic), passer à Continuous Deployment une fois la culture d'équipe et l'infra mûres. Pour un domaine régulé (banque, santé), Delivery suffit souvent. Question d'entretien DevOps : la réponse exhaustive distingue clairement Delivery vs Deployment.

Q3 : Vous voulez ajouter un test de performance dans votre CI : la latence du modèle doit rester < 100ms. Comment l'implémenter sans casser à chaque variation aléatoire ?

💡 Voir l'indice

moyenne et tolerance.

✅ Voir la réponse

Trois précautions essentielles. (1) Mesurer plusieurs runs : un seul run peut être lent à cause d'un GC, d'une page swap, ou d'un voisin bruyant sur le runner CI. Mesurer 100 runs et prendre la médiane ou la moyenne après warm-up.

Python
import time

def test_latency():
    # Warm-up
    for _ in range(10):
        model.predict(X_sample)

    # Mesure
    durations = []
    for _ in range(100):
        start = time.perf_counter()
        model.predict(X_sample)
        durations.append((time.perf_counter() - start) * 1000)

    p95 = sorted(durations)[95]
    assert p95 < 100, f"p95 latency was {p95:.2f}ms, expected < 100ms"

(2) Tolérance dans le seuil : ne pas mettre exactement 100ms si la moyenne tourne autour de 95ms — vous aurez des faux positifs. Mettre 150ms pour avoir de la marge. (3) Runner stable : utiliser un type d'instance constant (ex : ubuntu-latest). Les VMs cloud ont des perfs variables. Ou mieux, utiliser un self-hosted runner dédié pour les tests de performance. (4) Bench dans un job séparé : ne pas bloquer la CI sur les perfs si elles sont lentes ailleurs (un test fonctionnel qui passe ne devrait pas dépendre d'une CI saturée). (5) Tracker plutôt qu'asserter : pour des optimisations gradient, mieux que asserter, c'est de tracker la latence dans le temps (avec un dashboard) et alerter sur une dégradation > 20%. C'est plus robuste qu'un seuil absolu. Question d'entretien CI : "Comment testez-vous la performance dans la CI ?" — la réponse exhaustive révèle l'expérience pratique.
