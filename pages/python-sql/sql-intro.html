📋 Cheatsheet — SQL

SQL
-- ── STRUCTURE COMPLÈTE (ordre obligatoire) ───────────
SELECT col1, col2, COUNT(id) AS total   -- colonnes à afficher
FROM "Table" AS t                        -- table source
JOIN "Autre" AS a ON t.id = a.table_id  -- jointure
WHERE t.colonne = 'valeur'              -- filtre AVANT agrégation
GROUP BY t.colonne                       -- regroupement
HAVING COUNT(id) > 100                  -- filtre APRÈS agrégation
ORDER BY total DESC                      -- tri
LIMIT 10;                               -- limitation du nombre de résultats

-- ── PROJECTION ───────────────────────────────────────
SELECT *                   -- toutes les colonnes
SELECT col1, col2          -- colonnes spécifiques
SELECT col1 AS alias       -- renommer une colonne

-- ── FILTRES WHERE ────────────────────────────────────
WHERE col = 'valeur'               -- égalité exacte
WHERE col != 'valeur'              -- différent
WHERE col > 100                    -- comparaison numérique
WHERE col IN (1, 2, 3)             -- parmi une liste de valeurs
WHERE col LIKE 'John %'            -- % = 0 ou plusieurs caractères
WHERE col IS NULL                  -- valeur manquante
WHERE col IS NOT NULL              -- valeur présente
WHERE cond1 AND cond2              -- les deux conditions vraies
WHERE cond1 OR cond2               -- au moins une condition vraie

-- ── AGRÉGATIONS ──────────────────────────────────────
COUNT(id)        -- nombre de lignes non nulles
SUM(col)         -- somme des valeurs
AVG(col)         -- moyenne des valeurs
MIN(col)         -- valeur minimale
MAX(col)         -- valeur maximale

-- ── CASE WHEN ────────────────────────────────────────
CASE
    WHEN condition1 THEN 'résultat1'   -- si condition1 vraie
    WHEN condition2 THEN 'résultat2'   -- sinon si condition2 vraie
    ELSE 'défaut'                      -- sinon
END AS nom_colonne                     -- nom de la colonne calculée

-- ── WHERE vs HAVING ──────────────────────────────────
WHERE  -- filtre les lignes AVANT GROUP BY (sur données brutes)
HAVING -- filtre les agrégats APRÈS GROUP BY (sur résultats calculés)

⚠️ Récapitulatif — Erreurs clés à éviter

❌

Filtrer avec WHERE sur un agrégat au lieu de HAVING

SQL
SELECT COUNT(id), country_id
FROM "Match"
GROUP BY country_id
WHERE COUNT(id) > 1000;  -- ❌ SyntaxError : WHERE s'applique avant GROUP BY

HAVING COUNT(id) > 1000; -- ✅
❌

Oublier GROUP BY quand on utilise une agrégation

SQL
SELECT country_id, COUNT(id)
FROM "Match";  -- ❌ retourne une seule ligne incohérente

SELECT country_id, COUNT(id)
FROM "Match"
GROUP BY country_id;  -- ✅
❌

Confondre JOIN (INNER) et résultats manquants

SQL
-- Un INNER JOIN exclut les lignes sans correspondance des deux côtés
-- Si un pays n'a pas de ligue, il disparaît du résultat
-- ✅ Utilise LEFT JOIN pour garder tous les enregistrements du côté gauche
❌

Oublier UPPER() ou LIKE insensible à la casse

SQL
WHERE player_name LIKE 'john %'              -- ❌ peut rater "John" ou "JOHN"
WHERE UPPER(player_name) LIKE 'JOHN %'       -- ✅
❌

Utiliser = au lieu de LIKE pour les recherches partielles

SQL
WHERE player_name = 'John'         -- ❌ cherche exactement "John"
WHERE player_name LIKE 'John %'    -- ✅ cherche "John ..." (prénom)

9) Groupement : GROUP BY + HAVING

Regrouper par une colonne et filtrer les agrégats.

🤔 Nombre de matchs par pays

SQL
SELECT COUNT(m.id) AS match_count,
	m.country_id
FROM "Match" AS m
GROUP BY m.country_id;

🤔 Trier les résultats

SQL
SELECT COUNT(m.id) AS match_count,
	m.country_id
FROM "Match" AS m
GROUP BY m.country_id
ORDER BY match_count DESC;

🤔 Pays avec plus de 3000 matchs

SQL
SELECT COUNT(m.id) AS match_count,
	m.country_id
FROM "Match" AS m
GROUP BY m.country_id
HAVING match_count >= 3000
ORDER BY match_count DESC;

WHERE vs HAVING

  • WHERE filtre les lignes avant l'agrégation.
  • HAVING filtre les résultats après l'agrégation.

🤔 Pays avec plus de 700 matchs nuls

SQL
SELECT COUNT(m.id) AS match_count,
	m.country_id
FROM "Match" AS m
WHERE m.home_team_goal = m.away_team_goal
GROUP BY m.country_id
HAVING match_count >= 700
ORDER BY match_count DESC;

🤔 Compter les matchs par résultat (domicile, nul, extérieur)

SQL
SELECT COUNT(m.id) AS outcome_count,
	CASE
		WHEN m.home_team_goal > m.away_team_goal THEN 'home_win'
		WHEN m.home_team_goal = m.away_team_goal THEN 'draw'
		ELSE 'away_win'
	END AS outcome
FROM "Match" AS m
GROUP BY outcome
ORDER BY outcome_count DESC;

10) Joindre plusieurs tables : JOIN

Croiser des tables via leurs clés.

🤔 Ligues avec le nom de leur pays

SQL
SELECT League.name,
	Country.name
FROM League
JOIN Country
	ON League.country_id = Country.id;

🤔 Nombre de matchs par ligue (avec pays)

SQL
SELECT League.id,
	League.name AS league_name,
	COUNT(m.id) AS match_count,
	Country.name AS country_name
FROM "Match" AS m
JOIN League
	ON m.league_id = League.id
JOIN Country
	ON League.country_id = Country.id
GROUP BY League.id
ORDER BY match_count DESC,
	country_name ASC;

🧭 Introduction

SQL (prononcé "S-Q-L" ou "sequel") est le langage universel pour interroger des bases de données relationnelles. Il est utilisé partout : applications web, analyses de données, data science, finance, e-commerce... Si tu travailles avec des données structurées un jour, tu auras besoin de SQL.

Points clés à retenir

  • SQL signifie Structured Query Language — c'est un langage de requête, pas un langage de programmation classique.
  • Il permet de lire, filtrer, trier et agréger des données stockées dans des tables.
  • Il existe plusieurs "dialectes" (SQLite, PostgreSQL, MySQL...) mais la syntaxe de base est la même.
  • SQL est déclaratif : tu décris ce que tu veux, pas comment le calculer.
  • Les requêtes s'écrivent dans un ordre précis et obligatoire.

Analogie : une table = une feuille Excel

Imagine que ta base de données est un classeur Excel avec plusieurs onglets. Chaque onglet est une table. Chaque colonne a un nom (ex: id, nom, pays), et chaque ligne est un enregistrement (une entrée de données).

SQL te permet de poser des questions à ce classeur en langage presque humain : "Donne-moi tous les joueurs dont la taille est supérieure à 2 mètres, triés par poids."

1. Types de bases de données

Il existe deux grandes familles de bases de données.

Bases relationnelles (SQL)

Les données sont organisées en tables liées entre elles. C'est le modèle le plus répandu pour les données structurées.

Exemples : SQLite, PostgreSQL, MySQL, Microsoft SQL Server, Oracle.

Bases non relationnelles (NoSQL)

Pour des données moins structurées ou des besoins très spécifiques (performance, flexibilité).

  • Document : MongoDB, CouchDB — stocke des documents JSON
  • Clé-valeur : Redis, DynamoDB — comme un dictionnaire géant
  • Graphe : Neo4J — pour des réseaux de relations complexes
💡

Dans ce cours, on se concentre sur SQL avec SQLite. C'est la base idéale pour apprendre : simple, sans installation serveur, et identique dans sa syntaxe aux autres systèmes SQL.

2. Rappels : les bases relationnelles

Une base de données relationnelle organise les données en tables. Voici les concepts fondamentaux :

ConceptDéfinition simple
TableUn tableau avec des lignes et des colonnes (= une feuille Excel)
LigneUn enregistrement (ex: un joueur, un match)
ColonneUn attribut (ex: nom, poids, date)
Clé primaireUn identifiant unique par ligne (souvent appelé id)
Clé étrangèreUn id qui fait référence à une autre table, pour créer un lien

Exemple concret : Dans une base de données de football, on peut avoir :

  • Une table Player avec les colonnes id, player_name, height, weight
  • Une table Match avec les colonnes id, season, country_id, home_team_goal, away_team_goal
  • Une table Country avec les colonnes id, name

La colonne country_id dans Match est une clé étrangère qui pointe vers id dans Country. C'est ce lien qui permet de retrouver le nom du pays d'un match.

🧠

Les clés étrangères évitent la duplication : au lieu de répéter "France" dans chaque ligne de match, on stocke juste le chiffre 4769. Le nom complet est disponible dans la table Country.

3. SQLite — une base de données dans un seul fichier

SQLite est une base de données ultra-légère : toute la base est stockée dans un seul fichier sur ton ordinateur (extension .db ou .sqlite).

Avantages pour apprendre :

  • Aucun serveur à installer ou configurer
  • On peut ouvrir le fichier directement dans des outils en ligne
  • La syntaxe SQL est identique aux autres systèmes

Pour pratiquer en ligne sans rien installer, tu peux utiliser SQLiteOnline.com et y charger une base .db.

✅

Dans ce cours, on utilise la base de données "European Soccer Database" disponible sur Kaggle, qui contient des données de matchs de football européens.

4. Projection : choisir ses colonnes avec SELECT

La projection consiste à choisir quelles colonnes tu veux voir dans ton résultat. C'est le rôle de SELECT.

Récupérer toutes les colonnes

SQL
SELECT *          -- l'étoile * signifie "toutes les colonnes"
FROM Country;     -- depuis la table Country

Résultat : toutes les lignes de la table Country, avec toutes ses colonnes.

Récupérer des colonnes spécifiques

On veut uniquement id, season, stage et date des matchs, pas toutes les 100 colonnes de la table.

SQL
SELECT "Match".id,       -- la colonne id de la table Match
       "Match".season,   -- la colonne season
       "Match".stage,    -- la colonne stage
       "Match".date      -- la colonne date
FROM "Match";            -- depuis la table Match
💡

Les guillemets doubles autour de "Match" sont nécessaires car Match est un mot qui pourrait être confondu avec un mot-clé SQL selon le dialecte. Par prudence, on met les guillemets pour indiquer que c'est le nom d'une table.

Utiliser des alias pour simplifier l'écriture

Répéter "Match". devant chaque colonne est verbeux. On peut donner un alias (surnom) à la table avec AS :

SQL
SELECT m.id,       -- m est l'alias de "Match"
       m.season,
       m.stage,
       m.date
FROM "Match" AS m; -- on définit m comme alias de la table Match

C'est exactement la même requête, juste plus lisible !

🧠

Les alias sont une bonne pratique dès que tu travailles avec plusieurs tables. m pour Match, p pour Player, c pour Country... ça rend le code beaucoup plus clair.

5. Sélection : filtrer les lignes avec WHERE

WHERE te permet de ne garder que les lignes qui correspondent à une condition. C'est comme filtrer une feuille Excel, mais en code.

Filtrer sur une valeur exacte

On veut uniquement les matchs joués en France (dont l'id pays est 4769) :

SQL
SELECT *               -- toutes les colonnes
FROM "Match" AS m      -- depuis la table Match (alias m)
WHERE m.country_id = 4769;  -- ne garder que les lignes où country_id vaut 4769

Combiner plusieurs conditions avec OR

On veut les matchs de Belgique (id = 1) OU d'Angleterre (id = 1729) :

SQL
SELECT *
FROM "Match" AS m
WHERE m.country_id = 1        -- condition 1 : Belgique
   OR m.country_id = 1729;    -- OU condition 2 : Angleterre

Alternative plus lisible avec IN

Quand tu veux tester plusieurs valeurs pour une même colonne, IN est plus pratique :

SQL
SELECT *
FROM "Match" AS m
WHERE m.country_id IN (1, 1729);  -- dans la liste (1 ou 1729)

Ce code fait exactement la même chose que le OR ci-dessus, mais est plus lisible.

Recherche partielle avec LIKE

On veut tous les joueurs dont le prénom est John (John Smith, John Doe...) :

SQL
SELECT *
FROM Player
WHERE UPPER(Player.player_name) LIKE 'JOHN %';
-- UPPER() met tout en majuscules pour éviter les problèmes de casse
-- LIKE 'JOHN %' = commence par "JOHN " suivi de n'importe quoi
💡

Le % dans LIKE est un wildcard (joker) : il remplace zéro ou plusieurs caractères. Donc 'JOHN %' correspond à "JOHN SMITH", "JOHN DOE", "JOHN A. JOHNSON", etc. Le _ (underscore) remplace exactement un seul caractère.

⚠️

Ne jamais utiliser = pour une recherche partielle. WHERE player_name = 'John' cherche exactement la chaîne "John" et rien d'autre. Utilise LIKE pour les recherches avec jokers.

Récapitulatif des opérateurs de filtrage

SQL
WHERE col = 'valeur'       -- égalité exacte
WHERE col != 'valeur'      -- différent de
WHERE col > 100            -- supérieur à (aussi <, >=, <=)
WHERE col IN (1, 2, 3)     -- appartient à une liste
WHERE col LIKE 'Jo%'       -- commence par "Jo" (recherche partielle)
WHERE col IS NULL          -- valeur manquante (vide)
WHERE col IS NOT NULL      -- valeur présente (pas vide)
WHERE cond1 AND cond2      -- les deux conditions doivent être vraies
WHERE cond1 OR cond2       -- au moins une condition doit être vraie

6. Tri et limitation : ORDER BY et LIMIT

Trier les résultats

ORDER BY trie les résultats selon une colonne. DESC = ordre décroissant, ASC = ordre croissant (par défaut).

On veut les 10 joueurs les plus lourds :

SQL
SELECT *                         -- toutes les colonnes
FROM Player                      -- depuis la table Player
ORDER BY Player.weight DESC      -- trier par poids, du plus lourd au plus léger
LIMIT 10;                        -- ne garder que les 10 premiers résultats
💡

LIMIT est très utile en pratique pour ne pas afficher des milliers de lignes d'un coup. C'est comme dire "montre-moi juste les 10 premiers résultats".

7. Agrégations : calculer des statistiques

Les fonctions d'agrégation permettent de calculer une valeur à partir de plusieurs lignes : compter, sommer, faire une moyenne...

FonctionCe qu'elle fait
COUNT(col)Compte le nombre de lignes non nulles
SUM(col)Additionne toutes les valeurs
AVG(col)Calcule la moyenne
MIN(col)Retourne la plus petite valeur
MAX(col)Retourne la plus grande valeur

Compter des lignes

Combien de joueurs mesurent au moins 2 mètres (200 cm) ?

SQL
SELECT COUNT(Player.id)   -- compter le nombre de lignes (par leur id)
FROM Player               -- depuis la table Player
WHERE Player.height >= 200;  -- en ne gardant que ceux >= 200 cm

Résultat : un seul chiffre, par exemple 42.

Calculer une moyenne

Quel est le poids moyen de ces joueurs de grande taille ?

SQL
SELECT AVG(Player.weight)    -- calculer la moyenne de la colonne weight
FROM Player
WHERE Player.height >= 200;  -- uniquement les joueurs >= 200 cm

Faire un calcul dans la requête

Le poids des joueurs est stocké en livres (données américaines). On peut convertir en kilogrammes directement dans la requête :

SQL
SELECT AVG(Player.weight) / 2.20462   -- diviser la moyenne par 2.20462 pour avoir des kg
FROM Player
WHERE Player.height >= 200;
🧠

SQL peut faire des calculs mathématiques directement dans le SELECT. Les opérateurs +, -, *, / fonctionnent comme en arithmétique classique.

8. Groupement : GROUP BY et HAVING

GROUP BY est l'outil puissant qui permet de calculer des agrégations par groupe. Par exemple : compter les matchs par pays, calculer la moyenne des buts par saison, etc.

Compter par groupe

Combien de matchs a-t-on par pays ?

SQL
SELECT COUNT(m.id) AS match_count,   -- compter les matchs, nommer la colonne "match_count"
       m.country_id                   -- afficher aussi l'id du pays
FROM "Match" AS m                     -- depuis la table Match
GROUP BY m.country_id;               -- créer un groupe par valeur unique de country_id

Résultat : une ligne par pays, avec le nombre de matchs associé.

Trier les groupes

SQL
SELECT COUNT(m.id) AS match_count,
       m.country_id
FROM "Match" AS m
GROUP BY m.country_id
ORDER BY match_count DESC;   -- trier par nombre de matchs, du plus grand au plus petit

Filtrer les groupes avec HAVING

On veut uniquement les pays avec plus de 3000 matchs :

SQL
SELECT COUNT(m.id) AS match_count,
       m.country_id
FROM "Match" AS m
GROUP BY m.country_id
HAVING match_count >= 3000    -- filtrer les groupes APRÈS agrégation
ORDER BY match_count DESC;

WHERE vs HAVING — la distinction essentielle

C'est l'une des confusions les plus fréquentes chez les débutants.

WHEREHAVING
Quand ?Avant le GROUP BYAprès le GROUP BY
Sur quoi ?Sur les lignes brutesSur les résultats agrégés
ExempleWHERE country_id = 1HAVING COUNT(id) > 1000

Exemple combinant les deux : Pays avec plus de 700 matchs nuls (score identique à domicile et à l'extérieur) :

SQL
SELECT COUNT(m.id) AS match_count,
       m.country_id
FROM "Match" AS m
WHERE m.home_team_goal = m.away_team_goal  -- WHERE filtre les lignes AVANT le groupe
GROUP BY m.country_id                       -- regrouper par pays
HAVING match_count >= 700                  -- HAVING filtre les groupes APRÈS agrégation
ORDER BY match_count DESC;
⚠️

Erreur classique : utiliser WHERE sur un résultat agrégé. WHERE COUNT(id) > 1000 provoque une erreur de syntaxe. Pour filtrer sur un agrégat, c'est toujours HAVING.

⚠️

Autre erreur classique : oublier le GROUP BY quand on utilise une agrégation avec une autre colonne. Si tu sélectionnes country_id et COUNT(id), tu dois avoir un GROUP BY country_id, sinon SQL ne sait pas comment grouper les données.

Classer des résultats avec CASE WHEN

CASE WHEN est l'équivalent SQL du if/elif/else. Il permet de créer une nouvelle colonne calculée selon des conditions.

Compter les matchs par type de résultat (victoire domicile, nul, victoire extérieur) :

SQL
SELECT COUNT(m.id) AS outcome_count,   -- compter le nombre de matchs par résultat
       CASE
           WHEN m.home_team_goal > m.away_team_goal THEN 'home_win'   -- victoire domicile
           WHEN m.home_team_goal = m.away_team_goal THEN 'draw'       -- match nul
           ELSE 'away_win'                                             -- victoire extérieur
       END AS outcome                  -- nommer cette colonne calculée "outcome"
FROM "Match" AS m
GROUP BY outcome                       -- grouper par valeur de outcome
ORDER BY outcome_count DESC;
💡

CASE WHEN ... THEN ... ELSE ... END se lit comme : "Si telle condition, alors telle valeur. Sinon si... Sinon (par défaut)..."

9. Jointures : relier plusieurs tables avec JOIN

Dans une base de données bien conçue, les informations sont réparties entre plusieurs tables. Pour combiner ces informations, on utilise JOIN.

Principe du JOIN

Un JOIN fusionne deux tables en faisant correspondre des lignes via leurs clés. La syntaxe est :

SQL
JOIN AutreTable ON table1.cle_etrangere = AutreTable.cle_primaire

Exemple simple : ligues avec le nom de leur pays

La table League contient country_id mais pas le nom du pays. Ce nom est dans la table Country.

SQL
SELECT League.name,       -- nom de la ligue (depuis League)
       Country.name       -- nom du pays (depuis Country)
FROM League               -- table principale
JOIN Country              -- table à joindre
    ON League.country_id = Country.id;  -- lien : country_id de League = id de Country

SQL va "assembler" chaque ligne de League avec la ligne de Country dont l'id correspond.

Exemple avancé : plusieurs jointures enchaînées

On veut le nombre de matchs par ligue, avec le nom de la ligue et le nom du pays :

SQL
SELECT League.id,
       League.name AS league_name,         -- nom de la ligue (alias)
       COUNT(m.id) AS match_count,          -- nombre de matchs dans cette ligue
       Country.name AS country_name         -- nom du pays (alias)
FROM "Match" AS m                          -- table principale : les matchs
JOIN League                                -- joindre la table League
    ON m.league_id = League.id             -- via league_id
JOIN Country                               -- joindre aussi la table Country
    ON League.country_id = Country.id      -- via country_id
GROUP BY League.id                         -- grouper par ligue
ORDER BY match_count DESC,                 -- trier par nombre de matchs (décroissant)
         country_name ASC;                 -- puis par nom de pays (alphabétique)
⚠️

Un JOIN (aussi appelé INNER JOIN) exclut les lignes sans correspondance des deux côtés. Si un pays n'a pas de ligue, il disparaît du résultat. Pour garder toutes les lignes du côté gauche même sans correspondance, utilise LEFT JOIN.

🧠

On peut enchaîner autant de JOIN que nécessaire. Chaque JOIN ... ON ... ajoute une nouvelle table à la requête. L'important est de bien préciser la condition de jointure (ON) pour chaque table ajoutée.

10. L'ordre obligatoire des clauses SQL

En SQL, les clauses doivent être écrites dans un ordre précis. Tu ne peux pas les permuter librement.

SQL
SELECT col1, col2, COUNT(id) AS total   -- 1. Colonnes à afficher (et agrégations)
FROM "Table" AS t                        -- 2. Table source
JOIN "Autre" AS a ON t.id = a.table_id  -- 3. Jointures
WHERE t.colonne = 'valeur'              -- 4. Filtres sur les lignes brutes
GROUP BY t.colonne                       -- 5. Regroupement
HAVING COUNT(id) > 100                  -- 6. Filtres sur les agrégats
ORDER BY total DESC                      -- 7. Tri
LIMIT 10;                               -- 8. Limite du nombre de résultats
💡

Mémo pour retenir l'ordre : Select From Join Where Group Having Order Limit — "SF-JW-GHOL".

🧠

Important : l'ordre dans lequel SQL exécute les clauses est différent de l'ordre dans lequel on les écrit. En interne, SQL fait d'abord le FROM et les JOIN, puis le WHERE, puis le GROUP BY, puis le HAVING, puis le SELECT, et enfin le ORDER BY et LIMIT. C'est pourquoi on ne peut pas utiliser un alias défini dans SELECT dans un WHERE : le WHERE est exécuté avant le SELECT !

📌 TL;DR

  • SQL est le langage pour interroger des bases de données relationnelles (tables liées entre elles par des clés).
  • SELECT choisit les colonnes à afficher ; FROM indique la table source.
  • WHERE filtre les lignes selon une condition (avant toute agrégation).
  • ORDER BY trie les résultats ; LIMIT en restreint le nombre.
  • COUNT, SUM, AVG, MIN, MAX calculent des statistiques sur un ensemble de lignes.
  • GROUP BY regroupe les lignes par valeur unique d'une colonne pour appliquer des agrégations par groupe.
  • HAVING filtre les groupes après agrégation (l'équivalent de WHERE mais pour les résultats de GROUP BY).
  • JOIN ... ON ... fusionne deux tables en faisant correspondre leurs clés (clé étrangère = clé primaire).
  • L'ordre des clauses est obligatoire : SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT.
  • Erreur classique à éviter : utiliser WHERE sur un agrégat — utilise HAVING à la place.

A. Concepts du cours

Le modèle relationnel en 2 minutes

Une base relationnelle organise les données en tables (relations). Chaque table est un ensemble de lignes (tuples) qui partagent les mêmes colonnes (attributs). Une clé primaire identifie de manière unique chaque ligne. Les tables sont reliées entre elles par des clés étrangères qui pointent vers la clé primaire d'une autre table.

SQL
-- Table users
| id | name    | email           |
|----|---------|-----------------|
| 1  | Alice   | alice@x.com     |
| 2  | Bob     | bob@x.com       |

-- Table orders
| id | user_id | product   | amount |
|----|---------|-----------|--------|
| 10 | 1       | Laptop    | 1200   |
| 11 | 1       | Mouse     | 25     |
| 12 | 2       | Keyboard  | 80     |

orders.user_id est une clé étrangère qui référence users.id. C'est cette structure qui permet d'éviter la duplication : on ne stocke pas le nom d'Alice dans chaque ligne de orders, juste son id.

💡

Analogie : un tableur Excel avec un seul onglet est plat. Une base relationnelle est comme un tableur avec plusieurs onglets connectés par VLOOKUP — sauf que les liens sont déclarés explicitement, vérifiés à l'écriture (contrainte d'intégrité), et optimisés à la lecture (via des index).

SELECT, WHERE, ORDER BY, LIMIT

La requête fondamentale :

SQL
SELECT name, email           -- Colonnes à renvoyer
FROM users                   -- Table source
WHERE name LIKE 'A%'         -- Filtre sur les lignes
ORDER BY name ASC            -- Tri du résultat
LIMIT 10;                    -- Ne garde que les 10 premières

Décryptage de l'ordre logique d'évaluation (différent de l'ordre d'écriture !) :

  1. FROM : choisit la table source
  2. WHERE : filtre les lignes
  3. SELECT : choisit les colonnes à renvoyer
  4. ORDER BY : trie le résultat
  5. LIMIT : ne garde que les N premières

Cet ordre est crucial à comprendre : un alias de colonne défini dans SELECT n'est pas accessible dans WHERE (parce que WHERE est évalué avant), mais l'est dans ORDER BY.

Opérateurs courants :

SQL
WHERE age > 18
WHERE age BETWEEN 18 AND 65
WHERE city IN ('Paris', 'London', 'Berlin')
WHERE name LIKE 'A%'         -- commence par A
WHERE name LIKE '%son'        -- finit par "son"
WHERE name LIKE '_lice'       -- 5 caractères, finit par "lice" (le _ = un caractère)
WHERE email IS NULL           -- attention : pas '=' NULL !

Agrégations et GROUP BY

Les fonctions d'agrégation calculent une statistique sur un groupe de lignes :

SQL
SELECT COUNT(*) AS total,
       AVG(amount) AS avg_order,
       SUM(amount) AS total_revenue
FROM orders;
-- total=3, avg_order=435, total_revenue=1305

GROUP BY divise la table en sous-groupes avant d'appliquer l'agrégation :

SQL
SELECT user_id,
       COUNT(*) AS num_orders,
       SUM(amount) AS total_spent
FROM orders
GROUP BY user_id
HAVING SUM(amount) > 100;
-- user_id=1 : 2 commandes, 1225€ (gardé)
-- user_id=2 : 1 commande, 80€ (filtré car < 100)

HAVING filtre les groupes (après agrégation), WHERE filtre les lignes (avant). C'est une distinction qui revient à chaque entretien data.

SQL
-- ❌ Mauvais (SUM dans le WHERE)
SELECT user_id, SUM(amount)
FROM orders
WHERE SUM(amount) > 100        -- Erreur SQL : agrégat interdit dans WHERE
GROUP BY user_id;

-- ✓ Bon
SELECT user_id, SUM(amount)
FROM orders
GROUP BY user_id
HAVING SUM(amount) > 100;
⚠️

Piège classique : oublier GROUP BY quand on mélange agrégat et colonne non-agrégée. SELECT user_id, COUNT(*) FROM orders; plante (en SQL strict) parce que vous demandez user_id mais sans dire comment l'agréger. Règle : toutes les colonnes non-agrégées du SELECT doivent être dans le GROUP BY.

JOIN entre tables

SQL
-- INNER JOIN : seulement les lignes qui matchent dans LES DEUX tables
SELECT u.name, o.product, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
-- Alice | Laptop    | 1200
-- Alice | Mouse     | 25
-- Bob   | Keyboard  | 80
SQL
-- LEFT JOIN : TOUTES les lignes de gauche + correspondances à droite
SELECT u.name, o.product
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
-- Alice | Laptop
-- Alice | Mouse
-- Bob   | Keyboard
-- Charlie | NULL        -- Inclus car LEFT JOIN, même sans commande

Les quatre types de jointures :

TypeGardeCas d'usage
INNER JOINLignes qui existent dans les 2 tablesFiltrage strict
LEFT JOINToutes les lignes de gauche + matches"Tous les users avec leurs commandes éventuelles"
RIGHT JOINToutes les lignes de droite + matchesRare, on inverse plutôt l'ordre des tables
FULL OUTER JOINToutes les lignes des deuxComparaison complète
💡

Analogie : INNER JOIN est l'intersection (∩), LEFT JOIN est la table de gauche entière + ce qui matche à droite, FULL OUTER JOIN est l'union (∪). Si vous dessinez les diagrammes de Venn, vous ne vous tromperez plus jamais.

B. Compléments avancés

NULL et logique trois-valuée

NULL en SQL signifie "valeur inconnue ou absente", pas "vide" ni "zéro". Conséquence : la logique booléenne devient ternaire (TRUE, FALSE, UNKNOWN).

SQL
SELECT NULL = NULL;           -- NULL (pas TRUE !)
SELECT NULL = 0;              -- NULL
SELECT NULL <> NULL;          -- NULL
SELECT NULL IS NULL;          -- TRUE — la seule façon correcte
SELECT NULL IS NOT NULL;      -- FALSE

Cette propriété fait des ravages dans les WHERE :

SQL
-- Compte les users dont email est différent de 'admin@x.com'
SELECT COUNT(*) FROM users WHERE email <> 'admin@x.com';
-- Ce COUNT EXCLUT les users dont email IS NULL !

Pour inclure aussi les NULL :

SQL
SELECT COUNT(*) FROM users
WHERE email <> 'admin@x.com' OR email IS NULL;
⚠️

Question d'entretien data classique : "Pourquoi WHERE col != X peut-il donner un résultat différent de NOT (col = X) en SQL ?" Réponse : à cause des NULL. Les deux expressions renvoient NULL pour les lignes où col est NULL, et le WHERE exclut les lignes NULL. Conclusion : ces deux requêtes excluent silencieusement les NULL — si vous les voulez, ajoutez OR col IS NULL.

Sous-requêtes vs CTE

Une sous-requête est une SELECT imbriquée dans une autre :

SQL
-- Top 5 users par dépenses totales
SELECT * FROM users
WHERE id IN (
    SELECT user_id FROM orders
    GROUP BY user_id
    ORDER BY SUM(amount) DESC
    LIMIT 5
);

Les sous-requêtes deviennent vite illisibles dès qu'elles s'imbriquent. La CTE (Common Table Expression, syntaxe WITH) est plus claire :

SQL
WITH big_spenders AS (
    SELECT user_id, SUM(amount) AS total
    FROM orders
    GROUP BY user_id
    HAVING SUM(amount) > 1000
)
SELECT u.name, b.total
FROM users u
INNER JOIN big_spenders b ON u.id = b.user_id
ORDER BY b.total DESC;

Lire ça : "construis big_spenders (une vue temporaire), puis fais le SELECT final dessus". Cette discipline rend les requêtes complexes maintenables.

Index : pourquoi vos requêtes sont lentes

Une table sans index force la base à parcourir toutes les lignes pour filtrer (full scan). Sur 10 millions de lignes, c'est plusieurs secondes par requête.

Un index est une structure auxiliaire (généralement un B-tree) qui maintient les valeurs d'une colonne triées avec un pointeur vers la ligne correspondante. Avec un index sur users.email, une requête WHERE email = 'alice@x.com' se fait en $O(\log n)$ au lieu de $O(n)$.

SQL
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_orders_user_id ON orders(user_id);

Les bonnes pratiques :

  • Indexer les clés étrangères (presque toujours)
  • Indexer les colonnes utilisées dans WHERE et ORDER BY fréquents
  • Ne pas trop indexer : chaque index ralentit les INSERT/UPDATE/DELETE
  • Index composites : CREATE INDEX idx ON orders(user_id, created_at) pour des requêtes qui filtrent sur les deux

Pour comprendre ce que la base fait :

SQL
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 42;
-- Cherchez "Index Scan" (bon) ou "Seq Scan" (potentiellement mauvais)
👉

Bonne pratique : avant d'indexer, mesurez. Avant d'optimiser, comprenez le plan d'exécution. La majorité des "requêtes lentes" en data science sont causées par UNE table mal indexée. L'ajout d'un seul index peut multiplier les perfs par 1000.

C. Questions de compréhension

Q1 : Vous voulez compter les utilisateurs uniques qui ont passé au moins une commande. Quelle est la différence entre SELECT COUNT(user_id), COUNT(DISTINCT user_id), et COUNT(*) ?

💡 Voir l'indice

NULL et duplicates.

✅ Voir la réponse
  • COUNT(*) compte toutes les lignes, y compris celles avec des NULL.
  • COUNT(user_id) compte les lignes où user_id n'est PAS NULL (mais compte les doublons).
  • COUNT(DISTINCT user_id) compte les valeurs uniques non-NULL.

Pour la question posée — "utilisateurs uniques avec au moins une commande" — c'est COUNT(DISTINCT user_id) qui est correct, parce qu'un utilisateur avec 5 commandes ne doit compter que pour 1.

SQL
SELECT COUNT(DISTINCT user_id) FROM orders;

Sur de très grandes tables, COUNT(DISTINCT) peut être lent ; on utilise alors HyperLogLog (approximation probabiliste) — Snowflake et BigQuery exposent APPROX_COUNT_DISTINCT qui est ~100x plus rapide pour 0.5% d'erreur :

SQL
SELECT APPROX_COUNT_DISTINCT(user_id) FROM orders;  -- BigQuery, Snowflake

Q2 : Quelle est la différence entre INNER JOIN et LEFT JOIN, et quand l'un est-il préférable à l'autre ?

💡 Voir l'indice

Sémantique vs perte de données.

✅ Voir la réponse
  • INNER JOIN ne garde que les lignes qui ont une correspondance dans les deux tables. Si un utilisateur n'a aucune commande, il disparaît du résultat.
  • LEFT JOIN garde toutes les lignes de la table de gauche, et remplit avec des NULL quand il n'y a pas de correspondance à droite.

Choisissez LEFT JOIN quand la disparition d'une ligne serait un bug métier — par exemple "liste tous les utilisateurs avec leur nombre de commandes" doit inclure les utilisateurs à 0 :

SQL
SELECT u.name, COUNT(o.id) AS num_orders
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.name;
-- Charlie | 0   ← inclus car LEFT JOIN

Choisissez INNER JOIN quand vous voulez explicitement filtrer ceux sans correspondance — par exemple "liste les commandes avec leur utilisateur connu".

Petit piège : un LEFT JOIN suivi d'un WHERE right_table.col IS NOT NULL est sémantiquement équivalent à un INNER JOIN, mais souvent mal optimisé par la base. Préférez INNER JOIN directement.

Q3 : Vous avez une requête SELECT * FROM logs WHERE created_at > '2024-01-01' qui prend 10 secondes. Comment diagnostiquer et accélérer ?

💡 Voir l'indice

EXPLAIN ANALYZE.

✅ Voir la réponse

Trois étapes :

Étape 1 — Diagnostiquer avec EXPLAIN ANALYZE qui montre le plan d'exécution réel :

SQL
EXPLAIN ANALYZE
SELECT * FROM logs WHERE created_at > '2024-01-01';

Si vous voyez Seq Scan on logs, la base parcourt toute la table — probable absence d'index sur created_at.

Étape 2 — Solution n°1 : créer un index :

SQL
CREATE INDEX idx_logs_created_at ON logs(created_at);

La requête tombe de 10 secondes → quelques millisecondes.

Étape 3 — Solution n°2 : si vous filtrez aussi sur d'autres colonnes (WHERE created_at > '2024-01-01' AND user_id = 42), créer un index composite :

SQL
CREATE INDEX idx_logs_user_date ON logs(user_id, created_at);

L'ordre des colonnes dans l'index compte : la première colonne doit être celle qui filtre le plus drastiquement.

Solution n°3 — Partitionnement : si la table est massive (> 100 M lignes) et qu'on requête souvent par date, partitionner par mois — chaque mois devient une sous-table physique. C'est ce que fait BigQuery automatiquement avec le partitionnement temporel.
