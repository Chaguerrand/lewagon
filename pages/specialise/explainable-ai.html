Feature importance

SHAP
Valeurs de Shapley : décomposition équitable de la prédiction par feature.
LIME
Approxime localement le modèle par un modèle linéaire simple.
Global vs local
Expliquer le comportement moyen vs une prédiction individuelle.

📋 Cheatsheet — Explainable AI (SHAP)

SHAP — Explicabilité locale (une prédiction)

Python
## pip install shap xgboost
import shap
import xgboost

## 1. Entraîner un modèle
model = xgboost.XGBRegressor(n_estimators=100, max_depth=2)
model.fit(X_train, y_train)

## 2. Créer l'explainer SHAP
explainer = shap.Explainer(model)

## 3. Calculer les SHAP values pour UNE instance
row = X_test.iloc[[24]]                           # une seule ligne
shap_values = explainer(row)

## 4. Vérification : base_value + sum(SHAP) = prediction
print(shap_values.base_values[0])                  # moyenne des prédictions (baseline)
print(shap_values.values.sum())                    # contribution totale des features
print(model.predict(row)[0])                       # = base_value + sum(SHAP)

## 5. Visualisations locales
shap.plots.bar(shap_values[0])                     # barplot des contributions
shap.plots.waterfall(shap_values[0])               # cascade : base → prediction
shap.plots.force(shap_values[0])                   # diagramme de force

SHAP — Explicabilité globale (tout le dataset)

Python
## Calculer les SHAP values pour tout le test set
shap_values = explainer(X_test)

## Visualisations globales
shap.plots.bar(shap_values)                        # importance moyenne des features
shap.plots.beeswarm(shap_values)                   # distribution des contributions par feature

⚠️ Récapitulatif — Erreurs clés à éviter

❌

Confondre interprétabilité et explicabilité

L'interprétabilité = comprendre le mécanisme interne du modèle (possible pour linéaire, arbre). L'explicabilité = expliquer le comportement en termes humains (nécessaire pour les boîtes noires).

❌ Dire "mon Random Forest est interprétable parce que j'ai les feature importances"

✅ Les feature importances sont une forme d'explicabilité — le modèle reste une boîte noire

❌

Confondre explicabilité locale et globale

Les SHAP values d'une instance expliquent cette prédiction — pas le modèle en général.

❌ "La feature X est la plus importante" en regardant une seule instance

✅ Local = une prédiction (shap_values[0]). Global = tout le dataset (shap_values sur X_test)

❌

Interpréter les Shapley values comme des coefficients de régression

Un Shapley value mesure la contribution marginale d'une feature pour une instance, pas un effet fixe.

❌ "SHAP = 0.5 pour income → chaque dollar de revenu ajoute 0.5 au prix"

✅ "Pour cette instance, la valeur d'income a poussé la prédiction de 0.5 au-dessus de la baseline"

❌

Calculer les SHAP values sur le train set au lieu du test set

Les explications doivent porter sur des données que le modèle n'a pas vues pendant l'entraînement.

❌ explainer(X_train) → le modèle peut avoir mémorisé ces données

✅ explainer(X_test) → explications sur des données non vues

❌

Utiliser SHAP comme mesure de causalité

SHAP mesure la contribution à la prédiction, pas la relation causale entre les variables.

❌ "SHAP montre que la taille de la maison cause un prix plus élevé"

✅ "SHAP montre que la taille de la maison contribue positivement à la prédiction du prix"

1) Pourquoi l'Explainable AI ?

1.1 Le problème

Les modèles complexes (Random Forest, Deep Learning) sont des boîtes noires — ils font de bonnes prédictions mais personne ne comprend pourquoi.

1.2 Cas d'usage

  • Validation de modèle : vérifier que le modèle apprend les bons signaux (pas le fond de l'image !)
  • Debugging : comprendre pourquoi le modèle se trompe sur certains cas
  • Découverte de connaissances : identifier les facteurs clés d'un phénomène
  • Conformité légale : le RGPD 🇪🇺 limite les décisions automatisées sans explication humaine

3) Shapley Values — Intuition

3.1 Fondements en théorie des jeux

[Untitled](Explainable%20AI/Untitled%2033c4e4108246806b877ac4de12089d6b.csv)

En théorie des jeux, les Shapley values mesurent la contribution marginale de chaque joueur à un jeu coopératif. En ML, chaque feature est un joueur et la prédiction est le gain.

3.2 Définition

Pour une instance donnée :

$\text{Prediction} = \text{Average prediction (baseline)} + \sum_{j=1}^{n} \phi_j$

Où $\phi_j$ = Shapley value de la feature j = sa contribution à l'écart entre la prédiction et la baseline.

3.3 Calcul intuitif

Pour calculer $\phi_j$ :

  1. Construire toutes les coalitions de features (avec et sans la feature j)
  2. Pour chaque coalition, comparer la prédiction avec et sans j
  3. Moyenner les différences → Shapley value
💡

On ne peut pas simplement "retirer" une feature du modèle — on simule son absence en remplaçant sa valeur par des valeurs tirées aléatoirement du dataset.

⚠️

Le calcul exact est exponentiel en nombre de features. Le package SHAP utilise des approximations efficaces (TreeSHAP pour les arbres, KernelSHAP pour les modèles agnostiques).

4) SHAP en pratique

4.1 Installation et setup

Python
## pip install shap xgboost
import shap
import xgboost
from sklearn.model_selection import train_test_split

shap.initjs()                                      # nécessaire pour certaines visualisations

4.2 Entraîner un modèle

Python
X, y = shap.datasets.california()                  # prix des maisons en Californie (×$100K)
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2, random_state=13)

model = xgboost.XGBRegressor(n_estimators=100, max_depth=2)
model.fit(X_train, y_train)

5) Explicabilité locale — Une prédiction

Python
## Sélectionner une instance
row = X_test.iloc[[24]]
prediction = model.predict(row)[0]                 # → 1.558

## Calculer les SHAP values
explainer = shap.Explainer(model)
shap_values = explainer(row)

## Vérification : baseline + contributions = prediction
print(f"Base value:      {shap_values.base_values[0]:.3f}")   # 2.067
print(f"Sum SHAP values: {shap_values.values.sum():.3f}")     # -0.509
print(f"Prediction:      {prediction:.3f}")                    # 1.558

Visualisations locales

Python
shap.plots.bar(shap_values[0])                     # barplot des contributions par feature
shap.plots.waterfall(shap_values[0])               # cascade de la baseline à la prédiction
shap.plots.force(shap_values[0])                   # diagramme de force interactif
👉

Le waterfall plot montre comment chaque feature pousse la prédiction au-dessus ou en dessous de la baseline (moyenne). Les features rouges augmentent la prédiction, les bleues la diminuent.

6) Explicabilité globale — Tout le dataset

Python
shap_values = explainer(X_test)                    # SHAP values pour tout le test set

shap.plots.bar(shap_values)                        # importance moyenne de chaque feature
shap.plots.beeswarm(shap_values)                   # distribution : chaque point = une instance
💡

Le beeswarm plot montre pour chaque feature : la distribution des SHAP values (axe X) colorée par la valeur de la feature (rouge = haute, bleu = basse). On voit d'un coup quelles features sont importantes et dans quel sens elles influencent.

🧭 Introduction

Tu as passé des heures à entraîner un modèle de Machine Learning. Il prédit avec 95 % de précision. Mais si quelqu'un te demande pourquoi il a prédit "refus de crédit" pour ce client précis… tu ne sais pas quoi répondre. C'est exactement le problème que résout l'Explainable AI (XAI).

Analogie fil rouge — La boîte noire

Imagine un oracle dans une boîte noire : tu lui donnes des informations sur une maison (surface, quartier, étage…) et il te crache un prix. L'oracle est précis, mais tu ne sais pas comment il raisonne à l'intérieur. L'Explainable AI, c'est l'ensemble des outils qui permettent d'ouvrir cette boîte — ou du moins, d'interroger l'oracle de l'extérieur pour comprendre ses choix.

À retenir absolument

  • Un modèle complexe (Random Forest, réseau de neurones) est une boîte noire : il prédit bien, mais on ne sait pas pourquoi.
  • Interprétabilité = comprendre les mécanismes internes du modèle (possible seulement pour les modèles simples).
  • Explicabilité = expliquer le comportement du modèle en termes compréhensibles, depuis l'extérieur.
  • SHAP est la bibliothèque de référence pour l'explicabilité : elle dit combien chaque variable a "poussé" la prédiction vers le haut ou vers le bas.
  • Une explication peut être locale (pour une seule prédiction) ou globale (pour l'ensemble du dataset).

1. Pourquoi l'Explainable AI ?

1.1 Le problème des boîtes noires

Un modèle de régression linéaire est simple : tu peux lire ses coefficients et comprendre immédiatement l'effet de chaque variable. Mais un Random Forest avec 500 arbres, ou un réseau de neurones profond ? Impossible de lire les paramètres directement — il y en a des millions.

Pourtant ces modèles complexes sont souvent bien plus performants. Alors on les utilise… et on perd la compréhension.

1.2 Pourquoi c'est important en pratique

Exemple concret : Une banque utilise un modèle ML pour décider d'accorder ou refuser un crédit. Le modèle refuse. Le client demande : "Pourquoi ?". Sans Explainable AI, la banque ne peut pas répondre. En Europe, le RGPD impose d'ailleurs le droit à une explication pour toute décision automatisée.

Autres cas d'usage :

  • Validation : vérifier que le modèle apprend les bons signaux. Exemple : un modèle de détection de tumeur qui apprend à reconnaître... les règles d'image de l'hôpital plutôt que la tumeur elle-même.
  • Debugging : comprendre pourquoi le modèle se trompe sur certains cas.
  • Découverte de connaissances : identifier quelles variables sont vraiment importantes pour un phénomène.
💡

L'Explainable AI ne remplace pas un bon modèle — elle permet de l'auditer, de le corriger et de le justifier.

2. Interprétabilité vs Explicabilité

C'est une distinction subtile mais fondamentale.

2.1 Les modèles interprétables

Certains modèles sont interprétables par nature : leurs paramètres internes ont un sens direct.

ModèleComment on l'interprète
Régression linéaireLes coefficients — chaque coefficient = l'effet d'une variable
Arbre de décisionLes règles de décision — si age > 30 et revenu > 2000 alors...
KNNLa logique de voisinage — cette observation ressemble aux k plus proches

Exemple : En régression linéaire prix = 3.2 × surface + 150 × étage - 20000, le coefficient 3.2 signifie "chaque m² supplémentaire ajoute 3.2 au prix". C'est de l'interprétabilité directe.

2.2 Les modèles explicables (mais pas interprétables)

Les modèles complexes (Random Forest, XGBoost, réseaux de neurones) sont des boîtes noires. On ne peut pas lire leurs paramètres. Mais on peut les expliquer de l'extérieur avec des techniques post-hoc comme SHAP ou LIME.

⚠️

Attention à la confusion fréquente : les "feature importances" d'un Random Forest (score Gini) ne rendent pas le modèle interprétable — c'est une forme d'explicabilité globale. Le modèle reste une boîte noire.

QuoiComprendre les mécanismes internesExpliquer le comportement depuis l'extérieur
QuandModèles simplesModèles complexes (boîtes noires)
CommentLire les paramètresTechniques post-hoc : SHAP, LIME…
ExempleCoefficients d'une régressionSHAP values d'un XGBoost

3. Approches d'explicabilité

3.1 Intrinsèque vs Agnostique vs Dépendant du modèle

ApprochePrincipeExemples
IntrinsèqueLe modèle est lisible par natureRégression linéaire, arbre de décision
AgnostiqueFonctionne sur n'importe quelle boîte noire (on ne regarde que les inputs/outputs)SHAP (KernelSHAP), LIME
Dépendant du modèleOptimisé pour un type de modèle spécifiqueTreeSHAP (optimisé pour les arbres de décision)

3.2 Local vs Global

C'est une autre dimension clé :

  • Local : expliquer une seule prédiction. Exemple — "Pourquoi le modèle a-t-il prédit un prix de 250 000 € pour cette maison ?"
  • Global : comprendre le comportement général du modèle. Exemple — "Quelles variables sont les plus importantes pour le modèle sur l'ensemble des données ?"
🧠

On peut agréger des explications locales pour obtenir une vue globale — c'est exactement ce que fait SHAP.

4. Shapley Values — L'intuition mathématique

C'est le coeur conceptuel du cours. Accroches-toi — l'intuition est belle.

4.1 L'analogie du jeu coopératif

Imagine une équipe de foot qui marque 3 buts. Chaque joueur a contribué à la victoire. Mais comment répartir équitablement le mérite entre les 11 joueurs ? C'est exactement la question que pose la théorie des jeux coopératifs — et Lloyd Shapley y a répondu en 1953.

En Machine Learning, on adapte cette idée :

  • Les joueurs = les variables (features) du modèle
  • Le gain = la prédiction du modèle
  • La valeur Shapley d'une feature = sa contribution marginale à la prédiction

4.2 La formule

Pour une prédiction donnée, la Shapley value $\phi_j$ d'une feature $j$ vérifie :

$$\text{Prédiction} = \underbrace{\bar{y}}_{\text{baseline}} + \sum_{j=1}^{n} \phi_j$$

Où :

  • $\bar{y}$ = la baseline = la prédiction moyenne du modèle sur tout le dataset (le point de départ)
  • $\phi_j$ = la contribution de la feature $j$ pour cette instance précise (positive ou négative)
  • $n$ = le nombre de features

Exemple numérique concret :

Ton modèle prédit le prix d'une maison. La prédiction moyenne (baseline) est $\bar{y} = 206\,700$ €. Pour une maison précise, le modèle prédit 155 800 €. L'écart est $-50\,900$ €. Les Shapley values expliquent comment chaque variable contribue à cet écart :

FeatureValeur pour cette maisonShapley value $\phi_j$
Surface45 m² (petite)$-35\,000$ €
QuartierCentre-ville$+12\,000$ €
Étage0 (rez-de-chaussée)$-18\,000$ €
Nb. pièces2$-9\,900$ €
Total$-50\,900$ €

$155\,800 = 206\,700 + (-35\,000) + 12\,000 + (-18\,000) + (-9\,900)$ ✓

4.3 Comment ça se calcule ?

L'idée : on "enlève" une feature et on regarde comment la prédiction change. Mais on ne peut pas vraiment "enlever" une variable du modèle — il a été entraîné avec toutes. On simule son absence en remplaçant sa valeur par des valeurs tirées aléatoirement du dataset.

Le calcul exhaustif consiste à tester toutes les coalitions possibles de features (avec et sans la feature $j$) et à moyenner les différences. Si on a $n$ features, il y a $2^n$ coalitions — c'est exponentiel !

⚠️

Avec 20 features, il y a $2^{20} = 1\,048\,576$ coalitions. Le calcul exact est impossible. SHAP utilise des approximations efficaces : TreeSHAP pour les modèles à base d'arbres, KernelSHAP pour les autres.

5. SHAP en pratique — Installation et setup

shap est une bibliothèque Python qui implémente les Shapley values de façon efficace.

Python
# Installation (une seule fois)
# pip install shap xgboost

import shap
import xgboost
from sklearn.model_selection import train_test_split

# Initialise les visualisations interactives dans un notebook
shap.initjs()

Charger un dataset et entraîner un modèle

Python
# Dataset intégré à shap : prix des maisons en Californie (en centaines de milliers de $)
X, y = shap.datasets.california()

# Découper en train / test (bonne pratique : toujours expliquer sur le test set)
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, random_state=13
)

# Entraîner un XGBoost (modèle boîte noire très performant)
model = xgboost.XGBRegressor(n_estimators=100, max_depth=2)
model.fit(X_train, y_train)
💡

On utilise XGBoost car SHAP dispose d'une implémentation ultra-rapide pour ce type de modèle (TreeSHAP). Mais SHAP fonctionne avec n'importe quel modèle scikit-learn, keras, etc.

6. Explicabilité locale — Comprendre une seule prédiction

6.1 Calculer les SHAP values d'une instance

Python
# Sélectionner une instance précise (la ligne 24 du test set)
row = X_test.iloc[[24]]                           # double crochet pour garder un DataFrame

# Faire la prédiction
prediction = model.predict(row)[0]                # → 1.558 (soit ~155 800 $)

# Créer l'explainer SHAP (il analyse le modèle)
explainer = shap.Explainer(model)

# Calculer les SHAP values pour cette instance
shap_values = explainer(row)

# Vérification : baseline + somme des contributions = prédiction
print(f"Baseline (prédiction moyenne) : {shap_values.base_values[0]:.3f}")  # → 2.067
print(f"Somme des SHAP values :         {shap_values.values.sum():.3f}")    # → -0.509
print(f"Prédiction :                    {prediction:.3f}")                  # → 1.558
# 2.067 + (-0.509) = 1.558 ✓

La vérification montre que la formule tient bien :

$$1.558 = \underbrace{2.067}_{\text{baseline}} + \underbrace{(-0.509)}_{\sum \phi_j}$$

6.2 Visualiser les contributions locales

Trois types de graphiques pour explorer une prédiction individuelle :

Python
# --- Barplot : importance de chaque feature pour cette instance ---
shap.plots.bar(shap_values[0])
# Affiche les contributions en valeur absolue, triées par importance
# Les barres rouges = contributions positives (poussent la prédiction vers le haut)
# Les barres bleues = contributions négatives (tirent la prédiction vers le bas)

# --- Waterfall : cascade de la baseline à la prédiction ---
shap.plots.waterfall(shap_values[0])
# Montre pas à pas comment chaque feature déplace la prédiction
# Depuis la baseline (2.067) jusqu'à la prédiction finale (1.558)

# --- Force plot : diagramme de forces ---
shap.plots.force(shap_values[0])
# Vue compacte : les features rouges "poussent" vers la droite (hausse)
# Les features bleues "poussent" vers la gauche (baisse)
🧠

Le waterfall plot est souvent le plus pédagogique pour expliquer une décision à quelqu'un : "La superficie du logement a fait baisser la prédiction de X, mais la localisation a compensé de Y…"

7. Explicabilité globale — Comprendre le modèle en général

Une fois les SHAP values calculées sur tout le test set, on obtient une vue globale du comportement du modèle.

Python
# Calculer les SHAP values pour TOUTES les observations du test set
# (peut prendre quelques secondes selon la taille)
shap_values = explainer(X_test)           # shape : (n_samples, n_features)

# --- Barplot global : importance moyenne de chaque feature ---
shap.plots.bar(shap_values)
# Chaque barre = la moyenne des |SHAP values| de cette feature sur tout le dataset
# C'est l'importance globale de la variable pour le modèle

# --- Beeswarm plot : distribution des contributions ---
shap.plots.beeswarm(shap_values)
# Chaque point = une observation
# Position sur l'axe X = valeur de la SHAP value (contribution)
# Couleur du point = valeur de la feature (rouge = haute, bleu = basse)

Lire un beeswarm plot

Le beeswarm est l'outil le plus riche pour l'analyse globale. Voici comment l'interpréter :

  • Chaque ligne = une feature, triée par ordre d'importance (la plus importante en haut)
  • Chaque point = une observation du dataset
  • Position horizontale = la SHAP value (à droite = contribution positive au prix, à gauche = négative)
  • Couleur = valeur brute de la feature (rouge = grande valeur, bleu = petite valeur)

Exemple de lecture : Si pour la feature "surface", les points rouges (grande surface) sont à droite (SHAP positif) → les grandes surfaces augmentent la prédiction de prix. C'est logique !

✅

Le beeswarm permet de voir d'un coup d'œil : quelles features comptent le plus, et dans quel sens elles influencent la prédiction (relation positive ou négative).

8. SHAP sur des images — Bonus Deep Learning

SHAP ne se limite pas aux données tabulaires. Sur les images, il identifie les régions qui contribuent le plus à la prédiction d'un réseau de neurones.

Principe

Au lieu de "retirer" une feature, on floute des régions de l'image pour simuler leur absence, puis on observe comment la prédiction change.

Python
from tensorflow.keras.applications.resnet50 import ResNet50, preprocess_input

# Charger ResNet50 pré-entraîné sur ImageNet
model_deep = ResNet50(weights='imagenet')

# Fonction de prédiction compatible SHAP
def predict_fn(X):
    return model_deep(preprocess_input(X.copy()))   # prétraitement obligatoire pour ResNet50

# Masker : simule l'absence d'une région en la floutant (128x128 de flou gaussien)
masker = shap.maskers.Image("blur(128,128)", X[0].shape)

# Créer l'explainer pour images
explainer = shap.Explainer(predict_fn, masker, output_names=class_names)

# Calculer les SHAP values pour 4 images (500 évaluations chacune)
shap_values = explainer(X[1:5], max_evals=500, batch_size=50)

# Visualiser : zones rouges = contribuent positivement à la classe prédite
shap.image_plot(shap_values, pixel_values=X[1:5])
⚠️

L'explication d'images est beaucoup plus lente que pour les données tabulaires : 500 évaluations par image, c'est 500 passages dans le réseau de neurones. En prototypage, réduire max_evals à 100-200 pour aller plus vite.

9. Les pièges classiques à éviter

Piège 1 — Confondre interprétabilité et explicabilité

⚠️

"Mon Random Forest est interprétable parce que j'ai les feature importances" est faux. Les feature importances sont une forme d'explicabilité — le modèle reste une boîte noire.

Piège 2 — Confondre local et global

Une SHAP value calculée sur une seule instance n'est pas représentative du comportement général. Elle dit uniquement : "Pour cette observation, la feature X a contribué de +0.3". Pour conclure sur l'importance globale d'une feature, il faut agréger sur tout le dataset.

Piège 3 — Lire les SHAP values comme des coefficients de régression

Un coefficient de régression est fixe : "chaque euro de revenu ajoute 0.5 au score, toujours". Une SHAP value est locale : "pour cette instance, le revenu a contribué de +0.5 — mais pour une autre instance, ce sera différent".

Code
❌ "SHAP = 0.5 pour income → chaque euro de revenu ajoute 0.5"
✅ "Pour cette instance, la valeur d'income a poussé la prédiction de +0.5 au-dessus de la baseline"

Piège 4 — Calculer les SHAP values sur le train set

Python
❌ explainer(X_train)    # le modèle a mémorisé ces données → explications biaisées
✅ explainer(X_test)     # explications sur des données non vues = plus honnêtes

Piège 5 — Confondre contribution et causalité

SHAP mesure la contribution à la prédiction du modèle, pas une relation causale dans le monde réel.

Code
❌ "SHAP montre que la taille de la maison CAUSE un prix plus élevé"
✅ "SHAP montre que la taille de la maison CONTRIBUE positivement à la prédiction du prix"
🧠

La causalité est une question de philosophie et de design expérimental — pas de statistiques seules. SHAP explique le modèle, pas le monde.

🎯 Questions de vérification

Question 1. Un collègue te dit : "Mon arbre de décision est une boîte noire, j'ai besoin de SHAP pour l'expliquer." Est-ce correct ? Justifie ta réponse en utilisant la distinction interprétabilité / explicabilité.

Question 2. Tu calcules les SHAP values d'une observation et tu obtiens : base_value = 2.1, shap_values = [0.3, -0.8, 0.5, -0.2]. Quelle est la prédiction du modèle pour cette observation ? Montre le calcul.

Question 3. Sur un beeswarm plot, tu observes que pour la feature "distance_au_centre", les points bleus (faible distance) sont massivement à droite (SHAP positif). Qu'est-ce que cela signifie en langage courant ?

Question 4. Pourquoi faut-il calculer les SHAP values sur le test set et non sur le train set ?

📌 TL;DR

  • Les modèles complexes (Random Forest, XGBoost, Deep Learning) sont des boîtes noires : très performants, mais incompréhensibles sans outils dédiés.
  • Interprétabilité = lire les paramètres internes (possible pour régression linéaire, arbres simples). Explicabilité = expliquer le comportement depuis l'extérieur (technique post-hoc, nécessaire pour les boîtes noires).
  • Les Shapley values mesurent la contribution de chaque feature à l'écart entre une prédiction et la prédiction moyenne : $\text{Prédiction} = \bar{y} + \sum \phi_j$.
  • SHAP est la bibliothèque Python de référence : shap.Explainer(model) puis explainer(X_test).
  • Locale = expliquer une seule prédiction (shap_values[0], waterfall/force plot). Globale = comprendre le modèle en général (shap_values sur tout le test set, beeswarm/barplot).
  • Toujours calculer les SHAP values sur le test set.
  • SHAP ≠ causalité : il explique le modèle, pas les relations réelles entre les variables.
  • SHAP fonctionne aussi sur les images en floutant des régions pour simuler leur absence.

A. Concepts du cours

Pourquoi expliquer un modèle

Quatre raisons pour lesquelles l'interprétabilité est devenue critique en data science.

1. Confiance : un médecin ne va pas suivre une recommandation de modèle "boîte noire" sans comprendre pourquoi. Idem pour un juge, un banquier, un recruteur.

2. Conformité réglementaire : le RGPD donne le droit à l'explication pour toute décision automatisée. Un modèle qui refuse un crédit doit pouvoir expliquer pourquoi. L'AI Act européen (2024) impose des obligations encore plus strictes pour les systèmes "à haut risque".

3. Debug : un modèle qui se trompe doit être analysable. Si vous ne savez pas pourquoi votre modèle prédit "cancer", vous ne pouvez pas savoir s'il a appris des features pertinentes ou des artefacts dans les données.

4. Découverte de biais : un modèle peut être très précis tout en discriminant illégalement. Sans interprétabilité, ces biais restent invisibles jusqu'au procès.

💡

Cas célèbre : le modèle de recidive criminelle COMPAS, utilisé aux US, présentait un taux de faux positifs deux fois plus élevé pour les Afro-Américains que pour les Blancs, à risque réel équivalent. Sans audit explicite, ce biais structurel avait passé le filtre du déploiement. C'est exactement le genre de problème que l'XAI cherche à éviter.

Modèles intrinsèquement interprétables

Certains modèles sont interprétables par construction — pas besoin de techniques d'explication post-hoc.

ModèleInterprétation
Régression linéaire/logistiqueCoefficient × feature → contribution directe
Decision Tree (peu profond)Chemin de décision visible
Régression à coefficients sparses (Lasso)Seules quelques features comptent
Generalized Additive Models (GAMs)Une fonction par feature, somme
Rule-basedRègles explicites (si X > 5 alors Y)
Python
# Régression linéaire interprétable
from sklearn.linear_model import LogisticRegression
model = LogisticRegression()
model.fit(X, y)
print(dict(zip(X.columns, model.coef_[0])))
# {'age': 0.05, 'income': 0.0001, 'has_default': -2.3, ...}
👉

Bonne pratique : pour les domaines régulés (médical, finance, justice), commencer par un modèle interprétable comme baseline. Souvent, il atteint déjà 95% de la performance d'un modèle complexe — et il est validable, défendable, déployable. Le gain marginal d'un XGBoost peut ne pas justifier sa complexité légale.

Feature importance

Pour les modèles ensemble (Random Forest, XGBoost), trois mesures de l'importance des features :

1. Mean Decrease Impurity (MDI) : moyenne de la diminution d'impureté due à chaque feature, sur tous les splits.

Python
importances = model.feature_importances_

Avantages : gratuit, calculé pendant fit.

Inconvénients : biaisé envers les features avec beaucoup de valeurs uniques (cardinalité élevée).

2. Permutation Importance : on permute aléatoirement une feature et on mesure la chute de performance.

Python
from sklearn.inspection import permutation_importance

result = permutation_importance(
    model, X_test, y_test,
    n_repeats=10, random_state=42, n_jobs=-1,
)

importances = pd.DataFrame({
    "feature": X.columns,
    "importance": result.importances_mean,
}).sort_values("importance", ascending=False)

Avantages : agnostique au modèle, plus fiable que MDI.

Inconvénients : plus lent (1 fit par feature × n_repeats), peut être trompeur si les features sont corrélées.

3. SHAP values : la méthode moderne (cf. section suivante).

⚠️

Piège classique : MDI biaisée envers les features à haute cardinalité. Une colonne "ID utilisateur" peut apparaître la plus importante alors qu'elle ne contient aucune information utile. Toujours vérifier avec permutation_importance ou SHAP avant de tirer des conclusions sur les features importantes.

SHAP values

SHAP (SHapley Additive exPlanations, Lundberg & Lee, 2017) est devenu le standard de l'XAI moderne. Il décompose chaque prédiction en contributions par feature, basées sur la théorie des jeux coopératifs (Shapley values).

Idée : pour chaque feature, on calcule "combien cette feature a contribué à la prédiction par rapport à la moyenne des prédictions". La somme des contributions est exactement la différence entre la prédiction et la baseline.

$$\hat{y}(x) = \phi_0 + \sum_{i=1}^{p} \phi_i(x)$$

où $\phi_0$ est la valeur moyenne des prédictions (baseline) et $\phi_i(x)$ est la contribution de la feature $i$ pour cet exemple $x$.

Exemple chiffré — décision de crédit : un modèle prédit une probabilité de défaut de 0.72 pour un client. La moyenne sur tout le portefeuille est 0.15.

FeatureValeur du clientContribution SHAP ($\phi_i$)
(baseline $\phi_0$)—+0.15
credit_score520+0.35
income22 000 €+0.12
debt_ratio0.68+0.18
age45-0.05
years_employed2+0.03
Total prédiction= 0.78

Note : en pratique SHAP travaille dans l'espace des log-odds, on retrouve 0.72 en probabilité après transformation logistique. L'idée clé : la somme des contributions + la baseline donne exactement la prédiction. Le client a un score explicitement décomposé : "c'est surtout votre credit_score faible (+0.35) et votre debt_ratio (+0.18) qui font augmenter votre risque".

Propriétés :

  • Local accuracy : la somme des SHAP values reproduit exactement la prédiction
  • Consistency : si une feature contribue plus dans un modèle, sa SHAP value augmente
  • Symétrie : deux features avec la même contribution ont la même SHAP value
Python
import shap

# TreeExplainer : algorithme optimisé pour arbres, O(TLD^2) au lieu de O(2^M)
# Compatible XGBoost, LightGBM, RandomForest, sklearn GradientBoosting
explainer = shap.TreeExplainer(model)

# shap_values : matrice (n_samples, n_features) de contributions
# expected_value : la baseline phi_0 (moyenne des prédictions)
shap_values = explainer.shap_values(X_test)

# Visualisations classiques
shap.summary_plot(shap_values, X_test)              # beeswarm : importance ET direction
shap.summary_plot(shap_values, X_test, plot_type="bar")  # bar chart classique des |SHAP|
shap.dependence_plot("age", shap_values, X_test)    # effet marginal de age + interactions
shap.force_plot(explainer.expected_value, shap_values[0], X_test.iloc[0])  # 1 prédiction
shap.waterfall_plot(shap.Explanation(                # décomposition claire d'une instance
    values=shap_values[0],
    base_values=explainer.expected_value,
    data=X_test.iloc[0],
))
💡

Analogie : SHAP, c'est comme un audit comptable d'une décision. Pour chaque "décision" (prédiction) du modèle, vous obtenez un grand-livre où chaque feature a apporté X € (positif ou négatif) au résultat final. La somme balance exactement. C'est rigoureux, défendable en audit, et intuitif pour les non-statisticiens.

B. Compléments avancés

LIME

LIME (Local Interpretable Model-agnostic Explanations, Ribeiro et al., 2016) est l'autre méthode classique d'XAI.

Idée : pour expliquer une prédiction spécifique, on approxime localement le modèle complexe par un modèle simple (régression linéaire). On perturbe l'instance, on récupère les prédictions du modèle complexe sur ces variations, et on entraîne un linéaire dessus.

Python
import lime
import lime.lime_tabular

explainer = lime.lime_tabular.LimeTabularExplainer(
    X_train.values,
    feature_names=X.columns,
    class_names=["No", "Yes"],
    discretize_continuous=True,
)

explanation = explainer.explain_instance(
    X_test.iloc[0].values,
    model.predict_proba,
    num_features=10,
)
explanation.show_in_notebook(show_table=True)

LIME vs SHAP :

LIMESHAP
ThéorieApproximation localeShapley values (théorie des jeux)
GarantieAucuneLocal accuracy, consistency
VitesseLentTrès rapide pour les arbres (TreeExplainer)
StabilitéVariable selon les perturbationsDéterministe
Standard 2025Moins utiliséStandard de facto
👉

Recommandation : utilisez SHAP pour la production et la communication. LIME a une valeur historique et reste utile pour des cas spécifiques (texte, images), mais SHAP est plus rigoureux et plus rapide.

Partial dependence plots

Un PDP montre l'effet moyen d'une feature sur la prédiction, en marginalisant les autres.

Python
from sklearn.inspection import PartialDependenceDisplay

PartialDependenceDisplay.from_estimator(
    model,
    X_train,
    features=["age", "income", ("age", "income")],   # 1D et 2D
    grid_resolution=50,
)

Pour 1 feature : courbe "valeur de la feature → prédiction moyenne". Pour 2 features : heatmap qui montre les interactions.

Limite : suppose que les features sont indépendantes. Si "age" et "income" sont fortement corrélés, fixer "age" à 70 tout en gardant "income" à 100k$ crée une combinaison irréaliste qui fausse le PDP.

ICE plots (Individual Conditional Expectation) sont une variante qui trace une courbe par instance plutôt que la moyenne — révèle les hétérogénéités cachées par le PDP.

Counterfactual explanations

Plutôt que d'expliquer "pourquoi cette prédiction", expliquer "que faut-il changer pour avoir une prédiction différente".

Python
# Exemple : "Pourquoi ce crédit a-t-il été refusé ? Que faire pour qu'il soit accepté ?"
from dice_ml import Dice

d = Dice(data, model, method="random")
counterfactuals = d.generate_counterfactuals(query_instance, total_CFs=5, desired_class=1)
counterfactuals.visualize_as_dataframe()

Output type : "Si vous augmentez votre revenu de 20k€ et votre score de crédit de 50 points, votre demande aurait été acceptée". Beaucoup plus actionnable pour l'utilisateur final qu'une SHAP value.

C'est la forme d'explication la plus appréciée des utilisateurs non techniques, mais la plus difficile à générer correctement (il faut des contrefactuels réalistes : pas "augmente ton âge de -10 ans").

💡

Analogie : SHAP vous dit "voici les raisons de cette décision". Counterfactual vous dit "voici comment changer cette décision". Pour un utilisateur, le counterfactual est souvent plus actionnable. Pour un audit légal, SHAP est plus rigoureux. Les deux sont complémentaires.

C. Questions de compréhension

Q1 : Pourquoi la feature_importances_ d'un Random Forest peut-elle être trompeuse, et que faire à la place ?

💡 Voir l'indice

MDI bias.

✅ Voir la réponse

feature_importances_ utilise le Mean Decrease Impurity (MDI), qui est biaisé en faveur des features avec beaucoup de valeurs uniques (haute cardinalité). Une colonne "ID utilisateur" ou "code postal" peut apparaître très importante simplement parce qu'elle offre plus d'opportunités de splits, sans contenir d'information utile pour la généralisation. Solutions : (1) Permutation importance (sklearn.inspection.permutation_importance) : agnostique au modèle, plus fiable. Permute aléatoirement une feature et mesure la chute de performance. (2) SHAP values : encore mieux, donne aussi des explications par instance et capture les interactions. (3) Drop-column importance : retirer la feature et réentraîner le modèle, mesurer la chute. Le plus rigoureux mais le plus coûteux. Bonne pratique : croiser au moins deux méthodes. Si MDI dit que "user_id" est important mais que SHAP et permutation_importance ne le voient pas, c'est du bias MDI. Question d'entretien data science classique pour vérifier qu'un candidat sait les pièges.

Q2 : Vous voulez expliquer une décision de crédit refusée à un client. SHAP, LIME, ou counterfactual ?

💡 Voir l'indice

actionnabilité.

✅ Voir la réponse

Counterfactual est probablement le meilleur choix pour cet usage. SHAP et LIME diraient quelque chose comme : "Votre demande a été refusée principalement à cause de votre score de crédit (-2.3) et de votre revenu insuffisant (-0.8)". Informatif mais frustrant pour le client : qu'est-ce qu'il peut faire ? Counterfactual répond directement : "Si vous augmentez votre score de crédit de 580 à 650 ET votre revenu de 30k€ à 38k€, votre demande sera acceptée". C'est actionnable. En pratique, on combine les trois : (1) counterfactual pour la communication client, (2) SHAP pour l'audit interne et la documentation réglementaire, (3) LIME ou plot global de SHAP pour la documentation de modèle. Précaution éthique : les counterfactuals doivent être réalistes (pas "diminuez votre âge de 10 ans") et non discriminatoires (pas basés sur le genre, l'ethnie, etc.). Question d'entretien XAI moderne : la réponse révèle si le candidat pense aux besoins de l'utilisateur final.

Q3 : Vous avez un modèle XGBoost à 95% d'accuracy en classification médicale. Le médecin demande "pourquoi ?". Quelle approche d'explication choisir et pourquoi ?

💡 Voir l'indice

SHAP global et local.

✅ Voir la réponse

Approche en deux étapes. Étape 1 — explication globale (SHAP summary plot) : montrer au médecin quelles sont les features les plus importantes en moyenne sur l'ensemble du dataset, ainsi que la direction de leur effet. Cela répond à la question "comment ce modèle pense en général ?" et permet au médecin de vérifier que les features prises en compte sont médicalement pertinentes. Si le modèle accorde un poids majeur à "ID patient" ou "date d'admission", c'est un red flag. Étape 2 — explication locale (SHAP waterfall ou force plot) : pour chaque prédiction individuelle, montrer la décomposition exacte des features qui ont contribué à cette décision. "Le modèle a prédit cancer parce que : âge = +1.2, marqueur K-67 élevé = +0.8, antécédents familiaux = +0.4". Cela permet au médecin de valider la décision avec son expertise clinique : si les features pointées sont les bonnes, il fait confiance. Étape 3 — counterfactual (optionnel mais utile) : "qu'est-ce qui aurait fait basculer le diagnostic dans l'autre sens ?". Cela aide à quantifier la marge de la décision. Pourquoi SHAP plutôt que LIME : SHAP est plus rigoureux (théorie des jeux), plus rapide pour les arbres (TreeExplainer en O(secondes)), et est devenu le standard de la documentation médicale. Question d'entretien XAI dans un contexte régulé : la réponse exhaustive parle de transparence, validation clinique, et des trois niveaux d'explication.
